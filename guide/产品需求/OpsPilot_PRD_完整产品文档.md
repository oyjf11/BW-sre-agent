# OpsPilot Agent 产品文档（PRD）

版本: v1.0  
定位: 面试展示 + 可落地开发的生产级智能体应用产品文档  
产品名称: OpsPilot Agent（生产级运维工单排障与修复智能体）

---

## 1. 产品概述

### 1.1 一句话定位
面向研发/SRE/值班工程师的智能体系统，对故障工单或报警事件进行自动分诊、证据收集、根因分析、修复建议、人工审批、执行与复盘，并沉淀为可复用的知识与评测闭环。

### 1.2 解决的问题
企业运维与故障处理通常存在以下问题:
- 故障处理依赖资深工程师经验，路径不标准化
- 信息分散在日志、监控、发布记录、工单系统、知识库中
- 值班时上下文切换成本高，排障效率低
- 高风险动作不敢自动化，AI 只能给建议，无法进入真实闭环
- 复盘质量不稳定，经验难沉淀复用

### 1.3 产品目标
1. 缩短 MTTR（平均故障恢复时间）
2. 标准化故障处置流程（计划 -> 证据 -> 判断 -> 方案 -> 审批 -> 执行 -> RCA）
3. 在安全边界内引入智能体自动化（HITL 审批）
4. 建立可观测、可评测、可迭代的生产级 Agent 能力

## 2. 目标用户与使用场景

### 2.1 目标用户
- 一线值班工程师（On-call）
- SRE / 运维工程师
- 后端 / 全栈研发（自服务稳定性负责）
- 技术支持（L2/L3）

### 2.2 核心场景
1. **报警触发场景**
   - 支付服务 5xx 飙升
   - p95 延迟突增
   - 错误率突增

2. **工单投诉场景**
   - 用户反馈订单创建失败
   - 页面提交失败、接口超时

3. **发布后故障场景**
   - 新版本发布后接口错误显著增加
   - 部分实例异常、探针失败

4. **周期性异常场景**
   - 凌晨 CPU 打满
   - 定时任务冲突
   - 批处理堆积

## 3. 产品边界与版本规划

### 3.1 MVP（面试可演示版本）
- 输入工单文本或告警事件
- 自动生成排查计划
- 调用工具收集证据（日志、指标、发布记录、Runbook）
- 输出根因候选与修复建议
- 高风险动作触发人工审批
- 审批后执行 mock action（模拟执行）
- 生成 RCA（复盘报告）
- 可查看 trace / 执行路径

### 3.2 P1（建议实现，面试加分）
- 多 Agent 协作（Planner / Investigator / Critic / Executor）
- RAG 接入（Runbook、历史事故复盘、FAQ）
- 风险分级与策略引擎（低风险自动、高风险审批）
- 失败分级（可重试 / 转人工 / 终止）
- 持久化恢复（中断后续跑）
- 离线评测集 + 在线评测面板

### 3.3 P2（扩展方向）
- 接入真实监控/日志平台（Prometheus/Loki/ELK 等）
- 接入真实工单系统（Jira / ServiceNow / 自研）
- 真实受控执行动作（审批+审计+幂等）
- 多租户与权限体系完善
- 双人审批与策略规则增强

## 4. 核心价值主张

与普通 AI 聊天助手相比，OpsPilot Agent 是一个“生产级工作流智能体系统”:
- **有明确业务目标**: 缩短故障恢复时间
- **有状态**: thread / run / checkpoint
- **有工具编排**: logs / metrics / deployments / runbook / action
- **有人机协作**: 高风险动作 HITL 审批
- **有工程闭环**: trace / cost / latency / eval / RCA

---

## 5. 用户旅程与主流程

### 5.1 主流程（Happy Path）
1. 用户提交工单或告警（文本 + 服务名 + 时间范围）
2. 系统进行分诊（事件类型、严重级别、范围）
3. Agent 生成排查计划
4. Agent 并行收集证据（日志/指标/发布记录/知识库）
5. Agent 生成根因候选与置信度
6. Agent 生成修复建议与风险等级
7. 高风险动作进入人工审批
8. 审批通过后执行（mock 或真实受控执行）
9. 验证修复效果
10. 生成 RCA 报告并归档（可写入知识库）

### 5.2 异常与失败流程
- 工具超时 -> 重试 -> 降级（缩小时间窗/切换数据源）-> 转人工
- 证据不足 -> 请求补充信息（服务名、时间范围、环境）
- 审批超时 -> 挂起状态，等待恢复
- 执行动作失败 -> 补偿/回滚建议 -> 失败 RCA

## 6. 功能清单（PRD）

### 6.1 工单入口（Intake）
**能力**
- 手工录入工单文本 / 粘贴告警内容
- 可选结构化字段: service、env、severity、time_range
- 支持 ticket_id 模式（从数据库拉取工单详情）
- 实体抽取（服务名、错误码、发布时间、影响范围）

**输出**
- 标准化 `IncidentTicket`

### 6.2 智能分诊（Triage）
**能力**
- 判断事件类型（错误率、性能、资源、依赖、发布回归）
- 判断严重级别（P1/P2/P3）
- 给出初始排查范围与建议时间窗
- 判断是否需要立即人工介入

**输出**
- `triage_result`
- 初始排查范围（服务/上下游/时间窗）

### 6.3 排查计划生成（Planner Agent）
**能力**
- 基于工单 + 分诊 + 历史知识生成结构化排查计划
- 明确证据收集步骤及优先级
- 指定停止条件（何时可进入诊断）

**输出示例**
- 拉取错误日志 top patterns
- 拉取关键指标（5xx、latency、CPU、内存）
- 查询最近部署/配置变更
- 检索 runbook 与类似事故

### 6.4 证据收集（Investigator Agent）
**能力**
- 调用工具并行采集日志、指标、发布记录、runbook
- 对证据做摘要与去噪
- 记录证据来源、时间戳、可信度

**要求**
- 支持并发 fanout/fanin
- 节点级重试与超时
- 证据质量评分（完整性/新鲜度）

### 6.5 根因分析与方案生成（Diagnose + Remediation）
**能力**
- 输出 Top-K 根因候选（带置信度）
- 为每个候选附支持证据与反证
- 生成修复动作计划、预期影响、回滚预案

**输出**
- `root_cause_candidates[]`
- `recommended_action_plan`
- `risk_level`

### 6.6 批判与校验（Critic Agent）
**能力（P1）**
- 检查推理是否跳步
- 检查证据是否支撑结论
- 检查证据冲突/不足
- 要求补充证据或降低置信度

**价值**
- 提升可靠性，避免单 Agent 过度自信输出

### 6.7 人工审批（HITL）
**能力**
- 中高风险动作强制审批
- 审批人可: 批准 / 拒绝 / 修改参数 / 要求补证据
- 审批后恢复执行（resume）

**高风险示例**
- 重启服务
- 回滚版本
- 修改配置
- 批量降级
- 清理队列/缓存

### 6.8 执行器（Executor Agent）
**能力**
- 执行低风险自动动作或已审批动作
- 记录执行结果、耗时、回执
- 失败时生成补偿/回滚建议

**约束**
- 所有动作带 `idempotency_key`
- 所有动作带 `risk_level`
- 所有动作写审计日志

### 6.9 RCA 报告生成（Postmortem）
**输出内容**
- 事件概述
- 时间线（检测/诊断/执行/恢复）
- 根因分析
- 处置动作与结果
- 影响范围
- 后续预防项（Action Items）
- Agent 表现总结（证据不足/误判点）

### 6.10 知识沉淀（RAG 写回，P1）
**能力**
- 将人工确认的高质量 RCA 与处置经验写入知识库
- 为后续 Planner/Diagnose 提供检索证据

**规则**
- 不写入未经确认的推断
- 区分“已验证结论”与“候选假设”

## 7. 非功能需求（生产级关键点）

### 7.1 可控性
- 最大步骤数（step limit）
- 节点超时与重试
- 风险动作审批门禁
- 明确失败出口（转人工/终止）

### 7.2 可恢复性
- 每个 run 支持 checkpoint
- 系统重启后可恢复
- 审批挂起后可恢复
- 幂等保障避免重复副作用

### 7.3 可观测性
- 节点级 trace
- 工具调用日志
- token / latency / cost 指标
- 失败分类与回放能力

### 7.4 安全性
- 工具权限白名单
- 环境隔离（dev/staging/prod）
- 审批人身份校验
- 操作审计日志
- Prompt/Tool Injection 基础防护

### 7.5 性能
- 首次计划生成目标 < 5s（可根据模型/环境调整）
- 证据收集并行化
- 支持流式进展反馈（查日志/查指标/等待审批）

## 8. 产品架构（面试展示视角）

### 8.1 总体架构分层
1. **UI / API 层**
   - 工单提交、审批页面、执行态页面、RCA 页面
2. **Agent Orchestrator（LangGraph）**
   - 状态机、节点图、中断恢复、条件路由
3. **Tool Adapter 层**
   - 日志、指标、部署、Runbook、执行动作适配器
4. **RAG / Knowledge 层**
   - 历史 RCA、Runbook、FAQ 检索与写回
5. **Persistence & Observability 层**
   - Checkpoint、业务数据表、trace、metrics

### 8.2 为什么选 LangGraph（产品层面的理由）
- 场景是有状态、多步骤、可中断、可恢复的工作流
- 需要显式图编排和条件路由（非纯自由 ReAct）
- 需要 HITL 审批与 resume
- 需要并行证据收集和聚合
- 需要清晰失败路径与回退策略

## 9. 多 Agent 角色设计（P1）

### 9.1 Planner Agent
输入: ticket + triage + memory evidence  
输出: 结构化排查计划（steps、优先级、停止条件）

### 9.2 Investigator Agent
执行证据采集步骤，聚合工具结果并生成证据摘要，支持并行 fanout。

### 9.3 Critic Agent
检查证据-结论一致性，发现证据不足/冲突时要求补证据或降低置信度。

### 9.4 Executor Agent
将方案转为动作，走审批门禁，执行并验证结果，输出执行回执与失败补偿建议。

> 面试表达建议: 先逻辑角色拆分，再决定是否物理多 Agent/多模型化。

## 10. 数据源与平台集成（阿里云运维平台）

你已有阿里云运维平台（MySQL / K8s / 负载均衡 / OSS）可直接作为真实数据源，显著提升项目真实性。

### 10.1 数据源映射
- **MySQL**: 工单、告警、变更记录、服务元数据
- **K8s**: Pod/Deployment/Events/日志摘要（核心排障证据）
- **SLB/ALB**: 流量、错误率、后端健康状态（入口层证据）
- **OSS**: 日志归档、RCA 报告/证据包归档

### 10.2 分阶段接入策略（强烈建议）
- **Phase A（只读）**: MySQL + K8s + SLB + OSS Read/Write 报告
- **Phase B（模拟执行）**: 写建议动作与审批记录，不执行生产动作
- **Phase C（受控执行）**: 小范围接入非生产/白名单低风险动作

### 10.3 集成原则
- LLM 不直接拿到云凭证
- 所有平台能力通过 Internal Tool Gateway 暴露
- 环境显式化（dev/staging/prod）
- service / namespace / bucket prefix 白名单

## 11. 指标体系（产品侧成功标准）

### 11.1 核心业务指标
- 平均故障处置时长（MTTR）
- 首次建议采纳率
- 执行成功率（含 mock/真实执行）
- 审批平均等待时长
- 故障复盘产出率（有 RCA 的工单占比）

### 11.2 Agent 工程指标
- 根因 Top1 / Top3 命中率（离线评测）
- 风险等级判断准确率
- 漏审批率（严重）
- 不必要审批率（过于保守）
- 平均步骤数 / 平均延迟 / 平均成本

## 16. 风险与边界（产品视角）

### 16.1 主要风险
- 真实平台数据质量不一致（命名、时间戳、映射关系）
- 日志/事件数据过大导致上下文膨胀
- 权限与凭证管理不当带来安全风险
- 执行类动作误操作风险

### 16.2 控制策略
- Tool Gateway + 白名单 + 参数校验
- 规范化层（service metadata normalization）
- 日志摘要化、时序降采样、响应大小限制
- 高风险动作审批 + 幂等 + 审计

---

## 17. 未来演进方向
- 多租户与团队隔离（结合你现有平台能力）
- 更强的策略引擎（风险评分、双人审批）
- 图内 Planner/Executor 使用不同模型以优化成本
- 自动回滚与灰度策略（在强审计前提下）
- 失败案例自动入评测集，形成持续优化闭环

## 18. 结论

OpsPilot Agent 是一个非常适合面试“Agent 应用开发”的项目方向，因为它同时展示了:
- LangGraph 的核心价值（状态化编排、可中断恢复、条件路由）
- LangChain 的实用价值（工具、模型、RAG 集成）
- 生产级工程能力（安全、审计、可观测、评测、幂等）
- 真实平台集成能力（阿里云运维平台数据源）

如果按本文档落地到 MVP + P1，你的项目已经足以证明你不仅会“搭智能体”，而且理解如何把智能体做成可上线的系统。
