# 后端关键问题总结

本文总结当前后端实现中的关键问题。

## 范围

- Graph 编排与证据流
- Run 状态持久化
- 事件流可靠性
- 工具网关稳定性
- 配置一致性
- 文档状态与真实实现对齐

---

## 1. 证据采集节点已实现，但未接入当前主图

### 现状

- `app/graph/nodes/__init__.py` 中已有 `logs_node`、`metrics_node`、`deployments_node`、`runbook_node`。
- `app/graph/builder.py` 当前主链路是：
  - `intake -> triage -> planner -> evidence_aggregate -> diagnose -> critic -> remediation -> rca`
- 图上没有边把执行流路由到上述证据查询节点。

### 影响

- 证据列表经常为空或强度不足。
- 诊断与 RCA 更依赖 fallback 逻辑和模型推断，而不是实际证据。
- 端到端流程与设计目标不一致。

### 修复建议

- 在主图中接入证据节点（或使用 fanout/fanin 模式）。
- 确保 `evidence_aggregate` 在证据节点执行完成后再运行。
- 增加集成测试，断言正常 run 中包含关键证据类别。

### 优先级

- `P0`

---

## 2. Graph 执行过程中 Run 状态未持续回写数据库

### 现状

- `incident_runs` 会在创建时写入 DB。
- Graph 在内存中更新 state，但状态迁移没有持续回写到 `incident_runs`。

### 影响

- `/incidents/runs/{run_id}` 可能返回滞后状态。
- UI 与值班人员无法可靠追踪实时进度。
- 基于 DB 状态的恢复/重试逻辑不稳定。

### 修复建议

- 每个节点或关键阶段后持久化一次 run 状态。
- 在终态写入完成时间和失败原因。
- 增加状态流转测试（`NEW -> ... -> COMPLETED/FAILED`）。

### 优先级

- `P0`

---

## 3. 事件流仅存于进程内存（`events_db`）

### 现状

- `app/api/incidents.py` 中事件写入进程内列表 `events_db`。
- 没有落库，也没有外部事件存储。

### 影响

- 进程重启后事件全部丢失。
- 多实例部署无法共享事件历史。
- 长时运行与审计场景无法支撑。

### 修复建议

- 将事件持久化到 DB（新增 `incident_events` 表）或持久化流系统。
- 增加保留策略，并按 `run_id + timestamp` 建索引查询。
- 内存缓存仅作为可选读优化。

### 优先级

- `P1`

---

## 4. 工具网关审计日志存在函数签名不一致问题

### 现状

- `app/tools/gateway.py` 中 `_log_audit` 定义参数为：
  - `(request, latency_ms, success, result)`
- 成功路径调用时传了额外参数 `_log_audit(..., adapter_info)`。

### 影响

- 成功工具调用路径可能因 `TypeError` 失败。
- Graph 稳定性与证据采集受到影响。
- 调试信号被污染，因为审计流程本身报错。

### 修复建议

- 对齐 `_log_audit` 定义与所有调用点。
- 增加工具执行成功/失败路径单测。
- 以兼容方式记录适配器来源（`mock`/`real`）。

### 优先级

- `P0`

---

## 5. 数据库配置来源分裂（Settings 与仓储层不一致）

### 现状

- `Settings.database_url` 默认是 `sqlite+aiosqlite:///./opspilot.db`。
- 仓储层硬编码使用 `sqlite:///./incidents.db`。

### 影响

- 运行时可能读写不同数据库。
- 环境化配置部分失效。
- 本地调试与部署行为可能不一致。

### 修复建议

- 仓储层统一读取 `Settings.database_url`。
- 保留单一权威 DB URL 来源。
- 启动时校验并打印一次生效 DB URL。

### 优先级

- `P0`

---

## 6. 文档状态方向基本正确，但掩盖了关键运行时缺口

### 现状

- `docs/gaps/status.md` 将部分模块标记为 `production-ready`。
- 但主流程接线、持久化、工具网关稳定性仍有关键缺口。

### 影响

- 团队可能高估生产可用性。
- 规划与风险评估偏乐观。
- 新成员上手时容易忽略关键约束。

### 修复建议

- 为状态定义补充明确验收标准。
- 增加“阻塞问题”章节，关联具体文件和测试。
- 按 owner/date/pr 跟踪问题关闭状态。

### 优先级

- `P1`

---

## 建议执行顺序

1. 修复工具网关签名不一致问题（`P0`）。
2. 统一数据库配置来源（`P0`）。
3. 将证据节点接入主图（`P0`）。
4. 持久化 run 状态流转（`P0`）。
5. 让事件流可持久化（`P1`）。
6. 更新状态文档与验收标准（`P1`）。
