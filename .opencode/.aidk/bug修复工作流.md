# 分三层

## 1、输入层：两种入口

### DTS单号（DTS即缺陷单），通过调用MCP服务（getDTSDetail），自动拉取缺陷单的标题、描述、状态、附件等完整信息

### 自然语言描述

## 2、根因分析层

### 4个阶段：phase1 根因调查 -> phase 2 模式分析 -> phase 3 假设验证 -> phase 4 实施修复

### 具体手段

#### 代码搜索：提取bug中的类名、方法名、错误消息，用 Grep/Glob/语义搜索找到相关代码

#### 数据流追踪：从报错点回溯调用链（getRemoteCallChain），直到找到值来源

#### 对比分析：在代码库找到相似代码，对比差异点

#### 附件分析：缺陷截图，视觉理解

#### 假设驱动：必须先形成 “我认为X是根因，因为Y”的假设，做最小的改动验证

## 3、强制约束

### 明确“修修补补 = 失败”，必须找到根因才能动代码

### 自动熔断机制：3次修复失败，系统自动熔断
