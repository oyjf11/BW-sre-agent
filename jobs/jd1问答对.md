## Agent 基础与架构

1. 【高频核心】Q：你怎么理解 AI Agent？  
   A：Agent 不是单次问答模型，而是一个能围绕目标进行规划、调用工具、观察结果、更新状态并持续迭代的系统。核心包括 LLM、状态、工具、记忆、规划、执行和反馈闭环。

2. 【高频核心】Q：Agent 和普通 Chatbot 有什么区别？  
   A：Chatbot 主要生成文本回答，Agent 会主动拆解任务、调用外部系统、执行动作，并根据结果调整下一步。比如客服机器人是 Chatbot，而能查订单、判断原因、发起补偿流程的是 Agent。

3. 【高频核心】Q：Agent 系统的核心模块有哪些？  
   A：任务理解、规划器 Planner、工具调用 Tool Use、状态管理 Memory/State、执行器 Executor、观察器 Observation、反思/校验 Critic，以及日志监控和人工干预机制。

4. 【高频核心】Q：什么是 Agent Loop？  
   A：Agent Loop 是“思考 -> 行动 -> 观察 -> 再思考”的循环。模型先判断下一步，调用工具得到结果，再基于结果继续规划，直到任务完成或触发终止条件。

5. 【高频核心】【项目强相关】Q：Agent Loop 最大风险是什么？  
   A：无限循环、重复调用工具、错误动作、成本失控和状态漂移。因此需要最大步数、超时、预算限制、工具白名单、幂等控制和人工审批。

6. 【高频核心】【项目强相关】Q：什么是 Workflow Agent？  
   A：Workflow Agent 是把 Agent 行为约束在明确业务流程中的 Agent，例如客服退款流程、库存补货流程、广告投放优化流程。它比完全自由的 Agent 更可控，更适合生产环境。

7. 【高频核心】【系统设计】Q：什么时候用 Workflow Agent，什么时候用自由 Agent？  
   A：强业务约束、高风险、需要审计的场景用 Workflow Agent；探索性分析、低风险问答、开放式调研可以用更自由的 Agent。

8. Q：Agent 为什么需要状态管理？  
   A：因为 Agent 是多步任务，需要记录用户目标、历史决策、工具结果、中间证据和执行进度。没有状态，Agent 只能做单轮反应。

9. Q：Agent 状态一般包含什么？  
   A：任务目标、当前步骤、计划、工具调用记录、观察结果、错误信息、重试次数、最终输出、人工审批状态等。

10. 【高频核心】【系统设计】Q：如何设计一个可扩展 Agent 架构？  
    A：把 LLM、Planner、Tool Registry、Executor、Memory、Evaluator、Workflow Engine 解耦；工具通过统一 schema 注册；流程通过图或状态机编排；日志、指标、审计独立建设。

## LangGraph / Graph-based Agents

11. 【高频核心】【项目强相关】Q：LangGraph 适合解决什么问题？  
    A：适合复杂 Agent 工作流，尤其是有分支、循环、重试、人工中断、多节点协作的场景。

12. 【高频核心】Q：LangGraph 和 LangChain 的区别？  
    A：LangChain 更偏组件和链式调用，LangGraph 更偏状态图和流程编排。生产 Agent 通常更需要显式状态机，所以 LangGraph 更适合复杂工作流。

13. 【高频核心】【项目强相关】Q：为什么 Graph-based Agent 比 Chain 更适合生产？  
    A：Chain 多是线性流程，Graph 可以表达条件分支、循环、回退、审批、中断和恢复，更符合真实业务流程。

14. 【高频核心】Q：LangGraph 中 node 是什么？  
    A：node 是工作流中的执行单元，可以是 LLM 调用、工具调用、业务规则判断、数据处理或人工审批节点。

15. 【高频核心】Q：LangGraph 中 edge 是什么？  
    A：edge 表示节点之间的流转关系。普通 edge 固定流转，conditional edge 根据状态决定下一步。

16. 【高频核心】【项目强相关】Q：LangGraph 中 state 有什么作用？  
    A：state 是所有节点共享的数据上下文，节点读取 state，返回部分更新，Graph 根据状态继续路由。

17. 【高频核心】【项目强相关】Q：Graph 中如何防止死循环？  
    A：设置最大步数、最大循环次数、递归限制、状态去重、重复工具调用检测和失败终止条件。

18. 【高频核心】【项目强相关】Q：审批流怎么接入 LangGraph？  
    A：可以设计 risk_gate 节点判断风险，高风险进入 approval_interrupt 节点中断流程，审批通过后从安全节点恢复执行。

19. 【高频核心】【项目强相关】Q：LangGraph 如何支持 human-in-the-loop？  
    A：在关键节点保存状态并中断，等待人工输入后恢复图执行。适合高风险动作、低置信度诊断、重要业务决策。

20. 【系统设计】Q：多 Agent 协作适合怎么用图表达？  
    A：可以把不同 Agent 抽象为节点，例如 Planner Agent、Research Agent、Executor Agent、Critic Agent，由图控制它们之间的信息流和决策流。

## Planning / ReAct / 多 Agent

21. 【高频核心】【项目强相关】Q：什么是 Planning？  
    A：Planning 是让模型先把目标拆解成可执行步骤，再逐步执行。它能提升复杂任务的可控性和可解释性。

22. 【高频核心】Q：Planning 常见问题是什么？  
    A：计划过大、不可执行、忽略约束、缺少验证步骤。需要结合工具可用性、业务规则和执行反馈动态修正。

23. 【高频核心】Q：什么是 ReAct？  
    A：ReAct 是 Reasoning + Acting，模型一边推理一边调用工具，根据观察结果继续推理。

24. 【高频核心】【系统设计】Q：ReAct 和 Plan-and-Execute 的区别？  
    A：ReAct 是逐步思考和行动交替，适合探索性任务；Plan-and-Execute 是先规划再执行，适合流程明确的复杂任务。

25. Q：生产中会直接暴露模型的 Chain of Thought 吗？  
    A：不会。生产系统通常只保留结构化推理摘要、决策依据和工具结果，不暴露完整隐藏推理过程。

26. Q：多 Agent 系统的优点是什么？  
    A：可以分工，例如一个负责规划，一个负责检索，一个负责执行，一个负责审核，降低单个 Agent 职责复杂度。

27. Q：多 Agent 系统的缺点是什么？  
    A：通信成本高、协调复杂、延迟增加、容易互相放大错误，需要明确角色边界和终止条件。

28. Q：什么时候不建议上多 Agent？  
    A：任务简单、流程固定、单 Agent 加规则就能解决时，不应为了概念复杂化系统。

29. 【项目强相关】Q：Critic Agent 有什么作用？  
    A：检查计划、诊断或输出是否可靠，判断证据是否充分，发现矛盾并触发补充检索或重规划。

30. Q：Swarm / AutoGen 这类框架主要解决什么？  
    A：主要解决多 Agent 协作、角色分工、消息传递和任务交接问题，但生产落地仍需自己补可靠性、审计和权限控制。

## Tool Use / Function Calling

31. 【高频核心】Q：什么是 Function Calling？  
    A：Function Calling 是模型根据上下文生成结构化函数调用参数，由后端执行真实函数并把结果返回给模型。

32. 【高频核心】Q：Tool Use 和 Function Calling 有什么区别？  
    A：Function Calling 是实现方式，Tool Use 是更宽泛的能力，包括工具注册、权限、调用、结果观察和失败处理。

33. 【高频核心】【项目强相关】Q：为什么工具调用要用 schema？  
    A：schema 能约束参数类型、必填字段和结构，减少模型生成非法调用的概率。

34. 【高频核心】【项目强相关】Q：工具调用为什么不能让模型直接执行？  
    A：模型可能生成错误、危险或越权操作，必须由后端做校验、鉴权、审计和风险控制。

35. 【高频核心】【系统设计】Q：Tool Registry 应该包含哪些信息？  
    A：工具名称、描述、参数 schema、返回结构、超时时间、重试策略、风险等级、是否需要审批、权限范围。

36. 【高频核心】【项目强相关】Q：工具调用失败怎么处理？  
    A：区分临时错误和永久错误。临时错误可重试，永久错误要降级或终止，并记录错误上下文。

37. 【高频核心】【项目强相关】Q：工具调用为什么需要幂等？  
    A：Agent 可能重试或恢复执行，同一个动作不能重复产生副作用，比如重复发券、重复退款、重复修改库存。

38. 【高频核心】【项目强相关】Q：什么是工具白名单？  
    A：只允许 Agent 调用经过注册和授权的工具，防止模型访问未授权能力。

39. 【高频核心】Q：工具结果需要返回给模型原始数据吗？  
    A：不一定。应返回必要、脱敏、结构化摘要，避免 token 过大和敏感信息泄露。

40. 【高频核心】【项目强相关】Q：工具调用日志应该记录什么？  
    A：调用时间、调用人/Agent、工具名、参数摘要、结果、耗时、错误、trace id、审批信息。

## RAG / GraphRAG / 向量数据库

41. 【高频核心】【补短板】Q：什么是 RAG？  
    A：RAG 是 Retrieval-Augmented Generation，先从知识库检索相关信息，再把结果提供给模型生成答案。

42. 【高频核心】Q：RAG 解决什么问题？  
    A：解决模型知识过期、企业私有知识缺失、幻觉严重的问题。

43. 【高频核心】【补短板】Q：RAG 的核心链路是什么？  
    A：文档采集 -> 清洗切分 -> 向量化 -> 入库 -> 检索 -> 重排 -> 拼接上下文 -> 生成 -> 评估。

44. 【高频核心】【补短板】Q：chunk 切分太大或太小有什么问题？  
    A：太大噪声多、召回不精确；太小语义不完整、上下文断裂。需要按业务文档结构调参。

45. 【高频核心】【补短板】Q：向量检索和关键词检索区别？  
    A：向量检索基于语义相似度，关键词检索基于词匹配。生产中常用 hybrid search 结合两者。

46. 【高频核心】【补短板】Q：为什么 RAG 需要 rerank？  
    A：初召结果可能相关性不足，rerank 可以用更强模型重新排序，提高上下文质量。

47. 【高频核心】Q：RAG 常见幻觉来源是什么？  
    A：检索不到、检索错、上下文冲突、模型忽略证据、prompt 约束弱。

48. 【高频核心】【补短板】Q：如何评估 RAG 效果？  
    A：评估召回率、命中率、上下文相关性、答案忠实度、人工满意度、端到端任务成功率。

49. 【高频核心】Q：GraphRAG 和普通 RAG 区别？  
    A：GraphRAG 会构建实体和关系图谱，适合复杂关联分析；普通 RAG 更偏文档片段检索。

50. 【补短板】Q：电商场景中 RAG 可以用在哪里？  
    A：客服知识库、商品规则、平台政策、物流规则、售后 SOP、广告投放经验、选品分析报告。

## Prompt 策略与可控性

51. 【高频核心】Q：Prompt 工程在生产 Agent 中的作用是什么？  
    A：约束模型角色、任务边界、输出格式、工具使用规则和异常处理方式。

52. 【高频核心】【项目强相关】Q：为什么不能只靠 Prompt 保证可靠性？  
    A：Prompt 是软约束，模型仍可能违反。生产系统必须结合 schema、规则、权限、审批和后处理校验。

53. 【高频核心】Q：System Prompt 应该包含什么？  
    A：角色、目标、禁止行为、工具使用规则、输出格式、安全边界、业务优先级。

54. 【高频核心】Q：如何减少模型输出不稳定？  
    A：降低 temperature，明确 schema，提供 few-shot，使用 JSON mode 或 function calling，增加校验和重试。

55. 【高频核心】Q：Prompt 注入是什么？  
    A：用户或外部文档通过文本诱导模型忽略系统规则、泄露信息或执行越权动作。

56. 【高频核心】【项目强相关】Q：如何防 Prompt 注入？  
    A：隔离系统指令和外部内容，工具权限后端控制，输出校验，敏感操作审批，外部文档标记为不可信。

57. 【高频核心】【项目强相关】Q：Agent 干预机制有哪些？  
    A：人工审批、低置信度转人工、规则拦截、风险评分、最大预算、灰度开关、人工接管。

58. 【高频核心】【系统设计】Q：如何设计 Agent 置信度？  
    A：可以结合模型自评、证据数量、检索分数、工具成功率、规则命中、历史准确率综合计算。

59. 【高频核心】Q：低置信度时怎么办？  
    A：补充检索、请求用户澄清、转人工审核、只给建议不执行动作。

60. 【高频核心】【项目表达】Q：什么是可解释的 Agent 输出？  
    A：输出不仅给结论，还给依据、使用的数据、调用的工具、关键判断和风险提示。

## 电商 / 供应链 Agent 场景

61. 【补短板】【系统设计】Q：智能客服 Agent 需要哪些能力？  
    A：意图识别、订单查询、政策 RAG、工具调用、工单流转、情绪识别、转人工和安全边界。

62. 【补短板】Q：客服 Agent 最大风险是什么？  
    A：错误承诺、错误退款、泄露隐私、绕过规则。因此需要权限、审批、规则校验和日志审计。

63. 【补短板】【系统设计】Q：智能选品 Agent 可以怎么设计？  
    A：整合销量、搜索趋势、竞品价格、评论、广告成本、库存、物流时效，输出选品建议和理由。

64. 【补短板】Q：库存预测 Agent 需要哪些数据？  
    A：历史销量、季节性、促销活动、广告计划、供应周期、库存水位、缺货记录、退货率。

65. Q：定价策略 Agent 应考虑哪些因素？  
    A：成本、毛利、竞品价、库存压力、广告 ROI、销量弹性、平台规则、汇率和物流成本。

66. Q：智能广告 Agent 可以做什么？  
    A：分析投放效果、调整关键词、预算分配、出价策略、素材建议和 ROI 归因。

67. 【补短板】Q：供应链补货 Agent 的核心指标是什么？  
    A：缺货率、库存周转、滞销率、履约时效、补货提前期、资金占用和预测准确率。

68. Q：物流场景 Agent 可以解决什么？  
    A：异常包裹识别、路线优化、时效预测、赔付判断、客服解释和承运商评估。

69. 【补短板】Q：跨境电商 Agent 和普通电商 Agent 有什么额外难点？  
    A：多语言、多币种、汇率、关税、国际物流、平台规则差异、时区和本地化合规。

70. 【高频核心】【补短板】Q：业务 Agent 落地为什么要和产品、数据团队协作？  
    A：需要明确业务指标、数据口径、流程边界、用户体验和上线评估标准，不能只从技术角度设计。

## 推荐 / 预测 / NLP 算法能力

71. 【补短板】Q：推荐系统常见召回方式有哪些？  
    A：热门召回、协同过滤、内容召回、向量召回、规则召回、用户行为召回。

72. 【补短板】Q：召回和排序有什么区别？  
    A：召回从大规模候选中快速找出可能相关的集合，排序对候选进行精细打分。

73. Q：电商推荐常见特征有哪些？  
    A：用户画像、商品属性、价格、类目、点击、收藏、加购、购买、评价、库存、折扣、上下文时间。

74. 【补短板】Q：预测模型在库存场景怎么评估？  
    A：看 MAE、MAPE、RMSE，也要看业务指标，如缺货率、库存周转和滞销成本。

75. 【补短板】Q：NLP 意图识别怎么做？  
    A：可以用规则、传统分类模型、embedding 相似度或大模型分类。生产中常规则和模型结合。

76. Q：客服对话管理难点是什么？  
    A：多轮上下文、省略表达、情绪、意图切换、权限动作和转人工时机。

77. 【补短板】Q：RAG 和推荐系统中的召回有什么相似点？  
    A：都需要从大量候选中找相关内容，再进行排序或重排。

78. 【补短板】Q：大模型能替代传统推荐/预测模型吗？  
    A：不能完全替代。大模型擅长理解和生成，传统模型在大规模实时排序、数值预测、成本控制上仍有优势。

79. 【补短板】【系统设计】Q：Agent 和算法模型如何结合？  
    A：Agent 负责流程编排和决策解释，算法模型负责预测、排序、评分等专业子任务。

80. 【高频核心】【补短板】Q：业务指标和模型指标冲突怎么办？  
    A：以业务目标为准。模型 AUC 提升不代表 GMV、利润或客服满意度提升，必须做线上评估。

## 后端工程能力

81. 【系统设计】Q：Agent 后端为什么需要微服务思维？  
    A：LLM 服务、工具网关、检索服务、业务 API、任务调度、审计系统可能独立扩展和部署。

82. 【高频核心】Q：API 设计要注意什么？  
    A：资源建模清晰、请求响应稳定、错误码明确、幂等性、版本管理、鉴权和审计。

83. 【高频核心】【项目强相关】【系统设计】Q：长任务 API 怎么设计？  
    A：提交任务立即返回 task_id/run_id，后台执行，前端通过轮询、SSE 或 WebSocket 查询进度。

84. 【高频核心】【补短板】Q：消息队列在 Agent 系统中有什么用？  
    A：异步解耦、削峰填谷、任务重试、失败补偿、事件驱动和跨服务通信。

85. 【高频核心】【补短板】Q：缓存适合用在哪里？  
    A：热点知识、工具查询结果、模型配置、用户权限、商品基础信息、检索结果。

86. 【系统设计】【补短板】Q：缓存一致性怎么处理？  
    A：设置 TTL、主动失效、版本号、消息通知更新，关键数据仍以数据库为准。

87. 【高频核心】【项目强相关】【补短板】Q：并发下如何防止重复执行？  
    A：使用幂等 key、数据库唯一约束、分布式锁、状态机校验和事务。

88. 【高频核心】【项目强相关】Q：如何设计 Agent 日志？  
    A：记录 run_id、trace_id、节点名、输入摘要、输出摘要、工具调用、耗时、错误、审批人和最终结果。

89. 【高频核心】【项目强相关】【补短板】Q：Agent 监控指标有哪些？  
    A：任务成功率、平均耗时、工具失败率、LLM 调用成本、人工介入率、幻觉率、重试率、用户满意度。

90. 【高频核心】【补短板】Q：如何做 Agent 灰度上线？  
    A：按用户、业务线、场景或流量比例灰度；先建议模式，再半自动，再自动执行；实时监控和可回滚。

## 落地闭环与项目表达

91. 【高频核心】【项目表达】Q：一个 Agent 项目从 0 到 1 怎么落地？  
    A：先选低风险高价值场景，定义业务指标，梳理流程和数据，设计工具和状态机，做 Demo，离线评估，小流量灰度，再逐步自动化。

92. 【系统设计】【项目表达】Q：如何判断一个场景适不适合 Agent？  
    A：看是否需要多步决策、工具调用、上下文理解和动态规划。如果只是固定规则 CRUD，不一定需要 Agent。

93. 【高频核心】【项目表达】Q：如何把模糊需求拆成工程方案？  
    A：先明确目标和边界，再拆输入、输出、流程、依赖工具、数据源、异常分支、评估指标和上线策略。

94. 【高频核心】【项目表达】Q：Agent 生产环境最重要的三件事是什么？  
    A：可控、可靠、可观测。具体包括权限、审批、幂等、日志、回滚、监控和人工接管。

95. 【高频核心】【补短板】【项目表达】Q：如何评估 Agent 是否真的有效？  
    A：不能只看 demo，要看任务完成率、人工节省时长、错误率、用户满意度、成本、线上稳定性和业务收益。

96. 【补短板】【项目表达】Q：如果业务方说“做一个智能选品 Agent”，你会先问什么？  
    A：目标是什么，是提高 GMV、毛利还是降低库存？数据有哪些？决策频率多高？结果由谁审核？失败成本是什么？

97. 【高频核心】【项目强相关】【项目表达】Q：如果 Agent 给出错误建议，系统如何兜底？  
    A：保留证据链，输出置信度，重要动作人工审批，低置信度转人工，提供回滚和复盘机制。

98. 【高频核心】【项目表达】Q：Side Project 面试时怎么介绍更有说服力？  
    A：不要只说用了什么模型，要讲清楚业务问题、架构、工具调用、状态管理、评估指标、失败处理和上线方式。

99. 【补短板】【项目表达】Q：你这个 SRE Agent 项目如何迁移到电商 Agent？  
    A：底层模式类似：工单/业务目标输入 -> 规划 -> 工具取数 -> 诊断/决策 -> 风险控制 -> 执行/建议 -> 验证 -> 报告。只需要替换工具、数据源和业务策略。

100. 【高频核心】【项目表达】Q：如果面试官问“你为什么适合这个岗位”，怎么答？  
     A：可以说：我做过端到端 Agent 后端，从任务状态机、工具调用、LLM fallback、日志监控、审批和执行可靠性都有实践；虽然当前项目是 SRE 场景，但架构能力可以迁移到客服、选品、补货、广告和供应链自动化。重点不是会调模型，而是能把不确定的大模型能力落成可控的业务系统。
