# 01 — 项目架构

> 写给前端工程师看的 OpsPilot 项目架构说明。用 TS 世界的概念做类比，帮你快速建立心智模型。

---

## 1. OpsPilot 是什么

OpsPilot 是一个 **SRE 故障处置智能体（Agent）**。当系统出故障时，它能自动执行以下全流程：

```
收到告警/工单 → 分析问题 → 收集证据 → 诊断根因 → 制定修复方案 → 审批 → 执行 → 验证 → 输出报告
```

类比：就像前端的 CI/CD pipeline，只是这里编排的不是 build → test → deploy，而是 diagnose → remediate → verify。

## 3. 全栈数据流

用一句话概括：**用户（或告警系统）发起请求 → 触发一个 LangGraph 工作流 → 每个步骤的结果写数据库 → 前端轮询/SSE 展示进度**。

```
┌─────────────┐     POST /incidents/runs      ┌──────────────┐
│   前端 UI    │ ──────────────────────────────→│   FastAPI     │
│  (React)    │                                │   (Python)   │
└─────────────┘                                └──────┬───────┘
     ▲                                               │
     │  GET /runs/:id                       启动后台任务
     │  SSE /runs/:id/events                      ↓
     │                                      ┌──────────────┐
     │                                      │  GraphRunner  │
     │                                      │ (LangGraph)   │
     │                                      └──────┬───────┘
     │                                             │
     │                              节点执行 → 写 DB → 发事件
     │                                             │
     └──────────────────── 轮询 / SSE ←────────────┘
```

## 4. 三条核心主线

### 4.1 请求处理线

```
POST /incidents/runs （三种输入模式之一）
  → IntakeService 验证输入，创建 IncidentTicket
  → RuntimeService 在 DB 创建 run 记录
  → 后台启动 GraphRunner.run()
  → 返回 run_id + status 给前端
```

**三种输入模式（互斥）**：

| 模式 | 说明 | 类比 |
|------|------|------|
| 手工工单 | 用户在前端填表单，所有字段必填 | 手动创建 issue |
| ticket_id 拉取 | 只提供 ticket_id，后端从外部系统拉取详情 | 用 ID 查已有订单 |
| 告警事件 | 监控系统推送告警，包含 alert_name/service/env/severity | Webhook 回调 |

### 4.2 Agent 工作线

```
intake → triage → retrieve_memory → planner
  → evidence_fanout → evidence_aggregate → diagnose
  → critic → remediation → risk_gate
  → approval_interrupt / executor → verify → rca
```

每个节点是一个纯函数：`(state) => state`。就像 Redux reducer，但允许异步且可以有副作用（调用工具、写数据库）。

### 4.3 持久化线

每个节点执行完后，GraphRunner 自动做三件事：
1. **同步 run 进度** → UPDATE `incident_runs` 表
2. **保存 checkpoint** → INSERT `incident_checkpoints` 表（完整 state 快照）
3. **持久化 evidence** → INSERT `incident_evidence` 表（去重）

前端通过两个途径获取数据：
- **轮询**：GET `/runs/:id`、`/runs/:id/evidence` 等
- **SSE**：GET `/runs/:id/events` 实时接收事件推送

## 5. 关键技术栈

| 层 | 技术 | 说明 |
|----|------|------|
| 后端框架 | FastAPI | Python 异步 Web 框架，类似 Express/Koa |
| Agent 编排 | LangGraph | 有向图状态机，节点+条件边 |
| LLM 调用 | OpenAI / MiniMax / DeepSeek | 三供应商可切换 |
| 向量存储 | ChromaDB | 轻量级向量数据库 |
| 索引框架 | LlamaIndex | 文档加载、分块、索引 |
| 关系数据库 | SQLite（默认） | 可替换为 PostgreSQL |
| ORM | SQLAlchemy + Alembic | 类似 TypeORM + migrations |
| 前端 | React + TypeScript | Vite 构建 |
| 重试库 | Tenacity | 类似 p-retry |

## 6. 前后端接口速查

### 6.1 创建工单（核心入口）

```bash
# 手工工单模式
POST /incidents/runs
{
  "ticket": {
    "ticket_id": "INC-001",
    "title": "支付服务5xx升高",
    "description": "...",
    "service": "payment-service",
    "env": "staging",
    "severity": "P2",
    "source": "manual"
  }
}
```

### 6.2 查询运行状态

```
GET /incidents/runs/:run_id          → 运行详情
GET /incidents/runs/:run_id/events   → 事件流
GET /incidents/runs/:run_id/evidence → 证据列表
GET /incidents/runs/:run_id/diagnosis → 诊断结果
GET /incidents/runs/:run_id/remediation → 修复方案
GET /incidents/runs/:run_id/trace    → 调用链追踪
```

### 6.3 审批操作

```
GET  /approvals/pending              → 待审批列表
POST /approvals/:id/decision         → { decision: "approved"/"rejected", comment: "...", approver: "human" }
```

## 7. 环境配置核心变量

```bash
# .env 文件位置：backend/.env

# LLM 配置（三选一）
LLM_PROVIDER=minimax          # minimax | openai | deepseek
MINIMAX_API_KEY=xxx
MINIMAX_GROUP_ID=xxx

# 工具适配器模式
TOOL_ADAPTER_MODE=mock        # mock → 返回仿真数据 | real → 调用真实平台

# RAG 开关
RAG_ENABLED=false             # 默认关闭，按需开启

# 数据库
DATABASE_URL=sqlite+aiosqlite:///./opspilot.db
```

## 9. 小结

如果你是前端工程师，理解这个项目最关键的三个「锚点」：

1. **State** = 一个大的 TypeScript 对象，在整个工作流中流转（就像 Redux store）
2. **Nodes** = 纯函数 `(state) => state`，类似 Redux reducer 但支持异步
3. **Graph** = 编排这些节点的有向图，节点之间通过条件边连接
