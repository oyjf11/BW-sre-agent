# 05 — 数据库设计

> 从 TS 前端工程师视角，讲解 OpsPilot 的数据库设计。

---

## 1. 技术选型

| 组件 | 选型 | 说明 |
|------|------|------|
| ORM | SQLAlchemy 2.0 | Python 生态最主流的 ORM，类似 TypeScript 的 TypeORM / Prisma |
| 迁移工具 | Alembic | SQLAlchemy 官方迁移工具，类似 Prisma Migrate / Knex migrations |
| 默认数据库 | SQLite | 本地文件数据库，零配置，开发友好 |
| 生产可选 | PostgreSQL | 只需改 `DATABASE_URL` 环境变量 |

---

## 2. 启动自动迁移

```python
# app/main.py — FastAPI lifespan
@asynccontextmanager
async def lifespan(app):
    # 启动时自动执行数据库迁移
    from alembic.config import Config
    from alembic import command
    alembic_cfg = Config("alembic.ini")
    command.upgrade(alembic_cfg, "head")
    yield
```

应用启动时自动跑 `alembic upgrade head`，保证数据库 schema 始终最新。类比：`prisma migrate deploy` 在启动时执行。

**向后兼容**：对于早期没有用 Alembic 创建的 SQLite 文件，`repositories/__init__.py` 中有自动检测和补丁逻辑（`ensure_sqlite_schema()` — 检查缺失列并 `ALTER TABLE ADD`）。

## 4. 逐表详解

### 4.1 `incident_runs` — 诊断运行主表

```sql
-- 创建运行的简化 SQL
CREATE TABLE incident_runs (
    run_id           TEXT PRIMARY KEY,      -- 运行唯一 ID，UUID
    thread_id        TEXT,                  -- LangGraph thread ID
    ticket_id        TEXT,                  -- 关联工单 ID
    status           TEXT NOT NULL,         -- 枚举：NEW/TRIAGED/.../COMPLETED/FAILED
    severity         TEXT,                  -- P0/P1/P2/P3
    service          TEXT,                  -- 服务名
    env              TEXT,                  -- 环境 staging/production
    current_node     TEXT,                  -- 当前卡在哪个节点（用于监控）
    input_source     TEXT,                  -- manual/ticket_id/alert_event
    schema_version   INTEGER DEFAULT 1,
    step_count       INTEGER DEFAULT 0,     -- 已执行步数
    last_error_code  TEXT,                  -- 最后一次错误码
    last_error_message TEXT,                -- 最后一次错误信息
    requires_human   BOOLEAN DEFAULT FALSE, -- 是否需要人工介入
    summary_json     TEXT,                  -- 摘要 JSON
    created_at       DATETIME NOT NULL,
    updated_at       DATETIME NOT NULL,
    started_at       DATETIME,
    completed_at     DATETIME,
    started_by       TEXT                   -- 发起人
);
```

**TypeScript 等价视角**：

```typescript
interface IncidentRun {
  run_id: string;
  thread_id: string;
  ticket_id: string;
  status: RunStatus;  // enum
  severity: 'P0' | 'P1' | 'P2' | 'P3';
  service: string;
  env: string;
  current_node: string;
  step_count: number;
  created_at: string; // ISO 8601
  updated_at: string;
}
```

### 4.2 `incident_checkpoints` — 状态检查点

```sql
CREATE TABLE incident_checkpoints (
    checkpoint_id   INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id          TEXT NOT NULL REFERENCES incident_runs(run_id),
    node_name       TEXT NOT NULL,         -- 哪个节点
    state_json      TEXT NOT NULL,         -- 完整 state 的 JSON 序列化
    created_at      DATETIME NOT NULL
);
```

每个节点执行完后自动保存一份完整 state 快照。用途：
- 审批中断后恢复执行（从 checkpoint 还原状态）
- 调试时回溯「执行到这一步时 state 什么样」

类比：Redux DevTools 的时间旅行。

### 4.3 `incident_evidence` — 证据条目

```sql
CREATE TABLE incident_evidence (
    evidence_id       INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id            TEXT NOT NULL REFERENCES incident_runs(run_id),
    tool_name         TEXT,                -- 哪个 tool 产生的
    category          TEXT,                -- 证据类别 log/metric/deployment/...
    source_ref        TEXT,                -- 来源引用
    summary           TEXT,                -- 摘要（200 字符以内）
    raw_payload_json  TEXT,                -- 完整原始数据 JSON
    created_at        DATETIME NOT NULL
);
```

evidence 去重：`GraphRunner` 内部维护 `_persisted_evidence_ids` set，同一 run 内不重复插入。

### 4.4 `incident_approvals` — 审批记录

```sql
CREATE TABLE incident_approvals (
    approval_id      INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id           TEXT NOT NULL REFERENCES incident_runs(run_id),
    action_json      TEXT,                 -- 待审批的 action 详情 JSON
    risk_level       TEXT,                 -- LOW/MEDIUM/HIGH/CRITICAL
    status           TEXT DEFAULT 'pending', -- pending/approved/rejected
    reason           TEXT,                 -- 审批原因
    expected_impact  TEXT,                 -- 预期影响
    rollback_plan    TEXT,                 -- 回滚方案
    approver         TEXT,                 -- 审批人
    comment          TEXT,                 -- 审批意见
    created_at       DATETIME NOT NULL,
    updated_at       DATETIME NOT NULL
);
```

审批流程：
```
executor 需要审批 → agent 创建审批 → status=pending
                   → 人工在 UI 操作 → POST /approvals/:id/decision
                   → status=approved/rejected
                   → 如果是 approved → resume graph → executor
```

### 4.5 `incident_actions` — 执行动作审计

```sql
CREATE TABLE incident_actions (
    action_id                 INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id                    TEXT NOT NULL REFERENCES incident_runs(run_id),
    idempotency_key           TEXT UNIQUE NOT NULL, -- 幂等键
    action_type               TEXT,                  -- rollback/restart/scale/...
    params_json               TEXT,                  -- 参数
    execution_status          TEXT DEFAULT 'pending', -- pending/running/completed/failed
    result_json               TEXT,                  -- 执行结果
    approval_id               INTEGER,               -- 关联的审批 ID
    attempt_no                INTEGER DEFAULT 1,     -- 第几次尝试
    executor_name             TEXT,                  -- 执行器名称
    request_json              TEXT,                  -- 完整请求
    verification_status       TEXT,                  -- 验证状态
    verification_details_json TEXT,                  -- 验证详情
    created_at                DATETIME NOT NULL,
    started_at                DATETIME,
    completed_at              DATETIME
);
```

**幂等设计**：`idempotency_key` 唯一约束，同一操作不会重复执行。类比：前端提交按钮的防重复点击。

### 4.6 `incident_rca_reports` — RCA 报告

```sql
CREATE TABLE incident_rca_reports (
    run_id                       TEXT PRIMARY KEY REFERENCES incident_runs(run_id),
    report_markdown              TEXT,          -- Markdown 格式报告
    root_cause                   TEXT,          -- 根因
    resolution                   TEXT,          -- 解决方案
    prevention_items_json        TEXT,          -- 预防措施 JSON
    confirmed_by_human           BOOLEAN DEFAULT FALSE, -- 是否经人工确认
    timeline_summary             TEXT,          -- 时间线摘要
    impact_assessment            TEXT,          -- 影响评估
    supporting_evidence_ids_json TEXT,          -- 支撑证据 ID 列表
    executed_action_ids_json     TEXT,          -- 执行的动作 ID 列表
    archive_ref                  TEXT,          -- OSS 归档引用
    created_at                   DATETIME NOT NULL
);
```

和 `incident_runs` 是一对一关系（run_id 同样是 PK）。确认后的报告会通过 RAG 写回向量存储，供未来检索。

### 4.7 `incident_tool_audits` — 工具调用审计

```sql
CREATE TABLE incident_tool_audits (
    audit_id      INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id        TEXT NOT NULL,              -- 有索引
    tool_name     TEXT NOT NULL,
    adapter_mode  TEXT,                       -- mock 还是 real
    request_json  TEXT,                       -- 完整请求体
    response_json TEXT,                       -- 完整响应体
    success       BOOLEAN,
    error_message TEXT,
    latency_ms    INTEGER,                    -- 耗时（毫秒）
    created_at    DATETIME NOT NULL
);

CREATE INDEX idx_tool_audits_run_id ON incident_tool_audits(run_id);
```

每次 `gateway.call_tool()` 都记录一条审计日志。用于：
- 调试时定位工具调用问题
- 统计工具耗时和成功率
- 合规审计

### 4.8 `incident_knowledge_writebacks` — 知识写回日志

```sql
CREATE TABLE incident_knowledge_writebacks (
    writeback_id  INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id        TEXT NOT NULL,              -- 有索引
    target        TEXT,                       -- 目标（chroma/oss）
    content_json  TEXT,                       -- 写回内容
    metadata_json TEXT,                       -- 元数据
    status        TEXT DEFAULT 'pending',     -- pending/success/failed
    error_message TEXT,                       -- 错误信息
    created_at    DATETIME NOT NULL
);

CREATE INDEX idx_writebacks_run_id ON incident_knowledge_writebacks(run_id);
```

记录每次知识写回操作（RCA → RAG、证据 → OSS）的成功/失败状态。

## 5. 数据写入时机

```
时间线                      写入操作
───────                    ────────
POST /incidents/runs       → INSERT incident_runs
intake 节点完成             → UPDATE incident_runs.status + INSERT checkpoint
triage 节点完成             → UPDATE incident_runs.status + INSERT checkpoint
retrieve_memory 完成        → INSERT incident_evidence (RAG 检索结果)
evidence_fanout 完成        → INSERT incident_evidence (每个 tool 的返回)
evidence_aggregate 完成     → UPDATE incident_runs.status
diagnose 节点完成           → INSERT incident_evidence (诊断结果)
executor 执行              → INSERT incident_actions
approval_interrupt         → INSERT incident_approvals
verify 节点完成             → INSERT incident_evidence (验证结果)
rca 节点完成               → UPSERT incident_rca_reports
                            → INSERT incident_knowledge_writebacks
每次 tool 调用              → INSERT incident_tool_audits
每次节点执行               → INSERT incident_run_events (NODE_STARTED/NODE_FAILED)
```

## 7. Repository 模式

所有数据访问通过 Repository 类实现，每个 repo 接受 SQLAlchemy `Session`：

```python
# 示例：RunsRepo
class RunsRepo:
    def __init__(self, session: Session):
        self.session = session

    def create(self, run: IncidentRun) -> IncidentRun: ...
    def update(self, run_id: str, **kwargs) -> IncidentRun: ...
    def get(self, run_id: str) -> IncidentRun | None: ...
    def list(self, limit=20, offset=0) -> list[IncidentRun]: ...

# 其他 Repo 类似
EventsRepo      → list_by_run(), list_incremental()
CheckpointsRepo → save(), load_latest()
EvidenceRepo    → insert(), list_by_run()
ApprovalsRepo   → create(), get(), update(), get_pending()
ActionsRepo     → insert(), get_by_idempotency(), mark_started(), mark_completed()
RcaRepo         → upsert(), get(), confirm()
```

类比：每个 Repo 就像一个 `PrismaClient` 的 service 封装，提供类型安全的 CRUD。

## 9. 从 TS 视角的关键理解

| TS 概念 | Python 等价 |
|---------|------------|
| `interface` / `type` | Pydantic `BaseModel` / SQLAlchemy `Base` |
| `prisma.schema` | Alembic migration + SQLAlchemy models |
| `prisma.user.findMany()` | `session.query(User).all()` 或 Repo 方法 |
| `z.object({...})` | `ToolRequest` Pydantic model |
| `Promise.all()` | `asyncio.gather()` |
| `try/catch` | `try/except` |
| `const x = await db.user.create(...)` | `session.add(user); session.commit()` |

## 小结

数据库设计的核心：

1. **一张主表**（runs）+ **七张子表**，通过 `run_id` 外键关联
2. **迁移驱动**：Alembic 管理 schema 变更，启动时自动执行
3. **Repository 层**：封装所有 SQL 操作，上层只调 Repo 方法
4. **默认 SQLite**：零配置，开发即用，生产可切 PostgreSQL
