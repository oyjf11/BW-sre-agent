> 生成日期：2026-06-03 | 用途：面试准备 / 架构速查

---

## 一、核心工作流图（LangGraph，13+1 节点）

工作流定义在 `backend/app/graph/builder.py`，由 `GraphRunner`（`backend/app/services/graph_runner.py`）驱动执行。每个节点完成后自动持久化 checkpoint + evidence + 进度更新。

```
dispatcher → intake → triage → retrieve_memory → planner → evidence_fanout
    → evidence_aggregate → diagnose → critic
        ├─ PASS → remediation → risk_gate
        │     ├─ LOW_ONLY → executor → verify
        │     │     ├─ SUCCESS → rca → END
        │     │     ├─ RETRYABLE → executor (retry, max 2)
        │     │     └─ FATAL → rca → END
        │     ├─ NEEDS_APPROVAL → approval_interrupt → END (等待人工)
        │     └─ BLOCKED → rca → END
        ├─ NEED_MORE_EVIDENCE → evidence_fanout (loop)
        └─ REPLAN/CONTRADICTION → planner (loop)
```

- 最多 2 轮循环（critic loop guard，`max_loop_count=2`）
- `recursion_limit=50`
- 入口支持 resume：dispatcher 根据 `_resume_from_node` 跳过已完成节点，从 node_intake / node_executor / node_risk_gate / node_evidence_fanout 恢复

## 二、13 个节点详解（按职责分类）

### 2.1 输入阶段

#### Node 1: `intake` — 工单摄入

| 项目 | 内容 |
|------|------|
| 代码位置 | `graph/nodes/__init__.py:37-67` |
| 辅助服务 | `services/intake.py:1-83`（IntakeService） |
| 模型 | `models/incident.py`（IncidentTicket） |
| 依赖 | `normalizers/env.py`、`normalizers/service_alias.py`、`normalizers/time_range.py` |

**三种输入模式（互斥）**：
1. **手工工单** — 所有字段必填（ticket_id、title、description、service、env、severity、source）
2. **ticket_id 拉取** — 调用 `query_ticket_by_id` 工具从外部系统拉取
3. **告警事件** — 从 alert_name/service/env/severity 自动构造 ticket

**核心逻辑**：IntakeService 调用 normalizers 规范化 env → service → time_range 三个字段，然后构建 `IncidentTicket` Pydantic 对象。

**输出**：规范化后的 ticket + 初始化 state（evidence_items/root_cause_candidates 空列表，status=NEW）

#### Node 2: `triage` — 分诊分类

| 项目 | 内容 |
|------|------|
| 代码位置 | `graph/nodes/__init__.py:179-212` |
| 模型 | `models/triage.py`（TriageResult） |
| 依赖 | `llm_client.py` |

**三级分诊策略**：

```
规则匹配 → 命中则直接返回
   ↓ 未命中
LLM 增强分诊 → JSON 解析成功则返回
   ↓ 未命中/LM失败
兜底分诊 → service_degradation / P2
```

**规则匹配的关键词-类型映射**：
- `deploy/rollback/release/版本/发布/回滚` → `deployment_regression`
- `cpu/memory/oom/disk/内存/磁盘` → `resource_exhaustion`
- `timeout/connection refused/503/502/依赖/超时` → `dependency_failure`

**LLM 分诊**：prompt 要求 JSON 格式返回 6 个字段（incident_type、severity、suspected_services、suggested_time_window、requires_immediate_human、rationale）。

**输出**：`TriageResult`，其中 `requires_immediate_human=True`（prod + P0/P1）时会写入 `user_context` 供后续节点使用。

### 2.2 调查阶段

#### Node 4: `planner` — 调查规划

| 项目 | 内容 |
|------|------|
| 代码位置 | `graph/nodes/__init__.py:215-343` |
| 模型 | `models/planning.py`（InvestigationTask、InvestigationPlan） |
| 设计模式 | 策略模式——不同 incident_type 不同任务优先级 |

**根据 incident_type 动态生成任务列表**：

| 故障类型 | 优先级 1 | 优先级 2 | 优先级 3 | 优先级 4+ |
|----------|----------|----------|----------|-----------|
| `deployment_regression` | query_deployments | query_logs, K8s status/events | K8s pods, metrics, DB slow | runbook |
| `resource_exhaustion` | query_metrics | K8s status/pods, DB variables/processlist | K8s events, DB slow, logs | deployments, runbook |
| `dependency_failure` | query_logs | metrics, K8s status/events, DB processlist/slow | K8s pod logs, LB, runbook | deployments |
| 通用(unknown) | — | logs, metrics, K8s status/pods | K8s events, DB, deployments, runbook | — |

**任务属性**：每个 `InvestigationTask` 包含 task_id、category、tool_name、priority(1-10)、params（含 service/env）、degrade_on_failure、timeout_ms 等。

**输出**：`InvestigationPlan`（tasks list + rationale），同时维护向后兼容的 `plan` dict。

---

#### Node 5: `evidence_fanout` — 证据并行采集

| 项目 | 内容 |
|------|------|
| 代码位置 | `graph/nodes/__init__.py:346-449` |
| 依赖 | `tools/gateway.py`（ToolGateway）、`graph/evidence_utils.py`（classify_result） |

**核心机制**：

```
InvestigationPlan.tasks → asyncio.gather(*[_collect_one(...) for each task])
                                ↓
            ┌───────────────────┼───────────────────┐
            ↓                   ↓                   ↓
      ToolGateway            ToolGateway         ToolGateway
      .call_tool()           .call_tool()        .call_tool()
            ↓                   ↓                   ↓
       classify_result()    classify_result()    classify_result()
            ↓                   ↓                   ↓
      SUCCESS_USABLE?       FAILED_PARTIAL?      FAILED_RUNTIME?
            ↓                   ↓                   ↓
      EvidenceItem          (None, result)       (None, result)
            ↓                   ↓                   ↓
      append to             只记录结果，不产生     degrade=true 则不抛异常
      evidence_items         可用证据
```

**classify_result 分类逻辑**（`graph/evidence_utils.py`）：
- `SUCCESS_USABLE` — 工具成功 + 有结果数据 → 生成 EvidenceItem
- `FAILED_PARTIAL` — 工具成功但结构性不完整
- `FAILED_RUNTIME` — 工具抛出异常

**失败处理**：`degrade_on_failure=True` 的任务允许降级（不中断流程），`False`（如 execute_action）则抛 RuntimeError。

---

#### Node 6: `evidence_aggregate` — 证据聚合

| 项目 | 内容 |
|------|------|
| 代码位置 | `graph/nodes/__init__.py:577-620` |
| 依赖 | `graph/evidence_utils.py`（compute_quality_from_results） |

**聚合输出**：
- **分类统计**：按类别统计证据条目数（logs: 3, metrics: 2, k8s: 4 …）
- **质量评分**：`compute_quality_from_results()` 基于成功/失败/部分成功比例计算
- **缺失类别**：哪些预期类别没有收集到证据
- **矛盾检测**：同一摘要中出现 "error" + "healthy" → `contradiction_signals`

---

#### Node 7: `diagnose` — LLM 根因诊断

| 项目 | 内容 |
|------|------|
| 代码位置 | `graph/nodes/__init__.py:623-760` |
| 模型 | `models/root_cause.py`（RootCauseCandidate） |
| 依赖 | `llm_client.py` |

**诊断流程**：
1. 构建 evidence 摘要文本（按类别提取关键字段）
2. 构造 LLM prompt（含 Incident 上下文 + Collected Evidence）
3. LLM 返回 JSON 数组（每个 candidate 含 hypothesis/confidence/supporting_evidence/next_checks）
4. 正则提取 `[...]` → `json.loads` → 创建 RootCauseCandidate 列表

**兜底逻辑**：LLM 失败时生成两个静态假设：
- 「High resource usage causing degradation」（confidence=0.7）
- 「Recent deployment may have introduced the issue」（confidence=0.5）

### 2.3 决策阶段

#### Node 8: `critic` — 批判节点（4路路由）

| 项目 | 内容 |
|------|------|
| 代码位置 | `graph/nodes/__init__.py:763-822` |

**循环守卫**：`loop_count >= max_loop_count(2)` → 直接 NEEDS_HUMAN，设置 terminal_reason `EVIDENCE_LOOP_EXHAUSTED`

**4路决策矩阵**：

| 条件 | 路由 | 结果 |
|------|------|------|
| `loop_count >= 2` | NEEDS_HUMAN | 停止流程，进入 RCA 降级 |
| 有矛盾证据 (`contradicting_evidence_ids` 非空) | `CONTRADICTION` | 回到 planner |
| 质量分 < 0.4 或 缺失类别 >= 2 | `NEED_MORE_EVIDENCE` | 回到 evidence_fanout |
| 质量分 < 0.3 且 证据 < 2 | `REPLAN` | 回到 planner |
| 其他 | `PASS` | 进入 remediation |

---

#### Node 9: `remediation` — 修复方案生成

| 项目 | 内容 |
|------|------|
| 代码位置 | `graph/nodes/__init__.py:825-893` |
| 模型 | `models/remediation.py`(RemediationPlan)、`models/action.py`(ActionSpec) |

**动作决策**：

```
top_candidate.hypothesis 中是否包含 "deployment" +
  evidence_items 中是否有 category="deployments" 的证据？
    ├─ YES → rollback (HIGH risk, requires_approval=True)
    └─ NO  → restart (LOW risk, requires_approval=False)
```

**失败兜底**：无法生成任何动作时，设置 `NEEDS_HUMAN` + 错误码 `CANNOT_GENERATE_TRUSTED_ACTIONS`。

---

#### Node 10: `risk_gate` — 风险门禁

| 项目 | 内容 |
|------|------|
| 代码位置 | `graph/nodes/__init__.py:896-998` |
| 依赖 | `tools/policies/risk.py`、`policies/risk.py` |

**综合评估 6 个维度**：

1. 动作风险等级（HIGH / CRITICAL / LOW）
2. 环境（prod vs staging/dev）
3. 诊断置信度（top candidate.confidence）
4. triage 人工优先级（requires_immediate_human）
5. 循环守卫触发状态（loop_guard_triggered）
6. **能力预检**：`_can_execute_action()` 检查 execute_action 真实适配器是否可用

**4路输出**：

| 条件 | 路由 |
|------|------|
| execute_action 不可用 | `NEEDS_HUMAN`（AUTOMATION_CAPABILITY_UNAVAILABLE） |
| CRITICAL + prod + (低置信度 或 loop_guard) | `BLOCKED`（RISK_BLOCKED） |
| 需审批 或 HIGH 风险 或 loop_guard | `NEEDS_APPROVAL` |
| 其他 | `LOW_ONLY`（直接执行） |

### 2.4 执行阶段

#### Node 11: `approval_interrupt` — 审批中断

| 项目 | 内容 |
|------|------|
| 代码位置 | `graph/nodes/__init__.py:1001-1154` |
| 模型 | `models/approval.py`(ApprovalRequest、ApprovalResult) |

**核心流程**：
1. 从 RemediationPlan 第一个 action 构造 `ApprovalRequest`
2. **双路径持久化**（保证可靠性）：
   - 路径 A：直接 SQLAlchemy ORM 写入（`db.add(IncidentApproval)`）
   - 路径 B：ApprovalsRepo 写入（`approvals_repo.create()`）
   - 如果路径 A 失败，回退到路径 B
3. 状态置为 `WAITING_HUMAN` → graph 到达 END 暂停
4. 后续通过 `POST /approvals/{id}/decision` 或 `POST /incidents/runs/{id}/resume` 恢复

---

#### Node 12: `executor` — 执行修复

| 项目 | 内容 |
|------|------|
| 代码位置 | `graph/nodes/__init__.py:1157-1273` |
| 依赖 | `services/executor.py:1-291`（ControlledExecutor） |

**ControlledExecutor 执行流程**：

```
action_dict
    ↓
execute_plan() ──→ for each action:
    │                 1. 幂等检查（idempotency_key → 已 COMPLETED/EXECUTING 则跳过）
    │                 2. 前置条件检查（service_exists / deployment_healthy）
    │                 3. 创建 action_record（IncidentAction → INSERT）
    │                 4. 标记 started（attempt_no + started_at）
    │                 5. gateway.call_tool("execute_action", ...)
    │                 6. 标记 completed（COMPLETED / FAILED / ERROR）
    │                 7. 首次失败（非跳过）立即 break
    ↓
results (List[ExecutionResult])
```

**重试控制**：MAX_EXECUTOR_RETRIES = 2，由 verify 节点触发。

---

#### Node 13: `verify_outcome` — 结果验证

| 项目 | 内容 |
|------|------|
| 代码位置 | `graph/nodes/__init__.py:1276-1449` |

**三重验证（每次验证结果都作为新 EvidenceItem 追加）**：

| 检查 | 工具 | 通过条件 |
|------|------|---------|
| 1. K8s 部署状态 | `query_k8s_deployment_status` | status in (running, healthy, available) |
| 2. LB 健康检查 | `query_lb_health_status` | status in (healthy, ok, UP) |
| 3. 流量指标 | `query_lb_traffic_metrics` | error_rate < 0.05（CMS 不可用则标记 unavailable） |

**3路决策**：

| 条件 | 路由 |
|------|------|
| 全部检查通过 | `SUCCESS` → rca |
| 过半通过 或 全部无法运行 | `RETRYABLE_FAILURE` → 回到 executor（最多重试 2 次） |
| 多数失败 | `FATAL_FAILURE` → rca |

---

#### Node 14: `rca` — 根因分析报告（终点节点）

| 项目 | 内容 |
|------|------|
| 代码位置 | `graph/nodes/__init__.py:1452-1813` |
| 模型 | `models/rca.py`(RcaReport) |

**双模式**：

**A. NEEDS_HUMAN 降级模式**（无法完成自动化排查时）：
- 静态模板 Markdown 报告
- 包含：自动化停止原因（terminal_reason）、失败工具清单、缺失类别、manual_next_steps
- root_cause_status = "UNKNOWN"

**B. 正常 RCA 模式**：
- LLM 生成 JSON（root_cause / confidence / resolution / prevention_items / timeline / impact）→ 组装 Markdown
- `write_rca_to_oss` 归档报告到阿里云 OSS
- `write_evidence_to_oss` 归档原始证据 JSON
- 写入 `incident_rca_reports` 表

**RCA 确认后的知识写回**（`POST /runs/{id}/rca/confirm`）：
- `KnowledgeWritebackService.writeback()` → 写回 runbook
- `KnowledgeWritebackService.archive_rca()` → 归档

#### 9.2 RunStatus 枚举（12 种状态）

```
NEW → TRIAGED → PLANNED → GATHERING_EVIDENCE → DIAGNOSED
    → PENDING_APPROVAL → WAITING_HUMAN → NEEDS_HUMAN
    → EXECUTING → VERIFYING → COMPLETED / FAILED
```

#### 9.3 序列化（`graph/serde.py`）

**设计痛点**：Pydantic 对象经过 checkpoint 反序列化后变成 dict，导致 `ticket.service` 访问崩溃。

**解决方案**：
- `serialize_state()`：Pydantic → `.model_dump()` → dict → JSON
- `deserialize_state()`：JSON → dict（**不转回 Pydantic**）
- 所有节点代码使用防御性访问：`hasattr(obj, "xxx")` else `obj.get("xxx")`
- 遍历列表元素时防 None：`for a in actions: if a is None: continue`

### 十一、关键设计模式与已知陷阱

#### 设计模式

| 模式 | 应用位置 | 说明 |
|------|---------|------|
| **Strategy** | planner 节点 | 不同 incident_type 不同任务生成策略 |
| **Template Method** | GraphRunner.run() | 统一节点执行模板（事件 + checkpoint + evidence + RCA 持久化） |
| **Chain of Responsibility** | triage 节点 | 规则 → LLM → 兜底 三级传递 |
| **Observer** | EventBus | 异步事件发布/订阅，SSE 实时推送 |
| **Singleton** | ToolGateway | 全局单例工具网关 |
| **Adapter** | tools/adapters/ | mock ↔ real 双模式，统一接口 |
| **Repository** | repositories/ | 数据访问抽象，隔离 SQLAlchemy |
| **State Machine** | IncidentAgentState + RunStatus | 12 种状态 + 条件路由 |

#### 已知陷阱（来自 AGENTS.md）

1. **Pydantic vs dict 二义性** — checkpoint 反序列化后对象变 dict，必须用 `hasattr(obj, "field") or obj.get("field")` 兼容访问
2. **Evidence 持久化** — evidence_items 在 graph state 内存中，由 GraphRunner._persist_evidence() 在每个节点完成后写入 DB
3. **LLM 配置** — 默认 minimax，支持 openai/deepseek 切换；dev 环境不强制校验 key；LLM 调用失败有 fallback 逻辑
4. **Tool Adapter** — 默认 mock，real 模式下 4 个未实现的适配器会 fail-closed（抛 RuntimeError）
5. **app_env=DEV 不校验 LLM key** — `validate_for_production()` 在 dev 下跳过 API key 检查
