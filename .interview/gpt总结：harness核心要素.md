你说得对。你关注的不是“大企业怎么落地一套产研平台”，而是：

从这套大规模 Harness 架构中，抽象出不随规模变化的底层设计原则。

这些原则既适用于企业级研发平台，也适用于一个 Bug 修复 Agent、一个测试 Agent，甚至一次复杂工具调用。

我认为可以提炼为下面这套 Harness 不变量。

1. 先定义执行环境，再讨论 Agent 智能

Harness 的第一原则不是选模型，也不是写 Prompt，而是先回答：

Agent 知道什么
Agent 能做什么
Agent 必须遵守什么
Agent 如何被控制
Agent 如何证明自己做对了

对应你看到的架构：

认知层：知道什么
工具层：能做什么
契约层：必须遵守什么
编排层：如何执行
证据与评审：如何证明
知识沉淀：如何改进

这几个问题在任何规模下都成立。

一个小 Bug Agent 同样需要：

- 读取哪些代码和 Issue
- 允许使用哪些工具
- 禁止修改哪些目录
- 先复现还是先分析
- 修复完成的判定标准是什么
- 失败经验是否需要沉淀

所以 Harness 本质不是“包住模型的一层代码”，而是：

围绕模型建立一个受控的任务执行环境。

⸻

2. 将知识、能力、约束分离

这套架构非常重要的一个思想，是没有把所有东西都塞进 Prompt。

它把输入拆成三类：

知识

用于帮助 Agent 理解和判断：

代码
文档
历史经验
业务背景
运行数据

工具

用于让 Agent 执行动作：

读取文件
修改代码
调用 API
运行测试
查询数据库

契约

用于限制 Agent 行为和定义合格标准：

代码规范
安全规则
输出 Schema
质量门禁
权限边界

这三者不能混在一起。

例如：

“项目使用 Vue3”

可能是知识。

“必须使用内部表单组件”

是契约。

“可以调用组件检索工具”

是工具能力。

如果混在一个 Prompt 中，Agent 很难区分：

- 哪些是参考信息
- 哪些是强制要求
- 哪些是可执行动作

因此一个通用设计原则是：

Context、Capability、Constraint 必须分离建模。

无论系统大小，这个原则都适用。

⸻

3. 用确定性系统约束非确定性模型

这是 Harness 最核心的不变量。

LLM 是概率系统，它可能：

- 漏步骤
- 跳步骤
- 自行扩大范围
- 提前宣布完成
- 错误理解工具结果

所以不能让模型自己掌握全部流程控制权。

正确结构是：

确定性的 Harness
↓
调度非确定性的 Agent

Harness 负责：

- 顺序
- 状态
- 条件
- 权限
- 重试
- 超时
- 退出标准

Agent 负责：

- 理解
- 推理
- 生成
- 判断

一个小型修复 Agent 也应如此：

复现成功
→ 才允许根因分析
根因证据充分
→ 才允许修改代码
代码修改完成
→ 必须运行测试
测试未通过
→ 不允许宣布完成

所以 Harness 的核心作用不是帮助 Agent“更聪明”，而是：

防止 Agent 用不确定性破坏工程流程的确定性。

⸻

4. 将复杂任务拆成可编排能力

这套架构中，编排层并不是一个大 Agent，而是将工作拆成多个节点。

这是另一个不随规模变化的原则：

不要让一个 Agent 同时负责理解、决策、执行和验收。

应该拆成不同责任：

理解任务
→ 构建上下文
→ 制定方案
→ 执行
→ 验证
→ 评审

大型系统中可能是多个 Agent。

小型系统中，也可以只是一个 Agent 的多个阶段，但阶段之间必须有边界。

例如一个测试 Agent：

分析测试范围
→ 生成测试用例
→ 执行测试
→ 收集证据
→ 判断失败类型
→ 输出报告

即使只有一个进程，也不意味着它应该是一段连续自由推理。

所以“可编排”不一定等于复杂平台，它更本质地指：

任务具有显式阶段、依赖和状态转换。

⸻

5. 中间产物比对话更重要

Harness 中真正传递的，不应该只是自然语言消息，而是结构化产物。

例如：

需求分析结果
设计方案
任务清单
代码 Diff
测试报告
证据包
评审结论

这是因为自然语言对话：

- 不稳定
- 难校验
- 难复用
- 难审计
- 难作为下游输入

而结构化产物可以：

- 被程序校验
- 被其他 Agent 消费
- 被持久化
- 被版本管理
- 被重新执行

小 Agent 也一样。

比如根因分析节点不应只输出：

可能是缓存问题。

而应该输出：

{
"hypothesis": "缓存未失效",
"evidence": [],
"counter_evidence": [],
"confidence": 0.72,
"next_action": "检查缓存写入路径"
}

所以一个重要的不变量是：

Harness 设计围绕 Artifact，而不是围绕 Conversation。

⸻

6. 每个节点都必须有契约

这套架构中的契约层，不只是整个系统的全局规范，还应落实到每个执行节点。

一个节点至少应该定义：

输入是什么
输出是什么
允许做什么
禁止做什么
什么算成功
什么算失败
失败后怎么办

可以抽象为：

Node Contract
├── Input Contract
├── Tool Contract
├── Output Contract
├── Behavior Contract
└── Exit Contract

例如代码修复节点：

input:

- confirmed_root_cause
- affected_files
  allow:
- edit_affected_files
- run_unit_tests
  deny:
- change_public_api
- modify_unrelated_modules
  output:
- code_diff
- change_summary
- test_result
  success:
- build_passed
- target_test_passed

这套思想既适用于平台流水线，也适用于一个本地 Skill。

所以契约真正的价值不是“把所有场景提前穷尽”，而是：

为当前节点定义可执行边界和可判定结果。

⸻

7. 证据优先于模型结论

整个架构引入证据链、评审 Agent 和反证机制，反映出一个根本认知：

模型输出不是事实，只是待验证的假设。

因此，Harness 中应区分三类内容：

Claim：模型提出的结论
Evidence：支持或否定结论的事实
Decision：系统根据证据作出的判断

例如：

Claim：登录失败是接口超时造成的
Evidence：

- 请求耗时 15 秒
- 返回 504
- Console 无前端异常
- 重试后成功
  Decision：
- 归类为后端或网关问题

不能直接把 Claim 当 Decision。

小型 Agent 同样需要遵循：

模型认为修复完成
≠
修复真的完成

只有：

构建通过
测试通过
目标行为验证通过

才可以宣布完成。

因此一个重要不变量是：

Harness 必须把事实采集与语义判断分离。

⸻

8. 必须设计反证，而不只是验证

普通 Agent 工作流通常只寻找支持自己结论的证据。

成熟 Harness 会主动要求：

哪里可能错
有哪些反例
是否遗漏其他解释
什么条件下结论不成立

这就是反证机制的价值。

例如 Agent 判断某函数是根因，反证过程要检查：

- 是否存在另一个调用路径
- 是否有日志与结论冲突
- 是否能稳定复现
- 修复该函数后问题是否消失
- 是否只是相关而非因果

这个原则在任何规模都适用。

一个很小的 Agent，也可以要求输出：

{
"supporting_evidence": [],
"counter_evidence": [],
"remaining_uncertainties": []
}

所以反证不是一定要再启动一个 Agent。

它可以是：

- 独立 Critic Agent
- 第二轮 Prompt
- 规则检查
- 对照实验
- 测试验证

核心思想是：

Harness 不只要求模型回答“为什么是”，还要求回答“为什么不是其他原因”。

⸻

9. 状态必须外置

模型上下文不能作为唯一状态存储。

成熟 Harness 会显式维护：

当前阶段
已完成任务
待执行任务
失败原因
重试次数
已产生的产物
待确认事项

这是因为对话状态存在几个问题：

- 上下文可能丢失
- 模型可能忘记
- 会话不可恢复
- 状态难审计
- 并行任务难协调

因此，状态要存在 Harness 中，而不是存在模型“脑子里”。

大型系统可能存数据库和事件流。

小型 Agent 也可以用：

todo.json
state.json
checkpoint.json
run-result.json

所以不变量是：

LLM 负责推理，Harness 负责记住事实状态。

⸻

10. 运行时必须可恢复，而不是一次性调用

Harness 不应假设任务可以一次完成。

现实中会发生：

- 模型失败
- 工具超时
- 环境异常
- 用户需要补充信息
- 测试失败
- 人工审批等待
- 外部系统不可用

所以任务应该具备：

暂停
恢复
重试
回滚
重放
跳转
人工接管

大型系统中是工作流恢复。

小 Agent 中也可以表现为：

从上一次成功节点继续
而不是重新开始整个任务

例如测试已经生成，只是执行失败，那么恢复时不应重新生成测试。

因此另一个不变量是：

Harness 设计的是任务生命周期，而不是一次模型请求。

⸻

11. 能力必须可组合，而不是绑定单一流程

你看到的“各项目可以编排自己的流水线”，反映出的底层思想是：

Harness 提供能力，业务决定组合。

一个测试能力应当能够用于：

- 开发自测
- CI 回归
- 缺陷复测
- UAT
- 发布前验收

而不是写死在某一条流程里。

同样，一个代码评审 Agent 应该既能用于：

- PR 评审
- 代码生成后的自检
- Bug 修复后的审查
- 安全治理

所以 Harness 中最重要的资产不是某一条完整流程，而是：

可复用能力节点

- 可组合流程定义

这在小系统中也适用。

即使你只开发一个 Agent，也应该尽量让其核心能力：

- 输入独立
- 输出明确
- 无隐式依赖
- 可被其他流程复用

⸻

12. 记忆必须经过治理，不能直接累积

架构中的增量知识沉淀，也反映出一个重要原则：

运行记录不等于知识，Agent 结论也不等于知识。

知识沉淀必须经过：

提取
去重
验证
冲突检查
分类
版本管理
有效期管理

否则 Harness 会出现知识污染：

错误结论
→ 写入记忆
→ 后续检索
→ 再次影响判断
→ 错误被强化

这对小 Agent 也同样成立。

例如 Claude Code Memory 不应该记录：

登录模块有 Bug。

而应该记录稳定、可复用的事实：

该项目登录态由 localStorage 中的 access_token 和 HttpOnly Cookie 中的 refresh_token 共同维护。

所以记忆设计的核心不是“记得更多”，而是：

只沉淀经过验证、未来可复用的事实和经验。

⸻

13. Human-in-the-loop 是风险控制，而不是系统缺陷

成熟 Harness 不会追求所有节点都自动化。

它会根据风险决定执行方式：

低风险、确定性任务
→ 自动执行
高不确定性任务
→ Agent 提出方案
高风险决策
→ 人工审批
业务价值判断
→ 用户确认

无论系统大小，都应该先判断：

这个动作错了会有什么后果

然后决定：

- 自动执行
- 模拟执行
- 只给建议
- 人工确认后执行

因此 Human-in-the-loop 不是“Agent 不够先进”，而是：

Harness 对风险责任边界的显式设计。

⸻

14. 评测必须来自运行，而不是主观感觉

这套架构中的评测和反馈模块说明：

Harness 不是设计完成就结束，而是需要根据真实运行数据持续演化。

需要观察：

成功率
完成率
重试率
人工接管率
证据充分率
测试通过率
错误归因准确率
平均执行时长
Token 和工具成本
Bad Case 类型

一个小 Agent 也应该能回答：

- 哪一步最容易失败
- 哪种任务不适合它
- 哪类工具调用错误最多
- 哪些规则没有起作用
- 哪些场景需要人工处理

因此不变量是：

Harness 必须设计可观测性和反馈回路。

⸻

15. 最终可以归纳为七个核心不变量

我认为这套大型架构可以压缩成七个不会随规模变化的核心。

1. Context Engineering

给 Agent 正确、有限、可追溯的上下文。

它需要知道什么

2. Capability Engineering

将工具和能力标准化、权限化。

它能做什么

3. Contract Engineering

用契约约束输入、输出、行为和完成条件。

它必须遵守什么

4. Orchestration Engineering

用确定性流程管理非确定性执行。

它按什么顺序做

5. Evidence Engineering

所有结论必须有证据，并接受反证。

如何证明做对了

6. State and Memory Engineering

状态外置、任务可恢复、知识受治理。

如何持续执行和积累

7. Evaluation and Governance

通过指标、门禁和人工控制风险。

如何判断系统长期有效

⸻

大系统与小 Agent 的映射

核心原则 企业级 Harness 小型 Bug Agent
认知 代码知识图谱、文档库 相关文件、Issue、日志
工具 MCP、CI、测试平台 read、edit、bash、test
契约 企业规范、项目规范 只改目标模块、测试必须通过
编排 DAG、事件、状态机 复现 → 分析 → 修复 → 验证
证据 Trace、报告、审计链 日志、Diff、测试结果
反证 Reviewer/Critic Agent 检查替代根因和回归风险
状态 数据库、Checkpoint state.json、todo.md
记忆 知识库、图谱 项目规则、已验证经验
治理 权限、审批、评测 高风险操作先确认

规模变化的只是：

- 节点数量
- 并发规模
- 存储方式
- 工具复杂度
- 治理强度

不会变化的是：

输入要受控
能力要有界
过程要可编排
状态要外置
结果要有证据
错误要能反证
经验要能沉淀
系统要能评测

最本质的总结

Harness 设计并不是在 Agent 外面增加更多流程，而是对 Agent 做六种工程化改造：

把上下文变成可管理资源，把工具变成受控能力，把规则变成执行契约，把任务变成状态流程，把结论变成证据链，把经验变成受治理的知识。

这套思想与系统规模无关。

一个 Harness 是否成熟，不取决于它有多少 Agent，而取决于：

它是否能够让一个非确定性的智能体，在明确边界内，稳定地完成任务，并以可验证的方式证明结果。
