## 1. 核心命题

### 1.1 所有 Harness 都在解同一个方程

无论是阿里几万人的产研操作系统，还是一个只有三个节点的小 agent，它们要解的方程完全一样：

> **如何用一个确定、可信、可审计的系统，去包裹一个概率性、会幻觉、不可靠的算子（LLM）？**

系统的大小，只改变这个方程的**系数**（吞吐、租户、成本量级、组织协作），不改变方程的**形式**。Harness 设计的不变内核，就是这个方程形式本身的解。

这就是为什么：**抄一个大厂 harness 的功能列表是学不到东西的，因为功能随规模膨胀；但抄它的内核思维是通用的，因为内核来自建材，不来自规模。**

### 1.2 不变性从哪来——内核是建材属性的"对策"

Harness 的建材只有两种：**LLM** 和 **它所处的现实世界**。它们的固有属性是固定的，每一条 harness 内核，都是针对其中某条属性的直接对策：

| 建材的固有属性 | 它必然导致的风险 | 对应的不变内核 |
|---|---|---|
| **LLM 概率性**：同输入 ≠ 同输出 | 流程不可复现、不可控 | ① 控制权归确定性系统 |
| **LLM 会幻觉**：自信地给出错误 | 错误悄悄进入下游/生产 | ④ 证据高于断言 |
| **LLM 无界**：乐意永远循环下去 | 卡死、烧穿预算 | ② 约束前置（终止条件 / 预算） |
| **LLM 不透明**：无法内省其决策 | 无法调试、无法治理 | ③ 契约优先 + ⑦ 可观测内建 |
| **LLM 单点不可靠**：一步可能就错 | 一步错，全盘错 | ⑤ 分解胜于强提示 |
| **LLM 无状态**：调用之间无记忆 | 无法暂停 / 恢复 / 长跑 | ⑥ 状态外置且可恢复 |
| **世界：动作不可逆** | 误删、误发布无法挽回 | ① 之特例：人在不可逆之处 |
| **世界：外部环境会变** | 过期状态、过期决策 | ⑥ 状态外置 + 证据即时性 |
| **世界：资源有成本** | 成本失控 | ② 约束前置（quota / 熔断） |

**读这张表的方式**：左列在任何尺度下都不变（LLM 在小 agent 里也会幻觉，在大平台里也会幻觉），所以右列的内核在任何尺度下都适用。**这就是"尺度无关"四个字的全部证明。**

> 小团队可以"先跑通再补治理"；几万人尺度，治理没内建进架构，平台第一天就是定时炸弹。但治理的原则是一样的，只是实现的工程量不同。

## 2. 前置区分：Context / Capability / Constraint 三分法

> 七个内核讲的是"如何控制这个算子"。但在控制之前还有更基础的一步：**喂给 LLM 的输入，本身要分三类建模，绝不能糊在一个 prompt 里。**

| 类别 | 回答的问题 | 本质 | 例子 |
|---|---|---|---|
| **Context（知识）** | 它**知道**什么 | 参考信息 | 代码、文档、历史缺陷、运行数据 |
| **Capability（能力）** | 它**能做**什么 | 受权限约束的动作 | 读文件、改代码、调 API、跑测试 |
| **Constraint（契约）** | 它**必须遵守**什么 | 硬约束 | 输出 Schema、目录边界、安全规则、完成判定 |

**为什么必须分离**：同一句"项目用 Vue3"是 Context（参考）；"必须用内部表单组件"是 Constraint（强制）；"可以调组件检索工具"是 Capability（能力）。三者一旦混在一个 prompt 里，LLM 无法稳定区分哪些是参考、哪些是铁律、哪些是可执行动作——这正是它"不透明 + 概率性"两个属性叠加的必然后果。

**为什么尺度无关**：大平台用认知层 / 工具注册 / 契约层三套独立子系统分别承载；小 agent 至少要在 prompt 里用三个独立区块把它们写开。**混着喂，是新手最隐蔽的一类 bug——你以为给的是约束，模型当成了参考。**

### 延伸：Harness 的本质

围绕这三类输入，Harness 的本质不是"包住模型的一层代码"，而是：

> **围绕模型建立一个受控的任务执行环境。**

它首先回答六个问题，无论规模大小——这正是 GPT 从企业级架构中反推出的第一组不变量：

1. Agent 知道什么（Context）
2. Agent 能做什么（Capability）
3. Agent 必须遵守什么（Constraint）
4. Agent 如何被控制（Orchestration）
5. Agent 如何证明自己做对了（Evidence）
6. 失败经验如何沉淀（Knowledge）

## 3. 七个不变内核

每个内核按以下结构展开：一句话 → 对抗的属性 → 详细阐述（含从 GPT 不变量中提取的更细分裂变）→ 大系统 vs 小 Agent 映射。

### 3.1 控制权永远归确定性系统

**一句话**：LLM 是被编排的算子，不是编排者本身。"下一步做什么、是否结束、走哪条分支"永远由确定性的代码（或人）决定，绝不交给 LLM 自己。

**对抗的属性**：LLM 概率性 + 无界。

**详细阐述**：

确定性 Harness → 调度非确定性 Agent。编排引擎负责：任务顺序、依赖、并行、状态流转、事件触发、超时重试、失败恢复、权限切换、审批、审计。Agent 只负责：理解、分析、生成、归纳、判断、提方案。

- **大系统**：编排引擎（LangGraph / 状态机 / DAG）掌管控制流。Agent 是执行单元，不是决策单元。
- **小 agent**：哪怕只有一个 while 循环，也是你的代码在判断"要不要再来一轮"，而不是问 LLM"你觉得你做完了吗"。终止条件是代码里的 if，不是 prompt 里的祈使句。

**它的关键特例——人在不可逆之处**：当一个动作不可逆（删生产数据、线上发布、转账），控制权不是交给 LLM，而是交给人。人在环不是给全自动流程打的补丁，它是"控制权归确定性决策者"在世界不可逆性下的自然推论。

- 大系统：审批节点是一等公民，与 Agent / Tool 平级。
- 小 agent：rm -rf 之前停下来问一句"确认？"。

Human-in-the-loop 不是系统缺陷——**Harness 不追求所有节点都自动化，它根据风险决定执行方式**：低风险确定性任务自动执行，高不确定性任务 Agent 提方案，高风险决策人工审批，业务价值判断用户确认。

**违反会怎样**：让 agent 自己决定流程终态 → "测试通过"是它说的、不是产物证明的 → 幻觉直接进生产 / 死循环烧穿预算。这是所有 harness 失败模式里最致命的一种。

**GPT 独立展开——流程控制权的进一步分裂变**：

小型修复 Agent 的流程应严格受控：
```
复现成功 → 才允许根因分析
根因证据充分 → 才允许修改代码
代码修改完成 → 必须运行测试
测试未通过 → 不允许宣布完成
```

这背后的更根本原则：**模型输出不是事实，只是待验证的假设**。Harness 中应区分三类内容：

> **Claim**：模型提出的结论
> **Evidence**：支持或否定结论的事实
> **Decision**：系统根据证据作出的判断

不能直接把 Claim 当 Decision。

### 3.2 约束前置，而非事后纠偏

**一句话**：Agent 能做什么、必须遵守什么，是在它执行之前就划定的边界，不是跑起来再纠偏。

**对抗的属性**：LLM 无界 + 世界的成本属性。

**详细阐述**：

约束做成"准入门禁"，不是"事后巡检"——"防止"永远比"补救"便宜，这条经济学在任何尺度都成立。

- **大系统**：工具白名单（注册才能调）、权限边界、输出 Schema、quota 与熔断、状态转换条件。
- **小 agent**：限定这个 agent 只能调这三个工具、最多跑 10 轮、单次 token 上限 N。

**最危险的失败模式**：声明了约束，但运行时没人强制。工具 Schema 在控制面定义得很漂亮，可业务团队在执行面直接拿到了工具句柄、绕过了约束。

→ **声明式约束必须有命令式强制兜底**：工具调用时实时鉴权，而不是靠注册时的 Schema 声明。真正的解法是在控制面和执行面之间加一个**权限执行层**。

### 3.3 契约优先于对话

**一句话**：系统里流动的应该是结构化产物，不是自然语言对话。每个边界都定义机器可读的输入/输出 Schema。Schema 是组合的基本单位。

**对抗的属性**：LLM 不透明 + 概率性。

**详细阐述**：

自然语言对人友好、对机器敌对——下游无法稳定消费、无法评测、无法定位错误。这个困境与系统大小无关，只与"机器要不要消费它"有关。只要下游是机器，就需要契约。

- **大系统**：需求阶段出 `{requirements, acceptance_criteria, scope, risks}`，测试阶段出 `{test_results, evidence_pack, defect_records}`。Agent 只是产物的生产者或消费者，Agent 之间不靠自然语言交接。
- **小 agent**：强制 LLM 输出 JSON / 调用结构化工具，而不是返回一段需要正则去抠的散文。

**GPT 不变量展开——每个节点都必须有契约**：

一个节点至少应该定义：
```
Node Contract
├── Input Contract     — 输入是什么
├── Tool Contract      — 允许/禁止做什么
├── Output Contract    — 输出是什么
├── Behavior Contract  — 什么算成功/失败
└── Exit Contract      — 失败后怎么办
```

例如代码修复节点：
- **input**：confirmed_root_cause, affected_files
- **allow**：edit_affected_files, run_unit_tests
- **deny**：change_public_api, modify_unrelated_modules
- **output**：code_diff, change_summary, test_result
- **success**：build_passed, target_test_passed

契约真正的价值不是"把所有场景提前穷尽"，而是**为当前节点定义可执行边界和可判定结果**。

### 3.4 证据高于断言

**一句话**：永远不相信 LLM 的自我报告。决策必须基于可验证的产物（工具输出、测试结果、trace、截图）。

**对抗的属性**：LLM 幻觉——它会自信地告诉你它成功了。

**详细阐述**：

LLM 最危险的不是"会错"，而是**"会以高置信度错"**。一个会说"我不确定"的模型是安全的，一个永远说"已完成"的模型是地雷。证据是唯一能戳破自信幻觉的东西。

- **大系统**：质量门禁依据 `test-results.json / trace.zip / screenshots` 判断是否放行，而非依据归因 Agent 的一句"看起来没问题"。
- **小 agent**：agent 说"我跑了测试且通过了"——不信，去捕获真实的测试退出码和输出。agent 说"文件已修改"——去读文件确认。

**它的进阶形态——反证，不只验证**：

成熟的 harness 不止问"有什么证据支持结论"，还主动问"**有什么证据能推翻它**"——哪里可能错、有没有别的解释、什么条件下结论不成立。LLM 天生倾向于为自己已经给出的结论找支持（确认偏误），所以"找反例"必须由 harness 结构性地强制，而不能指望模型自觉。

反证的实现形式多样，但"必须找反例"这条不能省：
- 独立 Critic Agent（OpsPilot 的 critic 节点）
- 第二轮 Prompt
- 规则检查
- 对照实验
- 测试验证

无论哪种形式，核心思想是：**Harness 不只要求模型回答"为什么是"，还要求回答"为什么不是其他原因"**。

最小化的实现：哪怕只是在 prompt 里多要字段——`{hypothesis, supporting_evidence, counter_evidence, remaining_uncertainties}`——逼模型把"为什么可能不是它"也写出来。

**这是 harness 的灵魂**：这一条是整张表中唯一一条无法靠"模型进步"解决的内核。模型越聪明，它的幻觉就越逼真——所以 harness 必须永远把事实采集与语义判断分离。

### 3.5 分解胜于强提示

**一句话**：可靠性来自把任务拆成小的、可验证、可回退的步骤，不来自把 prompt 写得更强。

**对抗的属性**：LLM 单点不可靠——任何一次调用都有不低的出错概率。

**详细阐述**：

新手以为"agent 不靠谱 = prompt 不够好"，于是不断堆 prompt；老手知道"**agent 的可靠性是结构给的，不是 prompt 给的**"。把 0.9 可靠的一步，拆成五个各自 0.98 可靠、且每步可验证可回退的小步——整体可靠性反而上去了。这个数学在任何尺度都成立。

**它的终极推论**：**好的 harness，用一个平庸的模型也能跑出可靠结果。** 因为可靠性的来源是结构，而结构是你设计的、不是模型送的。

- **大系统**：把研发拆成需求理解 / 影响分析 / 设计 / 编码 / 单测 / E2E / 失败归因 / 评审… 每个单元独立、可复用、可单独评测。
- **小 agent**：与其写一个"帮我重构整个模块"的巨型 prompt，不如拆成"先列出要改的文件 → 逐个改 → 每改一个跑一次测试"。

**GPT 不变量展开——能力可组合，而非绑定单一流程**：

一个测试能力应当能用于：开发自测、CI 回归、缺陷复测、UAT、发布前验收——而不是写死在某一条流程里。Harness 最重要的资产不是某一条完整流程，而是**可复用能力节点 + 可组合流程定义**。

即使在单 agent 场景下，也应确保其核心能力：输入独立、输出明确、无隐式依赖、可被其他流程复用。

### 3.6 状态外置且可恢复

**一句话**：执行状态必须活在 LLM 之外，可被快照、可被恢复。LLM 的上下文窗口不是你的数据库。

**对抗的属性**：LLM 无状态（调用间无记忆）+ 世界会变（长任务期间环境漂移）。

**详细阐述**：

- **大系统**：checkpoint + 事件溯源 + 状态机持久化。任务跑一半等人审批，能从精确的节点恢复；崩溃了能重放。
- **小 agent**：把进度写进一个 dict / 一张表/`todo.json` / `state.json`，而不是指望"塞进 prompt 让它记住"。

**它带出的一个永恒难题**——恢复时世界可能已经变了，过期状态怎么处理？大系统叫 stale state 治理，小 agent 叫"恢复前重新校验一下环境"。**同一个问题，同一个答案方向。**

**Harness 设计的是任务生命周期，而不是一次模型请求**。任务应该具备：暂停、恢复、重试、回滚、重放、跳转、人工接管。大系统用工作流恢复；小系统也应能做到"从上一次成功节点继续，而不是重新开始整个任务"——例如测试已生成只是执行失败，恢复时不应重新生成测试。

**它的孪生陷阱——记忆必须治理，不能直接累积**：

状态外置解决"短期记不住"，但一旦想让 agent 跨任务积累经验（长期记忆），就冒出一个新风险：**运行记录 ≠ 知识，LLM 的结论也 ≠ 知识。**

如果把每次跑的结论无脑写进记忆库，错误结论会被后续检索出来、再次影响判断，把错误越强化越深——这是**记忆污染回路**。

所以长期记忆必须经过治理：提取 → 去重 → 验证 → 冲突检查 → 分类 → 版本/有效期管理。

- 大系统：知识入库要过质检和准入，bad case 单独管理，知识带版本和失效期。
- 小 agent：Memory 里不该记"登录模块有 bug"（一次性、会过期的结论），而该记"该项目登录态由 localStorage 的 access_token + HttpOnly Cookie 的 refresh_token 共同维护"（稳定、可复用的事实）。

> **记忆设计的核心不是"记得更多"，是"只沉淀经过验证、未来可复用的事实"。**

### 3.7 可观测与闭环内建

**一句话**：你必须能回放发生过什么（每次 LLM 调用、每次工具调用的输入输出、耗时、决策路径），并让结果反哺改进。可观测不是事后加的日志，是设计时就内建的能力。

**对抗的属性**：LLM 不透明 + 让系统能"越用越好"。

**两层含义，缺一不可**：
1. **可观测（看见）**：没有 trace，agent 出错你只能瞎猜。
2. **闭环（变好）**：结果不反哺，系统就停在第一天的水平。

- **大系统**：全链路 trace、证据链、离线评测数据集、Bad Case 库、指标体系；评测结果反哺控制面的 prompt / 工具 / 模型选择，形成闭环。
- **小 agent**：把每一次 LLM 调用和工具调用打进结构化日志，出问题能 debug；攒一个 bad case 集合，改完 prompt 用它回归。

**为什么内建而非附加**：可观测性补在事后，永远是残缺的（关键中间态没记下来就永远丢了）。

**评测必须来自运行，而非主观感觉**：

Harness 不是设计完成就结束，而是需要根据真实运行数据持续演化。需要观察：成功率、完成率、重试率、人工接管率、证据充分率、测试通过率、错误归因准确率、平均执行时长、Token 和工具成本、Bad Case 类型。

一个小 Agent 也应该能回答：哪一步最容易失败、哪种任务不适合它、哪类工具调用错误最多、哪些规则没有起作用、哪些场景需要人工处理。

> **评测是 harness 的控制论，不是 QA 的尾巴。**

## 4. 同一组真理的两种切法

> 面试官用哪套词不一定，你得能在两套词之间自由翻译。

**切法 A（属性切法）**：按"对抗 LLM / 世界的哪条属性"来切内核（如上文）。能**推导**出为什么必然这么几条，回答"你怎么知道清单是全的？"

**切法 B（学科切法）**：按"工程学科"切。好记、好讲、覆盖全，适合开场快速搭框架。

| 工程学科切法 | 回答的问题 | 对应属性内核 |
|---|---|---|
| **Context Engineering** | 它知道什么 | 1.5 节 Context + ⑥ 状态/记忆外置 |
| **Capability Engineering** | 它能做什么 | 1.5 节 Capability + ② 约束前置 |
| **Contract Engineering** | 它必须遵守什么 | ③ 契约优先 + ② 约束前置 |
| **Orchestration Engineering** | 它按什么顺序做 | ① 控制权归确定性系统 + ⑤ 分解 |
| **Evidence Engineering** | 如何证明做对了 | ④ 证据高于断言（含反证） |
| **State & Memory Engineering** | 如何持续执行与积累 | ⑥ 状态外置 + 记忆治理 |
| **Evaluation & Governance** | 如何判断系统长期有效 | ⑦ 可观测与闭环 + ① 之特例人在环 |

## 5. 企业级规模的专项关注：当系数变大的时候

> 前四节讲的是一切尺度通用的内核。本节只讲**当规模达到几万人产研、年 token 成本以亿计**时才会冒出来的问题。小 agent 场景可以跳过本节；但如果你想从"大厂怎么做的"学到东西，这里有它的精华。

### 5.1 五个关键工程动作

每个动作都配了取舍分析和失败模式——这是"知道怎么做"之外的第二个认知层次。

#### 动作一：能力原子化

把研发过程拆成独立、可复用的能力单元（需求理解 / 影响分析 / 架构设计 / 编码 / 单测 / E2E / 失败归因 / 代码评审 / 缺陷修复 / 知识沉淀…），每个单元可由 Agent / Skill / Tool / Script / CI Job / Human Task / External Service 任一执行者承载。

- **取舍**：牺牲"一个全能 agent 端到端搞定"的丝滑感，换来可组合、可复用、可替换、可评测。
- **失败模式**：建一个"全能开发 Agent"——不可观测、不可评测、不可治理，demo 惊艳，规模化即黑洞。
- **尺度意义**：原子化之后，一个资深专家封装的 Skill 三万人都能调用——"一人的能力上限，变成全组织的能力下限"。

#### 动作二：产物结构化（契约层 = 玩具与生产的分水岭）

- **核心判断**：Harness 里流动的不是对话，是标准化工程产物。
- **取舍**：结构化牺牲表达自由度，换来下游可自动消费、可评测、可审计。
- **失败模式**：agent 间用自然语言交接，后果是无法治理。

#### 动作三：规范变运行时契约（"治理即运行时"）

把"请遵循规范"这类文档约束，转换成可执行约束：必须过 Lint、核心路径必须有 E2E、接口变更必须生成兼容性报告、高风险变更必须人工审批。

- **取舍**：约束做成准入/门禁而非事后巡检——牺牲灵活性，换确定性。
- **最危险的失败模式**：声明了约束，但运行时没人强制。→ 声明式约束必须有命令式强制兜底。

#### 动作四：用确定性编排控制非确定性 Agent

```
编排引擎 = 确定性的控制系统     Agent = 非确定性的智能执行单元
```

- **铁律**：最终决策不能依据 agent 的一句"测试通过"，必须依据真实执行产物。
- **最反直觉、也最值钱的认知**：**可靠性来自结构，不来自模型。**

#### 动作五：知识回流闭环

一次任务结束不应只留下代码，还要沉淀需求↔设计↔代码↔测试↔缺陷↔修复↔发布的双向关联、失败模式、有效修复方案。

- **为什么这是护城河而非普通功能**：这构成一个复利飞轮——用得越多 → 沉淀越厚 → 检索/规划越准 → 更多人用 → 沉淀更厚。
- **CTO 视角的战略意义**：模型是租来的、会迭代的、谁都能买；但沉淀在 harness 里的组织知识与能力资产，是复利增长且无法被竞品复制的。

### 5.2 三层可配置——对康威定律的正面强攻

> 架构里**中心化才能治理，去中心化才能创新。两者必须同时成立。**

| 层级 | 负责 | 可配置度 |
|---|---|---|
| **平台层（通用编排调度）** | 编排引擎、状态管理、权限系统、模板中心、运行观测 | 不可改，一处约束三万人受益 |
| **团队层（项目配置）** | 选模板、配流水线、接项目知识、定产品契约、配质量门禁 | 可配置，差异化编排 |
| **任务层（执行单元）** | 各领域能力组件（需求/设计/代码/测试/安全/评审） | 可编排，业务定制流程 |

**它的政治意义**：康威定律说"系统结构映射组织结构"。这张图反过来用——先用三层可配置定义组织的协作契约，再让系统去承载它。**大厂中台的坟场里，死的全是没处理好这条线的：管太死→业务跑路自建；管不住→失控背锅。**

不同项目天然需要不同流水线：
```
内部管理系统：   需求 → 设计 → 开发 → 单测 → E2E → 验收
核心交易系统：   需求 → 架构评审 → 安全评审 → 开发 → 单测 → 集成 → 性能 → 灰度 → 观察 → 验收
紧急缺陷修复：   定位 → 最小修复 → 定向测试 → 人工审批 → 紧急发布 → 全量回归 → RCA
```

### 5.3 尺度经济学——Token / Quota / 路由是生存项

小项目里加缓存是"性能优化"。几万人尺度，多级缓存、LLM 路由——直接上 CFO 和 CISO 的桌子。

- **Token 经济学是 P0**：三万人 × 每天 N 次多轮调用，年成本以亿计。缓存命中率提几个点、把 80% 请求从旗舰模型路由到小模型——省的是真金白银。**模型路由在这个尺度不是技术选型，是成本结构设计。**
- **没有 Quota + 熔断，平台会被一个失控 agent 拖垮**："124 轮"说明系统允许长迭代，那就必须要有预算上限和熔断——一个陷入死循环的 agent 能烧穿一个团队的季度预算。
- **治理的 ROI 与组织规模成正比**：一条 prompt 注入防御、一个工具白名单、一条审计规则，在中心写一次，三万人全部受保护。

### 5.4 组织分工与落地路径

#### 四类团队

| 团队 | 负责 | 对应架构层 |
|---|---|---|
| **平台团队** | 编排引擎、状态管理、事件总线、权限系统、模板中心、版本管理、运行观测、审计 | 通用编排调度 |
| **能力建设团队** | 各领域能力组件（需求分析/设计/代码生成/前后端测试/安全/评审） | 执行单元 |
| **项目团队** | 选模板、配流水线、接项目知识、定产品契约、配质量门禁、处理人工审批 | 团队配置层 |
| **治理团队** | 通用规范、安全要求、质量标准、评测体系、Bad Case 管理、能力准入与版本发布 | 治理面（竖贯三层） |

#### 五阶段落地路径

顺序不能乱——这是一条组织能力建设路径：

1. **能力工具化** — 高频场景先封装成 Skills/Agents/Commands。解决"能不能用"。
2. **能力标准化** — 为每个能力定义输入/输出/权限/工具/契约/证据/失败状态。解决"能不能稳定复用"。
3. **流程编排化** — 把能力放进流水线，加状态、依赖、事件、人工审批。解决"能不能形成闭环"。
4. **治理与评测** — 质量门禁、证据链、离线数据集、Bad Case、指标体系、版本管理、审计追踪。解决"能不能规模化推广"。
5. **知识闭环** — 运行数据/代码/需求/测试/缺陷关联成持续更新的知识图谱。解决"能不能越用越好"。

> 注意第 4 步「治理与评测」是**规模化推广的前提，不是收尾**。这正是离线评测在项目里的定位——它不是"做完验收"，而是"让能力敢被三万人用"的准入条件。CISO 和合规敢不敢放行，看的就是这一层。

### 5.5 受控自动化与生产关系迁移

```
确定性任务（Lint/构建/CI）        → 脚本、工具执行
语义理解与方案生成               → Agent 执行
高风险决策（删生产数据/线上回滚） → 人工审批
最终业务验收                     → 需求方负责
```

Human Task 在编排面是**一等公民**，与 Agent / Tool / Script 平级——这不是给全自动流程打的补丁，是架构级设计。放到几万人尺度看，潜台词是一场**生产关系的迁移**：

> 三万工程师的主体工作，正从"自己写"迁移到"编排 AI 写、在关键节点审核与决策"。人退到三个位置：**约束的制定者**（控制面）、**流程的编排者**（编排面）、**关键节点的审核者**（Human Task）。

但这也带工程债：超时策略（人工 4 小时没响应怎么办）、stale state（审批期间环境变了）、审批原子性（人批准了但执行失败，回滚责任在谁）。

### 5.6 大厂 Harness 的七种死法

> 能讲清"做错会死在哪"，才显出深度。

1. **造全能 agent** → 不可观测、不可评测、不可治理。demo 惊艳，规模化即黑洞。（违反动作一）
2. **agent 间用自然语言交接** → 下游无法机器消费，错误无法定位。（违反动作二）
3. **约束只声明不强制** → 控制面写得漂亮，执行面随便绕过。治理形同虚设。（违反动作三）
4. **让 agent 自己决定流程终态** → "测试通过"是 agent 说的，不是产物证明的。幻觉直接进生产。（违反动作四）
5. **三层可配置失衡** → 管太死，业务跑路自建影子系统；管不住，成本与安全失控。（违反三层可配置）
6. **没有 quota / 计量** → 一个死循环 agent 烧穿预算；多团队成本无法归因。（违反尺度经济学）
7. **评测当尾巴而非内建** → 治理跟不上推广，平台带病规模化，最终被一次安全/合规事故叫停。（违反落地路径）

**七种死法，对应七个动作/原则。这套架构的每一个设计，都是在堵其中一个坑。**

## 6. 完整映射：大系统 ↔ 小 Agent

| 核心原则 | 企业级 Harness | 小型 Bug Agent |
|---|---|---|
| **认知层**（Context Engineering） | 代码知识图谱、文档库、RAG | 相关文件、Issue、日志 |
| **工具层**（Capability Engineering） | MCP、CI、测试平台、多工具注册中心 | read、edit、bash、test |
| **契约层**（Contract Engineering） | 企业规范、项目规范、Schema 注册 | 只改目标模块、测试必须通过 |
| **编排层**（Orchestration Engineering） | DAG、事件、状态机 | 复现 → 分析 → 修复 → 验证 |
| **证据层**（Evidence Engineering） | Trace、报告、审计链 | 日志、Diff、测试结果 |
| **反证** | Reviewer / Critic Agent | 检查替代根因和回归风险 |
| **状态** | 数据库、Checkpoint、事件溯源 | state.json、todo.md |
| **记忆** | 知识库、知识图谱、质检流程 | 项目规则、已验证经验 |
| **治理** | 权限、审批、评测、计量 | 高风险操作先确认 |

**规模变化的只是**：节点数量、并发规模、存储方式、工具复杂度、治理强度。

**不会变化的是**：输入要受控，能力要有界，过程要可编排，状态要外置，结果要有证据，错误要能反证，经验要能沉淀，系统要能评测。

## 7. 统一心智模型

七条内核可以收敛成一个比喻——比喻天然是尺度无关的：

> **把 LLM 想象成一个聪明、但不可靠、记不住事、不知疲倦也不知停止、做完不留痕迹的外包专家。**

你对外包的直觉规矩，逐条对应七个内核：

| 你对外包的规矩 | 对应内核 |
|---|---|
| 公章绝不给他，重要决定我拍板 | ① 控制权归确定性系统 |
| 先说清楚他能动哪些东西、预算多少 | ② 约束前置 |
| 要他交可验收的成果物，不是听他口头汇报 | ③ 契约优先 |
| 他说"搞定了"，我要看到东西才信 | ④ 证据高于断言 |
| 大活拆成里程碑，一个个验收 | ⑤ 分解胜于强提示 |
| 他记不住，所以进度和上下文我来存 | ⑥ 状态外置且可恢复 |
| 他做事不留痕，所以我给他装监控、复盘 | ⑦ 可观测与闭环 |

> 这个比喻是整套方法论的"压缩包"——你不需要记七条，你只需要记住"LLM 是个聪明但不可靠的外包"，七条内核都能从这句话当场推导出来。

## 8. 自检清单

无论面对组织级平台还是一个周末写的小 agent，用这七问体检。**全是"否"也能跑，但每个"否"都是一个已知的塌方点**：

1. **控制权**：流程的"下一步/是否结束"是代码决定的，还是 LLM 自己说了算？
2. **约束**：它能调的工具、能花的预算、能改的范围，是跑之前就锁死的吗？
3. **契约**：各环节交接的是结构化产物，还是一段需要猜的自然语言？
4. **证据**：放行/结束的依据是真实产物，还是 LLM 的一句"完成了"？
5. **分解**：靠的是任务拆解 + 逐步验证，还是赌一个巨型 prompt？
6. **状态**：进度活在 LLM 外面、可暂停可恢复吗？还是塞在上下文里？
7. **可观测**：出错时能回放它每一步做了什么吗？结果能反哺改进吗？

> 这七问，问一个 30000 人的平台，和问一个 300 行的 agent，字面一模一样。答案的工程量天差地别，但"该不该问、问的是什么"完全不变。

## 9. 收口

### 六个"变成"（体系认知的终极压缩）

```
把研发知识   变成   上下文
把研发规范   变成   契约
把研发能力   变成   组件
把研发流程   变成   编排
把研发结果   变成   证据
把研发经验   变成   知识
```

### Harness Engineering 的完整定义

> 以编排引擎为运行骨架，以知识、工具、契约为执行输入，以 Agent 和自动化工具为执行单元，以结构化产物和证据链为阶段交付，以质量门禁和人工审批控制风险，并通过运行数据与知识回流持续优化的 **AI 原生产研工程体系**。

### 六种工程化改造（GPT 总结的最终收敛）

Harness 是对 Agent 做六种工程化改造：**把上下文变成可管理资源，把工具变成受控能力，把规则变成执行契约，把任务变成状态流程，把结论变成证据链，把经验变成受治理的知识。** 这套思想与系统规模无关。

### 面试一句话（记忆点）

> 我判断一个 harness 好不好，不看它多大、用了什么框架，我看七件事——**控制权在谁手里、约束有没有前置、交接是不是契约、结论靠不靠证据、任务有没有分解、状态在不在外面、过程可不可回放**。这七条不随系统大小改变，因为它们不是关于系统的，是关于"如何驾驭一个会幻觉的概率算子"的。

> 几万人尺度下，AI 产研的胜负手不在模型，在 harness。模型是全行业共享、会被快速拉平的外部变量；harness 是把组织生产力、知识、治理沉淀成复利资产的内部护城河。它的落地顺序是**先约束、再分解、后编排、终评测与复利**——而最反直觉的一点是：**harness 做得好，用一个平庸的模型也能跑出可靠的产研结果。谁的 harness 强，谁就能在模型同质化的时代，靠工程把可靠性与效率的差距拉开。**

### 最后一句话

> **Harness 设计的不变内核，不是关于"如何造一个大系统"，而是关于"如何与一个概率性算子安全地协作"。** 系统的规模决定你要写多少代码、养多少团队、烧多少预算；但决定一个 harness 是否可信的那组原则，是从 LLM 和现实世界的固有属性里长出来的——它们不随规模缩放。
