产品化 Agent：30 道 Product Sense 面试题详解

以下答案统一采用三个层次：

- 核心判断：先给结论。
- 详细展开：解释产品逻辑。
- 结合项目表达：你在面试中可以怎么说。

⸻

一、场景与产品边界

1. 什么场景适合做 Agent？

核心判断

适合 Agent 的场景通常同时满足几个条件：

1. 任务不是单一步骤，而是多步骤链路。
2. 执行路径会根据上下文动态变化。
3. 过程中需要调用外部工具。
4. 存在一定语义理解和判断。
5. 最终结果可以验证。
6. 错误风险可以被控制。

例如：

运维故障诊断
代码 Bug 排查
复杂需求拆解
企业知识问答
应用生成
自动化测试与失败归因

这些任务不是单次文本生成，而是：

理解问题
→ 收集信息
→ 制定计划
→ 调用工具
→ 判断结果
→ 验证完成

不适合 Agent 的场景

以下场景优先使用规则、Workflow 或普通程序：

- 规则完全固定。
- 输入输出明确稳定。
- 不需要语义判断。
- 对确定性要求极高。
- 错误成本非常高且无法回滚。
- 单次 API 调用即可完成。

例如：

计算税率
字段格式转换
固定审批流
订单状态同步
数据库定时备份

判断框架

可以使用：

任务复杂度
× 路径动态性
× 工具调用需求
× 可验证性
× 风险可控性

Agent 不是越复杂越好，而是在确定性程序难以覆盖全部变化时，使用模型处理不确定部分。

结合你的项目表达

我判断一个场景是否适合 Agent，不是看能不能接大模型，而是看任务是否存在动态路径、多步骤决策和工具调用。例如前端自动化测试不是简单执行 Playwright，而是先根据需求和代码变更判断测试范围，再生成或选择用例、执行测试、分析失败原因、收集证据并触发缺陷流转。这里同时存在语义判断和确定性执行，所以适合采用 Agent 加 Workflow 的混合模式。

⸻

2. Agent 与 Workflow 有什么区别？

核心判断

Workflow 解决的是：

已知流程如何稳定执行。

Agent 解决的是：

面对不确定输入，如何动态理解、判断和选择行动。

Workflow 特征

- 路径预先定义。
- 状态转换确定。
- 输入输出较稳定。
- 可预测性强。
- 适合标准业务流程。

例如：

提交申请
→ 主管审批
→ 财务审批
→ 完成

Agent 特征

- 输入通常是自然语言。
- 需要动态决策。
- 路径可能变化。
- 可以调用不同工具。
- 输出存在概率性。

例如运维诊断：

收到工单
→ 判断涉及哪个服务
→ 根据现象选择日志、指标或数据库查询
→ 提出多个根因假设
→ 继续收集证据

正确产品形态

生产系统通常不是二选一，而是：

Workflow 管控整体流程
Agent 处理流程中的不确定节点

例如：

需求分析
→ 设计
→ 开发
→ 测试
→ 验收

这是 Workflow。

其中：

需求歧义识别
架构方案生成
代码生成
失败归因

可以由 Agent 处理。

结合你的项目表达

我们的企业 Harness 不是让 Agent 自主控制全部研发流程，而是由编排引擎维护需求、设计、开发、测试和验收的确定性流程，再把需求理解、代码生成、测试范围分析和失败归因这些非确定性任务交给 Agent。这样既保留智能能力，又避免流程失控。

⸻

3. 如何设计 Agent 的完整用户旅程？

核心判断

Agent 用户旅程不能只设计“输入问题—输出答案”，而要覆盖完整任务生命周期：

任务发起
→ 上下文补充
→ 方案确认
→ 执行
→ 过程干预
→ 结果验证
→ 异常恢复
→ 历史复用

典型阶段

任务发起

用户需要明确：

- 能做什么。
- 需要提供什么。
- 预计产出什么。
- 是否会执行实际操作。

上下文确认

系统主动识别：

- 信息是否完整。
- 是否存在歧义。
- 权限是否足够。
- 是否需要用户补充。

执行过程

用户应看到：

- 当前阶段。
- 已完成事项。
- 当前正在调用什么能力。
- 是否等待用户决策。
- 是否出现阻塞。

结果交付

结果不能只是一个答案，而应包括：

- 最终结论。
- 产物。
- 证据。
- 风险。
- 未解决事项。
- 后续操作。

异常恢复

用户应可以：

- 重试失败节点。
- 修改输入后继续。
- 人工接管。
- 终止任务。
- 回退到上一阶段。

结合你的项目表达

前端测试能力的用户旅程可以是：

开发者提交需求或代码变更
→ Agent 分析影响范围
→ 用户确认测试范围
→ 系统准备环境
→ 执行 Playwright
→ 实时展示进度
→ 失败后自动归因
→ 输出截图、Trace、Console、Network
→ 生成缺陷或建议
→ 修复后定向复测

面试表达：

我设计 Agent 用户旅程时，不会只关注对话入口，而是会把任务的发起、执行、异常和验收全部纳入。复杂 Agent 更像任务工作台，而不是单纯聊天机器人。

⸻

4. 如何向用户解释 Agent 的能力边界？

核心判断

能力边界必须在用户执行任务前就体现，而不是失败后才告诉用户。

边界主要包括：

- 任务边界。
- 数据边界。
- 工具边界。
- 权限边界。
- 自动化边界。
- 结果可信边界。

产品设计方式

明确支持范围

例如：

支持：

- Web 页面功能测试
- 登录流程测试
- CRUD 页面回归
- Playwright 证据采集
  暂不支持：
- 原生移动端测试
- 生产环境破坏性操作
- 无测试环境的项目

区分自动化等级

自动执行
需要用户确认
仅提供建议
不允许执行

展示结果性质

区分：

已验证事实
模型推断
风险提示
待确认内容

为什么重要

边界不清会导致：

- 用户预期过高。
- 错误使用。
- 结果不被信任。
- 高风险行为。
- 产品被少数 Bad Case 否定。

结合你的项目表达

我们不会把测试 Agent 描述成“自动保证所有页面没有问题”，而是明确它能验证哪些用户路径、在哪些环境运行、是否使用真实接口，以及哪些失败归因只是高概率判断。产品上需要把执行事实和模型判断分开展示。

⸻

5. 为什么 Agent 产品容易“Demo 惊艳，实际不好用”？

核心判断

Demo 通常只证明模型可以完成一次任务，产品需要证明它能：

稳定完成大量真实任务

常见原因

场景过于理想化

Demo 输入清晰、数据完整、环境正常；真实用户输入模糊、上下文缺失。

缺少状态和恢复

一旦工具失败或服务超时，整个任务重来。

缺少验证

模型生成结果看起来合理，但不能执行。

缺少权限治理

Demo 中工具无限制，生产中无法开放。

缺少产品交互

用户不知道：

- 当前进度。
- 为什么失败。
- 是否需要干预。
- 结果能否相信。

成本不可控

多轮推理、多 Agent、长上下文在 Demo 中不是问题，规模化后费用急剧上升。

缺少真实数据闭环

没有 Bad Case、人工反馈和评测集，无法持续优化。

结合你的项目表达

前端测试 Demo 很容易做到“根据一句需求生成 Playwright 脚本”，但真实产品要处理项目启动、登录态、动态端口、测试数据、选择器稳定性、环境异常、Flaky、证据收集、失败归因和缺陷流转。真正的产品价值不在第一次生成，而在整个测试闭环是否稳定。

⸻

6. 如何平衡 Agent 自主性和流程确定性？

核心判断

不是让 Agent 自主性越高越好，而是：

让 Agent 在需要智能判断的局部拥有自主性，让 Workflow 对整体过程保持控制。

分层处理

确定性部分

由代码和规则控制：

- 状态转换。
- 权限。
- 质量门禁。
- 工具参数校验。
- 超时重试。
- 审批。
- 成功条件。

非确定性部分

由 Agent 处理：

- 需求理解。
- 方案生成。
- 根因假设。
- 测试范围分析。
- 失败归因。
- 文档总结。

风险分级

低风险：Agent 可自动决定并执行
中风险：Agent 决定，系统验证后执行
高风险：Agent 只给建议，人工批准

结合你的项目表达

测试 Agent 可以自主判断需要运行哪些用例，但不能自主宣布测试通过。最终是否进入下一阶段，应由 Playwright 执行结果、阻断缺陷数量和质量门禁共同决定。Agent 负责判断，Harness 负责裁决。

二、Harness 与平台设计

7. Harness 对 Agent 产品价值的核心贡献是什么？

核心判断

Harness 的价值不是让模型更聪明，而是让模型能力变得：

可控、可执行、可验证、可恢复、可复用

具体作用

控制上下文

只给 Agent 当前任务需要的信息，避免噪音和错误知识。

控制工具

限制可调用能力、权限和参数。

控制流程

防止 Agent 跳步骤、漏步骤和无限循环。

控制质量

通过测试、规则、评审和证据判断结果是否合格。

控制状态

支持中断、恢复、重试和追踪。

控制经验沉淀

防止错误结论污染知识库。

产品意义

没有 Harness：

模型输出一个可能正确的结果

有 Harness：

系统在边界内完成任务，并提供证据证明结果

结合你的项目表达

对我们的开发套件来说，Harness 的核心价值是把 AI Coding 从个人自由使用转成团队可复制的研发能力。它通过 Rules、Skills、SubAgent、权限、测试和证据，把模型的不确定性控制在工程流程之内。

⸻

8. 如何平衡平台统一与项目个性化？

核心判断

采用：

平台统一底座

- 标准模板
- 项目级配置
- 强制治理规则

哪些必须统一

- 安全规范。
- 审计要求。
- 数据隔离。
- 工具权限模型。
- 质量结果格式。
- 核心状态模型。
- 高风险审批规则。

哪些可以项目自定义

- 流程阶段。
- 节点组合。
- 测试策略。
- 技术栈适配。
- 项目特殊规范。
- 门禁阈值。
- 角色与审批人。

模板机制

平台提供：

前端功能开发模板
Bug 修复模板
后端服务模板
紧急发布模板

项目基于模板调整，而不是完全从零配置。

防止失控

- 强制节点不可删除。
- 配置变更需审批。
- 流程版本管理。
- 运行实例绑定版本。
- 模板升级采用提示而非强制覆盖。

结合你的项目表达

前端自动化测试能力作为标准节点由平台提供，但具体项目可以配置登录方式、测试环境、Smoke 范围、是否执行全量回归、阻断阈值等。统一的是输入输出和证据格式，变化的是项目策略。

⸻

9. 如何将复杂业务流程拆成 Agent 节点？

核心判断

节点拆分应同时满足：

- 单一职责。
- 输入输出明确。
- 可以独立验证。
- 失败可以单独恢复。
- 有复用价值。

拆分方法

围绕任务中的认知变化拆分：

理解
→ 决策
→ 执行
→ 验证

例如测试流程：

分析测试范围
→ 生成测试计划
→ 准备测试环境
→ 执行测试
→ 分析失败
→ 输出报告

节点过大的表现

- 输入输出不明确。
- 一个节点执行时间过长。
- 失败后不知道从哪里恢复。
- 同时承担判断和执行。
- 无法复用。

节点过小的表现

- 节点之间频繁传递上下文。
- 编排复杂度高于业务价值。
- 每一步都需要模型调用。
- 状态和网络开销过高。

判断标准

一个节点通常应该有：

一个主要目标
一个明确责任主体
一组确定输入
一组标准输出
一套完成条件

结合你的项目表达

我不会把“前端自动化测试”做成一个大节点，而会拆成测试范围分析、环境准备、用例执行、证据采集、失败归因和结果门禁。这样不同项目可以组合，也方便失败后只重跑对应节点。

⸻

10. 哪些节点必须设置人工审批？

核心判断

人工审批不由“模型是否聪明”决定，而由：

风险 × 不可逆性 × 责任归属

决定。

必须人工审批的典型场景

- 生产环境写操作。
- 数据删除或批量修改。
- 发布和回滚。
- 权限调整。
- 对外发送正式信息。
- 涉及财务和法律责任。
- Agent 低置信度但影响重大。
- 用户验收和业务价值确认。

不适合频繁审批的节点

- 只读查询。
- Lint。
- 单元测试。
- 代码格式化。
- 生成草稿。
- 沙箱执行。

优化方式

采用分级审批：

低风险自动通过
中风险抽样或规则审批
高风险强制人工审批

结合你的项目表达

测试 Agent 的用例生成和沙箱执行不需要审批，但如果要自动创建阻断缺陷、修改测试环境数据或触发生产发布，就必须根据项目治理规则进入人工确认。

三、交互体验与人机协同

11. Agent 产品应该以 Chat UI 还是任务工作台为主？

核心判断

Chat 适合任务发起和补充上下文，任务工作台适合过程管理和结果验收。

复杂 Agent 最适合：

Chat + Task Workspace

Chat 适合

- 自然语言输入。
- 需求澄清。
- 临时追问。
- 结果解释。
- 轻量任务。

任务工作台适合

- 长任务。
- 多阶段任务。
- 多产物。
- 状态跟踪。
- 审批。
- 失败恢复。
- 证据查看。

设计形态

左侧：

任务、历史、项目

中间：

对话与任务状态

右侧：

产物、证据、代码 Diff、测试报告

结合你的项目表达

前端测试 Agent 不应该只放在聊天框里。开发者可以通过聊天描述测试目标，但执行过程应该进入测试工作台，展示测试计划、运行状态、失败用例、截图和 Trace。

⸻

12. 如何向用户展示 Agent 当前状态？

核心判断

状态展示应从技术状态转化为用户可理解的业务状态。

不要只展示：

Node 3 running

而应该展示：

正在分析本次代码修改影响的页面
正在准备登录态
正在执行核心路径测试
发现两个失败用例，正在判断失败原因

状态层级

任务级

等待执行
执行中
等待用户
已完成
部分完成
失败
已取消

阶段级

需求理解
测试规划
环境准备
测试执行
失败分析
结果验证

节点级

给高级用户展示：

- 工具调用。
- 耗时。
- 重试次数。
- 输入输出。
- 错误信息。

重要设计

必须明确区分：

- 系统正在执行。
- 系统正在等待。
- 系统已经阻塞。
- 用户需要操作。

结合你的项目表达

我会使用阶段状态和事件时间线，而不是展示模型思考。用户可以看到执行了哪些用例、哪些通过、哪些失败、当前是否正在重试，以及是否需要人工确认。

⸻

13. Human-in-the-loop 解决什么问题？

核心判断

Human-in-the-loop 主要解决三类问题：

1. 高风险决策。
2. 语义不确定性。
3. 责任归属。

它不是简单增加审批

Human-in-the-loop 可以出现在：

- 输入补充。
- 方案选择。
- 风险确认。
- 执行授权。
- 结果验收。
- 错误纠正。

产品价值

防止错误扩大

Agent 不确定时及时停下来。

保留业务判断

模型可以判断技术可行性，但用户决定是否符合业务预期。

形成反馈数据

用户修正可以形成高价值 Bad Case。

设计原则

人工节点必须说明：

- 为什么需要用户。
- 用户要决定什么。
- 不同选择有什么后果。
- 是否可以稍后处理。

结合你的项目表达

在测试场景中，Agent 可以自动判断失败更像代码问题还是环境问题，但置信度不足时应让测试人员确认。用户的确认结果会回流到失败归因数据集，而不是只停留在本次任务里。

⸻

14. Agent 置信度不足时，应该怎么办？

核心判断

不能只依赖一个统一置信度阈值，而要同时判断：

置信度
× 风险等级
× 补充证据成本
× 用户等待成本

四种处理策略

继续查证

适合：

- 可以低成本获取更多数据。
- 工具查询不会产生风险。

询问用户

适合：

- 缺少业务信息。
- 用户能快速回答。
- 问题无法通过系统自动获得。

输出多个候选方案

适合：

- 存在多个合理解释。
- 用户需要选择。

停止或人工接管

适合：

- 风险较高。
- 无法获得可靠证据。
- 继续执行可能产生损失。

结合你的项目表达

例如测试失败：

页面按钮未出现

Agent 可以继续检查：

- Console。
- Network。
- DOM。
- 截图。
- 重试结果。

如果仍无法区分环境和代码问题，再提示用户确认环境状态。

面试表达：

低置信度不是统一进入人工，而是先看能否通过低成本工具调用补充证据。只有系统无法自行消除不确定性时，才把最小问题交给用户。

⸻

四、可信结果与记忆

15. 为什么 Agent 产品需要证据，而不只是答案？

核心判断

Agent 的输出本质是推断，证据能够帮助用户判断：

- 结论从哪里来。
- 是否可信。
- 是否可以复现。
- 是否适合继续执行。

证据的三种价值

建立信任

不是“Agent 说有问题”，而是：

某接口返回 500
Console 出现异常
页面未进入预期状态

支撑决策

用户可以根据证据决定是否接受结果。

支撑审计和复盘

后续可以追踪：

- 当时看到什么数据。
- 为什么作出判断。
- 哪一步发生错误。

证据与解释的区别

解释是模型生成的语言。

证据是：

- 代码。
- 日志。
- 测试结果。
- 截图。
- Trace。
- 工具输出。

结合你的项目表达

在前端自动化测试中，Agent 的失败归因只是判断，截图、Trace、Console 和 Network 才是证据。我们需要让用户能从结论直接跳到原始证据，而不是只相信一段总结。

⸻

16. 是否应该向用户展示置信度？

核心判断

可以展示，但不能只展示一个没有解释的百分比。

“85% 置信度”容易让用户误解成统计准确率。

更好的方式

展示：

高可信
中等可信
待确认

同时说明原因：

高可信：

- 测试可稳定复现
- Console 和接口日志一致
- 修复后验证通过

什么时候适合展示数字

只有在：

- 置信度经过校准。
- 用户理解其含义。
- 有历史数据支持。
- 不会被误认为客观概率。

否则更适合展示等级和证据充分程度。

结合你的项目表达

测试失败归因中，我更倾向展示“高可信：代码缺陷”“待确认：环境异常”，并附上判断依据，而不是直接给用户一个 82%。数字如果没有校准，反而会制造虚假精确感。

⸻

17. 当执行结果和 Agent 判断冲突时，应该信哪个？

核心判断

优先级应该是：

真实运行事实

> 确定性规则
> 权威数据
> Agent 判断

示例

Agent 判断：

登录功能已经修复。

但 Playwright 仍无法登录。

最终结果必须是：

验证失败

而不是接受 Agent 的解释。

冲突处理方式

1. 标记结论与证据冲突。
2. 不允许任务通过质量门禁。
3. 重新分析原因。
4. 保留原始判断作为 Bad Case。
5. 必要时人工介入。

产品设计

应明确区分：

- Agent 提议。
- 工具执行。
- 系统裁决。

结合你的项目表达

测试 Agent 可以预测某次修复应该解决问题，但只要浏览器真实执行失败，就不能进入交付阶段。执行事实是质量门禁的最高优先级。

⸻

18. Agent 应该记住哪些信息？

核心判断

Agent 只应该记住：

未来仍然有价值
经过验证
用户授权
作用域明确

的信息。

适合记忆

用户偏好

- 输出格式。
- 常用项目。
- 默认测试方式。

项目事实

- 技术栈。
- 项目启动命令。
- 登录方式。
- 目录结构。
- 测试环境。

经过验证的经验

- 常见失败模式。
- 稳定选择器规则。
- 某模块特殊约束。

不适合记忆

- 未验证推断。
- 临时任务状态。
- 敏感数据。
- 一次性凭据。
- 模型生成但未确认的结论。
- 用户没有同意保存的信息。

作用域设计

会话记忆
用户记忆
项目记忆
组织知识

不能混在一起。

结合你的项目表达

测试 Agent 可以记住某项目使用 /saas/login、登录按钮定位方式和测试启动命令，但不能把一次测试失败后未经确认的“登录模块存在竞态”直接写进长期知识。

⸻

19. 如何防止错误信息进入长期记忆？

核心判断

记忆写入必须经过治理流程，而不是 Agent 自由写入。

推荐流程

候选记忆提取
→ 来源检查
→ 事实验证
→ 去重
→ 冲突检测
→ 作用域确认
→ 审核
→ 正式写入

记忆状态

candidate
verified
approved
deprecated
conflicted

权威层级

用户明确确认

> 系统真实数据
> 经过测试验证
> 多次重复观察
> Agent 单次推断

单次推断不能直接进入长期记忆。

版本与失效

知识需要：

- 创建时间。
- 来源。
- 版本。
- 生效范围。
- 失效条件。

结合你的项目表达

我们可以让 Agent 自动生成知识候选，例如“该项目使用动态端口启动”，但需要通过配置文件或多次运行验证后才能进入项目知识库。错误记忆比没有记忆更危险。

⸻

五、指标与评测

20. 产品化 Agent 的北极星指标是什么？

核心判断

Agent 的北极星指标不能简单使用：

- 调用量。
- Token 数。
- 对话轮次。
- 日活。

更合适的是：

有效任务完成率。

定义

有效完成的任务数
÷
用户发起的有效任务数

“有效完成”需要同时满足：

- 任务完成。
- 结果被验证。
- 用户接受。
- 没有额外返工。
- 没有严重副作用。

不同场景的北极星指标

运维 Agent

一次正确解决率

应用生成 Agent

从需求到可运行应用的成功交付率

前端测试 Agent

有效测试任务完成率

可以定义为：

成功完成测试范围分析

- 用例可执行
- 结果可判定
- 证据完整

结合你的项目表达

对前端自动化测试能力，我不会只看测试执行次数，而会看有效测试任务完成率，以及失败结果中有多少能够被准确归因并进入后续处理。

⸻

21. 如何定义 Agent 的任务成功？

核心判断

任务成功不是 Agent 返回了内容，而是：

目标完成

- 结果验证
- 用户接受
- 风险可控

四层成功标准

技术完成

工作流执行结束，没有系统错误。

产物完成

产生了规定的输出。

结果正确

通过规则、工具或人工验证。

业务有效

用户实际使用结果，没有重新完成同一任务。

示例

测试 Agent：

不能只定义：

Playwright 命令执行结束

应定义：

测试范围已确认
测试用例成功运行
结果能够分类
证据完整
阻断问题被识别

结合你的项目表达

Agent 的完成状态和业务成功状态必须分开。系统运行结束不代表任务成功。例如测试执行完成，但因为环境异常无法判断结果，只能算部分完成，不能算成功。

⸻

22. 如何评估 Agent 的准确率？

核心判断

必须先定义“准确”的对象是什么。

Agent 系统通常有多种准确率：

- 意图识别准确率。
- 工具选择准确率。
- 参数准确率。
- 根因判断准确率。
- 测试用例有效率。
- 失败归因准确率。
- 最终任务成功率。

评估方法

建立标注集

使用历史任务和人工确认结果建立 Ground Truth。

分阶段评估

不要只看最终输出。

例如失败归因：

代码问题
环境问题
测试问题
数据问题
网络问题

分别计算：

- Precision。
- Recall。
- F1。
- 混淆矩阵。

在线验证

观察：

- 用户修正率。
- 人工推翻率。
- 重新执行率。
- 最终采纳率。

结合你的项目表达

测试 Agent 的准确率不能用“测试通过率”衡量。更有意义的是测试范围命中率、有效用例率、失败归因准确率和缺陷确认率。每个环节需要独立评测，才能定位问题在检索、生成、执行还是判断阶段。

23. 如何设计 Bad Case 管理机制？

核心判断

Bad Case 不是简单收集失败对话，而是结构化定位：

在哪个节点

如何修复
是否回归

至少包括：

- 用户输入。
- 项目上下文。
- 模型版本。
- Prompt 版本。
- Workflow 版本。
- 工具结果。
- 错误节点。
- 预期结果。
- 实际结果。
- 人工修正。
- 根因分类。

分类方式

需求理解错误
上下文缺失
检索错误
工具调用错误
模型推理错误
契约缺失
环境异常

收集

结合你的项目表达

对前端测试 Agent，我会把 Bad Case 分成测试范围遗漏、选择器失效、环境异常误判、代码缺陷误判、Flaky 和证据不足等类型。不同类型对应不同优化手段，不能统一归因成 Prompt 不好。

24. 如何平衡效果、响应速度、成本和体验？

核心判断

这是一个多目标优化问题，不存在单一最优解。

需要根据任务价值和风险进行分级。

分级策略

低价值、低风险任务

- 小模型。
- 少轮推理。
- 简单检索。
- 快速返回。

高价值、高风险任务

- 高质量模型。
- 多源证据。
- 反证。
- 人工审批。
- 更长执行时间。

产品策略

渐进式结果

先返回：

任务已接受
初步分析
完整结果稍后更新

模型路由

分类、小结 → 小模型
复杂规划、评审 → 大模型

上下文裁剪

避免每次传整个历史。

结果缓存和资产复用

相同项目规则、测试环境和历史用例可以复用。

结合你的项目表达

对测试场景，我会把 Smoke 和全量回归分开。开发阶段优先快速 Smoke，交付前再做完整回归。这样既保证开发反馈速度，也控制浏览器执行和模型分析成本。

六、结合你的开发套件项目

25. 为什么要做开发套件，而不是直接推广 OpenCode 或 Claude Code？

核心判断

通用 Code Agent 提供基础能力，但企业需要解决：

- 知识不统一。
- 规范不统一。
- 工具接入困难。
- 权限风险。
- 流程不一致。
- 结果不可验证。
- 能力无法沉淀和复用。

通用工具的问题

开发者个人使用时可以自由：

- 写 Prompt。
- 调工具。
- 选择流程。
- 判断结果。

但企业规模化后会出现：

- 每个人用法不同。
- 产出质量不稳定。
- 安全边界不明确。
- 无法审计。
- 经验无法复用。
- 新人学习成本高。

开发套件的价值

把个人使用经验
→ 变成团队标准能力

通过：

- Command。
- Agent。
- Skill。
- Rule。
- MCP。
- 自动验证。
- 证据。
- 模板。

面试表达

我们不是重新做一个 Code Agent，而是在 OpenCode 这类 Runtime 之上建设企业研发 Harness。通用 Agent 解决“模型可以做什么”，开发套件解决“在公司研发环境中应该怎么做、能做到什么程度、如何验证和治理”。

26. 为什么前端自动化测试值得成为独立能力域？

核心判断

因为前端测试不是一个工具调用，而是一整套领域能力：

测试范围分析
测试设计
测试数据
环境准备
浏览器执行
证据采集
失败归因
缺陷闭环

独立能力域的必要性

技术复杂度高

涉及：

- Playwright。
- 浏览器。
- DOM。
- 网络请求。
- 登录态。
- 测试环境。
- 截图和 Trace。

领域知识强

需要理解：

- 用户旅程。
- 页面交互。
- 验收条件。
- Flaky。
- 稳定选择器。
- 测试隔离。

可复用价值高

不同流程都需要测试：

- 开发自测。
- Bug 修复。
- MR 检查。
- SIT。
- UAT。
- 发布前回归。

是重要质量门禁

前端自动化测试能提供接近用户真实行为的执行证据。

面试表达

我们把前端自动化测试独立出来，不是因为 Playwright 复杂，而是因为它同时承担验证、证据和缺陷流转职责。它是研发 Harness 中连接代码生成和交付验收的关键质量能力。

⸻

27. 测试 Agent 的核心价值是生成脚本还是验证代码结果？

核心判断

核心价值是：

验证代码结果。

生成脚本只是手段。

为什么不能把脚本生成作为核心

- 脚本可能不可运行。
- 可能没有正确断言。
- 可能覆盖错误路径。
- 可能只是形式上存在测试。
- 生成数量不代表质量。

完整价值链

理解验收目标
→ 设计测试场景
→ 生成或选择用例
→ 真实执行
→ 判断结果
→ 提供证据

产品指标

不应只看：

生成多少测试

而应看：

有效用例率
执行成功率
缺陷发现率
失败归因准确率
回归覆盖率

面试表达

我们不把测试 Agent 定义成代码生成器，而是前端验证系统。Playwright 脚本只是执行资产，最终价值是能否证明本次需求或修复真的满足用户验收条件。

⸻

28. 测试失败后，如何告诉用户是代码、环境、数据还是用例问题？

核心判断

需要独立的 Failure Triage 流程，而不是只把错误堆栈返回给用户。

证据来源

- 测试断言。
- 页面截图。
- Playwright Trace。
- 浏览器 Console。
- Network。
- 服务健康状态。
- 测试数据状态。
- 重试结果。
- 历史失败记录。

分类逻辑

代码问题

- 页面行为稳定不符合预期。
- 接口返回业务错误。
- 同一环境多次复现。
- 修复后问题消失。

环境问题

- 服务不可访问。
- 依赖接口超时。
- 多个无关用例同时失败。
- 健康检查异常。

数据问题

- 前置数据缺失。
- 账号状态不符合要求。
- 测试数据被其他任务污染。

用例问题

- 选择器过期。
- 断言与需求不一致。
- 等待时机错误。
- Mock 配置错误。

Flaky

- 重试结果不一致。
- 失败集中在时序和异步加载。
- 没有稳定产品缺陷证据。

产品展示

不能只展示分类，还要展示：

分类
可信等级
判断依据
推荐下一步

面试表达

失败归因不应该只靠 LLM 看错误信息，而要建立证据组合。比如 Console、Network 和 Trace 一致指向接口 500，同时环境健康正常，就更可能是产品代码问题。如果重试后通过且失败点不稳定，则更像 Flaky。

⸻

29. 如何衡量前端自动化测试能力的真实效果？

核心判断

不能只看用例数量、测试通过率或代码覆盖率。

应该从五个维度衡量。

1. 测试设计质量

- 验收条件覆盖率。
- 核心用户路径覆盖率。
- 风险场景覆盖率。
- 有效用例率。

2. 执行质量

- 一次执行成功率。
- 环境准备成功率。
- 平均执行时长。
- Flaky 率。

3. 缺陷发现价值

- 有效缺陷发现率。
- 阻断缺陷发现数。
- 缺陷确认率。
- 缺陷逃逸率。

4. 失败归因质量

- 归因准确率。
- 人工修改率。
- 自动分流成功率。
- 平均定位时间。

5. 研发效率价值

- 人工编写测试时间降低。
- 回归耗时降低。
- 缺陷复测时间降低。
- 发布前人工投入降低。

北极星指标建议

有效自动化验证任务完成率

辅助指标：

测试任务人工干预率
失败归因准确率
缺陷逃逸率

面试表达

我不会把“测试通过率”当作效果指标，因为通过率高可能只是没有覆盖风险。真正需要看验收条件覆盖、有效缺陷发现、Flaky 率、失败归因准确率和人工投入变化。

30. 你的前端测试能力如何接入上层编排平台？

核心判断

前端测试能力需要以标准化、可组合节点的形式接入，而不是以一个固定大流程接入。

能力节点

例如：

面试表达

我负责的测试能力不会直接绑定某一条固定流水线，而是封装成标准节点。上层编排平台决定在开发自测、SIT 或 UAT 哪个阶段调用，以及失败后流转到缺陷修复还是环境恢复；测试能力负责稳定执行并返回统一结果和证据。

面试回答的统一结构

面对这些 Product Sense 问题，可以统一使用以下回答框架：

1. 先明确目标

用户是谁
要完成什么任务
当前最大问题是什么

2. 划分确定性和非确定性

哪些适合 Agent
哪些适合 Workflow、规则或工具

3. 设计完整任务闭环

输入
→ 执行
→ 验证
→ 异常
→ 人工
→ 沉淀

4. 明确风险与边界

自动执行到什么程度
什么时候停止
什么时候需要人工

5. 用指标验证

是否真的完成任务
是否降低人工投入
是否提高质量
是否控制成本和风险

⸻

最后可用于面试的总结

我理解产品化 Agent 的 Product Sense，不是设计更多 AI 功能，而是围绕用户真实任务，判断哪些环节需要模型智能、哪些环节必须保持确定性，并通过流程、工具、证据和人工协同形成完整闭环。

对开发套件来说，我们不是追求让 Agent 自动写更多代码，而是把需求理解、开发、测试、评审和交付组织成可控流程。我的前端自动化测试能力，核心也不是生成 Playwright 脚本，而是把需求验收条件转化为可执行测试，通过真实浏览器结果和证据判断代码是否可以进入下一阶段。
