## 一、设计阶段

### ICCR 核心设计原则（OpenAI 内部验证的黄金法则）

| 原则 | 英文 | 含义 |
|------|------|------|
| I | Inform | 精准上下文按需注入 |
| C | Constrain | 架构规则代码化 |
| C | Control | 工具与环境最小权限控制 |
| R | Reflect | 自动化反馈闭环 |

### 风险分级管控

| 风险等级 | 示例操作 | 管控规则 |
|---------|---------|---------|
| 低风险 | 读取文件、查询信息 | 可自动执行 |
| 中风险 | 写入文件、发送消息 | 记录日志，可自动执行 |
| 高风险 | git push、数据修改 | 必须人工确认 |
| 极高风险 | 删除数据、系统配置 | 禁止自动接入 |

## 二、Skills 标准化接入四步法

**Step 1：元数据注册**

- 在 Harness 注册中心录入完整元数据
- 明确适用/不适用场景、输入输出规范、风险等级

**Step 2：标准化封装**

- 遵循统一三层架构（元数据 + 指令 + 资源）
- 内置异常处理、重试机制、输出规范

**Step 3：自动化测试**

- 功能测试 + 边界测试 + 安全测试
- 通过率 100% 才可上线

**Step 4：灰度发布**

- 先小流量灰度测试
- 验证效果、稳定性、安全性后全量发布

---

## 三、CLI 标准化接入四步法

**Step 1：白名单配置**

- 仅允许任务所需最小命令集合
- 禁止 `rm`、`dd` 等高危命令

**Step 2：沙箱封装**

- 所有 CLI 命令在 Docker 沙箱中执行
- 限制文件系统、网络访问权限

**Step 3：参数校验**

- 内置严格参数校验规则
- 禁止注入式参数、越权参数

**Step 4：审计留痕**

- 完整记录输入、输出、时间、执行者
- 不可篡改，满足合规要求

## 五、企业级规模化落地最佳实践

### 组织与治理

1. **公私分离**：通用能力封装成公共 Skills 库，部门专属封装成私有 Skills 库
2. **先试点后推广**：先在 1-2 个核心场景打磨体系，验证后推广全公司
3. **熵管理机制**：定期清理冗余、过期、低使用率的 Skills / CLI——这是最容易被忽略的核心点
4. **成本分摊**：按部门、项目统计调用量和 Token 消耗，实现成本可追溯管控

### 调度与编排

1. **需求解析 RAG 化**：Skills 数量超过 10 个时，用语义检索实现需求与 Skills 的精准匹配，避免 Prompt 膨胀导致的匹配错误，准确率可提升 40% 以上
2. **确定性优先，自主性为辅**：核心业务流程优先用确定性工作流编排，仅在模糊决策环节引入大模型自主判断，避免完全自主规划导致不可控
3. **异常处理与熔断机制**：预设所有可能的异常场景，制定重试、回滚、降级、人工升级方案；单次任务失败超过 3 次，自动熔断，避免无效循环与 Token 浪费
4. **成本最优调度策略**：简单任务优先调用轻量化 Skills / 本地 CLI 命令，复杂任务再调用大模型 / 云端 API，实测可降低 80% 以上的推理成本
5. **检查点机制**：长任务设置多个检查点，自动保存执行进度，任务中断后从检查点恢复，无需从头执行，大幅提升执行效率

### 安全与合规

1. **最小权限原则**：每个 Skill / CLI 仅授予完成任务所需的最小权限，绝对禁止一次性开放管理员 / root 权限
2. **不可逆操作双确认**：所有涉及文件删除、数据修改、系统配置的不可逆操作，必须设置人工确认节点，预览执行影响，100% 确认后才可执行
3. **全链路审计留痕**：所有操作必须记录不可篡改的审计日志，必填字段：操作时间 / 操作人 / Skill ID / CLI 命令 / 输入参数 / 执行结果 / 耗时 / Token 消耗 / 风险等级，留存时间不低于 6 个月
4. **动态权限回收**：任务执行完成后，自动回收临时权限，避免权限长期留存导致的安全风险
5. **安全左移**：在 Skills / CLI 的开发、测试阶段，就完成安全扫描、漏洞检测、合规校验，不符合要求的能力禁止上线

### 可观测与迭代优化

1. **核心指标体系搭建**：建立可量化的核心指标，覆盖：执行成功率、输出准确率、平均执行时长、Token 消耗、单任务成本、用户满意度、异常率
2. **全链路追踪**：用 OpenTelemetry 等工具，实现任务执行的全链路追踪，给每个任务分配唯一 Trace ID，可快速定位失败根因、性能瓶颈、成本浪费点
3. **自动化评估闭环**：搭建自动化评估套件，每次 Skills / CLI 更新都自动执行批量测试，量化评估效果，避免人工评估的主观性
4. **Bad Case 专项治理**：建立 Bad Case 闭环管理流程，自动收集执行失败的案例，拆解根因，针对性优化 Skills / CLI，再回归测试，形成完整优化闭环
5. **灰度发布与 A/B 测试**：Skills / CLI 的新版本先小流量灰度发布，与老版本做 A/B 测试，验证效果、稳定性、成本优于老版本后，再全量切换

## 六、避坑指南 ⚠️

1. **不要把 Harness 做成黑盒系统**：严格遵循三层解耦，不要把业务和执行逻辑都塞进去
2. **不要跳过标准化直接规模化**：先制定统一接入规范，否则系统会碎片化失控
3. **不要为了自主性牺牲确定性**：核心业务流程优先确定性编排，仅决策环节引入大模型
4. **不要忽略熵管理**：定期清理冗余 Skills / CLI，否则匹配准确率下降，维护成本飙升
5. **不要放松安全管控**：绝不给 AI 开放管理员权限，高危操作必须人工确认
6. **不要只搭建不迭代**：必须建立完整的可观测、评估、Bad Case 闭环迭代机制

## 📝 本节小结

**核心要点：**

1. ICCR 是 Harness 设计的黄金法则：**Inform**（精准注入）· **Constrain**（规则代码化）· **Control**（最小权限）· **Reflect**（反馈闭环）
2. 风险分级管控是安全红线：低风险自动执行，高风险人工确认，极高风险禁止自动接入
3. Skills 接入四步法：注册 → 标准化封装 → 自动化测试 → 灰度发布
4. 熵管理是最容易被忽略的核心点：定期清理冗余 Skills / CLI

**关键词**：ICCR · 风险分级 · 熵管理 · 公私分离 · 灰度发布 · Harness + Skills + CLI
