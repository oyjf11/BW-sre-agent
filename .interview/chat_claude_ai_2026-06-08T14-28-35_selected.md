```
这类"AI 生成全栈应用"产品(Lovable / v0 / Bolt 等)的多租户,通常要分两层来看,因为它们其实有两个层面的"租户",很多人会混淆:
一、平台自身的多租户(SaaS 层)
也就是你的用户(开发者)之间的隔离。这部分是常规 SaaS 多租户,主流方案:
1. 共享数据库 + 共享 Schema(Row-Level)
所有租户数据在同一套表里,用 tenant_id / org_id 列区分,配合数据库行级安全(如 Postgres RLS)强制隔离。这是绝大多数同类产品的默认选择,成本最低、运维最简单。
2. 共享数据库 + 独立 Schema
每个租户一个 schema。隔离性更好,但 schema 数量上千后元数据膨胀、迁移痛苦,一般只在中大客户场景用。
3. 独立数据库 / 实例
每个租户(或大客户)独占数据库。强隔离、合规友好,成本和运维最重,通常作为 Enterprise 套餐的选项,而不是默认。
实践里普遍是 混合策略:中小租户走共享库 + RLS,签了合规要求的大客户单独拎出来独立库。
二、用户生成应用的运行隔离(真正的难点)
这才是这类产品区别于普通 SaaS 的地方——你不只是存数据,还要跑用户生成的代码 / 应用。这里的"多租户"指的是隔离每个被生成应用的运行时:
1. 代码生成与构建隔离
每次生成/构建在独立沙箱里执行(容器 / microVM,如 Firecracker、gVisor、E2B 这类),避免用户代码互相影响或攻击宿主。
2. 预览环境(Preview)隔离
每个应用一个隔离的预览运行时,常见做法:

每个 preview 一个容器 / pod,用子域名路由(app-xxx.yourdomain.dev)
临时容器 + 空闲回收(冷启动换成本)
或者把生成的应用直接部署到边缘 / serverless(WebContainer、Cloudflare Workers 等)做轻量预览

3. 用户应用自带的后端 / 数据库
如果生成的应用需要数据库(Lovable 用 Supabase),通常是为每个生成的项目单独 provision 一个 Supabase 项目 / 实例,或者一个共享实例下按项目隔离。这相当于"租户的租户"——你的用户的最终用户的数据,又是一层隔离。
给你的几个关键决策点
由于你已经做出了产品,真正值得明确的是这几条边界:

平台数据(用户、项目元信息、对话历史)——共享库 + RLS 基本够用,先别过度设计。
代码执行 / 构建——必须强隔离(沙箱),这是安全红线,不能图省事。
生成应用的运行时——按成本和体验权衡:容器池 vs serverless vs 边缘。
生成应用的数据后端——决定是"每项目独立实例"还是"共享实例逻辑隔离",直接影响成本结构和你给最终用户的隔离承诺。

要不要我针对其中某一层展开?比如你现在最纠结的是平台层的数据隔离,还是用户生成应用的运行时 / 数据后端方案?这两类的取舍逻辑差别很大。

```
基于自托管开源版 Supabase 做这类产品的多租户,核心矛盾是:Supabase 开源版本质上是"单项目"设计的(一个 Postgres + 一套 Auth/Storage/Realtime/Kong),官方的 Supabase Cloud 那种"一个账号管多个 project"的控制平面(Platform)并不开源。所以你要自己补上这层。
主流有三条路线,从轻到重:
路线 A:单 Postgres + Schema 隔离(逻辑多租户)
整个平台共用一套 Supabase 自托管栈,每个"用户生成的应用"分配一个独立 Postgres schema。

隔离手段:每个租户一个 schema(tenant_abc.*),配合 RLS;PostgREST 通过 db-schemas 暴露多个 schema,请求时用 Accept-Profile / Content-Profile 头指定 schema。
Auth:GoTrue 默认所有用户在 auth.users 同一张表,没有原生组织/项目概念。要么自己加 tenant_id 到 JWT 的 claims(用 GoTrue 的 hooks / 自定义 access token claims),RLS 里读 auth.jwt() ->> 'tenant_id';要么干脆每租户独立 GoTrue 实例(见路线 B)。
优点:一套基础设施,成本最低,适合海量小应用。
痛点:schema 数量上千后,迁移、连接数、search_path、PostgREST schema 缓存都会变痛;一个坏租户能拖垮共享实例;Storage / Realtime 这些组件不像 Postgres 那样天然按 schema 分。

适合:最终用户应用都是小流量、对隔离要求不高的场景。
路线 B:每项目独立 Supabase 栈(物理多租户)
复刻 Supabase Cloud 的模型——每个用户生成的应用 = 一套独立的 Supabase(独立 Postgres + 独立 GoTrue + PostgREST + Storage + Kong),用容器编排起来。

怎么 provision:Supabase 官方提供 docker-compose,你把它模板化,每个项目一组容器(或一个 namespace / 一组 pod on K8s)。用 Kong 或上层网关按子域名 / 路径路由到对应项目的栈。
数据库:可以每项目一个独立 Postgres 容器(最强隔离、最重),或者多项目共享一个 Postgres 实例但每项目一个 database(CREATE DATABASE),省资源同时仍有 database 级隔离。
控制平面:这是你必须自研的部分——管理项目的创建/销毁、密钥(每个 GoTrue 需要自己的 JWT secret、anon/service key)、域名、连接信息、计费、休眠回收。本质上你在重建 Supabase 没开源的那层 management API。
优点:隔离干净,直接对应"每个生成应用有自己的后端",和 Lovable 的体验最接近,出问题影响面小。
痛点:资源占用大(每套栈基础内存就不低),冷启动/休眠策略必须做好,运维复杂度高。

适合:你要给最终用户提供"真·独立后端"的承诺,且愿意投入编排和成本。
路线 C:共享 Postgres,每项目一个 database(折中)
介于 A 和 B 之间,实践中很常用:

一个(或几个)大 Postgres 实例,每个项目 CREATE DATABASE,database 级隔离比 schema 干净(权限、备份、删除都更利落)。
上层组件(GoTrue / PostgREST / Storage)可以按项目动态指向不同 database 的连接串,或每项目轻量起一组无状态组件(它们比 Postgres 轻得多)。
用 PgBouncer / Supavisor(Supabase 自己的连接池)做连接复用,缓解"多 database × 多组件"带来的连接数爆炸。

适合:既想要比 schema 更强的隔离,又不想为每个项目都跑一整套带独立 Postgres 的栈。

落地时几个具体的坑(自托管开源版特有)

GoTrue 没有 org/project 概念:这是开源版最大的缺口。要么自定义 JWT claims 注入 tenant_id 走 RLS,要么每租户独立 GoTrue。别指望开源 GoTrue 帮你做多项目用户隔离。
密钥管理:Supabase 的 anon key / service_role key 是基于每个实例的 JWT secret 签出来的。多实例方案下,你得为每个项目生成、存储、轮换这套密钥——放进你的控制平面和密钥管理(Vault 之类)。
Storage:开源 Storage 后端默认对接 S3 兼容存储。多租户下用 bucket 前缀 / 独立 bucket 隔离,RLS policy 同样要按 tenant 写。
Realtime / Edge Functions:Realtime 按 schema/表订阅,多租户下要确认订阅过滤是否按 tenant 生效;Edge Functions(Deno)的执行隔离要单独考虑沙箱,别和数据隔离混为一谈。
连接数:无论哪条路线,Postgres 连接数是硬天花板。Supavisor(Supabase 官方多租户连接池,本身就是为这场景写的)值得优先用上。
迁移 / 版本管理:每个生成应用的 schema 是动态变化的(AI 不断改表)。你需要一套自己的 migration 跟踪,不能依赖单一全局 migration。

我的建议:如果你的目标是对标 Lovable 那种"每个应用有独立后端"的体验,路线 C(共享 Postgres + 每项目 database)通常是性价比最高的起点——隔离够用、成本可控、不至于像路线 B 那样一上来就被运维拖垮,后期大客户再升级到独立实例。
你们现在生成的应用,最终用户的数据量级和隔离要求大概是什么样?另外你们更看重的是成本可控,还是给最终用户一个"独立后端"的强承诺?这两点会直接决定 A/B/C 怎么选。

路线 C 详解:共享 Postgres + 每项目独立 Database
这条路线的核心思想是:用 Postgres 的 database 作为隔离单元,而不是 schema(太弱)或独立实例(太重)。下面从架构到落地一层层拆开。

一、整体架构
                          ┌─────────────────┐
                          │   控制平面       │  ← 你自研的核心
                          │ (Control Plane)  │     管项目生命周期/密钥/路由
                          └────────┬────────┘
                                   │
            ┌──────────────────────┼──────────────────────┐
            │                      │                      │
      ┌─────▼─────┐         ┌──────▼──────┐        ┌──────▼──────┐
      │  网关层    │         │ 共享无状态   │        │  Supavisor  │
      │  (Kong/    │────────▶│ 组件池       │───────▶│ (连接池)     │
      │  自研路由)  │         │ GoTrue/REST/ │        └──────┬──────┘
      └───────────┘         │ Storage      │               │
                            └─────────────┘                │
                                                    ┌───────▼────────┐
                                                    │  共享 Postgres  │
                                                    │  实例           │
                                                    │  ├ db_project_a │
                                                    │  ├ db_project_b │
                                                    │  ├ db_project_c │
                                                    │  └ ...          │
                                                    └────────────────┘
关键点:Postgres 是有状态的、贵的、要隔离的;它上面的组件(GoTrue/PostgREST/Storage)是无状态的、轻的、可共享或按需起的。 路线 C 的全部技巧都围绕这个不对称性展开。

二、数据库层:每项目一个 database
创建
每当用户新建一个应用,控制平面执行:
sql-- 1. 创建项目专属 database
CREATE DATABASE db_project_a;

-- 2. 创建项目专属的 owner 角色(最小权限边界)
CREATE ROLE project_a_owner LOGIN PASSWORD '<随机生成>';
GRANT ALL ON DATABASE db_project_a TO project_a_owner;

-- 3. 切到该 database,初始化 Supabase 必需的 schema/扩展
-- (auth schema、storage schema、extensions 等)
每个 database 内部,你需要把 Supabase 那套基础结构装好:auth schema(GoTrue 用)、storage schema、extensions(pgcrypto、pgjwt、uuid-ossp 等)、以及 PostgREST 需要的 anon / authenticated / service_role 角色。
实践做法:做一个"模板 database"(类似 Postgres 的 template1 机制),把所有 Supabase 基础结构预装好,新项目直接 CREATE DATABASE db_x TEMPLATE supabase_template,秒级完成且保证一致性。
隔离边界

database 之间天然隔离:跨 database 不能直接 join / 查询(除非用 FDW,你不会用),权限、连接、备份、删除都是 database 粒度。
删除干净利落:用户删项目 → DROP DATABASE db_project_a,一行搞定,不像 schema 方案要小心翼翼清理残留。
比 schema 强在哪:schema 方案里所有租户共享同一套 auth.users、同一套角色、同一个 search_path 上下文,稍有不慎就串数据;database 方案里这些全是各自独立的。

三、连接管理:Supavisor 是命门
这是路线 C 能不能跑起来的决定性因素。
为什么连接数是头号敌人
Postgres 每个连接 ≈ 一个进程,内存开销不小,实例的 max_connections 通常几百到一两千就到顶。而你的场景是:
N 个项目 × M 个组件(GoTrue/PostgREST/Storage)× K 个并发请求
项目一多,直连必然爆炸。绝不能让组件直连 Postgres。
Supavisor 怎么解决
Supavisor 是 Supabase 官方写的多租户连接池(Elixir 实现),就是为这个场景设计的:

多租户感知:一个 Supavisor 集群可以代理到多个不同的 database,按租户路由,这正好契合"一个 Postgres 多个 database"。
事务级连接池(transaction mode):上千个客户端连接复用少量真实 Postgres 连接,这是把连接数压下来的关键。
横向扩展:Supavisor 本身可以多节点,扛住大量前端连接。

连接串大致形如:postgres://project_a_owner.<tenant_key>:<pwd>@supavisor:6543/db_project_a,Supavisor 根据 tenant key 路由到正确的后端 database。

注意:事务模式下不支持会话级特性(prepared statements 跨语句、SET 会话变量等),组件配置要适配。这是用池子换连接数必付的代价。

四、无状态组件:共享 vs 按项目起
GoTrue、PostgREST、Storage 这些是无状态的,有两种部署策略,可以混用:
策略 1:组件按项目独立起(进程轻、隔离清晰)
每个项目起自己的一组 GoTrue + PostgREST + Storage 容器,各自连到自己的 database。

优点:配置隔离干净,每个项目的 JWT secret、Storage bucket、PostgREST 暴露的 schema 完全独立;一个项目改配置不影响别人。
缺点:容器数量 = 项目数 × 组件数,即使每个很轻,数量大了编排和内存也吃不消 → 必须配合休眠回收(见第六节)。

策略 2:组件共享,运行时按请求路由(省资源)
少量 PostgREST / GoTrue 实例,根据请求里的项目标识动态切换目标 database 连接。

优点:组件数量恒定,资源占用低。
缺点:实现复杂——PostgREST 单实例对应单 database 连接,要"共享"得在前面加路由层按项目分发到不同上游,或者用 PostgREST 的多 schema 能力(但那就退化成 schema 方案了)。GoTrue 同理,单实例绑单 database。

实务建议
混合:活跃项目用策略 1(独立组件,体验好),空闲项目休眠掉只留 database;Storage 这类可以做成共享一组实例 + 按项目 bucket 前缀,因为 Storage 横向共享相对容易。

五、Auth 与密钥管理(开源版最大的坑)
每个 database 有独立的 auth schema 和独立的 GoTrue 配置,意味着:
每个项目独立的密钥体系
Supabase 的 anon key / service_role key 是用该项目的 JWT secret 签出来的 JWT。所以每建一个项目,控制平面要:

生成一个随机 JWT secret(每项目唯一)。
用它签出该项目的 anon 和 service_role key。
把这三样存进密钥管理系统(Vault / KMS / 加密的元数据库)。
GoTrue 和 PostgREST 启动时注入对应项目的 JWT secret,这样它们才能验签该项目的 token。

控制平面元数据(存在你自己的平台库里,加密):
project_a → { jwt_secret, anon_key, service_key, db_conn_string, gotrue_config }
用户体系
每个 database 的 auth.users 独立 → 天然实现了"每个生成应用有自己独立的最终用户体系",这正是你想要的:用户 A 在应用甲注册,和应用乙完全无关。这比 schema 方案(共享 auth.users 再用 RLS 切)干净太多,也是路线 C 相对路线 A 的一大优势。
密钥轮换
要支持轮换 JWT secret(安全事件、定期轮换),控制平面需要能重签 key 并热更新组件配置。设计元数据结构时预留这个能力。

六、生命周期管理:休眠与回收(成本核心)
路线 C 能否经济运行,取决于空闲项目不烧钱。绝大多数生成的应用是僵尸项目(用户建了就不管)。
休眠机制

组件休眠:项目一段时间无请求 → 停掉它的 GoTrue/PostgREST/Storage 容器,只保留 database(database 不占 CPU/内存,只占磁盘)。
冷启动唤醒:请求进来(网关检测到目标项目组件未运行)→ 拉起组件(秒级)→ 转发请求。第一个请求有冷启动延迟,可以加"唤醒中"提示。
完全回收:长期不活跃(比如 90 天)→ 通知用户 → DROP DATABASE + 清 Storage + 删元数据。

这一层就是你的控制平面要做的核心活
监控每个项目活跃度、按策略起停组件、管理资源配额(防止单个项目把共享 Postgres 拖垮——per-database connection limit、statement_timeout、资源 group 等)。

七、网关 / 路由层
用户的最终应用通过子域名访问,比如 project-a.yourapp.dev:

请求到网关(Kong 或自研)。
网关从子域名 / Host header 解析出项目标识。
查控制平面拿到该项目的路由信息(组件地址、是否需要唤醒)。
若组件休眠 → 触发唤醒 → 等就绪 → 转发。
转发到对应的 PostgREST / GoTrue / Storage。

Kong 是 Supabase 自托管栈里默认的网关,可以复用并扩展;但多项目动态路由这套逻辑,通常你会在 Kong 前面或里面接自己的控制平面逻辑。

八、Schema 迁移(AI 场景特有)
你的产品里,AI 会不断帮用户改表结构。这意味着:

每个 database 的业务 schema 是动态、各不相同的,不能用一套全局 migration。
但 Supabase 基础结构的 migration 是统一的(GoTrue、Storage 升级时,所有 database 的 auth/storage schema 都要升)。

所以要分两层管理:

平台层 migration:升级 Supabase 组件时,批量对所有项目 database 的基础 schema 跑迁移(要做好幂等、分批、失败回滚)。
项目层 migration:每个项目的业务表变更,由 AI 生成的 DDL 驱动,你需要记录每个项目的 schema 版本 / 变更历史,支持回滚。

九、路线 C 的代价(诚实地说)

控制平面是必须自研的重活:项目 provision、密钥、路由、休眠、迁移、配额——这套东西本质上是在重建 Supabase 没开源的 Management 层,工作量不小。
冷启动体验:休眠项目的第一个请求有延迟,需要产品上做好预期管理。
共享 Postgres 的爆炸半径:虽然 database 隔离了数据,但它们共享同一个 Postgres 实例的 CPU / 内存 / 连接 / WAL。一个项目跑垮实例,所有项目受影响 → 必须做资源限制 + 监控,且要规划好"单实例承载多少项目就该分片到下一个 Postgres 实例"。
单实例容量上限:Postgres 单实例能开的 database 数量虽大,但实际受连接、内存、shared_buffers、备份时间等约束。要提前设计分片策略:实例满了就开新 Postgres 实例,控制平面记录"项目 → 实例"映射。

十、什么时候该从 C 升级

某个项目流量大到影响邻居 → 把它单独迁到独立 Postgres 实例(升级为路线 B 的待遇),控制平面改一下映射即可。
大客户要合规 / 数据驻留 → 同样拎出来独立实例甚至独立部署。

路线 C 的好处正在于此:它和路线 B 是连续的,不是非此即彼。共享实例 + 每项目 database 起步,需要时把重点项目"提级"到独立实例,平滑演进。

14. 🟢 Router 和 Planner 怎么分层?各自负责什么?
核心答案:
这是 Agent 系统里很常见的两层分工,关键区别是粒度和职责:

Router(路由器):轻量、快、做分类决策。它判断"这个请求该走哪条路"——比如用户说的是"新建一个页面"还是"改个按钮颜色"还是"问个问题",然后分发到对应的处理流程。它不解决问题,只决定交给谁。通常用小模型甚至规则就能做,追求低延迟低成本。
Planner(规划器):重一些,做任务拆解。当任务复杂(比如"做一个带登录的待办应用"),Planner 把它拆成有序的子步骤:建数据表 → 写后端接口 → 做登录页 → 做列表页……再交给下游执行。它负责怎么一步步做。

一句话:Router 决定"走哪条路",Planner 决定"这条路怎么一步步走"。 Router 在前(分流),Planner 在后(对复杂任务做编排)。

为什么要分层:
不是所有请求都需要 Planner。"改个颜色"这种简单请求,走 Router 直接到一个轻量流程就行,没必要启动昂贵的规划。分层是为了简单任务走快车道、复杂任务才走重流程,省 token 省时间。
可能的追问:

"Router 用大模型还是小模型?" → 小模型/分类器,追求快和便宜;甚至高频意图用规则兜底。
"Planner 拆错了怎么办?" → 需要有反馈/重规划机制(re-plan),执行失败的步骤能回到 Planner 调整,这就引出 LangGraph 的循环图(见 17 题)。

15. 🟢 场景化 RAG 的语料怎么积累?效果怎么评估?
核心答案 —— 语料积累:
"场景化"指的是针对这个产品的特定场景(生成 React/Supabase 应用)去喂料,而不是通用知识。来源主要几类:

官方文档/最佳实践:用到的框架和库(React、Tailwind、Supabase API 等)的文档,保证生成代码用法正确。
高质量代码模板/范例:平台沉淀的优质组件、页面、常见功能(登录、CRUD、表单)的标准写法,作为生成参考。
历史成功案例:用户成功生成、且没被反复修改的代码,反向说明"这种写法是 work 的",可以回流进语料。
常见错误 → 修复对:用户报错、平台自动修复的 case,沉淀成"遇到 X 错误该怎么改"的知识。

效果评估:
分两层:

检索质量:召回的语料和当前任务相不相关(命中率、相关性)。检索不准,后面全白搭。
最终生成质量(更重要,因为 RAG 是手段不是目的):

能不能跑通:生成的代码能否编译/运行不报错(这个可以自动化大批量跑)。

人工/模型打分:用更强的模型或人工对生成结果按"正确性、规范性、是否符合需求"打分。

答这题的加分点:强调 "评估 RAG 不能只看检索准不准,要看它最终有没有让生成代码更好——能跑通、用户少改,才是真有用。" 这体现你分得清手段和目标。

16. 🔴 为什么把 Agent 逻辑写在前端静态代码里?这有什么隐患?(重点题)
这是红色高危题,面试官大概率想听你说出隐患,所以答题结构是:先讲为什么有人这么做(给个合理动机),再重点讲隐患。
为什么会这么做(动机):

快、省事:早期 demo / MVP 阶段,把调用模型的逻辑直接写在前端,不用搭后端,迭代飞快。
少一跳:前端直接调 AI API,省掉自己服务器中转,延迟低、架构简单。

隐患(这才是得分点,要多讲):

密钥泄露(最致命):前端代码是用户浏览器里能直接看到的。如果把 AI API Key、数据库密钥写在前端,任何人打开开发者工具就能拿到,然后盗刷你的额度、攻击你的数据库。这是安全红线。
逻辑全暴露:你的 Agent 怎么运作、prompt 怎么写、调了哪些工具,全暴露在前端,容易被抄袭,也容易被人逆向后构造恶意输入绕过你的限制。
没法做服务端管控:限流、鉴权、用量计费、审计、内容安全过滤,这些必须在服务端做才可靠。写在前端,用户改改代码就绕过了,等于没有。
不可控、难统一升级:逻辑散在前端,改个 prompt、换个模型都要重新发布前端,而且老用户缓存的旧代码还在跑旧逻辑。

标准回答收口:"前端只应该负责展示和交互,凡是涉及密钥、计费、鉴权、限流、内容安全的核心逻辑,必须放到服务端。前端调用应该走自己的后端做代理(BFF),由后端去持有密钥、调用模型,前端永远拿不到敏感信息。"

这一题答出"密钥泄露 + 必须用后端代理(BFF)"就稳了,这是面试官最想听到的。

17. LangGraph ts 版本和 Python 版本有什么坑?
先讲 LangGraph 是干嘛的(垫一句显得你懂):它是用图(graph)来编排 Agent 流程的框架,节点是步骤、边是流转,支持循环(比如"执行失败 → 回到 Planner 重新规划"),比线性的 chain 更适合复杂 Agent。正好接上第 14 题的 re-plan。
两个版本的坑(诚实点,这题如果没真用过,就讲共性 + 生态差异,别硬编细节):

生态成熟度差异:Python 版是"亲儿子",功能更全、更新更快、文档和社区案例更多;TS 版往往功能跟进慢一拍,某些新特性、集成(尤其各种工具/向量库的连接器)在 TS 端缺失或滞后。这是最实际的坑。
类型系统的双刃剑:TS 有静态类型,理论上更安全,但 LangGraph 的状态(state)在图里流转,state 的类型定义在 TS 里会比较啰嗦/繁琐,改图结构时类型容易报一堆错;Python 反而灵活随意(但也更容易运行时才发现问题)。
流式 / 异步行为差异:streaming、并发、中断恢复(checkpoint)这类能力,两个版本的实现和稳定度可能不一致,迁移时不能假设行为完全等价。
选型现实:很多团队前端是 TS、AI 编排却用 Python,就是因为 Python 生态成熟,这本身会带来多语言/多服务的架构成本。

诚实策略:如果你没深用过,可以说 "我对 TS 版踩得不算深,但据我了解和实践,主要差异在 Python 版生态更成熟、TS 版功能跟进偏慢,以及 TS 里图状态的类型定义比较繁琐。" —— 承认边界比硬编细节安全,面试官反而觉得你靠谱。

18. 增量修改怎么识别变更范围?
核心答案:
用户说"把登录按钮改成蓝色",你不该重新生成整个项目,而要精准定位到要改的那一小块。识别变更范围的思路:

理解意图 + 定位目标:先让模型/解析层判断这次修改涉及哪些文件、哪些组件。比如"登录按钮"→ 定位到登录页组件文件。可以靠语义检索(在代码里找相关片段)+ 结构信息(组件树、文件依赖)。
依赖分析:改一个地方可能牵连别处。比如改了某个组件的 props,要找出所有用到它的地方。靠依赖图 / import 关系分析出影响范围,避免改了 A 漏了引用 A 的 B。
最小化改动:只把相关文件喂给模型,让它做局部修改(理想情况是输出 diff/patch,而不是整文件重写),既省 token 又减少"改一处错一片"的风险。
变更后校验:改完跑一下编译/类型检查/测试,确认没把别的地方搞坏。

一句话:靠"语义检索定位 + 依赖关系分析"圈定最小变更范围,只改相关文件、最好以 diff 形式增量更新,改完做校验。

可能的追问:

"怎么保证不破坏其他功能?" → 依赖分析 + 改完跑测试/类型检查(接第 19 题)。
"全量重写不行吗?" → 慢、贵、不稳定(每次重写都可能引入新随机性),还会丢掉用户之前的手动微调。

20. Sandpack 在线预览有没有遇到性能问题?
先讲 Sandpack 是啥:CodeSandbox 开源的浏览器内代码运行/预览方案,能直接在浏览器里把用户的前端代码跑起来、实时预览,不用真服务器。Lovable 这类产品常用它做"边写边看"的即时预览。
性能问题(这题想听你说出真实瓶颈):

首次加载慢 / 资源重:它要在浏览器里把一套打包运行环境跑起来,初始化和装依赖比较耗时耗内存,项目大了更明显。
浏览器内存吃紧:整个运行环境在浏览器里,大项目或长时间使用容易内存占用高,低端设备卡顿甚至崩。
依赖安装/解析慢:引入第三方库时,要在浏览器侧解析安装依赖,包多了等待明显。
热更新延迟:代码一改就重新编译预览,改动频繁或项目大时,更新会有可感知的延迟。

优化思路(给了优化才显得你真用过):

依赖缓存 / 预构建:常用依赖提前缓存,别每次重装。
懒加载 / 按需编译:只编译变动部分(接第 18 题的增量思路),不每次全量。
debounce:用户连续输入时不要每个字符都触发重编译,停顿一下再编译。
大项目降级方案:超过一定规模,考虑服务端预览(真容器)兜底,而不是硬扛在浏览器里。

收口:"Sandpack 的核心瓶颈在于整个运行环境都在浏览器里,所以问题集中在初始化慢、内存占用高、依赖和热更新延迟。优化主要靠依赖缓存、增量编译、输入防抖,大项目则降级到服务端预览。"
