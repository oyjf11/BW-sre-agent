基于这套架构，我认为这些架构专家真正想表达的，不是“怎样把 Agent 接入研发流程”，而是：

把企业原本依赖人、文档和经验驱动的产研流程，重构成一套可执行、可编排、可验证、可审计、可持续学习的工程系统。

这就是 Harness Engineering 在产研中的核心落地形态。

⸻

一、Harness 落地的本质：把产研流程变成“可执行系统”

传统研发流程通常是：

需求文档
→ 人理解需求
→ 人设计方案
→ 人写代码
→ 人安排测试
→ 人判断是否交付

流程虽然存在，但大量规则只存在于：

- 团队成员经验
- Wiki 和规范文档
- 项目负责人的脑子里
- 口头协作习惯
- CI 脚本和零散工具中

Harness 要做的是把这些内容工程化：

知识

- 工具
- 契约
- 流程
- 状态
- 证据
- 评审
- 记忆

最终形成：

需求输入
→ 系统自动加载相关知识和约束
→ 编排引擎调度 Agent、工具和人工任务
→ 每个阶段产生标准化产物
→ 质量门禁基于证据决定是否继续
→ 运行结果沉淀为新知识

所以 Harness 的第一价值不是“模型更聪明”，而是：

研发过程从松散的人工作业，变成一个受控的运行时。

⸻

二、它在产研中通过五个动作落地

1. 把能力原子化

不能只建设一个“全能开发 Agent”。

要把研发过程拆成独立、可复用的能力单元：

需求理解
需求澄清
影响范围分析
架构设计
数据设计
接口设计
编码实现
单元测试
构建打包
测试范围分析
E2E 执行
失败归因
代码评审
缺陷修复
用户验收
知识沉淀

每个能力单元可以由不同执行者完成：

- Agent
- Skill
- Tool/MCP
- Script
- CI Job
- Human Task
- External Service

例如你的前端自动化测试能力，不应该只是一个 /run-playwright 命令，而应拆成：

测试范围识别
→ 测试用例生成
→ 环境准备
→ 测试执行
→ 证据采集
→ 失败归因
→ 缺陷流转
→ 修复复测

原子化之后，项目才能自行组合流水线。

⸻

2. 把产物结构化

Harness 不能依赖 Agent 输出一大段自然语言。

每个阶段都要定义机器可读的中间产物。

例如需求分析输出：

{
"requirements": [],
"acceptance_criteria": [],
"open_questions": [],
"scope": [],
"out_of_scope": [],
"risks": []
}

设计阶段输出：

architecture.md
data-model.json
openapi.yaml
decision-records/
risk-analysis.json

开发阶段输出：

code_diff
unit_test_report
build_report
change_summary

测试阶段输出：

test_plan
test_cases
test_results
evidence_pack
defect_records

这意味着：

Harness 中真正流动的不是对话，而是标准化工程产物。

Agent 只是产物的生产者或消费者。

⸻

3. 把规范变成运行时契约

传统规范通常只是文档：

请遵循代码规范
请补充测试
请注意接口兼容

Harness 中要把它们转换成可执行约束：

必须通过 Lint
必须通过类型检查
核心路径必须有 E2E
接口变更必须生成兼容性报告
高风险变更必须人工审批
测试失败必须生成 Trace
租户数据必须携带 tenant_id

因此契约需要落到：

- 输入 Schema
- 输出 Schema
- Agent 权限
- 工具白名单
- 质量门禁
- 状态转换条件
- 人工审批规则
- 失败和回滚策略

例如：

node: frontend-e2e-test
entry_conditions:

- build_passed
- environment_ready
  required_outputs:
- test-results.json
- trace.zip
- screenshots
  exit_conditions:
  passed: - blocker_failures == 0
  rejected: - environment_unavailable
  manual_review: - flaky_cases > 3

这就是“规范即代码”和“治理即运行时”。

⸻

4. 用确定性的编排控制非确定性的 Agent

这是整套架构最核心的工程原则。

Agent 输出天然具有不确定性，所以不能让 Agent 自己决定整个流程。

正确关系是：

编排引擎：确定性的控制系统
Agent：非确定性的智能执行单元

编排引擎负责：

- 任务顺序
- 依赖关系
- 并行调度
- 状态流转
- 事件触发
- 超时重试
- 失败恢复
- 权限切换
- 人工审批
- 执行审计

Agent 负责：

- 理解
- 分析
- 生成
- 归纳
- 判断
- 提出方案

例如：

构建完成事件
→ 编排引擎触发 E2E 测试
→ 测试 Agent 生成或选择用例
→ Playwright 真实执行
→ 失败归因 Agent 分析结果
→ 质量门禁根据证据判断
→ 通过则进入 SIT
→ 失败则进入缺陷管理

最终决策不能仅依据 Agent 的一句“测试通过”，而要依据真实执行产物。

⸻

5. 建立知识回流闭环

一次任务执行结束，不应只留下代码。

还应沉淀：

- 需求和代码的对应关系
- 设计决策
- 测试覆盖关系
- 失败模式
- 缺陷原因
- 有效修复方案
- 项目隐含规则
- Flaky 测试特征
- 用户验收反馈

形成双向关联：

需求
↕
设计
↕
代码
↕
测试
↕
缺陷
↕
修复
↕
发布与运行结果

下一次 Agent 执行时，不再只是检索几个相似代码块，而是可以知道：

这个需求涉及哪些模块
这些模块历史上出现过哪些问题
有哪些测试覆盖
修改会影响哪些调用方
过去采用过什么设计决策
当前项目有哪些特殊规范

这才是所谓“代码和知识双向索引”的实际价值。

⸻

三、完整产研链路如何运行

可以将它抽象为下面这条主链路：

1. 需求进入

输入包括：

- 自然语言需求
- PRD
- Issue
- 设计稿
- 约束条件
- 优先级
- 项目信息

Harness 自动加载：

- 项目代码
- 业务知识
- 历史需求
- 历史缺陷
- 设计规范
- 产品特有规范

⸻

2. 需求分析

执行：

需求理解
→ 歧义识别
→ 需求澄清
→ 范围确认
→ 验收条件提取

质量门禁检查：

- 是否存在未解决歧义
- 是否有明确验收条件
- 是否定义了范围和非范围
- 是否识别关键风险

不满足则停留在需求阶段，而不是直接生成代码。

⸻

3. 设计

按任务依赖进行编排：

架构设计
↓
数据设计 ↔ 接口设计
↓
一致性评审

评审 Agent 和反证 Agent 负责：

- 是否违反现有架构
- 是否存在更小改动方案
- 是否影响兼容性
- 是否遗漏异常路径
- 是否符合产品特有规范

产出经过批准后才进入开发。

⸻

4. 开发

可以并行调度：

前端实现 后端实现
前端单测 后端单测
↘ ↙
联调构建

开发 Agent 不能自由修改整个仓库，而要受以下约束：

- 修改范围
- 工具权限
- 代码规范
- 设计产物
- 测试要求
- 目录边界

⸻

5. 测试

测试不是开发后的单一节点，而是贯穿全过程。

需求阶段：提取验收条件
设计阶段：检查可测试性
开发阶段：单元测试和自测
测试阶段：集成、SIT、回归
验收阶段：UAT 和交付验证

特别是前端自动化测试，应提供浏览器真实执行证据，而不是仅靠模型判断。

⸻

6. 用户验收与交付

最终交付需要同时具备：

- 需求确认记录
- 代码版本
- 测试报告
- 缺陷关闭记录
- UAT 结果
- 证据包
- 发布记录
- 回滚方案

Harness 不是“生成完代码就结束”，而是以用户验收和交付完成作为终态。

⸻

四、为什么要支持项目级自定义流水线

企业研发不存在一条适合所有项目的流程。

例如：

普通内部管理系统

需求确认
→ 设计
→ 开发
→ 单测
→ E2E
→ 验收

核心交易系统

需求确认
→ 架构评审
→ 安全评审
→ 开发
→ 单测
→ 集成测试
→ 性能测试
→ 灰度
→ 观察
→ 验收

紧急缺陷修复

缺陷定位
→ 最小修复
→ 定向测试
→ 人工审批
→ 紧急发布
→ 全量回归
→ RCA

所以平台提供的是通用能力，而项目负责配置：

- 哪些阶段存在
- 哪些节点必选
- 哪些节点可跳过
- 哪些任务并行
- 哪些门禁必须通过
- 哪些节点需要人工参与
- 哪个版本的模板生效

最终实现：

平台统一底座，项目差异化编排，组织规范统一治理。

⸻

五、组织层面如何分工

这套架构通常要求四类团队协作。

平台团队

负责：

- 编排引擎
- 状态管理
- 事件总线
- 权限系统
- 模板中心
- 版本管理
- 运行观测
- 审计能力

能力建设团队

负责不同领域的能力组件：

- 需求分析能力
- 设计能力
- 代码生成能力
- 前端测试能力
- 后端测试能力
- 安全检查能力
- 代码评审能力

你的工作属于这一层。

项目团队

负责：

- 选择模板
- 配置流水线
- 接入项目知识
- 定义产品特有契约
- 配置质量门禁
- 处理人工审批任务

治理团队

负责：

- 通用规范
- 安全要求
- 质量标准
- 评测体系
- Bad Case 管理
- 流程合规
- 能力准入与版本发布

⸻

六、Harness 落地不是“全自动化”

这张图里一个很重要但容易被忽略的结论是：

Harness 的目标不是消灭人，而是明确什么由系统执行、什么由 Agent 判断、什么必须由人负责。

典型分工：

确定性任务
→ 脚本、工具、CI 执行
语义理解与方案生成
→ Agent 执行
高风险决策
→ 人工审批
最终业务验收
→ 需求用户负责

例如：

- Lint 不需要 Agent 判断
- 架构方案可以由 Agent 辅助生成
- 生产数据删除必须人工审批
- 用户是否接受最终效果必须由用户确认

因此它追求的是：

受控自动化，而不是无人化研发。

⸻

七、真正的落地路径

企业不可能一次建设完整体系，通常应分阶段实施。

第一阶段：能力工具化

先将高频场景封装成 Skills、Agents、Commands：

- 代码生成
- 测试生成
- Bug 排查
- 代码评审
- 文档生成

解决“能不能用”。

第二阶段：能力标准化

为每个能力定义：

- 输入
- 输出
- 权限
- 工具
- 契约
- 证据
- 失败状态

解决“能不能稳定复用”。

第三阶段：流程编排化

将能力放进研发流水线：

需求 → 设计 → 开发 → 测试 → 验收

增加状态、依赖、事件和人工审批。

解决“能不能形成闭环”。

第四阶段：治理与评测

建立：

- 质量门禁
- 证据链
- 离线数据集
- Bad Case
- 指标体系
- 版本管理
- 审计追踪

解决“能不能规模化推广”。

第五阶段：知识闭环

将运行数据、代码、需求、测试和缺陷关联起来，形成持续更新的知识图谱。

解决“能不能越用越好”。

⸻

八、对你“开发套件”项目的最终理解

你的开发套件并不是整个 Harness。

它更像是：

企业 AI Harness 中的研发能力组件库。

编排平台把产研流程跑起来，开发套件向流程提供具体能力。

而你负责的前端自动化测试，是开发套件中的：

前端验证与证据能力域。

它不仅负责写 Playwright 脚本，还应该提供：

需求验收条件解析
代码变更影响分析
测试范围选择
用例设计与生成
测试环境控制
浏览器自动执行
失败归因
证据采集
缺陷流转
修复复测
UAT 支撑
测试知识沉淀

⸻

最终结论

从这套架构中，可以总结出 Harness Engineering 在产研中的完整定义：

Harness Engineering 是以编排引擎为运行骨架，以知识、工具和契约为执行输入，以 Agent 和自动化工具为执行单元，以结构化产物和证据链为阶段交付，以质量门禁和人工审批控制风险，并通过运行数据和知识回流持续优化的 AI 原生产研工程体系。

更简洁地说：

把研发知识变成上下文
把研发规范变成契约
把研发能力变成组件
把研发流程变成编排
把研发结果变成证据
把研发经验变成知识

这六件事，就是 Harness 真正在企业产研中的落地。
