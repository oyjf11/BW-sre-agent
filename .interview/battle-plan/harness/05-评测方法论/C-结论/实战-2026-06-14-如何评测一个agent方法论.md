# 如何设计一个最小框架来评测 Agent

> 🎯 **一句话总纲（背这句）**：评测一个 agent 的复杂度，由"对有多客观"逼出来——"对"客观能判的，一层代码就够、不需要 gold set；"对"一旦含主观，才被迫上模型判 + gold set。
>
> 本文既是方法论文章，也是**执行守则**：照着第 0→7 步走，就能从零搭出一个最小但可信的 agent 评测框架。前端自动化测试（自愈 agent）是贯穿全文的实例，但守则本身可迁移到任何 agent。

---

## 为什么需要"框架"，而不是"跑几个例子看看"

给 agent 打分最容易犯的错，是**拿几个 case 跑一遍、看着顺眼就说"行"**。这不叫评测，叫演示。问题有三：

1. **没有标准答案的题，等于没考**——你凭感觉说好，换个人就说不好。
2. **一个总分藏住一切**——"综合 85 分"既会被刷，又看不出到底哪段弱。
3. **今天过明天挂**——世界一直在变，同一道题结果不稳，分数没有可比性。

一个最小评测框架，就是用**最少的部件**把这三个洞堵上：**让"对"有据可依、让分数分段可读、让结果可重复。** 下面 8 步，每步一条守则。

---

## 第 0 步 · 先确认这个能力"可被验证"

> **守则：没有裁判的能力，再重要也先不评。**

评测的前提是存在某种"自动对错判定手段"——能跑的测试、有结论的历史数据、可校验的事实。如果一个 agent 的输出连人都说不清对错，框架无从搭起。先找裁判，再谈评测。

*前端例*：自愈修完的脚本能丢进沙箱重跑（过/不过是客观裁判），历史"改版导致修复"的提交带着人工修复版（标准答案白送）——所以这个能力可评。

---

## 第 1 步 · 把 agent 能力按生命周期拆段

> **守则：多阶段 agent 绝不给一个综合分；按生命周期拆段，每段单独评。**

一个真实的 agent 几乎都是多阶段的。把它拆成几个职责单一的段，分别评——因为每段"对"的含义不同，揉成一个数既会被刷、又看不出短板在哪。

*前端例*：前端测试 agent 的完整能力域是一个闭环——

```
编写(生成用例) → 执行(跑) → 修正(分诊/自愈) → 守护(回归发现力)
```

self-heal（自愈）只是"修正"这一段。只评自愈，等于拿"修车"代表"开车"。

---

## 第 2 步 · 拆解每段的"对"：尽量客观化，残留才上人【全框架的枢纽】

> **守则：把每段的"对"拆成「客观可判」+「主观残留」；用代码判和"真-bug-回放"尽量把东西推进客观内核，只给消不掉的主观残留建 gold set。**

这是整个框架最关键的一步，**做对了能省掉一半工作量**。注意：客观/主观**不是"段"的标签，是段内的一条边界**——同一段往往既有客观内核、又有主观边缘。逐段把"对"拆开：

| 段 | 客观内核（代码判/跑/回放，无需 gold set） | 主观残留（需人写标准 或 模型判+gold set） |
|----|------|------|
| 编写 | 断言非空壳(AST) + 绿 on 正确代码 + 抓得住已知 bug(回放/变异) | 全新行为"该测什么、够不够全"（无 bug 可回放时） |
| 执行 | 用例真跑了没/有没有漏跑(结果列表) + 多次跑一致(可重复) | flaky 归因(环境噪声 vs 真竞态) + 冻结边界够不够 |
| 修正 | 沙箱重跑过没过 + 断言 AST 没变 + 选择器唯一 | 修复"语义上合不合理" |

**关键一刀——覆盖率不是裁判。** 判"生成的测试覆盖够不够"时，**绝不能用代码覆盖率**（行/分支跑没跑到）：agent 能生成一堆空壳测试让覆盖率冲到 100%、却什么都不查（"任何被 AI 优化的指标都会被玩坏"）。客观化的正确办法是**"抓不抓得住真 bug"**——历史缺陷回放 + 变异测试，它把开放主观的"该测什么"换成封闭二值的"能不能抓住这个真实坏过的 bug"。

为什么这一步是枢纽？它决定 gold set 建在哪：**能推进客观内核的，代码/回放判到底、不建 gold set；推不动的主观残留，才请模型当裁判、再用 gold set 校准。** 拆错了，要么白建 gold set（浪费），要么漏建（主观分数没人信）。

> **gold set 跟着"主观判定"走，不跟着"段"走**——哪一段冒出了模型判，哪一段就得配 gold set，编写段也不例外（比如全新行为、连已知 bug 都没有可回放时）。

---

## 第 3 步 · 按"客观程度"配裁判，三层优先级

> **守则：能用代码判的绝不用模型判；能跑的就直接跑；只有真主观的才上模型判。**

裁判分三层，优先级从低成本到高成本：

1. **代码判**（首选，零成本零抖动）：答案命中没命中、关键词在不在、AST 结构变没变、沙箱重跑过没过。
2. **跑一遍**：编译器/测试说了算。
3. **模型判**（只给真主观的）：且必须降噪——随机性关到零（temperature=0）、只分两三档（别打 1–10 分制造噪声）、判三次取多数、**裁判模型 ≠ 被考模型**。

*前端例*：编写段=纯代码判（覆盖/空断言）；执行段=跑多遍比一致性；修正段=代码判（沙箱重跑 + swc 断言锁 + 选择器唯一性）**加** 模型判（修复语义是否合理）。

---

## 第 4 步 · 建题库（考 agent 的卷子）

> **守则：题库一道题 = 藏住结论的历史输入 + 冻结快照 + 标准答案 + 故意埋的陷阱。**

题库是考 agent 的卷子，物理上就是仓库里的一个文件，每道题有负责人。一道题包含：

- **输入**：藏住结论的历史案例 + 当时的数据**快照**（见第 6 步，快照是可重复的关键）。
- **期望**：标准答案 / 可接受范围 / **故意埋的"不可接受"陷阱选项** / 证据里必须出现的关键词。
- **三类题**：**业务题**（判得对不对，答案来自历史）、**行为题**（过程规不规矩：该查的查了吗、有没有越权、信息不足时老实说"建议补充"还是硬猜）、**安全题**（输入里藏指令注入，必须 100% 不中招）。
- **陷阱题**：每次抓到 agent 作弊未遂，就做成一道陷阱题永久留库站岗。

标准答案哪来？**从历史白送**——历史工单的结案结论、git 里人工修复的那一版，都是现成的标准答案。

---

## 第 5 步 · 凡是用了模型判的地方，建 gold set（考判卷器的卷子）

> **守则：哪里用了模型判，哪里就建 gold set——跟着"主观判定"走，不跟着"段"走。**

模型判会抖会错，所以判卷器本身也要被考。gold set 就是几十条由**人类专家亲手打分**的记录 + 一段算一致率的代码。建法：

1. 三位专家**各自独立打分，不许商量**。
2. 先看专家之间一致不一致（用 **kappa** 算，它扣掉了瞎蒙的水分）。
3. 专家都打不齐 → **开"对齐会"修评分标准，不是修人**；全员一致的题才入册，吵不拢的边界题剔除。
4. 判卷器任何改动（提示词改一行、裁判模型升一版）→ **先在 gold set 上重新及格才许上岗。**

为什么不会无限套娃：gold set 是**死数据 + 算术**，跑一万次结果都一样，套娃到此为止。终点是人类白纸黑字的共识——和计量体系最终追溯到物理定义一个道理。

> **这一步是"最小框架"最省力的地方**：客观能判的部分一律不碰 gold set；只给消不掉的主观残留建（修正的"语义合理"、编写无 bug 可回放的"新行为覆盖"、执行的 flaky 归因）。把昂贵的人类标注只花在真正需要的那一处。

**金标的"及格"本身也要讲统计**（常被忽略）：判卷器是活的、会抖，"它过没过金标"是一次带噪声的测量，所以会出现"第一轮过、第二轮不过"。四条治法：① **钉死裁判模型版本 + 评分提示词**（禁用自动最新版），否则厂商悄悄升级会让分数一夜漂移；② 判卷器对每条金标**判三次取多数**，吸收单次抖动；③ **kappa 报区间不报裸数**，及格看区间下界/多轮稳定，不看单轮点估；④ **严禁重跑金标到过**（p-hacking），分数在及格线晃就**加金标量**，不是重判到运气好那次。一句话：套娃到金标（死数据+算术）为止，但"被它考的判卷器会抖"，所以统计纪律上移一层照样适用。

---

## 第 6 步 · 冻结世界（可重复性的地基）

> **守则：跑评测前先把外部世界冻成快照，否则前面全白搭。**

世界一直在变（接口返回、时间戳、随机值），同一道题今天过、明天挂，分数无法比较。把某一时刻的外部世界录下来冻结，评测时 agent 查到的是快照而非实时系统。

**可重复性 = 把世界冻结。** 这一步不产出分数，但它是前面所有分数能不能信的地基。

*前端例*：录制回放 + 请求指纹归一化——按路径剥离时间戳/随机 ID，**且序列化前把字段名排序**（否则 `{user,role}` 和 `{role,user}` 算出两个哈希、回放全穿透真后端）。

---

## 第 7 步 · 汇总成北极星向量 + 跑对照

> **守则：把 agent 当分类器报 P/R + 成本，永远当向量、绝不揉成一个数；对照要单变量配对比较，结论写约束式。**

最后把分数汇总成**给人看的成绩单**，两条纪律：

**① 北极星当向量，不当标量。** 把被评的能力当成一个"出事没出事"的分类器：

| 北极星 | 分类器术语 | 含义 |
|--------|----------|------|
| 信噪比 | Precision | 报警里真问题占比（结果可不可信） |
| 发现力 | Recall | 真问题被抓住的比例（= 1 − 漏报率） |
| 有效覆盖 | — | 需求被**非空壳**测试覆盖的广度 |
| 维护成本 | — | 工时 / 元 |

四个**互相咬死**：想刷信噪比（删抖动测试）→ 发现力或覆盖立刻掉。所以结论永远写**约束式**：**"发现力/覆盖不降的前提下，成本降 X%、信噪比 a→b。"** 绝不说"综合分 85"。flaky、kappa、各段成功率降为**诊断指标**，解释北极星为什么动。

**② 对照单变量、配对比较、守统计纪律。** 要证明"比基线（人工/现有体系）好"：同一批输入、同一批事件，**一次只改一个变量**；看"几道由错变对、几道由对变错"的翻转明细，而不是只看总分差；报告带误差范围；**严禁重跑到绿**（分数在及格线晃就加题再考，不是反复重跑撞运气）。

**③ 误差范围是算出来的，不是画的。** 比例类指标（命中率/准确率）95% 误差半宽 ≈ `1.96 × √(p(1−p)/n)`。取 p=0.8：n=200 → **±5.5pp**；n=400 → ±3.9pp；n=1000 → ±2.5pp。**开平方关系——误差砍一半、样本翻四倍**，所以加题比加遍数划算。报告写 `79% ± 5.5`、不写 `79%`。样本量没有"标准数字"，按目标误差反推：比例指标 ~200 常见够用；**金标（kappa）更吃样本，几十条只够演示、想稳要 50–100、要窄区间得低几百**。kappa 及格线用 Landis-Koch 经验表（0.41–0.60 中等 / 0.61–0.80 可用 / 0.81+ 几乎完美），按风险定，**无全行业统一阈值**。

> 诚实边界：kappa、置信区间、防 p-hacking 是教科书铁标准；"裁判判三次取多数/版本钉死"是 2024 年后 LLM-as-judge 社区收敛的做法、非成文规范，各家阈值不一。

## 框架运转起来：什么时候跑 + 怎么用结果改进

**跑的时机（两个对象、两套节奏）：**

| 跑什么 | 触发 | 频率 |
|---|---|---|
| agent 跑评测集 | agent 变了（提示词/知识/工具/模型版本/编排） | 高频：smoke 每次改 → 全量合入前(门禁) → 藏题定期 |
| 判卷器跑 gold set | 判卷器变了（评分提示词/裁判模型/标准）+ 定期巡检 | 低频：一改先过 gold set 才许上岗 |

两条规矩：① 判卷器没过 gold set 前，它打的评测分不算数；② 评测分诡异时先复检 gold set，分清"agent 真退步"还是"裁判在抖"。

**用结果改进（短板 → 飞轮）：** 分数低不是短板，"哪类、为什么低"才是。定位四招——分层看分 / 配对翻转明细 / **翻最差题的执行轨迹**（最差题清单=改进待办）/ 失败按根因归类。改进**主战场是喂信息、不是换模型**：缺知识→补知识包（skill），过程不规矩→加行为准则（提示词），会作弊→加确定性闸门（不靠提示词），漏跑/顺序错→改编排。改完→重跑评测验证分涨→涨分题进回归、作弊未遂做成陷阱题——**而"改完"本身又 = agent 变了 = 触发上面那次评测，闭成飞轮。**

## 一页执行清单（照着打勾就能搭出最小框架）

```
□ 0. 这个能力有没有自动裁判？没有就先别评。
□ 1. 把 agent 拆成几段（按生命周期），列出来。
□ 2. 拆每段的"对"=客观内核+主观残留；覆盖好不好用"抓真bug(回放/变异)"判、别用覆盖率。 ← 枢纽
□ 3. 配裁判：客观→代码判；能跑→跑；主观→模型判(temp=0/两三档/判三次/裁判≠被考)。
□ 4. 建题库：历史输入(藏结论)+快照+标准答案+陷阱；业务/行为/安全三类题。
□ 5. 只给"主观残留"建 gold set(跟判定走不跟段走)：3人独立标→kappa→对齐会→全员一致入册→判卷器改动先及格。
□ 6. 冻结世界：快照回放，判等要严（排序/剥离易变字段）。
□ 7. 北极星向量(P/R+覆盖+成本)，约束式结论；对照单变量配对比、带误差范围、不重跑到绿。
□ 8. 跑的时机：agent 改→评测集(smoke/全量/藏题)；判卷器改→先过 gold set 才上岗。
□ 9. 用结果：翻最差题 trace 定短板→主要靠喂信息改→重跑验证→涨分进回归、作弊进陷阱题(改完=agent变了=再触发评测)。
```

**最小框架到此为止。** 再往上是"扩到企业级"——CI 评测门禁（提示词/配置改了也跑题库回归）、模型网关、题库负责人/金标负责人两个常设岗位。这些是规模和组织问题，不属于最小框架；**评测体系本质是组织产物，单人 demo 在"组织"那一维天然到不了企业级。**

---

## 一句话收口

> **题库考 agent，gold set 考判卷器；客观的活交给代码，主观的活才请模型、再用人类共识校准模型。整座评测塔，是把一次昂贵的人类共识，廉价地复制到一百万次自动判决里。**

**最高频追问 → 一句话答：**

- *怎么评测一个 agent？* → 拆段、每段配裁判、裁判轻重看"对有多客观"，总分当北极星向量绝不揉成一个数。
- *什么时候要 gold set？* → 只有引入模型判（主观）的段才要；客观段代码判到底、不需要。
- *谁监督裁判、会不会无限套娃？* → 到 gold set（死数据+算术）为止，终点是人类共识。
- *凭什么比传统/人工好？* → 同战场单变量配对比，赢=成本↓且发现力/覆盖不降，定位是接管苦力不替换人。
- *这算企业级 / 算测试框架吗？* → 都不算。是建在 Playwright 之上的"AI 测试 harness + 评测体系"，且是 demo 级方法论演示——内核企业级，缺规模/组织/运行时治理。

**诚实边界**：这是一套**方法论 + 一份设计 spec**，不是已部署、已实测的系统。前端自动化测试/自愈是我真趟过的域（能往深讲 swc 闸门、teleport 坑、请求指纹排序那行）；但这份"评测框架"是我和 Claude 在对话里推演、我负责判断每个框架立不立得住并落成 spec 的**设计**——里面的数字（60→83、kappa 0.55→0.81、22%→3%）是复刻自方法论母本的**目标量级/方案推演，尚未真跑实测**。讲的时候说清"这是我完整设计、准备真跑的方案"，不假装是交付后的实测曲线。
