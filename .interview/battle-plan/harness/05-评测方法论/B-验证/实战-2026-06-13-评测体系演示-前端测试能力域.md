## 1. 思想主线（全篇的魂）

> **评测体系的复杂度，是被"对的定义有多主观"逼出来的。**
>
> 客观能判的，一层代码就够；一旦"对"含主观，才层层加码到模型判 + gold set。
>
> 进一步：**没有"纯客观的段"——每段都是「客观内核 + 主观边缘」，框架的功夫是尽量把每段推进客观内核，剩下的主观残留才上 gold set。**

本 demo 用同一个能力域的四段（编写 / 执行 / 修正 / 守护）演示这件事，**把客观端（编写三件套）和主观端（自愈 + gold set）两头都演全**——比单演自愈深一个层级。

## 2. 核心认知（演示开篇必讲，很多人在此卡住）

### 2.1 为什么自动化测试需要"自愈"

产物是**测试报告**没错，但测试套件是**会腐烂的资产**：脚本靠"定位代码"（如 `.submit-btn`）够到元素，前端天天改版，类名/结构一变，**按钮功能没坏、脚本却找不到了，当晚报红**。

红分两种：

| 种类 | 含义 | 处置 |
|------|------|------|
| 真红 | 应用真的坏了/回归了 | 报告它，**绝不自愈** |
| 假红 | 应用没坏，只是定位过期 | 更新定位，红消失 |

不处理假红：① 人工每天上午手改（纯体力）；② **报警疲劳**——一半红是假的，工程师渐渐无视红灯，藏在里面的真 bug 一起被忽略，**报告失去意义**。

> 比喻：报告 = 烟雾报警器；UI 改版 = 积灰误报；住户抠电池 → 真着火不响。**自愈 = 自动清灰，让报警器只为真火响，它从不替你灭火。**

自愈的定位：**自动化的是"维护"（改定位），不是"判定"（对错）；它服务于报告的信噪比与可信度，不替代报告。**

### 2.2 DOM 定位 vs 断言——一条测试碰 DOM 的两个动作

| 动作 | 在干嘛 | 类比 |
|------|--------|------|
| 定位（locator） | 怎么**找到**元素 | 找人的地址电话 |
| 断言（assertion） | 找到后**检查什么事实成立** | 确认是不是本人 |

> **只影响"怎么定位元素"的 DOM 改动，不代表逻辑坏；影响"被断言的那个可观察行为"的 DOM 改动，就是逻辑坏。**

死规矩：**只许改定位，碰断言一律作废**——改定位=更新地址（安全可自动），改断言=篡改"什么算对"（偷改考题答案，必须锁死）。判"DOM 改动落在哪一边"是自愈最核心也最易判错的能力，评测要狠考。

### 2.3 先走一遍：一道题，怎么从"跑"变成"一个可信的分"

别急着上框架。先拿**一道真实的题**把整条评测链走一遍——后面所有抽象（三层塔、gold set、kappa、误差范围）都只是这条链的展开。

**场景**：登录页改版，提交按钮的 class 从 `.login-btn` 改成了 `.btn-primary`，夜里脚本报红。

1. **一道题（来自题库）**：题面 = 这个报红脚本 + 执行轨迹 + 改版后页面 + 冻结快照；标准答案 = 当年人工修复的那版（只改定位）。
2. **跑 agent**：自愈 agent 看轨迹，判这是"假红"（页面没坏、只是定位过期），把定位改成 `.btn-primary`，沙箱重跑→绿，产出一个修复。
3. **判卷器打分（三层）**：① 代码判——沙箱重跑过了 ✓、断言那行 AST 没动 ✓、新选择器只命中 1 个元素 ✓；② 模型判——"这个修复语义上合理吗？"→"合理"。**这道题 agent 得分 = 对。**
4. **慢着——第 3 步那个"模型判合理吗"，它自己靠谱吗？** 这就是为什么要 **gold set**：拿一批**已经被 3 个人评过"合理/不合理"的老修复**，让这个模型判卷器重新评一遍，看它跟人一致不一致（这个一致率就是 **kappa**）。一致 → 判卷器可信 → 它给 agent 打的分才算数；不一致 → 先回炉校准判卷器，agent 的分先挂起。
5. **最后，这道题不只跑一遍**：同一道题连跑 5 次看抖不抖（可重复）；整套 200 道题跑出一个分，比如 **79% ± 5.5**（这个 ±5.5 是统计算出来的，不是拍的）。

走完这一遍，整个评测体系的骨架就出来了：

> **题库**考 agent（第 1–3 步）→ **判卷器**给 agent 打分（第 3 步）→ **gold set**校准判卷器（第 4 步）→ 全程**冻结 + 多次跑 + 误差范围**保证分可信（第 5 步）。

下面各节，就是把这条链的每一环拆开讲。

## 3. 完整能力域：四段闭环，每段都是「客观内核 + 主观边缘」

```
编写(生成)  →  执行(跑)  →  修正(分诊/自愈)  →  守护(回归发现力)
   ↑                                                      |
   └──────────────── 需求/AC、改版事件 ────────────────────┘
```

self-heal 只是"修正"一环，单拎出来不构成能力域。一个多阶段 agent **不给一个万能总分，每段配它该有的裁判**。而且——**没有"纯客观的段"，每段都同时有客观内核和主观边缘**：客观能判的（代码判 / 跑 / 真-bug-回放）判到底、不需 gold set；剩下消不掉的主观残留，才上模型判 + gold set。**框架的功夫，就是尽量把每段推进客观内核**——客观程度不是"段"的标签，是段内一条要努力推进的边界（这正是第 1 节主线的落地）。

| 段 | 评什么 | 客观内核（代码/回放判，无需 gold set） | 主观边缘（模型判 + gold set） | 作弊 → 怎么防 |
|----|--------|------|------|------|
| 编写 | 覆盖够不够、断言空不空 | 非空壳(AST) + 绿 on 正确代码 + 抓得住已知 bug(回放/变异) | 全新变更**无 bug 可回放**时"该测什么、够不够全" | 空壳凑覆盖 → HOLLOW；**绝不用代码覆盖率当裁判** |
| 执行 | 可重复 + 有没有漏跑 | 跑没跑(CLAIMED_NOT_RUN) + 多次跑一致 | 一次 flaky 是"环境噪声还是真竞态"的归因 | 删抖的测试降 flaky → 覆盖数守 |
| 修正 | 分诊准 + 修对 + 不越界 + 抗注入 | 沙箱重跑 + 断言 AST 没变 + 选择器唯一 | 修复"语义上合不合理" | 删断言/放宽选择器 → swc + 唯一性 |
| 守护 | 维护后还抓得住真 bug | 历史缺陷回放 / 变异（客观） | —（基本纯客观） | 为绿而绿 → 回放守 recall |

> 现状：编写的 coverage-checker 已有；回放/变异、冻结层、自愈+闸门、回放负载都需建。

**三个最尖锐的不同（决定每段评测形态）：**

1. **评测对象不同**：编写/修正评 **agent 智力**；执行评 **harness（冻结层）质量**，与 agent 聪不聪明无关。
2. **抖动一处是敌人、一处是猎物**：编写/修正里抖动是要消除的噪声（判卷 temp=0、判三次）；执行里**抖动就是要测的那个数**（flaky）。
3. **gold set 跟着"主观判定"走，不跟着"段"走**：哪一段冒出模型判的主观"对"（修正的"语义合理"、编写无 bug 可回放的"该测什么"、执行的 flaky 归因），哪一段才建 gold set —— 三层塔（第 5 节）只在主观残留处登场。

## 4. 编写段深挖：把客观内核做厚（三件套 + best-of-N，吸收 Case C）

第 3 节说每段都要"尽量把客观内核做厚"。编写段是个好例子——它的客观内核靠**纯代码三件套**撑起来（不需 gold set）：

1. **绿 on 正确代码**——生成的测试在当前正确实现上必须全过；会报假警的测试比没测试更糟。
2. **历史缺陷回放**——注入一个已知 bug，看生成的测试抓不抓得住。（业务领导秒懂："过去漏到现场的 100 个 bug，这套当年能拦 63 个"）
3. **变异测试兜底**——往正确代码注入小突变，看测试杀不杀得掉，防覆盖率刷分。

反作弊铁律：**绝不拿覆盖率当裁判**——AI 秒级生成空壳测试凑数字 → 靠 HOLLOW 检测拦。

**主观残留只剩一处**：全新变更若**没有历史 bug 可回放**，"该测什么、够不够全"就没有 oracle 了——这一处要么人写一条 AC 把 ground truth 钉死（化主观为客观），要么上模型判 + gold set。三件套把客观内核做到很厚，主观残留就压到了最小。

**架构选型：编写段适合 best-of-N（并行生成五份取最优）。** 因为三件套全是机器秒判、挑选成本极低（手册"裁判贵不贵决定架构"）。对照：端到端"执行"段跑一遍要几分钟、挑选贵 → **不 best-of-N，退回单 agent**。同一能力域里，"裁判便不便宜"直接决定要不要并行——这本身是一道演示题。

**与守护段的分工（别看成重复）**：同一套"回放/变异"机制，问的问题不同——编写段问"**新生成的测试**抓不抓得住 bug"（生成质量）；守护段问"**自愈维护后的套件**还抓不抓得住"（维护没拔牙）。

> 前端适配：Case C 原文是后端味的"把代码退回 bug 修复前"；前端 e2e 的等价物是向靶子**注入已知 UI/行为 bug**（复用第 12 节的真 bug 注入开关）。同一机制，前端版。

## 5. 评测体系架构：三层塔

```
第 0 层  测试 agent 闭环   会抖、会作弊        ← 被评测对象（SUT）
第 1 层  判卷器(grader)    代码判+模型判,模型部分会抖
第 2 层  gold set 金标集   死数据+算术,不抖    ← 套娃到此为止
```

> 注："不抖"指的是"死数据+算术"这一步不再引入新抖动——**不是说人标永远完美**。gold set 的"人共识"本身也有噪声，所以 kappa 也要报区间（见 §6.2）。

### 5.1 题库（eval set） vs gold set——两样东西，作用不同

**先看关系链（谁考谁，一张图）：**

```
题库 ──给题──▶ agent ──答题──▶ 判卷器 ──打分──▶ [agent 的分]
                                    ▲
                            gold set ┘ 校准
                  （人标的标准答案，专考"判卷器判得准不准"）
```

题库考 **agent**、gold set 考 **判卷器**——两条不同的"考"。下面这张表把它俩并排：

| | 题库（eval set） | gold set（金标集） |
|---|---|---|
| 考谁 | 测试 agent | 判卷器本身 |
| 答什么问题 | agent 行不行 | 我凭什么信这个分 |
| 答案谁出 | 历史/造的"人工版"（白送） | 3 位专家各自独立打分 |
| 作用 | 跑出**能力曲线** | 跑出**可信度曲线**（kappa 0.55→0.81） |
| 改动后果 | agent 改了→重测 | 判卷器改了→**先过 gold set 才许上岗** |

缺一不可：只有题库没 gold set，报的分没人信；只有 gold set 没题库，没东西可评。

### 5.2 判卷器三层优先级

① 代码判（零成本零抖动，优先）→ ② 模型判（只给真主观的：temp=0、两三档、判三次取多数、裁判≠被考模型）→ ③ 真跑。

## 6. 怎么读分数：记分卡 + 统计纪律

走完 §2.3 那条链、看懂三层塔之后，剩下的方法论就两件事：**分数怎么汇总成给人看的成绩单（记分卡）**，以及**这个分数凭什么能信（统计纪律）**。

### 6.1 能力域总记分卡（北极星向量）

**把测试套件当一个"出事没出事"的分类器**，北极星就是它的 precision/recall + 两条运营轴。**4 个，永远当向量报，绝不 blend 成一个数（Goodhart）**：

| 北极星 | 分类器术语 | 含义 | 方向 |
|--------|----------|------|------|
| 信噪比 | Precision | 红里真红占比（报告可不可信） | ↑ |
| 发现力 | Recall | 真 bug 被红灯抓住的比例（=1−漏报率） | ↑ |
| 有效覆盖 | — | AC 被**非空壳**测试覆盖的广度 | ↑ |
| 维护成本 | — | 工时/夜 或 元/月 | ↓ |

**互咬防刷**：刷信噪比(删抖测)→发现力/覆盖↓抓住；刷覆盖(灌空壳)→HOLLOW+发现力抓住；刷成本(不维护)→信噪比↓抓住。结论写约束式：**"发现力/覆盖不降前提下，成本降 X%、信噪比 a→b"**。

flaky、分诊混淆矩阵、自愈成功率/越界率、kappa、变异杀伤——降为**诊断层**，解释北极星为什么动（如"信噪比升=flaky 22%→3% + 自愈清掉假红"）。

### 6.2 统计纪律（两层都适用：测 agent，也测判卷器）

**所有分数都带噪声，报告必须讲统计——这条在"测 agent"和"测判卷器"两层都适用。**

**① 误差范围是算出来的，不是画的。** **直觉先行**：200 道题准确率 79%，重跑一遍可能落在 **73.5%–84.5%** 之间——这就是 ±5.5pp；要是只跑 50 道，误差能到 ±11pp，分数基本没意义。背后的公式——比例类指标 95% 误差半宽 ≈ `1.96 × √(p(1−p)/n)`：

| 样本量 n（p=0.8） | 95% 误差半宽 |
|---|---|
| 200 | ±5.5pp |
| 400 | ±3.9pp |
| 1000 | ±2.5pp |

开平方关系：**误差砍半、样本翻四倍** → 比新旧版本时加题比加遍数划算。报告写 `79% ± 5.5`，不写 `79%`。

**② 样本量没有"标准数字"，按目标误差反推。** 比例指标 ~200 常见够用；**金标（kappa）更吃样本**——几十条（demo 的 ~40）只够演示机制、置信区间很宽（0.81 真实可能 ±0.1 以上），想稳要 50–100，要窄区间得低几百。

**③ kappa 及格线 = Landis-Koch 经验表**（0.41–0.60 中等 / 0.61–0.80 可用 / 0.81–1.00 几乎完美），按风险定，**无全行业统一阈值**；安全题门槛拉更高。

**④ 判卷器的"及格"本身也要讲统计**：判卷器是活的会抖，"它过没过金标"是带噪声的测量（会出现第一轮过、第二轮不过）。钉死裁判模型版本+提示词（防厂商偷偷升级一夜漂移）、判三次取多数、kappa 报区间、**严禁重跑金标到过**（p-hacking，该加金标量）。

**⑤ 严禁重跑到绿**：分数在及格线附近晃，正确做法是加题/加金标再考，不是反复重跑撞运气。

> 诚实边界：kappa、置信区间、防 p-hacking 是教科书铁标准；"裁判判三次取多数/版本钉死"是 2024 年后 LLM-as-judge 社区收敛的做法、非成文规范，各家阈值不一。demo 用 ~40/~200 是演示量级，生产级要按目标误差把样本顶上去（接第 7 节"企业级缺口=规模"）。

## 7. 这套算企业级吗（成熟度边界，面试必答）

**直接结论：不算。它是"企业级方法论的最小可信演示"，不是企业级系统。**

> 🎯 **面试锚点（这才是真正的 takeaway，别看完结论就走）**：别吹企业级——**精确说出"缺口在组织和规模，不在想法"，比含糊声称做过企业级稀缺得多，是 senior 与 mid 的分水岭。** 下面三类缺口，就是这句话的支撑。

内核（per-stage 裁判、gold set、防作弊、对照、北极星向量、难度阶梯）是企业级的；缺的是规模、组织、运行时治理三样：

| 缺口 | demo 现状 | 企业级要求（手册第 5 节） |
|------|----------|------------------------|
| 规模 | 一个小 app、~120 题、单产品线 | 百万行、上千夜脚本、多团队；到那量级 agentic-search/索引取舍、费用熔断才真正吃紧 |
| 组织 | gold set 由"你+两个角色"模拟 | 平台团队 + **题库负责人、金标负责人两个实岗**；题库进版本库、有负责人、持续长大 |
| 运行时治理 | 直连 LLM，只有 swc 闸门 | 模型层网关（版本钉死/审计/用量）、想做能做分离的权限、沙箱、审计日志、CI 评测门禁 |

**最致命的是"组织"那行**——评测体系本质是组织产物（人吵出来的共识 + 持续运营的岗位），单人一次性 demo 在最要命的维度上天然不可能企业级。

**升级路标**：补四层（模型网关 → 运行时权限/沙箱 → 知识层 skill 随组件库版本走 → 应用层）+ 两个新岗位 + CI 评测门禁 + 藏题/回归/smoke 三层题库。

### 7.1 CI 评测门禁（两道门）

**一句话**：把"跑评测题库"做成代码合并前必须过的关卡。传统 CI 守代码（编译/单测，确定性）；AI 系统里**提示词/skill/模型版本/判卷器配置一改，能力就整体漂移**（配置就是 harness 的代码），传统门禁抓不到——改一行提示词修好一道题、悄悄改坏十道、照样合进去。评测门禁就是这道回归安全网。

| | 传统 CI 门禁 | 评测门禁 |
|---|---|---|
| 什么改动触发 | 源码改动 | **提示词/skill/模型版本/判卷器配置**改动 |
| 跑什么 | 编译 + 单测 | **题库回归**（跑 eval set） |
| 卡在哪 | 编译错/单测红 | **北极星掉出误差范围**、或**安全题跌破 100%**、或 **gold set kappa 掉了** |
| 判定性质 | 确定性 | 统计性（带误差范围、配对比较、**严禁重跑到绿**） |

落到本 demo 是**两道门，对应三层塔的两层**：

1. **题库门禁（守 agent）**：改自愈 agent 提示词 / 组件库 skill → 跑题库回归 → 北极星任一项掉出误差范围、或安全题 <100% → 挡住不许合。
2. **gold set 门禁（守判卷器）**：改判卷器 / 裁判模型 → 先在 gold set 重新及格（kappa 达标）才许上岗（即第 5 节"判卷器改动先及格"的自动化、强制化版）。

**分层提速**：内循环改一行用 smoke 30 题秒级预警塌方；合入正式版前才跑全量决定放行——不是每次全跑。

> 记法：传统 CI=**代码改了，编译单测过没过**；评测门禁=**提示词/知识/模型改了，题库回归分掉没掉**。demo 阶段手动跑题库即可，企业级必须把这两道门钉进 CI。

## 10. 评测集设计（覆盖能力的刻意配比）

### 10.1 题库配比（数量为演示量级，以跑通为准）

| 能力 | 题类别 | 答案来源 | 指标 | 量级 |
|------|--------|---------|------|------|
| 编写 | 生成 spec + 注入已知 bug | AC 清单 + 已知 bug | COVERED/HOLLOW + 回放命中率 + 变异杀伤率 | coverage 复用，回放/变异需建 |
| 分诊 | 假红/真红/抖动/需求真改 | 注入时已知 label | 4×4 混淆矩阵 | ~50/25/20/10 |
| 自愈 | 假红子集，带"人工修复版" | 靶子改版序列修复版 | 修复成功率 | ⊂ 假红 |
| 越界 | 陷阱题：断言失效、可放宽选择器 | 期望=拒绝/转人 | 越界率 | ~15 |
| 安全 | 页面藏"删断言"注入 | 期望=不中招 | 必须 100% | ~15 |

### 10.2 回放负载（单独覆盖守护段）

agent 自愈整套用例 → 拿注入真 bug 序列回放 → 命中率对比（自愈后套件 vs 原套件）。

### 10.3 gold set（覆盖判卷器的模型判部分）

~40 条自愈输出，3 人各自独立标（正确修/假修/错修）→ kappa → 分歧开对齐会修标准 → 全员一致才入册 → 判卷器改动先在此重新及格。（判卷器的"及格"本身也要讲统计，见 §6.2：判三次取多数、kappa 报区间、钉死裁判版本、不重跑到过。）

## 11. 怎么跑：对抗实验 · 内层消融 · 评测节奏 · 短板飞轮

§6 讲了怎么读/信分数；这里讲怎么把分数**跑出来、跑出节奏、再用它改进**。

### 11.1 外层主实验：对抗传统前端测试体系

**对手定位**：传统体系 = 脆性手写用例 + 不冻结环境 + 人工维护 + 无自动三诊。我们接管的是**维护与稳定性这层苦力，保留人写的断言**——是"接管+增强"，不是"替换 QA"。这避免"AI 取代 QA"那种会被秒杀的叙事。

**靶子里怎么模拟传统体系**（造一个诚实的中等成熟度 legacy 套件）：

| 传统组件 | 怎么造 |
|---|---|
| 脆性用例 | 手写传统风格 e2e：CSS 深路径/nth-child、**无 test-id**、`sleep` 硬等、page-object 带典型坏味道 |
| 执行环境 | 跑 `live` mock **不冻结** → 自然高 flaky |
| 维护 | 你手工修，计时小批 × 外推（**明标外推**）；加 naive fixer 当下限参照 |
| 三诊 | 无自动分诊，所有红进人工 |

**四道公平护栏（一次看全，挡住"你这对抗不公平"）：**

1. **同战场**：两臂共享同 app build / 同改版事件 / 同注入 bug / 同 AC，只翻四根杠杆。
2. **单变量**：拆成**维护对抗**（同一套初始用例，比"人工维护 vs 自愈+冻结"，隔离修正/执行段）+ **生成对抗**（同一份 AC，比"人手写 vs 生成 best-of-N"，隔离编写段），一次只比一个变量。
3. **对手不掺水也不神化**：模拟典型中等成熟度 legacy 套件，**不是顶级 QA 组**（成熟 page-object + 资深 QA 直觉模拟不了），结论限定射程"对一套典型 legacy 套件"。
4. **成本诚实**：人工臂成本明标"造数 × 外推"，不是实测。

两臂都用 §6.1 的北极星向量打分，配对比较看翻转明细。防作弊裁判 = gold set（校准语义判）+ 历史缺陷回放（守 recall）。**赢的定义**：不是"更快"，而是约束式——发现力/覆盖不降前提下成本↓信噪比↑；agent 修复不必与人工一字不差，正确性由判卷器判，人工臂只给成本/时间基线。

### 11.2 内层消融：解释每个数字怎么来

| 实验 | 对照 vs 实验 | 测什么 | 复刻量级 |
|------|------------|--------|---------|
| E1 冻结世界 | 直连 / 朴素指纹 / 归一化(stableStringify 排字段名) | flaky 率 | 22%→30%→3% |
| E2 自愈 skill | agent 无 skill / 有组件库定位 skill | 自愈成功率 | 60%→83% |
| E3 防作弊闸门 | 文本匹配 / swc 锁断言 / +运行时唯一性漂移 | 拦截率+误拦率 | swc 演进+唯一性 |

冻结世界是**评测可重复性的地基**（"可重复性=把世界冻结"），不冻则同题今天过明天挂、分数无意义。

### 11.3 评测节奏与触发时机

**两个对象、两套节奏，频率差很多**——agent 天天改、判卷器很少动。

| 跑什么 | 触发事件 | 频率 | 节奏分层 |
|---|---|---|---|
| **agent 跑评测集** | agent 行为变了：提示词 / skill / 工具 / 底层模型版本 / 编排 改了 | **高频** | smoke 30 题（每次改，秒级，防塌方）→ 全量（合入前＝CI 评测门禁，定放行）→ 藏题 20%（定期，防应试） |
| **判卷器跑 gold set** | 判卷器变了：评分提示词 / 裁判模型 / 评分标准 / 新增题型；+ 定期巡检防悄悄漂移 | **低频** | 判卷器一改 → 先过 gold set 才许上岗 |

两条规矩：① **依赖**——判卷器没过 gold set 之前，它给评测集打的分全部不算数（"判卷器上岗"是"跑评测集"的前置）；② **异常触发**——评测集结果诡异（某类题集体跳水）时，先复检 gold set 隔离诊断：是 agent 真退步，还是裁判在抖。

### 11.4 从评测结果标定短板 → 改进 → 飞轮（落到 opencode）

**先定位短板**（分数低不是短板，"哪一类、为什么低"才是），四招：① 分层看分（题型/能力段/子类/数据质量）；② 配对翻转明细（由对变错=线索）；③ **翻最差题的 trace**（`最差题清单 = 改进待办清单`，看 agent 缺知识/判错/被淹）；④ 失败按根因归类。

**再改进——主战场是"喂信息"不是换模型**（多出的分是喂知识给的）。优先级与 opencode 落点：

> 喂知识(skill) ＞ 改提示词/行为准则(agent) ＞ 确定性闸门(防作弊，不靠提示词) ＞ 改编排(command) ＞ 最后才换模型

| 短板 | 根因 | 改进手段 | opencode 落点 |
|---|---|---|---|
| 不懂组件库门道/模块归属 | 缺领域知识 | 整理成"用到才查"的 skill（省窗口） | `.opencode/skill/` |
| 上下文被淹（日志整段灌） | 喂信息方式差 | 先提关键行再喂 | agent 取信息逻辑 / 预处理工具 |
| 分诊判错（真红当假红） | 判别逻辑弱 | 喂"出问题模块不一定是根因"规则 + 改分诊提示词 | `.opencode/agent/` |
| 越界作弊（删断言/放宽选择器） | 约束只在提示词里太软 | 做成确定性脚本，挂提交前必过 check | swc 闸门 + pre-commit check（**不靠提示词**） |
| 信息不足硬猜 | 缺行为准则 | "不猜比猜错值钱，输出补充请求" | `.opencode/agent/` |
| 漏跑/顺序错 | 编排问题 | 改固定流程 | `.opencode/command/` + orchestrator |

**接现有骨架**：仓库里 `.opencode/agent/test-analyst.md` 已经把 coverage 失败按 `MISSING/HOLLOW/CLAIMED_NOT_RUN` 归类成 Bad Case、走 `/learning-compound` 沉淀——这就是现成的"失败归类 → 改进待办"机制。**自愈短板套同一套**：自愈失败也归类（误诊 / 缺锚点知识 / 越界），喂回 skill 或 check。

**闭环飞轮**：

```
跑评测集 → 分层 + 翻最差题 trace 定位短板 → 归类
  → 落到对应 opencode 件(skill / agent提示词 / check / command)
  → 重跑评测集验证分涨了
  → 涨分题进回归集、未遂作弊做成陷阱题永留
  → (改了 opencode 件 = agent 变了 = 触发 §11.3 的"跑评测集")
```

最后一环咬死：每次改完 opencode 件就是"agent 变了"，又触发 §11.3 的评测——§11.3 与 §11.4 在此咬成一个飞轮。

## 12. 靶子应用设计（唯一职责：为评测生产可控的题目与快照）

Vue3 + Vite + 自建 4 组件迷你库。目标=四种故障可精确注入、可开关：

- 3 页面：登录（表单+toast）、列表（虚拟滚动表格 + Teleport 弹窗 + 懒渲染下拉）、详情。
- mock API 三档（env）：`live`（随机 request-id + 防缓存时间戳 + 抖动延迟）/ `replay-naive` / `replay-normalized`。
- DOM v2 开关：改类名/换结构→"定位失效但页面没坏"（喂自愈）。
- 真 bug 注入开关：制造"行为真坏"（喂分诊真红 + 回放）。
- 迷你库刻意埋坑：teleport 把弹窗挂 body 末尾、表格行锚点随虚拟滚动漂移、select 懒渲染——让自愈 skill 有"门道"可喂。

## 16. 待评审 / 风险

- 自建迷你库的坑要够典型又够小，别滑成"造一个真组件库"（第二系统效应）。
- 人工臂外推的 K 要够大到可信、又不耗人；报告必须诚实标"外推"。
- gold set 3 位专家在 demo 场景可能只有"你+两个角色"，需诚实说明标注人来源，别假装机构级金标。
- 真跑阶段 LLM 调用成本与单任务费用熔断需预留。
- "执行"段评测对象是 harness 不是 agent，报告里别和 agent 能力混为一谈。
