这是个特别好的问题，而且你这个困惑本身就解释了为什么 Case B 读着乱——它一上来就讲"自愈
83%"，却没先把"为什么需要自愈"这个前提讲透。我把它讲清楚。

你的前提没错，但漏了一个事实

你说"自动化测试的目的是输出测试报告"——完全对，自愈不改变这一点，报告永远是产物。
但漏了一件事：测试套件是会"腐烂"的资产。

测试脚本里有大量"定位代码"——告诉浏览器"点那个 class 叫 .submit-btn 的按钮"。前端 UI
天天改版，一个类名改了、DOM 套了一层，按钮功能一点没坏，但脚本按旧地址找不到了，当晚报红。

所以"红"分两种，这是全部的关键：

┌──────────┬──────────────────────────────────┬────────────────────────────────────┐
│ 红的种类 │ 含义 │ 该怎么办 │
├──────────┼──────────────────────────────────┼────────────────────────────────────┤
│ 真红 │ 应用真的坏了/功能回归了 │ 报告它！这是测试在干正事，绝不自愈 │
├──────────┼──────────────────────────────────┼────────────────────────────────────┤
│ 假红 │ 应用没坏，只是脚本的"定位"过期了 │ 把定位地址更新一下，红消失 │
└──────────┴──────────────────────────────────┴────────────────────────────────────┘

Case B 的数据：夜里的红灯里，45%
是纯定位失效（页面没坏、元素还在）。也就是说，每天早上近一半的红灯是假警报。

没有自愈会怎样——报告会失去意义

假红不处理，两件坏事：

1. 维护成本：以前三个人每天上午就干一件事——手动把过期的定位地址改对。纯体力活。
2. 更致命的：报警疲劳。 工程师天天看到一半红灯是假的，慢慢就养成"红了？八成又是定位漂了，不看"的习惯
   ——然后某个藏在红灯堆里的真 bug 也被一起忽略了。报告一旦充满假红，就没人信了，等于废了。

打个比方：测试报告 = 烟雾报警器。 但 UI
改版像屋里积灰，灰多了报警器天天误报，住户嫌烦把电池抠了——下次真着火它也不响了。自愈 =
自动清灰，让报警器只为真火响。它从不"替你灭火"。

所以自愈的定位是

▎ 自愈不是让测试通过，更不是替代报告。它是维护——自动清掉"定位过期"这一类假红，让剩下的红灯每一个都真
▎ 的意味着"出事了"，从而保住报告的信噪比和可信度。

它自动化的是人工维护（改地址），不是判定（对错）。而且它动手前先分诊"真红还是假红"——只修假红，真红原
样报给人。

这就是为什么必须配一道"防作弊闸门"

自愈对报告的威胁恰恰最大：AI 最省事的"修复"是把那行断言删掉——红变绿了，但那是把真红伪装成假绿，正是
对报告最严重的污染。所以"只许改定位、碰断言一律作废"这条死规矩，本质是守护报告的诚信。

这反过来把我们的评测方案磨得更锋利

你这一问其实直接定义了"被评测对象到底该被考什么"。自愈 agent
的评测，核心就是考它这三件事——而这正好对应三类题：

- 真红 vs 假红 分得清吗（分诊准确率）→ 业务题
- 只修了假红、没碰真红、没偷删断言吗（不污染报告）→ 行为题 + 防作弊闸门当判卷器
- 页面藏注入让它删断言，它扛住了吗 → 安全题

所以"前端自动化测试为什么要自愈"——答案是为了让报告继续可信；而"评测体系"要回答的，正是**"这个负责维
护报告可信度的 agent，它自己可不可信"**。教具和主题在这里闭环了。
