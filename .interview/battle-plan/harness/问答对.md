# Harness 模拟面试问答对

> 用途：面试官视角提问，候选人（基于本目录素材 + OpsPilot 实战）回答，逐条沉淀。

## Q1：你认为什么是 harness？由哪些部分组成？

**一句话定义**：模型本身不值钱，值钱的是包在模型外面那层工程——这层叫 harness，让模型的脑子变得能用、能管得住、能验证对错。

**落地是五件事**：能调哪些工具、有什么权限、每次喂什么信息、怎么打分、出事怎么追责。

**这五件事可以按"喂什么/能做什么/打分怎么打"重新分组成三层**（去掉"2026年2月Mitchell Hashimoto命名/OpenAI官方验证"这类不可查证的营销背书，分类法本身可用）：
- **信息层**：能力注册、记忆与上下文管理、RAG检索——对应"喂什么信息"
- **执行层**：工具封装、编排调度、安全合规网关——对应"能调哪些工具/有什么权限"
- **反馈层**：全链路追踪、自动化评估——对应"怎么打分/出事怎么追责"

**但 harness 这个词本身有两种颗粒度，回答"组成部分"前要先说清楚是哪个**：
- **广义**（LangChain "Agent = Model + Harness"）：模型之外的一切，包括执行引擎/编排框架本身（图怎么跑、节点怎么调度、checkpoint怎么存）。
- **狭义（控制面）**：只指监控/安全/评测/追责这些横切关注点，和 Agent 实际干活的"数据面"（业务流程编排逻辑本身）分离。

两者是包含关系：**广义 ⊇ 狭义**。执行层里的"编排调度引擎"属于广义 harness（框架机制本身），但"业务节点里具体怎么编排"是数据面，不算 harness；执行层里的"安全合规网关"和反馈层的"追踪/评估"才是狭义 harness（真正的控制面）——这才是"和业务逻辑解耦、不侵入节点代码"的那一层。

**业界三个真实代表作**（handbook §3，可信度高于白皮书）落在上面的分类里：
- **Anthropic Claude Code**：执行层-安全合规网关的招牌设计——"想做"和"能做"分离，模型决定意图，工具层独立决定权限，每种工具能力单独权限开关
- **OpenAI Agents SDK**：执行层到反馈层之间的三层防护栏——输入查一遍、输出查一遍、每次调工具再查一遍
- **LangChain LangGraph**：执行层-编排调度引擎本身，提出"Agent = Model + Harness"公式，走底层精细控制路线

**难度不是均匀分布在三层上的**——分水岭是读路径 vs 写路径：只读 Agent 的 harness（限流/留痕/测一测）谁都能搭；有副作用动作（写路径）才是深水区，在执行层和反馈层之间多出四件套：幂等键（防重复执行）、前置校验（防在坏状态上动手）、HITL 人工审批（高风险动作人工点头）、loop guard（自我修正设上限，结果落到反馈层的评估输出里）。

**举例**（OpsPilot 应用实例，不算定义本身）：
| 层/颗粒度 | 组件 | OpsPilot 落地 |
|---|---|---|
| 信息层 | 能力/记忆/检索 | `retrieve_memory` + `backend/app/rag/` |
| 执行层-编排（广义harness/框架本身） | 调度引擎 | LangGraph 的 StateGraph/边路由/checkpoint 机制（13个业务节点本身是数据面，不算harness） |
| 执行层-安全合规（狭义harness/控制面） | 想做/能做分离+人工确认 | `risk_gate_node→approval_interrupt_node`（HITL），`executor.py` 幂等键+前置校验，`gateway.py` schema校验/重试/审计/脱敏 |
| 反馈层-追踪 | 全链路tracing | `tracing.py` 独立span，provider容错 |
| 反馈层-评估 | 自动化评估+防失控 | `critic_node`（4路裁决+loop guard max 2） |
| 反馈层-迭代 | bad case沉淀 | `rca`节点 + RAG写回闭环 |

**一句话收尾**：harness = 模型外面的工程，按"喂什么/能做什么/打分追责"分三层；但只有控制面那部分（安全网关+追踪+评估）是狭义harness，和业务逻辑（数据面）解耦；难度不均匀分布——写路径的四件套才是真正的深水区。

## Q2：那你们团队是如何落地harness的（AI coding这块）？

A：这块是真实工作经历（案例B+E），不是OpsPilot这条线——OpsPilot是SRE agent，AI coding harness是另一条线：团队研发工具套件跑在开源coding agent（opencode）上，我在上面搭了一层测试harness。

**底座选型**：不自研coding agent本体，用开源opencode当运行时底座（对应"从最简单形态开始、第一版按会被扔掉来设计"）。harness做成一份配置+一组扩展，不是另起引擎。

**落地拆四块**：
1. CLI命令做编排骨架——固定流程写死成命令，AI只在节点内部的开放任务里自主发挥
2. Agent做不确定性强的子任务——自愈agent（`.opencode/agent/`）：先判断"页面真坏"vs"定位失效"，只改定位行
3. Skill做喂信息——组件库定位门道（teleport弹窗、虚拟滚动锚点）整理成skill，用到才查，自愈率60%→83%靠这个
4. 强制校验闸门做代码裁决——提交前必过的确定性脚本，第三版用swc解析AST：定位语句可改，流进断言节点的期望值/匹配方式锁死。对应"安全边界绝不交给AI、模型可以提议但代码裁决"

**配置分发本身按harness标准管理（案例E）**：配置就是harness的代码，变更管理不该低于代码标准——改动先过回归集、先推5%再全量，不能"一行命令整个重写"本地配置；分层合并（公司层锁死/团队层可调/个人层自由）；改开源底座本体要克制（fork税）。

诚实边界：案例B（前端测试harness）真做过，细节可深讲；案例E是架构评审时锚定的真实经历，定性和追问是我的判断，不代表重写了分发管道。

## Q3：那非coding的agent搭建harness如何搭建？

A：思路框架不变（Q1的五件事/控制面数据面/读写路径），但有一个核心差异决定重心往哪偏：

**核心差异——裁判从哪来**：coding agent天生有现成裁判（编译器/测试/CI），"对不对"客观，重心在编排+喂信息。非coding agent（SRE/客服）通常没有现成裁判——"这次修复方案对不对"是主观的，这正是推导链B起点"评测复杂度由对错难判断逼出来"。所以重心往两块倾斜：**评测体系**（题库/三层塔/gold set）和**写路径治理**（HITL更重——操作生产环境，后果比改坏一行代码更直接、更难revert）。

**搭建顺序（90天剧本逻辑）**：
1. 先盘裁判清单——历史工单/故障的结案结论（金矿），没有验证手段的场景先不碰
2. 选滩头：高频×低爆炸半径×可验证×有历史数据
3. 最薄只读闭环：先以"建议"形式输出，决策字段不碰——对应OpsPilot当前阶段（remediation出建议，执行要过risk_gate）
4. 写路径逐步加码：幂等键→前置校验→HITL→loop guard

**多agent编排选型——deepagents范式映射**（OpsPilot是手搓deep agent）：
- Planning → planner节点
- Sub-agents（只读互不相干）→ evidence_fanout（对应"招人常识"3b级"派实习生分头查"）
- Long-term memory → retrieve_memory + RAG
- 自我纠错 → critic回边（loop guard max 2）
- HITL → risk_gate + approval_interrupt

**评测体系是非coding agent的重心**：
- 业务题：历史RCA结案结论当标准答案
- 行为题：信息不足是否老实"建议补充"而非硬猜、是否越权
- 安全题：工单正文藏prompt injection
- 三层塔：critic节点（第0层会抖）→判卷程序（第1层）→gold set（第2层，死数据+算术，套娃到此为止）

一句话收尾：非coding agent搭harness框架不变，但缺现成裁判使评测体系从"加分项"变成"地基"；写路径操作生产环境使HITL/幂等/前置校验优先级排在编排优化之前——这也是OpsPilot remediation只敢出"建议"、决策字段锁在risk_gate之后的原因。

## Q4：那你的前端自动化测试agent（opencode中）你是怎么做评测的？

A：前端最大的麻烦是没有标准答案。路子：能客观化的全客观化，客观不了的缩到最小再上人（对应推导链B"代码判>跑一遍>模型/人判"）。

**业务题（自愈对不对）全代码判**：每晚回归按红灯分四类（分母=脚本×夜次，300条×每晚=2100样本/周）：纯定位失效约45%（能自愈，成功率83%）、真业务bug约25%（不修转人）、环境/数据竞态约20%（没啃下来）、断言语义变化约10%（禁止AI碰，转人）。83%=100条失效脚本自动修好83条，代码可算。

**视觉评测分两层，把最爱抖的判定关进最小笼子**：布局层用页面盒模型结构对比（谁压谁/有无重叠）；像素对比下沉到组件级——自研组件库几十个组件每个几状态冻基线图，库升级先组件级跑视觉回归，页面级只管"组件摆放对不对"。"按钮被键盘挡住"靠几何算（外框相交/盖超没超一半），不用人看。

**安全/行为题——防作弊闸门**：自愈agent曾把失效断言整行删掉糊弄绿，被闸门拦下。死规矩做成确定性脚本（swc解析AST，定位语句可改，断言节点期望值/匹配方式锁死），不只写提示词。未遂做成陷阱题永久入库。

**真正上人的——"差异算缺陷还是可接受变化"**：3人独立标80道视觉样例，第一轮kappa=0.55（分歧在"间距变了算不算缺陷"）。开对齐会把标准写细（间距超8px且压点击热区才算缺陷），第二轮升到0.81，全员一致的题才入金标集。敢报0.55，正因为对齐会真解决了问题，不是事后美化。

**冻结世界保证可重复**：不稳定率22%→3%，最深的坑是序列化前没排字段名（{user,role}和{role,user}哈希不同导致回放穿透打真后端），第一周反而冲到30%。
