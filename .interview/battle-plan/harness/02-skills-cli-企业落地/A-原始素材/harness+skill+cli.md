## 一、层级定位：金字塔式完整落地体系

三者是从上到下的管控—业务—执行层级关系，形成 AI 应用从「系统管控」到「业务封装」再到「原子执行」的完整工业化落地体系，缺一不可：

| 层级 | 技术 | 核心定位 | 核心角色 |
|------|------|---------|---------|
| 上层管控中枢 | Harness Engineering | AI 能力的统一管控、调度、观测体系 | 大脑中枢 + 神经系统，制定规则、管控全局、指挥调度 |
| 中间业务层 | AI Skills | 标准化、可复用的业务能力封装单元 | 专业技能包，定义「做什么、做成什么样、遵循什么规则」 |
| 底层执行层 | AI CLI | 原子化的系统 / 工具操作执行单元 | 手脚，完成「具体怎么干、和系统 / 工具交互、输出结果」 |

### 通俗类比：月度财务结账

以「月度财务结账」任务为例理解三者关系：

> - **Harness** = 公司 CEO + 管理体系：拆解任务、调度对应岗位、管控流程、审批高危操作、监控全流程
> - **Skills** = 公司财务专家：定义结账的业务流程、合规规则、输出规范、异常处理逻辑
> - **CLI** = 公司财务操作员：执行具体系统操作——拉取账单、核对发票、生成报表、提交审批

## 二、端到端 9 步协同流程

以「个人周报自动化生成系统」为例，展示三者如何协同完成一次任务：

1. **需求触发**：用户输入「生成本周工作周报」→ 进入 Harness 系统
2. **需求解析**：调度引擎解析需求，匹配「周报自动化生成 Skill」，拆解为 5 个步骤
3. **Skill 加载**：动态加载 Skill 的业务规则、执行流程、输出规范，注入大模型上下文
4. **CLI 调度**：按照 Skill 流程，依次调度 CLI 命令：
   - Git CLI → 拉取本周代码提交记录
   - 飞书 CLI → 拉取本周会议记录
   - 待办 CLI → 拉取本周完成的工作项
5. **安全校验**：所有 CLI 命令经过安全网关的白名单校验、权限校验，仅开放读权限
6. **业务处理**：CLI 执行结果返回给 Skill，Skill 整合数据生成符合规范的周报初稿
7. **人工确认**：触发确认节点，将周报初稿预览给用户，确认后继续执行
8. **最终执行**：用户确认后，调度 CLI 命令将周报保存到本地、推送飞书文档
9. **全链路观测**：记录全流程日志、核心指标，评估执行效果，对 Bad Case 触发优化

## 三、互补关系：各自的能力边界

| 技术 | 单独使用的核心局限 | 与另外两者结合的核心价值 |
|------|-------------------|---------------------|
| Harness Engineering | 无具体业务与执行能力，只是管控体系，是「空架子」 | 通过 Skills 获得标准化业务能力，通过 CLI 获得可靠底层执行，让管控真正落地 |
| AI Skills | 无统一管控、调度、安全机制，零散部署容易失控；无底层执行能力 | 通过 Harness 实现统一管控、安全合规、可观测；通过 CLI 获得底层执行能力 |
| AI CLI | 无业务逻辑封装，只是原子命令；无统一安全管控，极易造成安全风险 | 通过 Harness 实现统一安全管控；通过 Skills 获得业务逻辑，组合成完整业务解决方案 |

### 行业实测数据（相同模型 + 相同任务场景）

| 指标 | 优化前 | 优化后 | 提升 |
|------|-------|-------|------|
| 任务执行成功率 | 52% | 92% | +76.9% |
| Token 成本 | — | — | 降低 78% |
| 安全风险事件 | — | — | 下降 100% |
