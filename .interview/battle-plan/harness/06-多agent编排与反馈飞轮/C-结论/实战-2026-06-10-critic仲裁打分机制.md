# 实战沉淀 · critic 仲裁与打分机制（多 Agent 证据聚合）

> 一句话：**critic 是"材料审核员"不是"方案评委"——只打证据质量分、按矛盾优先做 4 路裁决，转 2 圈摆不平就转人工。**

## 30 秒开口

我刚看这块的时候用了一个很朴素的比喻：机房里多个调度员抢机器，最后领导拍板。带着这个比喻去读 OpsPilot 代码，发现**对了一半**——

- **错的一半**：这些专科 Agent（DB / K8s / 日志 / 指标…）是**分头取证、只读**，根本不抢同一台机器。所以不存在"抢资源"的硬冲突，只剩下"结论打架"的软冲突。
- **对的一半**：确实有个"领导"节点叫 critic。但它干的事比我一开始以为的窄多了——只审"证据够不够下结论"，不评"哪个修复方案更好"。

钉死一句话：**critic 是材料审核员，不是方案评委。** 下面所有细节都挂在这条线上。

## 面试 4 连问速记

**Q1：多 Agent 结论打架怎么解？**
矛盾优先级最高，靠**退回重查**不靠投票。critic 4 路裁决：CONTRADICTION > NEED_MORE_EVIDENCE > REPLAN > PASS。
→ 追问"矛盾怎么检测的？"往下翻第 3 节，一张跨层差分表。

**Q2：critic 怎么打质量分？**
**打分与决策解耦**：会计算分、领导拿分裁决。质量 = avg(置信度) × 截断惩罚 × 覆盖度。
→ "置信度会被吹高吗？"——`min(_,0.9)` 封顶。

**Q3：这算"领导择优"吗？**
**不算。** 方案择优现在是缺口。remediation 出单方案 + risk_gate 只做过/不过。
→ "要补在哪？"——`_score_remediation()`。

**Q4：怎么防无限重排？**
loop guard 转 2 圈 → 转人工 + 压低置信度。PASS 不计数，只有重查类裁决 +1。
→ "为什么还主动降智？"——防反复横跳误导后续修复。

## 详细复盘

### 1. 这个"领导"其实是两个人

会计和领导拆在两个文件里，纯粹为了**解耦**：

- **会计**（`aggregator.py:159` `_compute_weighted_quality`）：把各专科 Agent 的置信度揉成一个 0~1 的质量分。
- **领导**（`__init__.py:1074` `critic_node`）：自己不算分，直接读上一节点算好的 `evidence_quality_score`，拿分 + 矛盾信号做裁决。

换评分公式不用动决策逻辑，反过来也一样。

### 2. 质量分怎么算：三个因子，各管各的

`质量 = avg × penalty × coverage`

- **avg（材料硬不硬）**：平均置信度。单个 Agent 的 confidence 经过两道卡口——正常出结论 `min(raw, 0.9)` 封顶，不许吹满分；降级兜底 `partial=True` 再 ×0.5；彻底挂掉 `LLM_FAILED` 直接不计入。
- **penalty（有没有被腰斩）**：ReAct 跑到步数上限被截断的，每个 ×0.9，封底 0.5。
- **coverage（人到齐没）**：`min(active/5, 1)`，预期 5 个专科到齐才满分。只来 2 个直接腰斩——"证据没收全别急着下结论"。

### 3. 矛盾检测：一张跨层差分定位表

先正名：它叫"矛盾检测"，但真正干的事是**跨层差分定位**——不是"两个 Agent 吵架要劝架"，而是看哪层异常、哪层正常，这个极性组合本身就把故障范围缩小了。

算法三步（`aggregator.py:138` `_detect_cross_agent_contradictions`，纯函数、无 I/O）：

**① 给每个专科贴标签**（`_get_conclusion_prefix`）

把每个专科 Agent 的结论压成 4 选 1：`正常 / 异常 / 部分 / 失败`。优先认专科结论文本的前缀（它们按约定写成 `异常: …` / `正常: …`），命中哪个算哪个；没写前缀就兜底——`LLM_FAILED/TIMEOUT` → 失败，`partial=True` → 部分，否则默认 → 正常。

**② 拿有向规则表去比**（`CONTRADICTION_RULES`，6 条硬编码）

每条是 `(A 层, B 层, 定位结论)`，**有方向**。命中条件只有一条：A 层异常或部分，且 B 层正常。

| A：异常/部分 | B：正常 | 定位结论 |
|---|---|---|
| logs | k8s | 应用层故障，非基础设施 |
| k8s | metrics | 可能非资源级故障 |
| k8s | deployments | 故障可能非发布导致 |
| db | k8s | DB 自身问题，非容器层 |
| db | metrics | 慢查询/锁争用等非资源瓶颈 |
| metrics | k8s | 资源压力未导致 Pod 级故障 |

**③ 命中就产出一条 `{异常层, 正常层, 定位结论}`**，汇进全局摘要。

本质上是在做**排除法**："日志炸了但 K8s 健康 → 别查基础设施，往应用层看。"一条规则就是一条领域经验。

被追问时能甩的四个特性：

- **确定性、非 LLM**：纯规则，可解释、可复现，不烧 token。
- **有方向、不对称**：表里有 `(logs, k8s)` 不意味着有 `(k8s, logs)`。极性反过来语义就变了，所以只 curated 有意义的那一向。
- **粗粒度、有耦合**：只认"异常/部分 vs 正常"的极性，不量化幅度；而且依赖各专科守 `异常:/正常:` 的书写约定，不守约定就退到默认正常。
- **"首条命中即返回"会漏报**（我专门核过这个边界）：`_get_conclusion_prefix` 按顺序扫一个专科的证据项，返回第一条带前缀证据的标签，后面的不看。所以如果某专科"首条证据正常、后面才异常"，会被整体标成正常，这条跨层矛盾就漏检了。好处是取标签够简单，代价是漏报偏保守。

**和 critic 的关系（接口先行，刻意分两步走）**：这张表的产出目前进了 `state["contradiction_signals"]` + 全局摘要，做**观测**，还没**晋升**到 critic 真正读的 `contradicting_evidence_ids`。而 critic 的 CONTRADICTION 分支读的正是后者（`__init__.py:1110`）。这一步"信号灌装"是有意留的下一刀——仲裁策略（critic 优先级 + 退两步回 planner）和数据契约（候选字段）都已就位，哪天真把规则表结果映射进候选字段，CONTRADICTION 一行不改就点亮了。

### 4. 4 路裁决 + 回退深度 + 为什么不用投票

裁决按优先级短路（`critic_node:1109+`）：

```
loop_count >= 2         → NEEDS_HUMAN      保险丝，不再裁决，候选置信度压到 ≤0.5
有矛盾候选              → CONTRADICTION    策略已就位，信号灌装后才会触发
quality < 0.4 或缺≥2类  → NEED_MORE_EVIDENCE
quality < 0.3 且证据<2   → REPLAN
兜底                    → PASS
```

裁决 → 跳转的回退深度（`builder.py:115` `_route_after_critic`）：

- NEED_MORE_EVIDENCE → 退一步回 `evidence_fanout`（计划不变，再取一轮证）
- REPLAN / CONTRADICTION → 退两步回 `planner`（重做调查计划）
- PASS → `remediation`；NEEDS_HUMAN → `rca` 收尾转人工

**为什么不用投票**：因为这帮专科 Agent 是分头取证、只读，不抢排他资源，不需要乐观锁、先到先得那套。会出问题的只有"结论打架"，退回重查就行。

## 面试问答底稿

**Q：你的多 Agent 系统里几个 Agent 结论冲突了怎么办？**

**A：** 通用做法是仲裁/投票/角色优先级，单 Agent 内部用 Reflection 对自己的输出做批判再改。回到 OpsPilot——它的专科 Agent 是分头取证、只读、互不抢资源，所以没有抢资源的硬冲突，只有结论打架的软冲突。处理上不投票：critic 做质量门控 + 矛盾检测，矛盾优先级最高、命中就退回 planner 重做调查计划；同一冲突转 2 圈没解决就转人工，同时把候选根因置信度压到 ≤0.5 防止误导。

**诚实边界（这是加分项）**：这段代码是 coding agent 生成的、我审过取舍。矛盾检测这条线是有意做的接口先行——跨层差分表已经在算了、产出观测信号了，仲裁策略和数据契约也都就位，只差把信号晋升到 critic 读的候选字段。所以 CONTRADICTION 当前不在真实链路触发，这是我清楚并且有意留的下一刀。被追问"那不就是没做完"，我答："核心策略和契约先落地、信号灌装放下一刀，仲裁逻辑就能独立验证了。要点亮就差把规则表结果映射进 `contradicting_evidence_ids` 这一件事。"

**Q：critic 是怎么打这个质量分的？**

**A：** 通用做法是置信度加权 + 覆盖度/完整度惩罚。项目里做了打分与决策解耦——会计算 `avg × penalty × coverage`，critic 只拿这个分做 4 路裁决，换公式不碰决策。置信度 `min(_, 0.9)` 封顶防自吹，降级结论 ×0.5，覆盖度按"5 个专科到齐"归一逼着证据收全。

**诚实边界**：这是只读分析得出的理解，没动过代码。评分阈值 0.4/0.3 是否合理，我还没用离线评测验证过。

**Q：那这算"领导对多个方案择优"吗？**

**A：** 不算。我比喻里最精彩的"方案评委"角色——按统一评分给多个修复方案打分、分差近就 dry-run 模拟——现在的 critic 不干这个。它只评"证据够不够下结论"，不评"哪个修复方案更好"。remediation 出单方案、risk_gate 只做通过/不通过的风险闸门。真正的方案择优是缺口，要补就在 remediation 出多方案 + `_score_remediation()`（按恢复速度/影响面/回滚成本加权），让 critic 或新 arbiter 节点变成方案评委。
