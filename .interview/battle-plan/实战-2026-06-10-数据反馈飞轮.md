## 详细复盘

### 系统全貌：四层数据飞轮

```
在线打分（人工）
    ↓ 覆盖率低
LLM-as-Judge（补全覆盖）
    ↓ 基准固定
离线 Golden Set 评测（防回退）
    ↓ 有了好经验
写回知识库（RAG 闭环）
    ↓ 下次故障检索到历史经验
更好的诊断 → 更多好的 RCA → ...
```

### 第一层：写回闭环（RAG 知识库）

- `retrieve_memory` 节点：用故障标题+描述+服务名做语义+过滤检索，把历史 RCA 作为 context 注入 planner
- `writeback_knowledge_node`：只有 `confirmed_by_human=true` 的 RCA 才写回向量库
- **关键取舍**：宁可覆盖率低，也不自动写回。早期无门控时，LLM 生成的噪声 RCA 污染知识库，导致下次检索到错误经验，诊断质量下降。加门控后稳定了。
- 核心文件：`backend/app/rag/writer.py`、`backend/app/graph/nodes/writeback_knowledge.py`

### 第二层：在线人工打分

- run 结束后 SRE 可对诊断/修复方案打分，存 DB，追踪不同服务和故障类型的质量趋势
- **现实问题**：覆盖率低（20-30%），SRE 忙时基本不打分

### 第三层：LLM-as-Judge（补覆盖率）

- 用另一个 LLM 自动给 Agent 输出打分，补足人工覆盖率
- **分维度打分**（不用总分）：根因定位准确性、修复方案可操作性、证据链完整性、风险评估合理性
  - 原因：总分会掩盖短板（比如根因对但修复方案太激进）
- **偏差校准**：Judge LLM 倾向于给长答案打高分。用有人工打分的样本做对齐，算相关性；相关性低于阈值告警，说明 Judge 打分标准漂移了，需要重调 prompt

### 第四层：离线 Golden Set 评测（防回退）

- 从历史故障中挑：①confirmed_by_human=true ②覆盖最高频故障类型（发布后5xx、数据库慢查、K8s OOMKill 等）
- **用途**：每次换模型/改核心 prompt/新增节点，跑一次离线评测，用 LLM-as-Judge 打分，看各维度有无回退
- **核心坑：Golden set 腐烂** — 服务架构变了，老 case 的参考答案就不对了
  - 解法：每条 case 打版本标签，架构变更时同步 review 受影响 case，不是一套永久不动

### 服务治理四个方向（追问备弹）

**A. 可观测性**：各层有埋点（写回成功/失败/跳过计数；Judge 超时单独监控；离线评测分数 diff 告警）。原则：反馈链路的健康本身要可观测。

**B. 数据隔离**：RAG 向量库每条文档带 service+env 字段。retrieve_memory 默认带过滤。**冷启动问题**：新服务历史经验少 → 两阶段检索：先带过滤，结果少于阈值时 fallback 到宽检索。

**C. 降级与容错**：各层是旁路设计，不在关键路径上。RAG 检索失败 → 空 memory_hits，Agent 继续跑（只是没历史加持）。Judge 失败 → 跳过打分。写回失败 → 记日志人工重试。**反馈链路的故障不能影响主流程可用性。**

**D. 权限与安全**：写回门控（confirmed_by_human）。Golden set 存 git，改动走 code review。写入收窄，读取放开。

## 面试问答底稿

**Q：你的数据反馈机制怎么建的？**

**A（两层）：**

先给全貌：「我们建了一个四层的数据飞轮，核心思路是让每次故障处置的经验能沉淀回来、让下一次更好。四层是：写回知识库、在线人工打分、LLM-as-Judge 补覆盖、离线 golden set 评测。」

然后逐层说问题和取舍：

- 写回：最核心是门控——只有人工确认的 RCA 才写回。我们早期没门控，LLM 输出的噪声 RCA 把向量库污染了，下次检索反而召回了错误经验。加了 confirmed_by_human 这道门之后才稳定。

- LLM-as-Judge：人工打分覆盖率太低（SRE 忙的时候根本不打），所以用 Judge 补。关键设计是分维度打分而不是总分，这样能暴露具体短板。Judge 本身有偏差（偏向长答案），要定期和人工打分做相关性校准。

- Golden set：换模型或改 prompt 之前必须跑一次，防止回退。最大的维护坑是 case 腐烂——架构变了老 case 就不对了，所以每条 case 要打版本标签。

收尾：「四层串起来，在线监测质量，离线保证不回退，写回让好经验沉淀。飞轮转起来后同类故障的平均诊断时间和人工干预次数都有下降。」

**诚实边界：** 这套回答框架是我和 Claude 一起梳理出来的，背后的代码（writer.py、writeback_knowledge.py、retrieve_memory.py）是 AI 生成的，我负责审查设计决策是否合理、确认各个取舍（特别是门控这个核心决策）、并补充了服务治理的维度。"我实现了"这个说法不准确，准确的是"我验证了这套机制的设计合理性，并能对其中的取舍给出判断"。
