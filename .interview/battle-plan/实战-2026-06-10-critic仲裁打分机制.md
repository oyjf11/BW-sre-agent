## 详细复盘

**背景**：从一个"多调度员抢机器"的大白话比喻出发，去验证 OpsPilot 到底实现了多 Agent 冲突调和的哪些部分。核心落点在 critic 节点。

**这个"领导"其实是两道工序，分在两个文件：**
- **会计**（`aggregator.py:159` `_compute_weighted_quality`）：把各专科 Agent 的置信度揉成一个 0~1 质量分。
- **领导**（`__init__.py:1074` `critic_node`）：本人不打分，第 1079 行直接读上一节点算好的 `evidence_quality_score`，只拿分 + 矛盾信号做裁决。→ **打分和决策解耦，换评分公式不用动决策逻辑。**

**打分公式逐项**：`质量 = avg × penalty × coverage`
- `avg` 平均置信度：材料本身有多硬。单个 Agent 的 confidence 被卡两道——LLM 正常出结论 `min(raw, 0.9)` 封顶（不许吹 100%）；降级兜底 `partial=True` 再 ×0.5；彻底挂 `LLM_FAILED` 直接 `continue` 不计入。
- `penalty` 截断惩罚：ReAct 跑到上限被腰斩（`truncated`）每个 ×0.9，封底 0.5。
- `coverage` 覆盖度：`min(active/5, 1)`，预期 5 个专科到齐才满分，只来 2 个直接腰斩——**逼系统"证据没收全别急着下结论"**。

**矛盾是另一条独立的线（不进打分公式）**：
- `CONTRADICTION_RULES`（`aggregator.py:15`）是硬编码专科对照表：A 说"异常/部分"、B 说"正常"就记一条（如 logs 异常 + k8s 正常 → 应用层故障非基础设施）。
- ⚠️ 但 critic 真正拍板用的不是这张表，而是 `critic_node:1110` 看每个根因候选的 `contradicting_evidence_ids` 是否非空。**两套矛盾信号，被追问"谁说了算"是个坑。**

**4 路裁决严格按优先级短路**（`critic_node:1109`+）：
```
第0: loop_count>=2 → NEEDS_HUMAN（保险丝，不再裁决）
第1: 有矛盾候选     → CONTRADICTION
第2: quality<0.4 或 缺≥2类 → NEED_MORE_EVIDENCE
第3: quality<0.3 且 证据<2  → REPLAN
兜底:               → PASS
```
矛盾优先级最高——材料再多，结论打架也先解决矛盾。

**裁决→跳转的"回退深度"设计**（`builder.py:115` `_route_after_critic`）：
- NEED_MORE_EVIDENCE → 退一步回 `evidence_fanout`（计划不变，再取证）
- REPLAN/CONTRADICTION → 退两步回 `planner`（重做调查计划）
- PASS → `remediation`；NEEDS_HUMAN → `rca` 收尾转人工

**关键取舍**：用"质量门控 + 回退重查"而非"多 Agent 投票/加权表决"。因为这些专科 Agent 是**分头取证再会诊**（只读、互不写共享资源），不是抢排他资源——所以不需要乐观锁/先到先得那套，真正会撞的只有"结论打架"，用退回重查解决即可。

## 面试问答底稿

**Q：你的多 Agent 系统里几个 Agent 结论冲突了怎么办？**
**A：** 分两层。
- 通用：多 Agent 冲突常见解法是仲裁/投票/角色优先级，单 Agent 内则用 Reflection（对自己输出做批判再改）。
- 但 OpsPilot 实际上：专科 Agent 是**分头取证、只读、互不抢资源**，所以没有"抢同一资源"的硬冲突，只有"结论打架"的软冲突。处理上不投票——critic 节点做质量门控 + 矛盾检测，矛盾优先级最高，命中就**退回 planner 重做调查计划**而非当场表决；同一冲突转 2 圈未决就转人工，并把候选根因置信度压到 ≤0.5 防误导。
**诚实边界：** 这套 critic/aggregator 代码是 coding agent 生成的，我这次的工作是**审查它到底实现了多 Agent 调和的哪些部分、哪些是名不副实**——我自己第一遍读未必能 5 分钟内定位到具体行号，是我带 Claude 一起过、由我判断"它是材料审核员不是方案评委"这个结论立不立得住。

**Q：critic 是怎么打这个质量分的？**
**A：**
- 通用：质量评估常用置信度加权 + 覆盖度/完整度惩罚。
- 但这个项目实际上做了**打分与决策解耦**：aggregator 当会计算 `avg×penalty×coverage`，critic 当领导只拿这个分做 4 路裁决，换评分公式不用碰决策。置信度还被代码层 `min(_,0.9)` 封顶防 Agent 自吹，降级结论 ×0.5，覆盖度按"5 个专科到齐"归一逼着证据收全。
**诚实边界：** 这是只读分析得出的理解，没有改动代码；评分阈值（0.4/0.3）是否合理我还没用离线评测验证过。

**Q：那这算"领导对多个方案择优"吗？**
**A：** 坦白说**不算**。我那个"领导按统一评分给多个方案打分、分差近就 dry-run 模拟"的比喻里最精彩的"方案评委"角色，现在的 critic 不干——它只评"证据够不够下结论"，不评"哪个修复方案更好"。remediation 出单方案、risk_gate 只做过/不过的风险闸门。**真正的方案择优是缺口**，要补就在 remediation 出多方案 + `_score_remediation()`（按恢复速度/影响面/回滚成本加权），让 critic 或新 arbiter 节点变成方案评委。
