# 实战沉淀 · critic 仲裁与打分机制（多 Agent 证据聚合）

> 🎯 一句话 kernel（背这句）：**critic 是"材料审核员"不是"方案评委"——只打证据质量分、按矛盾优先做 4 路裁决，转 2 圈摆不平就转人工。**

## 30 秒开口（面试张口第一段，顺着比喻往下捋）

我一开始用一个大白话比喻去理解这块：好比一个机房里多个调度员抢机器、最后由一个领导拍板。带着这个比喻去读 OpsPilot 的代码，发现它**对了一半**：

- **错的一半**：这些专科 Agent（DB / K8s / 日志 / 指标…）是**分头取证、只读、互不写共享资源**，根本不抢同一台机器——所以没有"抢资源"的硬冲突，只剩"结论打架"的软冲突。
- **对的一半**：确实有个"领导"节点叫 critic。但它干的事比我以为的窄——只审"证据够不够下结论"，不评"哪个修复方案更好"。

一句话钉死：**critic 是材料审核员，不是方案评委。** 下面所有细节都挂在这条线上。

## 详细复盘

### 1. 这个"领导"其实是两道工序

会计和领导分在两个文件，核心是**解耦**：

- **会计**（`aggregator.py:159` `_compute_weighted_quality`）：把各专科 Agent 的置信度揉成一个 0~1 的质量分。
- **领导**（`__init__.py:1074` `critic_node`）：自己不打分，直接读上一节点算好的 `evidence_quality_score`，只拿分 + 矛盾信号做裁决。

意义：换评分公式不用动决策逻辑，反之亦然。

### 2. 质量分怎么算：三因子，各管一件事

`质量 = avg × penalty × coverage`

- **avg（材料硬不硬）** —— 平均置信度。单个 Agent 的 confidence 被卡两道：正常出结论 `min(raw, 0.9)` 封顶（不许吹满分）；降级兜底 `partial=True` 再 ×0.5；彻底挂掉 `LLM_FAILED` 直接不计入。
- **penalty（有没有被腰斩）** —— ReAct 跑到步数上限被截断（`truncated`）的，每个 ×0.9，封底 0.5。
- **coverage（人到齐没）** —— `min(active/5, 1)`，预期 5 个专科到齐才满分；只来 2 个直接腰斩，逼系统"证据没收全别急着下结论"。

### 3. 矛盾检测怎么算的：一张"跨层差分定位"表（这块是实打实在跑的）

先把名字正过来：它叫"矛盾检测"，真正干的却是**跨层差分定位**——不是"两个 Agent 吵架要劝架"，而是"看哪一层异常、哪一层正常，这个**极性组合**本身就把故障位置缩小了"。

算法三步（`aggregator.py:138` `_detect_cross_agent_contradictions`，纯函数、无 I/O）：

**① 给每个专科贴一个粗标签**（`_get_conclusion_prefix`）
把每个专科 Agent 的结论压成 4 选 1 的健康极性：`正常 / 异常 / 部分 / 失败`。怎么压：优先认专科结论文本的前缀（它们按约定写成 `异常: …` / `正常: …`），命中哪个算哪个；没写前缀就兜底——`LLM_FAILED/TIMEOUT` → 失败，`partial=True` → 部分，否则默认 → 正常。

**② 拿一张有向规则表去比**（`CONTRADICTION_RULES`，6 条硬编码）
每条是 `(A 层, B 层, 定位结论)`，**有方向**。命中条件只有一个：**A 层 异常或部分，且 B 层 正常**。

| A：异常/部分 | B：正常 | 推出的定位结论 |
|---|---|---|
| logs | k8s | 应用层故障，非基础设施 |
| k8s | metrics | 可能非资源级故障 |
| k8s | deployments | 故障可能非发布导致 |
| db | k8s | DB 自身问题，非容器层 |
| db | metrics | 慢查询/锁争用等非资源瓶颈 |
| metrics | k8s | 资源压力未导致 Pod 级故障 |

**③ 命中就产出一条 `{异常层, 正常层, 定位结论}`**，汇进全局摘要。

本质是**排除式定位**："日志炸了但 K8s 健康 → 别查基础设施，往应用层看。"一条规则 = 一条领域经验，把"该往哪查 / 不该往哪查"硬编码下来。

被追问时能甩的三个特性：
- **确定性、非 LLM**：纯规则，可解释、可复现、不花 token。
- **有方向、不对称**：表里有 `(logs, k8s)` 不等于有 `(k8s, logs)`——极性反过来语义就变了，所以只 curated 有意义的那一向。
- **粗粒度 + 有耦合**：只认"异常/部分 vs 正常"的极性，不量化幅度；且依赖专科守 `异常:/正常:` 的书写约定，不守约定就退到默认正常。
- **"首条命中即返回"会漏报**（我专门核过的边界）：`_get_conclusion_prefix` 按顺序扫一个专科的证据项，**返回第一条带前缀证据的标签，后面的不看**。所以某专科若"首条证据正常、后面才异常"，会被整体标成正常 → 这条跨层矛盾就漏检了。换来的是取标签够简单，代价是漏报偏保守。

**和 critic 的关系（接口先行，刻意分两步走）**：这张表的产出现在进了 `state["contradiction_signals"]` + 全局摘要做**观测**，还没**晋升**到候选根因的 `contradicting_evidence_ids`——而 critic 的 CONTRADICTION 分支读的正是后者（`__init__.py:1110`）。这步"信号灌装"是**有意留的下一增量**：仲裁策略（critic 优先级 + 退两步回 planner）和数据契约（候选字段）都已就位，哪天把这张表的结果映射进候选，CONTRADICTION 一行不改就点亮。

### 4. 4 路裁决 + 回退深度 + 为什么不投票

裁决严格按优先级短路（`critic_node:1109+`）：

```
loop_count>=2          → NEEDS_HUMAN     保险丝，不再裁决，候选置信度压 ≤0.5
有矛盾候选             → CONTRADICTION   策略已就位，信号灌装后才会触发，见第 3 节
quality<0.4 或 缺≥2类  → NEED_MORE_EVIDENCE
quality<0.3 且 证据<2   → REPLAN
兜底                   → PASS
```

裁决 → 跳转的"回退深度"设计（`builder.py:115` `_route_after_critic`）：

- NEED_MORE_EVIDENCE → 退一步回 `evidence_fanout`（计划不变，再取证）
- REPLAN / CONTRADICTION → 退两步回 `planner`（重做调查计划）
- PASS → `remediation`；NEEDS_HUMAN → `rca` 收尾转人工

**关键取舍**：用"质量门控 + 回退重查"而非"多 Agent 投票 / 加权表决"。因为这些专科是分头取证、只读、不抢排他资源，不需要乐观锁 / 先到先得那套；真正会撞的只有"结论打架"，退回重查即可。

## 面试问答底稿

**Q：你的多 Agent 系统里几个 Agent 结论冲突了怎么办？**

**A（两层）：**
- 通用：多 Agent 冲突常见解法是仲裁 / 投票 / 角色优先级，单 Agent 内则用 Reflection（对自己输出做批判再改）。
- 项目：OpsPilot 的专科 Agent 是分头取证、只读、互不抢资源，所以没有抢资源的硬冲突，只有结论打架的软冲突。处理上不投票——critic 做质量门控 + 矛盾检测，设计上矛盾优先级最高、命中就退回 planner 重做调查计划；同一冲突转 2 圈未决就转人工，并把候选根因置信度压到 ≤0.5 防误导。

**诚实边界（这是加分点）：** 代码是 coding agent 生成、由我审过取舍。矛盾这条线是**有意的接口先行**：检测（那张跨层差分表）已经在算、并产出观测信号，仲裁策略和数据契约也都就位，只差把信号**晋升**到 critic 读的候选字段——所以 CONTRADICTION 当前不在真实链路触发，这是我清楚、且有意留的下一增量。被追问"那不就是没做完"，我答："核心策略和契约先落地、信号灌装留作下一刀，这样仲裁逻辑能独立验证；要点亮就差把规则表结果映射进 `contradicting_evidence_ids` 这一步。"

**Q：critic 是怎么打这个质量分的？**

**A（两层）：**
- 通用：质量评估常用置信度加权 + 覆盖度 / 完整度惩罚。
- 项目：做了**打分与决策解耦**——会计算 `avg×penalty×coverage`，critic 只拿这个分做 4 路裁决，换公式不碰决策。置信度 `min(_,0.9)` 封顶防自吹，降级结论 ×0.5，覆盖度按"5 个专科到齐"归一逼着证据收全。

**诚实边界：** 这是只读分析得出的理解，没有改动代码；评分阈值（0.4 / 0.3）是否合理，我还没用离线评测验证过。

**Q：那这算"领导对多个方案择优"吗？**

**A：** 坦白说**不算**。我比喻里最精彩的"方案评委"角色——按统一评分给多个修复方案打分、分差近就 dry-run 模拟——现在的 critic 不干。它只评"证据够不够下结论"，不评"哪个修复方案更好"。remediation 出单方案、risk_gate 只做过 / 不过的风险闸门。**真正的方案择优是缺口**，要补就在 remediation 出多方案 + `_score_remediation()`（按恢复速度 / 影响面 / 回滚成本加权），让 critic 或新 arbiter 节点变成方案评委。
