## 详细复盘

**起点**：用户问"强化学习是不是在微调模型？"——把 RL 和 fine-tuning 混为一谈。

**根因/关键转折**：

RL 和微调的根本区别不是"参数有没有更新"（都有），而是**调的是哪个网络的参数**：

```
┌────────────────┬────────────────────────────┬──────────────┐
│                │        调的参数            │   参数量级   │
├────────────────┼────────────────────────────┼──────────────┤
│ 大模型预训练   │ Transformer Q/K/V/FFN 权重 │ 7B～700B     │
│ 微调（SFT）    │ 上述权重的一小部分         │ 亿级         │
│ RL Policy 训练 │ 策略网络（决策头）的权重   │ 可以只有几万 │
│ Prompt 工程    │ 没有，导航 LLM 走不同路径  │ 0            │
└────────────────┴────────────────────────────┴──────────────┘
```

**RL Agent 的结构**（以 OpsPilot planner 为例）：

```
┌────────────────────────────────┐
│         RL Agent               │
│                                │
│  Policy Network（策略网络）     │  ← 这层在学，调它的参数
│    入参：State 特征向量         │
│    出参：各 action 的概率分布   │
│                                │
│  ┌──────────────────────────┐ │
│  │  LLM（大模型）            │ │  ← 这层通常冻住，不动
│  │  负责理解文本、压缩特征   │ │
│  └──────────────────────────┘ │
└────────────────────────────────┘
```

**Policy Network 具体入参出参**（OpsPilot planner 的假设实现）：

```python
# 入参：把 IncidentAgentState 压缩成特征向量
state = [
    0.9,   # severity（P1=1.0, P2=0.8, P3=0.5）
    1.0,   # 是否有 deployment 变更记录
    0.0,   # 是否已查过 metrics
    0.0,   # 是否已查过 logs
    0.3,   # 当前 evidence 数量 / 最大值归一化
]

# 出参：各 action 的概率分布
{
    "query_deployment": 0.72,   # ← argmax，选这个
    "query_metrics":    0.18,
    "query_logs":       0.07,
    "call_runbook":     0.03,
}
```

训练过程：执行 query_deployment → verify 节点：故障解决 → 奖励 +10 反向传播 → 下次同类状态 query_deployment 概率从 0.72 涨到 0.81。

**OpsPilot 现在为什么是纯 LLM，不是 RL**：

```
故障状态 → [LLM] → "我觉得应该先查 deployment" → 执行
```

每次都靠 LLM 现场推理，没有 Policy 参数在更新。代价：历史经验不沉淀。但这是合理的起点。

**关键取舍**：为什么现在不上 RL

| 缺点 | OpsPilot 的具体影响 |
|------|---------------------|
| 需要模拟环境 | 要先建故障注入平台，比 RL 本身还难 |
| 奖励函数难设计 | verify 节点准确率不够高，"好"的定义都还没量化 |
| 样本效率低 | 历史故障数量不够，一次处理几分钟，反馈太慢 |
| 训练不稳定 | 上线风险高 |
| 可解释性差 | SRE 需要知道"为什么这么操作"，黑盒 Policy 推广不了 |

**最现实的未来路径**（若 Phase 8 发现某类故障 LLM 选工具准确率只有 60%）：

```
LLM 出 3 个候选工具 → 小 Policy Network 做 reranking → 选最优一个
```

RL 不是替换 LLM，是在 LLM 决策上再套一层修正。

## 面试问答底稿

**Q：你们项目里有没有用强化学习？**

**A（两层）**：
- 通用层：强化学习不是微调模型——它训的是一个决策策略（Policy Network），LLM 冻住当感知器，Policy 通过与环境交互、最大化累积奖励来学策略。Policy Network 可以很小，就是一个决策头，几万个参数。
- 项目层：OpsPilot planner 节点目前是纯 LLM，每次现场推理，没有 Policy 参数在更新。不上 RL 的原因很具体：没有故障模拟环境（RL 要试错几万次，不能拿真实集群练）、奖励信号稀疏（一次故障几分钟才出结果）、verify 节点的准确率还没量化（连奖励函数都定义不清楚）。先把 Phase 8 评测跑通、把 MTTR/准确率基线立起来，那才是考虑 RL 的前提。

**Q：如果以后加 RL，是替换 LLM 还是叠加？**

**A**：最现实是叠加——LLM 负责理解非结构化的故障描述、生成候选工具列表，小 Policy Network 负责从候选里按历史经验挑最优的。纯 RL 替换 LLM 的问题是泛化差，处理不了新型故障。

**诚实边界**：强化学习的原理和适用场景我能说清楚，包括为什么 OpsPilot 现在不适合上 RL。但我没有亲手建过 Policy Network 或故障模拟环境——这是知道方向但未动手实现的领域。
