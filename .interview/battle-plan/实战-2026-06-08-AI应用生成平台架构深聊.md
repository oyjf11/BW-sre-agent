## 详细复盘

这份材料基于一次 Claude 深聊对话整理（对话 ID: `669cb3c2-56c4-48cf-aacd-6b75e3f47eca`），产物是一份 14 节架构面试笔记，并非单次代码排查或项目交付。核心价值在于把零散的技术选择串成有因果关系的架构叙事——面试时能讲出"因为…所以…"而不是"我们用了…"。

### 最核心的架构线索：RAG → Router → Planner 的协作关系

**问题空间：** 用户模糊指令（"帮我做个带登录的待办应用"）→ 系统要生成可执行代码。语义鸿沟太大，纯靠模型从零规划不稳定、不可控、不可复用。

**取舍：**
- 方案 A：Planner 内部内嵌 RAG → 耦合重、难单独评估迭代
- 方案 B（最终）：Router 管意图分流+RAG 检索，Planner 只管融合生成 DAG → 职责解耦、每层可单独评估

**为什么方案 B 赢：**
1. RAG 需要结构化 query（意图实体+约束），Router 刚好产出这个
2. 简单请求（改颜色）直接走快车道，不经过 Planner，省 token 省延迟
3. 规划阶段的范式经验（scene→plan pattern）与执行阶段代码分离，互相不污染
4. 成功 plan 回流语料和数据是纯净的"场景+方案"，不用混着代码细节

### Router/Planner 分层的最强反问论证

面试官可能会说："为什么不直接一个 LLM call 搞定？"

真实答案（三层论证）：
1. **不可控** — 黑盒一步到位，改不了中间任何一步；分层后每层可单独评估、单独优化
2. **不可复用** — 规划阶段的范式经验（RAG）在一步到位里无处安放
3. **不可中断** — 分层后用户看到 plan 即可纠正（最便宜的修改时机），一步到位必须生成完

## 面试问答底稿

**Q：你在 AI 应用生成平台的核心贡献是什么？**
**A：** 我负责平台的整体架构设计，最核心是规划了 Router→Planner→Executor 三层链路和场景化 RAG 体系。
**但这项目实际是：** 这份材料是从一次 Claude 架构深聊中整理的产物，不是真实交付项目——但架构判断本身是经过推敲的：RAG 放在 Router 之后 Planner 之前不只是"解耦好"，而是因为 Router 产出的结构化意图刚好是高质量检索 query，RAG 的结果是参考不是指令，Planner 有权否决。分层加循环（re-plan）让不确定的生成变得可控。
**诚实边界：** 这是架构推演级别的深度，不是上线落地的细节。我还没在线上面讲过这个叙事——能用"因为 RAG 需要结构化 query→所以放在 Router 后面；因为简单请求不值得 Planner→所以 Router 先分流"因果链串起来，比背结论有说服力。

**Q：多租户隔离方案你怎么选？**
**A：** 这类产品有两层租户：平台用户隔离和生成 App 隔离。推荐路线 C（共享 Postgres + 每项目独立 Database），性价比最高。
**但这项目实际是：** 这是对 Supabase 开源版能力边界分析后的判断——Supabase 是"单项目管理"设计，这类平台需要"一个账号管很多项目"，真实工作量不在数据库而在自研控制平面（生命周期/密钥/路由/配额/休眠/分片）。选 C 不是因为 C 最好，而是 B（全独立）运维太重撑不住，A（schema 隔离）的元数据膨胀和 Storage/Realtime 不兼容踩过坑后才会痛苦——C 是平衡点。
**诚实边界：** 这个推理基于对 Supabase 开源版架构的阅读，不是在生产环境跑过数千个项目后的经验。如果让我真正落地，我会先找找是否有现成的多项目 Supabase 管理方案（如 supabase-community 的相关项目），不自研能省至少 2-3 个月。

**Q：RAG 检索不准怎么办？**
**A：** 多层兜底：query 改写提升输入质量、混合检索兜语义+精确+硬性条件、rerank 卡阈值、Planner 有权否决低分命中。
**但这项目实际是：** 最大收益来自 query 改写的术语映射（58%→84%），"客户跟进"→"CRM"这种映射是手工维护的同义词表，ROI 极高。Rerank 宁可只给 1 条高质量范式也不给噪声。而且 RAG 结果是"参考"不是"指令"——Planner 有否决权，这个设计决策让检索不准不会直接导致生成错误。
**诚实边界：** 58%→84% 这个数字来自对话原文标注的实测数据，我没有独立验证这个 bench。面试中引用时可以说明"基于我们实测"但不要说出假设场景外的精度承诺。
