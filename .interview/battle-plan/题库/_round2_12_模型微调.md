# Round 2 · 模型微调 — 44 题答案

## 模型微调（44 题）

### Q1 大规模意图识别（300-500意图）如何克服上下文长度限制

**核心**：语义向量召回 + 分层级联，非把所有候选塞进 context。

- 将 300-500 个意图的 description+few-shot 离线 embedding 入库；在线 query 先向量检索 top-N（如 20）候选，再注入 context 做最终分类；层级路由分派——先粗分领域（5-10 大领域），再在领域内精分意图

> 🏷️ M3/M5 | ⭐ 了解

### Q2 如何训练 Agent 的规划(Planning)能力

**核心**：行为克隆（SFT 学规划轨迹）+ 强化学习（PPO/GRPO 环境反馈微调）。

- 构建「任务描述 → 子任务序列 + 执行结果」的训练数据，SFT 让模型学会拆解和排序；再用环境交互 RL 微调，reward 含任务完成度+步骤效率+是否卡循环

> 🏷️ M10/M11 | ⭐ 了解

### Q3 多轮对话的上下文理解与记忆连贯性

**核心**：长期记忆外部化（向量库存历史）+ 训练侧多轮会话数据 + 推理侧压缩摘要。

- 训练数据构造多轮对话样本（含指代消解、话题切换后的回顾），SFT 让模型学会维持对话状态；推理时对超长历史做阶段性压缩摘要，关键事实持久化到外部记忆

> 🏷️ M14 | ⭐ 了解

### Q4 工业界大模型主流技术范式对比（Prompt/ICL/微调/RAG/Agent）

**核心**：五者按「改动成本 × 灵活性」光谱排列——Prompt 零改快但上限低，微调高改稳但更新慢，RAG/Agent 居中。

- Prompt/ICL 改动零成本、适合快速验证；SFT 改变模型行为分布、适合风格/格式对齐；RAG 注入外部知识、适合知识密集型；Agent 编排多步推理、适合长链路任务。实际落地常混合：RAG+SFT 保证知识准确+风格一致

> 🏷️ M1/M10/M17 | ⭐ 了解

### Q5 RAG 中新文档与历史文档语义差异引发检索偏差的对策

**核心**：增量 embedding + query 改写 + 元数据过滤，防止旧分布主导检索。

- 新文档单独 embedding 追加入库，不要求与旧文档同分布；query 端做术语映射和扩展（HyDE 生成假想文档），拉近与目标文档分布的距离；元数据按时间/来源过滤，让新文档获得公平曝光

> 🏷️ M8/M9 | ⭐ 了解

### Q6 外部工具调用（Tool Use / Function Calling）技术路径

**核心**：三路径——Prompt 工程（few-shot 示例教调工具）→ 微调（SFT Function Call 格式）→ Agent 框架（ReAct 循环编排）。

- Prompt 路径最快但参数易填错，需加 schema 校验兜底；微调路径输出格式更稳定、幻觉工具名更少；Agent 框架在工具调用上加规划-执行-反思闭环。核心挑战：参数合法性、工具失败自愈、幂等安全

> 🏷️ M15 | ⭐ 了解

### Q7 Reranker 微调训练数据的系统构造

**核心**：正样本 = query 与相关 doc 对（人工标注+用户点击），负样本 = 强负（语义近但不相关/hard negative）+ 弱负（随机不相关）。

- 强负样本从粗排 top-k 中挑「被 bi-encoder 打高分但实际不相关」的文档——它们是 cross-encoder 最需要学会区分的边界 case；标注可走人工+LLM 辅助，数据来源含业务日志、开源 IR 数据集

> 🏷️ M7 | ⭐ 了解

### Q8 Prompt 设计与优化

**核心**：三板斧——角色设定 + 任务指令（做什么/不做什么）+ 输出格式。迭代靠评测集闭环。

- 每条 prompt 必须有「不做什么」约束，比「做什么」更重要；Few-shot 选例覆盖成功 case + 常见错误 + 边界 case；迭代方法：写→跑 10+ golden case→看 badcase→改→全量重跑直到指标不再涨

> 🏷️ M17 | ⭐ 了解

### Q9 缓解 LLM 幻觉的技术方法

**核心**：RAG 治「知识缺失型」幻觉（补事实）；CoT/RL 治「推理型」幻觉（纠正逻辑）。

- 具体手段：检索增强注入外部事实 + 引用溯源标注来源 + 反幻觉 prompt 约束（「不知道就说不知道」）+ 二次校验用另一 LLM 检查；RAG 门槛：检索到但内容错误比没检索到更危险

> 🏷️ M9 | ⭐ 了解

### Q10 AI Agent 记忆中向量数据库的设计

**核心**：离线存结构化摘要（非整段对话），在线混合检索 + 时间衰减 + rerank。

- Embedding 模型一般不需微调，选领域匹配的通用模型（BGE/text-embedding-v4）；chunk 粒度按「事件单元」切（一次故障/一次交互），而非固定长度；检索策略：语义 + 关键词混合，近期加权，rerank 精排取 top-n

> 🏷️ M3/M4/M14 | ⭐ 了解

### Q11 为何引入 RAG？哪些场景优于 SFT

**核心**：RAG 知识可热更新不重训、答案可溯源，适合高频变化的事实型知识；SFT 适合稳定模式对齐。

- 优势场景：① 知识更新频繁（周/日级），重训成本不可接受；② 需要精确溯源、每条结论引用出处；③ 事实性要求高（医疗/法律/金融），编造成本巨大。RAG 不能替代 SFT——前者管「说什么」，后者管「怎么说」

> 🏷️ M1/M9 | ⭐ 了解

### Q12 为何引入 RAG？相比仅 SFT 的优势场景

**核心**：同上题——RAG 在知识更新灵活性、事实准确性、可溯源性三方面优于纯 SFT。

- 知识更新：RAG 追加文档即生效，SFT 需重新训练；事实准确：RAG 检索真实文档后再生成，降低幻觉；推理能力：RAG 提供外部信息支撑推理链，使模型「站在巨人的肩膀」上推理

> 🏷️ M1 | ⭐ 了解

### Q13 意图识别结构化输出不理想的优化策略

**核心**：三层递进——Prompt 给 JSON Schema + few-shot 示例 → 约束解码/JSON mode → 后处理校验+重试。

- Prompt 层：提供 2-3 个完整输入→输出 JSON 示例，比文字描述格式有效得多；解码层：开启 JSON mode 从源头保证语法合法；微调层：用大量「输入→期望 JSON」对做 SFT，让模型内化输出格式

> 🏷️ M17 | ⭐ 了解

### Q14 Few-shot vs Zero-shot vs Fine-tuning 的选择时机

**核心**：Zero-shot 适合简单任务/first cut；Few-shot 适合中等复杂度、需示范输出格式；Fine-tuning 适合高复杂度、风格/格式一致性要求高、大量标注数据可得。

- Few-shot 优势：不改模型、快速迭代；局限：窗口限制（塞不下太多示例）、示例选择若分布不均会造成负迁移。Fine-tuning 在行为一致性上完胜，但需要训练数据和算力

> 🏷️ M17 | ⭐ 了解

### Q15 向量检索系统保证准确性的技术

**核心**：混合检索（Dense+BM25）+ 精排 rerank + 元数据过滤 + query 改写。

- 混合检索解决向量盲区（精确匹配 ID/术语），RRF 融合免调权重；BM25 抓精确词项，向量抓语义关联；两阶段召回-精排：粗排 bi-encoder 快但糙，精排 cross-encoder 准但慢，算力换精度

> 🏷️ M5/M7/M8 | ⭐ 了解

### Q16 RAG 系统提升检索精度和生成质量的手段

**核心**：检索器→混合检索+精排+元数据过滤；生成器→反幻觉 prompt+引用约束+二次校验；架构→gate 门控仅注入高置信度内容。

- 检索器：BM25+向量混合 + cross-encoder rerank + small-to-big 分块；生成器：要求标注引用来源、低置信度拒答；整体：低分命中不注入 context（Planner 否决权），避免噪声污染生成

> 🏷️ M5/M7/M9 | ⭐ 了解

### Q17 RAG 中 Embedding 模型选型的关键因素

**核心**：四要素——领域匹配（通用 vs 专用）、维度（精度 vs 成本）、语言（中/英/多语言）、MTEB 榜表现。

- 领域匹配最重要：中文场景 bge-large-zh 比 OpenAI ada 强 10-20%；维度越高越准但存储和计算成本大；核心铁律：query 和 doc 必须用同一模型，换了模型必须全量重建旧向量

> 🏷️ M3 | ⭐ 了解

### Q18 RAG 工作原理、系统架构和关键优化

**核心**：离线索引 + 在线检索生成两段式。离线：清洗→分块→embedding→向量库；在线：query 改写→混合检索→粗排→精排→拼 context→LLM 生成。

- 关键优化：chunking 用语义切分+overlap 防断义；混合检索 Dense+BM25 互补；两阶段粗排召回+精排 rerank；生成侧引用溯源+反幻觉约束

> 🏷️ M1 | ⭐ 了解

### Q19 RAG 基本原理、架构、工作流程及优势

**核心**：同上——RAG 在模型推理时将外部知识注入上下文窗口，用检索换记忆。

- 相比纯生成模型优势：知识可实时更新不重训、答案可溯源有据可查、减轻幻觉（事实型）。离线建向量索引，在线检索 top-k 相关文档拼入 prompt 再生成

> 🏷️ M1 | ⭐ 了解

### Q20 RAG 技术原理、架构及最佳实践

**核心**：同 M1 全流程展开。

- 最佳实践：chunk 走 small-to-big（小块检索、大块喂 LLM）；query 改写 ROI 最高（术语映射 58%→84%）；精排 rerank 拉大 margin 做置信度阈值；门控写回知识库防污染

> 🏷️ M1/M2/M7/M9 | ⭐ 了解

### Q21 专业领域 RAG 系统搭建完整流程

**核心**：数据准备（领域文档采集清洗）→ 向量化（领域匹配的 embedding 模型）→ 检索策略（结构切分+元数据过滤+混合检索）→ 模型集成（拼 context+反幻觉约束）→ 效果评估（分成检索和生成两层）。

- 领域特殊处理：按专业结构切分（医疗 SOAP/法律条款/API endpoint）、术语映射做 query 改写、人工标注 Golden 集做评测

> 🏷️ M1/M2/M8 | ⭐ 了解

### Q22 RAG 技术流程和工作原理

**核心**：两段式——离线把文档转成可检索向量，在线把用户问题检索相关文档后注入 LLM 生成答案。

- 离线：清洗→分块（含 overlap）→ embedding→向量库建 ANN 索引；在线：query 改写→混合检索→粗排 top-k→精排 top-n→拼 prompt→LLM 生成+引用

> 🏷️ M1 | ⭐ 了解

### Q23 特定专业领域 RAG 应用链路的系统性搭建

**核心**：同 Q21——从数据采集到评估的端到端链路，每个环节针对领域特征定制。

- 数据：领域文档采集+结构化元数据标注；检索：按领域知识结构分块+元数据硬过滤；生成：领域反幻觉 prompt+引用校验；评估：业务相关 metric（不只用通用 RAG 指标）

> 🏷️ M1/M2 | ⭐ 了解

### Q24 大模型知识库构建完整流程

**核心**：数据收集→清洗（去标签/去重/格式化）→ 分块（语义切+overlap）→ embedding 向量化 → ANN 索引构建→入库。

- 清洗是关键瓶颈：规则方法（正则去 HTML/页眉页脚）+ LLM 辅助（判断内容质量）；分块要保语义完整；embedding 选与领域匹配的模型；索引选 HNSW（查询快）或 IVF（省内存）

> 🏷️ M1/M2/M3/M4 | ⭐ 了解

### Q25 常用文本嵌入模型及对比

**核心**：开源 BGE 系（bge-m3/大/小，中文强）、E5/multilingual-e5；商业 OpenAI text-embedding-3（多尺寸）、Cohere Embed、阿里 text-embedding-v4。

- BGE-M3 支持多语言+8192 token 长文本；text-embedding-3 系列可调维度（256-3072），性价比灵活；选型核心：领域适配 > MTEB 分数 > 维度/成本

> 🏷️ M3 | ⭐ 了解

### Q26 Embedding 模型选型因素

**核心**：领域匹配（有无医疗/法律专用模型）> 语言支持（中/英/多语）> 维度与存储成本 > MTEB 指标 > 推理延迟与吞吐。

- 不同模型在领域特定任务上差异可达 20%+；高维更准但存储翻倍；实测不同模型同 query 的 top-k 召回差异显著

> 🏷️ M3 | ⭐ 了解

### Q27 RAG 中 Embedding 模型的技术指标和性能因素

**核心**：召回率（Recall@k）、MRR（平均倒数排名）、NDCG（归一化折损累积增益）、推理延迟（ms/条）、最大输入长度 token 数。

- Recall@k 看能不能把正确答案捞进候选集；MRR 看第一个正确答案排第几；NDCG 看整体排序质量；推理延迟影响在线 QPS；最大 token 决定单 chunk 上限

> 🏷️ M3 | ⭐ 了解

### Q28 RAG 检索上下文不准确或噪声的对策

**核心**：数据端→清洗+结构切分；检索端→精排 rerank+阈值过滤；后处理端→context 压缩去噪+Planner 否决权。

- 训练评估数据集：MS MARCO、BEIR、NFCorpus、自然问题（NQ）、TriviaQA；强负样本从粗排 top-k 中挑「语义近但不相关」的作为 reranker 训练数据

> 🏷️ M7/M8/M9 | ⭐ 了解

### Q29 RAG 深度与局限性分析

**核心**：深度——知识更新、可溯源、成本低；局限——检索到≠答对，推理型幻觉治不了，检索质量上限受 chunk/embedding 制约。

- 优化方向：Agentic RAG（多步检索+推理+校验循环）、多模态 RAG、知识图谱+RAG 融合；检索精度天花板后，margin 置信度是下一阶段优化重点

> 🏷️ M1/M9 | ⭐ 了解

### Q30 RAG 典型实现方式、架构和应用场景

**核心**：实现方式——向量检索式（基础）、混合检索式（+BM25）、Agentic RAG（多步推理）、多模态 RAG。

- 架构：离线索引+在线检索生成两段式，核心模块含 chunker/embedder/retriever/reranker/generator；典型场景——客服知识库、法律案例检索、医疗文献 QA、企业文档搜索引擎

> 🏷️ M1 | ⭐ 了解

### Q31 Agent 系统预训练后的调优及优秀案例

**核心**：SFT（任务规划/工具调用的指令微调）+ RL（环境反馈优化决策质量）+ 人类反馈对齐。

- 优秀案例：Devin（代码 Agent，端到端执行+环境交互）、OpenAI Deep Research（多步检索+推理+报告生成）、Anthropic Claude Code（工具调用+文件系统操作）。共同特点：环境闭环反馈+工具调用+长链推理

> 🏷️ M10/M11 | ⭐ 了解

### Q32 大模型输出结构化 JSON 的技术手段

**核心**：四层兜底——Prompt Schema + Few-shot → JSON mode / 约束解码 → 后处理校验 → 失败重试。

- Prompt 层给完整 JSON Schema + 2-3 个输入输出示例；解码层开启 JSON mode（API 参数 response_format={"type":"json_object"}）从源头保证语法合法；后处理用 jsonschema 库校验字段完整性和类型；校验失败构造错误信息重试

> 🏷️ M17 | ⭐ 了解

### Q33 限制 LLM 输出到封闭标签集合的技术方案

**核心**：三级保障——Prompt（枚举 + few-shot）→ Logit Bias（限定候选 token）→ 后处理 fallback。

- Prompt 明确列出所有允许标签 + 给出每个标签的判定标准 + 2-3 示例；Logit Bias 把非候选标签的 token 概率压到极小（-100），从生成源头限制；后处理若 LLM 仍输出越界值，用规则匹配最近标签或 fallback 默认值

> 🏷️ M17 | ⭐ 了解

### Q34 Prompt Engineering 概念与设计关键原则

**核心**：Prompt 是 LLM 的编程语言——通过角色设定、任务指令、输出格式约束模型行为。

- 关键技巧：角色设定定行为边界、Few-shot 示例教格式（2-3 个完整输入→输出）、CoT 思维链引导分步推理（适合≥3 步推理的任务）；「不做什么」比「做什么」更重要；Lost in the Middle 效应——重要规则放开头或结尾

> 🏷️ M17 | ⭐ 了解

### Q35 SFT 与 RAG 的全面比较

**核心**：SFT 改变模型行为分布（风格/格式/推理模式），RAG 注入外部知识（事实/数据/文档）。两者互补不互斥。

- SFT 优势：行为一致性高、一次训练持续生效；劣势：知识固定不可热更新、更新需重训。RAG 优势：知识实时更新、可溯源；劣势：检索质量决定上限、增加延迟。通常 RAG 做知识层 + SFT 做行为层是最佳组合

> 🏷️ M1/M9 | ⭐ 了解

### Q36 检索正确但生成不理想的原因和优化

**核心**：问题在生成段不在检索段——原因含 context 组织不佳、prompt 约束不足、模型能力局限。

- 优化：Prompt 增强（加反幻觉约束+引用要求+CoT）；context 组织（按相关性排序+去重去噪+关键信息前置）；模型微调（用高质量检索-生成对做 SFT，让模型学会基于给定 context 精确作答）

> 🏷️ M1/M17 | ⭐ 了解

### Q37 输出严格限定在封闭标签集的技术手段

**核心**：同 Q33——Prompt 枚举 + 解码约束 + 后处理三段式。

- 分类场景最稳方案：微调加分类头（模型最后一层输出 logits 只覆盖候选标签），比纯 prompt 方式准确率更高；Prompt 方式适合快速验证、动态标签集场景

> 🏷️ M17 | ⭐ 了解

### Q38 检索返回正确但生成质量差的调优

### Q39 检索模块输出正确后生成仍不准确的对策

**核心**：可能原因——context 信息过载注意力稀释、LLM「先入为主」忽略 context、prompt 未强制要求基于文档回答。

- 优化策略：截断 context 到最相关 top-3 块 + Prompt 前置「只基于以上文档回答」+ 生成后校验与文档一致性。若仍不行，考虑 SFT 微调让模型内化「遵从 context」的行为

### Q40 提升 ChatGPT 类对话模型事实准确性的方法

**核心**：四维度——模型架构（更大参数+更好预训练）→ 训练策略（SFT+RLHF 对齐）→ 推理优化（CoT+Self-Consistency）→ 外部知识（RAG+校验）。

- RAG 补事实，CoT 补推理链，Self-Consistency（跑 N 次取多数）补偶然性；三者组合比任一单独用效果都好

### Q41 提升大模型输出准确性的方案

### Q42 动态 Prompt 机制下训练数据的构建

**核心**：核心挑战——动态 prompt 导致「同一任务、不同 prompt 形态」的分布，训练数据需覆盖这种多样性。

- 样本生成：同一任务用多套 prompt 模板生成变体，每个变体对应相同的标准输出；标签定义：输出格式固定（JSON Schema 约束），内容按任务语义标注；上下文建模：保留历史对话轮次，确保模型在动态上下文中维持行为一致性

> 🏷️ M17 | ⭐ 了解

### Q43 检索增强生成（RAG）基本工作流程

**核心**：用户 query→检索系统从知识库召回相关文档→将文档与 query 拼接注入 LLM→LLM 基于文档生成含引用的回答。

- 简化五步：提问 → 向量检索相似文档 → 重排序取最相关 → 拼成上下文 → LLM 生成答案（五字口诀：「问查排拼答」）

> 🏷️ M1 | ⭐ 了解

### Q44 RAG 技术解决了大模型的哪些核心问题

**核心**：三大问题——知识截止（训练数据有终点）、幻觉（编造不存在事实）、不可溯源（答案无据可查）。

- RAG 通过检索把外部可信知识注入上下文，让 LLM「有据可依」；价值场景：金融研报基于最新财报、医疗问诊引用文献、法律咨询援引法条——这些场景不允许编造，RAG 是刚需而非锦上添花
