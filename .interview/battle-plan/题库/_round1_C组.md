## M24 · GraphRAG

**Kernel**：传统 RAG 在文档里搜片段，GraphRAG 多一步——先把文档抽成实体-关系图，检索时走图遍历而非纯向量匹配。

**展开**：

### Layer 1 — 为什么传统 RAG 不够用

向量检索本质是「找语义相似的段落」。这天然应付不了**多跳推理**类型的查询——例如「支付服务依赖的 Redis 集群最近一次故障的根因是什么？」这个问题需要跨多个文档：先定位支付服务 → 查到它依赖 Redis → 再查到 Redis 故障事件 → 最后找到根因段落。向量检索会把「支付服务 502」和「Redis 内存溢出」都召回，但无法建模「支付→依赖→Redis」这个关系链路——召回的是孤岛，不是路径。

GraphRAG 的价值就在于把文档里的实体和关系结构化：`支付服务 --[依赖]--> Redis 集群 --[发生]--> 内存溢出事件 --[根因]--> Key 过期策略错误`。检索时沿图遍历，能直接拿到完整推理链。

### Layer 2 — 怎么建、怎么查

**离线建图（成本大头）**：
1. **实体识别**：LLM/NER 模型从文档中抽取实体（服务名、资源、告警类型等）
2. **关系抽取**：LLM 判断实体间关系（依赖、导致、包含、替代等）
3. **图构建**：实体→节点，关系→有向边，挂上原文引用和置信度
4. **可选**：每个实体节点再挂一个社区摘要（Microsoft GraphRAG 的 communities）

**在线检索（查时走图）**：
1. 用户 query 先做实体识别，定位到图中的**起始实体**
2. 从起始实体出发，按关系边做多跳路径遍历（BFS/DFS + 剪枝）
3. 路径上收集到的关联事实拼成 context 注入 LLM
4. LLM 基于结构化事实链生成回答，引用来源

与纯向量 RAG 的核心差异：向量检索返回「相似片段集合」，图检索返回「事实推理链」。前者对单事实查询够用，后者对需要缝合多文档信息的复杂查询明显更强。

### Layer 3 — 坑 / 边界 / 追问预判

**追问 1：GraphRAG 成本高在哪？**
建图阶段是最大开销——每个文档都要过 LLM 做实体/关系抽取，N 个文档 × 每文档调用 2-3 次 LLM。对比传统 RAG 只要一次 embedding（便宜 100 倍以上）。所以一般在**知识量适中、关系密集、查询复杂度高**的场景才上 GraphRAG，不会拿它替代基础 RAG。

**追问 2：不用图数据库能叫 GraphRAG 吗？**
能。核心是图的查询能力，不是存储引擎。用 Neo4j 自然最好（原生图遍历优化），但用 NetworkX + 邻接表存内存也能跑，只是规模受限。如果数据量不大（万级节点），图数据库不是硬门槛。

**追问 3：和知识图谱的关系？**
GraphRAG 的「图谱」通常是**文档级、轻量、非完备**的——建图目的是辅助检索，不追求完整性和严格本体。传统知识图谱（如 Wikidata）追求本体完备、严格推理，是另一个维度的事。GraphRAG 的图和传统 KG 之间有重叠但不相等。

**项目印证**：
- 项目中未直接使用 GraphRAG。理解来自对 RAG 体系的深入研究——OpsPilot 当前用传统向量 RAG（BM25 + embedding + rerank）做证据检索，暂不需要多跳推理能力。若未来需要处理「某服务的依赖链路故障溯源」类查询，GraphRAG 的图遍历模式是比向量召回更合适的方案。

## M25 · 知识图谱增量更新

**Kernel**：新知识进来→实体识别+关系抽取→与现有图谱做实体对齐→冲突解决→增量写入。最大挑战：实体对齐不准会污染图谱。

**展开**：

### Layer 1 — 为什么增量更新是硬骨头

知识不等人。运维手册会加新告警类型、服务依赖会变、故障根因库会累积新案例。每次全量重建图谱在数据量小时可行，百万级文档后重建一次可能耗时数小时——这期间图的版本是冻结的，用户查到的是过时信息。

但增量更新不是简单的「往数据库插一条」——新信息里的实体（如「Redis 集群 A」）可能已经在图中存在，如果直接新增节点，同一个 Redis 集群就成了两个节点，后续查询走图遍历会断链。这就是**实体对齐**要解决的问题。

### Layer 2 — 四步流水线

1. **增量抽取**：新文档进来，用同样的实体/关系抽取管线产出新实体和关系
2. **实体对齐（Entity Linking）**：把新实体和图里已有实体做匹配。方法分两种：
   - 精确匹配：名称 + 类型完全一致 → 直接合并
   - 模糊匹配：名称相似但不完全相同（「Redis-prod-01」vs「redis-prod-01」）→ 用 embedding 相似度 + 属性匹配打分，超过阈值归并
3. **冲突解决**：新关系和旧关系矛盾时（比如之前记录 `A--[依赖]-->B`，新文档说 `A--[替代]-->B`），按策略裁决：
   - 时间戳优先（新信息覆盖旧信息，适合运维场景「现在依赖关系变了」）
   - 置信度优先（保留抽取置信度更高的那一条）
   - 人工审核（高影响决策标出来等人确认，类比 HITL）
4. **增量写入**：新实体→新增节点，匹配上的实体→更新属性+追加关系边，写操作做好事务/版本控制

### Layer 3 — 坑 / 边界 / 追问预判

**追问 1：实体对齐出错的影响有多大？**
是级联的。一个实体对齐错误（本该合并的两个节点分成两个），会导致走图遍历时丢失穿过这个实体的所有路径——后续查询看起来「查不到」不是因为信息不存在，而是图断了。更糟的是，如果错误归并（把两个不同的实体合成一个），会造成「虚假关联」——A 的故障被关联到完全不相关的 B 上，LLM 基于错误事实链生成答案。

**追问 2：怎么保证时效性和一致性？**
- 时效性：事件触发（新文档入库 → 自动触发增量管道）+ 定时全量校验（每周一次全量重建做一致性校验，发现漂移）
- 一致性：写入事务保证图结构不被破坏 + 版本号标记每条边和节点的更新时间，查询时可指定版本

**追问 3：和传统 RAG 的增量更新有什么区别？**
传统 RAG 增量更新简单——新文档 embed 完直接追加到向量库。GraphRAG 多出实体对齐这一步，复杂度高一个数量级。这也是 GraphRAG 运维成本高的核心原因。

**项目印证**：
- 项目中未直接使用知识图谱。但 OpsPilot 的 evidence 持久化设计中包含类似思想：每轮诊断产出的 evidence 会 append 到 `evidence_items` 列表并通过 GraphRunner 自动写入 DB，且 critic 回边可以触发重新搜集 evidence——这本质是一种「增量追加 + 冲突修正」的简化模式。关系型证据管理没有图谱复杂，但「增量写 + 版本管理」的意识是通的。

## M26 · 推理优化框架

**Kernel**：vLLM（PagedAttention 高吞吐）、TensorRT-LLM（NVIDIA 绑硬件极致优化）、TGI（易用优先）。核心优化：KV-Cache 管理 + continuous batching。三角权衡：吞吐 vs 延迟 vs 易用性。

**展开**：

### Layer 1 — 推理为什么需要专门框架

LLM 自回归推理有一个核心矛盾：每生成一个 token，都要把之前所有 token 的 Key/Value 矩阵重新算一遍。如果不缓存，每步都是 O(n²) 重算；如果缓存，KV-Cache 的显存随序列长度线性增长——1000 token 可能吃几 GB，批处理时轻松爆显存。

通用深度学习框架（PyTorch raw）不懂 KV-Cache 的显存管理，就会在废弃和新建间反复碎片化，吞吐极低。推理框架的核心工作就是**把 KV-Cache 当一等公民管理**，减少显存浪费 + 最大化 batch 利用率。

### Layer 2 — 三大框架怎么做的

| 框架 | 核心技术 | 核心优势 | 适用场景 |
|------|---------|---------|---------|
| **vLLM** | **PagedAttention**：把 KV-Cache 按 block 分页管理，像操作系统管理虚拟内存一样，消除显存碎片 | 吞吐最高，开源社区活跃，支持模型最广 | 高并发在线推理，你的自建推理首选 |
| **TensorRT-LLM** | NVIDIA 专有：算子融合 + FP8 优化 + inflight batching，针对 NVIDIA GPU 做到底层极致 | 单卡极致性能，延迟最低 | 绑 NVIDIA GPU、追求单卡极致吞吐的场景 |
| **HuggingFace TGI** | 对 HuggingFace 生态最友好，内置 continuous batching + 量化 | 上手最快，5 行代码起服务 | 原型验证、小规模部署 |

三者的共同点：都做了 **continuous batching**——不等一个 batch 里所有请求生成完才组新 batch，而是随时有请求完就补新请求进来，让 GPU 一直忙。这是吞吐提升最关键的技术。

**KV-Cache 的优化维度**（框架底层都在做的事）：
- **MQA/GQA**（Multi-Query/Grouped-Query Attention）：多个 head 共享 KV，减少 KV-Cache 体积（LLaMA 2/3 已用）
- **Flash Attention**：分块计算 attention，减少 HBM 读写量，提速 + 省显存
- **PagedAttention**：vLLM 的杀手锏，KV-Cache 分页避免了线性分配导致的碎片

### Layer 3 — 坑 / 边界 / 追问预判

**追问 1：三选一怎么选？**
- 预算有限、开源优先、GPU 型号杂 → vLLM
- 绑 NVIDIA A100/H100 且追求极限 → TensorRT-LLM
- 团队后端人手少、先跑起来再说 → TGI

**追问 2：continuous batching 为什么能提升吞吐？**
传统 static batching：凑 16 个请求一起推理，但其中一个生成了 3 token 就完了，剩下 15 个生成 200 token——那 197 步里 batch size 只有 15，GPU 算力浪费。continuous batching：第 3 步就把已完成的请求踢出 batch，塞一个新请求进来，让 batch size 始终保持满的。

**追问 3：自建推理 vs 调 API 的成本线在哪？**
粗略算：单卡 A100 月租 ~$1000，vLLM 大概能撑 50-100 QPS（7B 模型）。如果 API 费用超过月租，且你需要长期稳定的推理量，自建划算。但如果流量波动大（白天高晚上零），API 的弹性更省钱。关键变量：QPS、模型大小、SLA 要求、数据合规（某些场景必须私有部署）。

**项目印证**：
- OpsPilot 当前使用 API 模式调用 LLM（MiniMax/OpenAI/DeepSeek），未涉及自建推理框架。但理解推理优化的原理对工程决策有直接价值——例如选择模型尺寸时知道 7B vs 70B 不只是准确度差异，直接决定能否在一张消费级 GPU 上自建推理。

## M27 · 模型量化

**Kernel**：把 FP16/FP32 权重压成 INT8/INT4，模型变小、推理加速、显存省。代价：INT4 通常损失 1-3% 精度但模型缩小 4×。GPTQ/AWQ 是主流方案。

**展开**：

### Layer 1 — 为什么要量化

一个 13B 模型 FP16 存储 ≈ 26GB（13B × 2 bytes）。一张 A100 80GB 勉强能跑，但消费级 4090 24GB 跑不了。推理时模型权重 + KV-Cache + 中间激活一叠加，显存是最大瓶颈。

量化解决两件事：**装得下**（同一硬件能跑更大模型）+ **跑得快**（INT8/INT4 运算比 FP16 快 + 显存带宽压力减小）。

以 13B 为例：
- FP16：26GB 存储
- INT8：13GB（缩小 2×）
- INT4：6.5GB（缩小 4×）

### Layer 2 — 两类量化路线

**A. 权重量化（Weight-only Quantization）**——只压权重，激活仍走 FP16

代表方案 **GPTQ**：用一批校准数据跑一遍模型，逐层贪心量化权重，同时调整剩余未量化权重的值来补偿精度损失。优势：一次量化永久生效，推理时反量化开销小。弊端：不感知激活的分布特征，某些 outlier 通道量化后精度掉得厉害。

**B. 激活感知量化（Activation-aware Quantization）**——兼顾权重和激活分布

代表方案 **AWQ**：核心发现——不是所有权重同样重要。某些通道的激活值特别大（outlier channels），量化这些通道的权重损失会很大。AWQ 的做法是：识别出这些关键通道 → 把这些通道的权重乘以略微放大的缩放因子 → 再量化 → 推理时缩放因子转移到激活侧修正。本质是「把精度损失从重要通道转移到不重要通道」。

**量化粒度**：
- Per-tensor（整个张量一套 scale）→ 最粗
- Per-channel（每个通道一套 scale）→ 最细，精度最好
- Per-group（每 128 个元素一组 scale）→ 折中，GPTQ/AWQ 常用

### Layer 3 — 坑 / 边界 / 追问预判

**追问 1：INT4 精度损失 1-3% 这个数字可靠吗？**
取决于任务。语言模型生成任务（续写、闲聊）INT4 损失确实很小，但在需要精确数值推理的数学任务上可能掉 5%+。关键教训：量化后必须在你的真实任务上评测，不能只看 benchmark 论文数字。

**追问 2：量化 vs 蒸馏怎么选？**
- 量化：压同一个模型，不掉架构能力，精度损失可控——适合「我就想用这个模型但装不下」
- 蒸馏：用大模型教小模型，模型本身变了——适合「我接受换小模型但要保持效果」
- 省显存层面，量化更直接（省的是权重存储），蒸馏省的是模型本身的参数量

**追问 3：量化感知训练 (QAT) vs 训练后量化 (PTQ)？**
GPTQ/AWQ 都是 PTQ——拿训练好的模型直接量化，不需要再训。QAT 是在训练时就模拟量化效果，让模型学会在量化精度下工作——精度更高但代价大（要再训一轮）。PTQ 性价比最高，当前主流。

**追问 4：量化后的模型能微调吗？**
QLoRA 就是干这个的——把原模型量化为 INT4，LoRA 旁路用 FP16 训练。核心思想：冻结量化底座（便宜），只训旁路矩阵（轻量）。这是目前消费级 GPU 微调大模型的标准姿势。

**项目印证**：
- 项目中未直接使用模型量化部署。但理解量化的「精度-速度-内存」三角权衡对 Agent 工程决策有意义——例如 OpsPilot 选 LLM 时，知道 mini 模型（7B）量化后在一张消费级卡就能跑，是自建推理的成本门槛关键参数。
