## 项目实战（25 题）

### Q1 当一个大模型在特定任务上表现不佳时，有哪些系统性的排查和优化策略？

**核心**：主战场是「喂信息」不是「换模型」——知识(skill) ＞ 提示词 ＞ 确定性闸门 ＞ 编排 ＞ 最后才换模型。

**展开**：
- 先分层定位短板，不靠端到端总分猜——翻最差题的 trace，看 agent 是缺知识、判错、上下文被淹还是约束太软。
- 数据层：query 改写做术语映射（如「客户跟进」→「CRM」），应用生成平台实测 58%→84%，ROI 最高的单一手段。
- 架构层：加确定性闸门替代 prompt 软约束——前端测试域自愈的 swc 锁断言，改一行 prompt 修好一道、悄悄改坏十道，闸门拦住。

**项目印证**：前端测试域自愈 skill 消融实验——无组件库定位 skill 自愈率 60%，有 skill → 83%。多出的 23pp 是喂知识给的，不是换模型给的。短板飞轮：跑评测 → 分层看分 + 翻最差题 trace → 落在 opencode 件(skill/agent/check/command) → 重跑验证。

> 🏷️ M10/M19

### Q2 请反思你所参与项目的局限性或未解决的问题，分析其根本原因并提出可能的改进方向。

**核心**：OpsPilot 有三处诚实缺口，共同特点是「核心骨架和契约先焊死、能独立验证，适配/增强层留作下一刀」。

**展开**：
- 入口 NL 抽取没做——前端是结构化表单，插入点明确（intake 前加 `nl_parse` 节点，下游一行不改），是有意取舍，不是没做完。
- CONTRADICTION 仲裁分支——aggregate 的规则表在算、信号在灌，但没晋升到 critic 读的候选字段，当前那条分支不触发。改进：把 `contradiction_signals` 接入 critic 的裁决逻辑。
- 方案择优——remediation 只出单方案（当前 rollback / restart 二选一），没有「多方案按恢复速度/影响面打分+dry-run」的方案评委。

**项目印证**：三处缺口的共识——先焊死骨架 + 接口契约，适配层留作下一刀。前端测试域同样有缺口：企业级卡在组织和规模（题库负责人/金标负责人两个常设岗位），不是想法问题，这是 senior 和 mid 的分水岭。

> 🏷️ M21

### Q3 请介绍一个你认为最有代表性或最有趣的实习项目，包括项目背景、你的角色和技术实现要点。

**核心**：OpsPilot——一个用 LangGraph 编排 13 节点的 SRE 故障处置 Agent，我负责整体架构设计和关键取舍。

**展开**：
- 背景：SRE 故障处置需要可控、可解释、能中途叫人。裸 ReAct 循环面对这三个要求会失控——步骤一长上下文塞满、跑偏无法回头、高危操作无法中断。
- 我的角色：架构设计（13 节点 + 5 段路由）、critic 门控的设计取舍（打分与决策解耦、矛盾检测用规则表不用 LLM）、HITL 审批中断的自建方案（绕过 LangGraph 0.0.55 无原生 interrupt 的限制）。
- 技术要点：分解胜于强提示，0.9¹³ ≈ 0.25 vs 0.98¹³ ≈ 0.77——13 个节点各司其职，每步可验证。critic 门控是「材料审核员」不是「方案评委」。

**项目印证**：一条报障 → intake 归一化 → triage 分类 → retrieve_memory 捞历史 → planner 排调查计划 → evidence_fanout 并行 4 路只读取证 → aggregate 打质量分+测跨层矛盾 → diagnose 下根因 → critic 证据不够退回重查 → remediation 出方案 → risk_gate 风险闸门 → HITL 审批中断 → executor 幂等执行 → verify 验真 → rca 复盘归档。

> 🏷️ M21

### Q4 请分享你在学术或工作中参与的机器学习项目，重点说明是否包含特征工程、模型选择、训练调参与评估的完整流程。

**核心**：前端测试域的评测体系构建，不是训模型而是「评估模型」——但同样覆盖了数据构建、指标设计、参数校准、评测的完整周期。

**展开**：
- 特征工程：不是传统 ML 特征，而是「冻结世界」的录制回放 + 指纹归一化——请求指纹排字段名（`stableStringify`），把 `{user,role}` 和 `{role,user}` 归一为同一哈希，否则回放全穿透真后端。
- 模型选择：按「客观程度配裁判」——代码判（沙箱重跑 + swc AST 锁断言 + 选择器唯一性）→ 模型判（修复语义合理性）→ gold set 校准判卷器。不是比谁分高，是比谁可信。
- 调参与评估：E1 消融实验——直连 22% flaky → 朴素指纹 30% → 归一化 3%。统计纪律：200 题 79% ± 5.5，误差砍半样本翻四倍。

**项目印证**：整条评测链——题库考 agent → 判卷器打分 → gold set 校准判卷器（kappa ≥ 0.61 上岗）。这本质是把 ML 的特征/模型/评估三件套映射到 Agent 评测域。

### Q5 请介绍一个你参与过的深度学习项目，包括项目目标、所用模型架构、数据处理流程以及最终成果。

**核心**：OpsPilot 的 RAG 两阶段检索——目标不是训检索模型，而是把 LLM 嵌入确定性编排中，让检索 + 生成形成可控链。

**展开**：
- 目标：SRE 故障处置中，新故障进来先查「以前遇到过没」，用 RAG 检索相似历史案例注入上下文，降低 LLM 幻觉。
- 模型架构：两阶段——粗排（向量检索做 embedding 召回 top-20）→ rerank（cross-encoder 精排 top-5）。对 embedding 模型不做 fine-tune，而是用 query 改写（术语映射）提升输入质量，最大单一收益 58%→84%。
- 数据处理：把历史故障的「工单特征+证据指纹+最终根因+处置方案」embedding 入库，不是存整段对话，存结构化摘要。门控写回——只有 `confirmed_by_human=true` 的 RCA 才进向量库。
- 成果：命中率打满天花板但 margin 翻 2.4 倍（裸向量 0.21 → rerank 0.51），能设 0.4 置信度阈值做拒答。

**项目印证**：OpsPilot `retrieve_memory` 节点走两阶段检索，命中率看「召回对不对」，margin 看「敢不敢下结论」。

> 🏷️ M18/M21

### Q6 在当前技术水平下，AI智能体的多种能力中，哪一项最有可能率先实现大规模商业落地？

**核心**：Coding Agent 的「维护与自愈」——因为它的「对」最客观（沙箱重跑绿了就是绿了），不需要复杂的 gold set 校准就能验证。

**展开**：
- 技术成熟度：前端测试域自愈——Playwright 真跑 + 截图取证，浏览器截图不会撒谎（证据高于断言），这是 coding 类能力天然的确证优势。
- 工程成本：自愈只改定位不改断言，死规矩靠 swc AST 闸门（代码判，零 token 成本），不需要 LLM 判模型判就能验 80% 的正确性——判卷成本极低。
- 市场需求：任何前端团队都有「改版 → 脚本红一片 → 人工改 → 报警疲劳」的死循环，自愈直接拆这个循环。

**项目印证**：前端测试域从 2.2 天 → 4 小时的回归周期，10+ 团队试点。自愈 = 自动清灰，让报警器只为真火响——这个叙事对业务领导秒懂。

> 🏷️ M22

### Q7 在当前技术条件下，你认为AI Agent的哪一个具体应用场景最有可能率先实现规模化商业落地？

**核心**：前端自动化测试的「自愈」——切入点窄、判据客观、不改业务逻辑（纯维护，安全边界清晰）。

**展开**：
- 用户需求强度：一份烟雾报警器一样重要的测试报告，一半红灯是误报 → 工程师抠电池 → 真 bug 漏报。自愈消灭假红，报告信噪比从 50% 拉到 95%+。
- 成本效益：传统人工工期 2.2 天/改版周期 → 4 小时。自愈不替代 QA 的判断，只接管「改定位」这种纯体力维护——不抢饭碗，安全叙事。
- 技术成熟度：编码能力 + GUI 操作 + 确定性验证（沙箱重跑）三合一，但最关键是「验证便宜」——代码判就够了，不用 gold set。

**项目印证**：自愈 skill 消融 60→83%、冻结世界消融 22%→3%。四段闭环：编写 → 执行 → 修正(自愈) → 守护(回放回归)，每段有独立裁判。

### Q8 在技术面试的第二轮和第三轮中，面试官通常会针对候选人的实习项目深入追问哪些类型的技术实现细节？

**核心**：追问一定是「你的判断为什么对」而非「用了什么技术」——选 LangGraph 不选 ReAct、矛盾检测用规则表不用 LLM、自建 checkpoint 不用原生。

**展开**：
- 架构取舍类：「OpsPilot 为什么 13 个节点不嫌多？」→ 取证和处置间硬插 critic、处置里硬插审批、执行后硬插验真——这些「多余」节点本质都是闸门，拿编排复杂度换不闯祸。
- 数据驱动判断类：「critic 矛盾检测为什么不用 LLM？」→ 跑了 28 组 mock 工单对比，LLM 裁判不一致率 39%、跨层传播准确率 53%，数据不支持上线，所以选确定性规则表。
- 自研 vs 开箱类：「checkpoint 怎么做的？」→ LangGraph 0.0.55 没原生 interrupt，审批中断是自建的——compile 时 `checkpointer=None`，自己逐节点落 `IncidentCheckpoint` 表，resume 从自建 checkpoint 读回 state。

**项目印证**：三个判断全是 OpsPilot 的真实设计取舍。面试官要追的是 tradeoff 背后的数据和推理链。

> 🏷️ M21

### Q9 请选择一个你参与过的技术项目，用非技术人员（如HR）能够理解的语言，清楚地介绍项目。

**核心**：前端测试域——把「AI 写完代码后人工跑一遍验收」变成「AI 自己跑、自己看截图、自己判断对错」。

**展开**：
- 背景：前端网页三天两头发版改样式，测试脚本要人每天手工改定位——纯体力。一半红灯是误报，工程师渐渐不当回事，藏里面的真 bug 一起被忽略。
- 挑战：AI 不能改「什么算对」（断言），只能改「怎么找到这个按钮」（定位）。但 AI 有时候分不清自己改的是定位还是断言——如果把「按钮必须是红色」偷偷改成「蓝色也行」，那就是篡改考题答案。
- 方案：做一个「只改定位、不碰断言」的自动维护员。改完先在沙箱重跑一遍——跑绿了才算修好；同时用一个代码锁（swc AST 闸门）保证它的修改没动断言那一行。

**项目印证**：2.2 天的手工回归工作 → 4 小时自动完成。比喻：烟雾报警器积灰误报，我们不叫人每天去擦——我们装一个自动清灰的小装置，让报警器只为真火响。

> 🏷️ M22

### Q10 请结合你在实习或项目中的实际经历，描述一个你遇到的具体技术或协作难题。

**核心**：OpsPilot 的 HITL 审批中断——LangGraph 0.0.55 没有原生 `interrupt()`，人审批完怎么从断点续跑？

**展开**：
- 难题定位：LangGraph 0.0.55 只有基础 `StateGraph`，没有 `interrupt()`/`Command`/`PostgresSaver`/`Send`。但 SRE 高危操作必须「人不点头不动手」——这要求图能在 `approval_interrupt` 节点挂起、等人审批完再续跑。
- 解决方案：自建 checkpoint 机制。compile 时 `checkpointer=None`，自己逐节点把 `state_json` 落 `IncidentCheckpoint` 表。审批时 `approval_interrupt → END` 图挂起。人点头后 `POST /runs/{id}/resume`，从自建 checkpoint 读回 state、设 `_resume_from_node` 重新 astream。
- 推动落地：白名单路由（只允许从 approval 下一跳走 executor）+ executor 幂等防重跑（`idem_key = sha256(run_id+action_type+params)`）。

**项目印证**：这个自建方案有真实的约束——Pydantic 对象从 checkpoint 回来变 dict，访问属性必须 `hasattr(x, 'f') else x.get('f')`，否则直接抛 AttributeError。

> 🏷️ M21

### Q11 在数据预处理与信息提取任务中，如何权衡使用基于规则的方法与基于大语言模型的自动化处理？

**核心**：规则先兜底、LLM 再补——便宜、确定、可解释的先上，LLM 只兜长尾。不是选一个，是分层。

**展开**：
- 规则适用场景：成本敏感 + 边界清晰 + 不能错。OpsPilot triage 先用关键词规则粗分故障类型（deployment_regression/resource_exhaustion/dependency_failure），命中即出结果，不烧 token。
- LLM 适用场景：长尾 + 模糊 + 需要语义理解。规则没命中才掉 LLM。critic 的矛盾检测更是——用 12 条有向规则表零 token 成本，LLM 裁判不一致率 39% 不能用。
- 成本考量：LLM 单次调用几美分看似便宜，但诊断类 agent 一个工单可能调 10+ 次 LLM，累积成本不可忽略。规则前置能砍掉 60%+ 的 LLM 调用。

**项目印证**：OpsPilot triage → critic 矛盾检测 + 前端测试域 swc 闸门——规则在「不能错」和「成本敏感」两处做硬兜底，LLM 在「语义模糊」处补。价值观一致：「约束要在执行前生效，不能只靠 prompt」。

> 🏷️ M21/M22

### Q12 在面对结构化决策任务时，相比传统的规则引擎，大模型有哪些优势和劣势？

**核心**：LLM 的优势在「语义模糊的分类和生成」，劣势在「一致性差、成本高、不可审计」。OpsPilot 的 critic 矛盾检测用规则表不用 LLM 就是最佳注脚。

**展开**：
- LLM 优势：能处理规则覆盖不到的长尾 case、能生成自然语言解释、对模糊输入有鲁棒性。triage 阶段规则命中不了的长尾故障类型，LLM 能根据工单描述合理推断。
- LLM 劣势（致命）：不一致——同一输入判三次可能三个结论（我们实测 LLM 裁判不一致率 39%）；不可审计——黑盒概率输出，没法说「因为规则 3 所以结论是 X」；成本——每个判断都要烧 token。
- 选型原则：确定性高、边界清晰、错了代价大 → 规则引擎（critic 矛盾检测、前端测试 swc 断言锁）；语义模糊、长尾、需要推理 → LLM（diagnose 根因推断、planner 调查计划）。

**项目印证**：critic 矛盾检测——LLM 裁判准确率 71%、跨层传播掉到 53%、不一致率 39%，12 条有向规则表零 token 成本零抖动，数据不支持上 LLM。

> 🏷️ M21

### Q13 在构建机器学习模型时，验证集的类别分布应如何设计？应与训练集、测试集保持怎样的关系？

**核心**：验证集和测试集都应保持真实分布（stratified），但与训练集的关键区别在于「不被模型看到、不被调参泄露」。

**展开**：
- 分布一致原则：验证集和测试集的类别比例应反映真实生产环境的频率分布——如果生产中 P1 故障只占 5%，验证集里 P1 占 50% 就是自欺欺人。前端测试域题库按真实改版频率配比（假红多、真红少、注入攻击少）。
- 验证集 vs 测试集：验证集用于调 prompt/阈值/rerank 参数（可以看），测试集用于最终报告（绝对不能看）。一旦用验证集做决策，它就从「验证集」变成了「训练集」——必须再分出一块新的 blind test set。
- 对抗样本：不仅要有分布一致的题，还要刻意埋陷阱——前端测试域题库含 ~15 题安全题（页面藏指令注入），必须 100% 不中招，否则不上线。

**项目印证**：前端测试域题库配比——分诊 50 假红/25 真红/20 抖动/10 需求真改 + 越界陷阱 15+ 安全题 15，覆盖能力的刻意配比而非随机抽样的自然配比。

> 🏷️ M19/M22

### Q14 在当前技术背景下，Agent的多种能力如coding、GUI、Search等正在快速发展。请分析这三种能力中，哪一种最有可能在短期内实现大规模商业落地。

**核心**：Coding 能力——因为验证成本最低（代码跑绿就是绿）、错误代价可控（沙箱内跑、不碰生产）、产品边界清晰（补全/自愈/生成，不是自主决策）。

**展开**：
- Coding vs GUI：GUI 操作需要理解像素级别的变化，验证需要视觉 diff——像素级的「对」天然主观，判卷成本高。Coding 的「对」是客观的：编译过没过、测试绿不绿、AST 有没有被非法修改。
- Coding vs Search：Search 的「对」含大量主观成分——「返回的这 10 条文档够不够好」没有唯一的 ground truth。RAG 评估需要 LLM-as-Judge + gold set 双层校准，成本高。
- 应用场景广度：Coding 覆盖补全（GitHub Copilot 已验证）、自愈（前端测试域）、生成（应用生成平台）、CR（代码评审），每个子场景的验证手段都便宜。

**项目印证**：前端测试域自愈——验证只需沙箱重跑（代码判），swc AST 锁断言（代码判），选择器唯一性（代码判）。三层验证全部零 token 成本、零抖动，这是 coding 类能力独有的规模优势。

> 🏷️ M22/M23

### Q15 如何对纯图像扫描生成的PDF文件进行内容解析？请说明涉及的关键技术流程。

**核心**：OCR 识别 → 版面分析（表格/段落/标题分离）→ 结构化后处理（去噪/纠错/语义合并），关键难点在版面还原而非文字识别。

**展开**：
- OCR 层：用 PaddleOCR/Tesseract 逐行识别，产出行级文本 + bounding box。瓶颈在不规则字体和低分辨率扫描，预处理（去底色、二值化、倾斜校正）比换模型更见效。
- 版面分析：PDF 的挑战是排版信息（表格、多栏、页眉页脚）在扫描件里全部丢失。用 LayoutLM 或规则（横纵投影）做物理版面分割，把「一页三栏」拆成三个阅读流。
- 后处理：合并跨行断句、按阅读顺序重排（不是 y 坐标排序就完事——右栏比左栏高一行的情况必须用版面信息纠正）。

**项目印证**：虽然不是我的核心项目域，但应用生成平台的 RAG 检索——如果知识源包含扫描 PDF，OCR + 版面分析就是检索质量的前置瓶颈。和 query 改写一样，预处理 ROI 远高于换模型。

> 🏷️ M23

### Q16 智能体（Agent）的各项能力（如coding、search、memory、planning等）中，哪一项最有可能率先实现规模化商业落地？

**核心**：Coding（代码自愈/生成）+ 确定性验证——因为验证闭环最便宜，不需要 gold set 就能判对错。

**展开**：
- Memory/Planning 的验证成本高：Memory 检索质量需要 LLM-as-Judge 判 faithfulness + gold set 校准 kappa——和前端测试域自愈的「沙箱跑绿就判对」比，判卷成本差一个数量级。
- Coding 的验证天然便宜：编译过不过、测试绿不绿、AST 锁断言有没被改——三层代码判，零 token 成本、结果确定、不抖。coding 是唯一「对的定义基本客观」的 Agent 能力域。
- 应用生成平台的验证是「生成→跑→截图 diff」的 pipeline，比纯文本 coding 多一层 GUI 不确定性，但核心验证（编译 + 功能跑绿）仍然是便宜的。

**项目印证**：前端测试域三层判卷器优先级——① 代码判（零成本零抖动，优先）→ ② 模型判（只给真主观的：temp=0、判三次取多数）→ ③ 真跑。能代码判的判到底，主观残留才上模型判 + gold set。

### Q17 在众多智能Agent应用场景中，你认为哪一个方向最有可能在未来2-3年内实现规模化商业落地？

**核心**：前端自动化测试的「自愈 + 生成」——市场疼（每个前端团队每天在手工改定位）、技术成熟（验证便宜）、安全可接受（只改定位，不碰业务逻辑）。

**展开**：
- 市场刚需：「改版→脚本红一片→人工改→报警疲劳→红灯失去意义」是每个有前端测试团队的通用痛点，不分行业。
- 技术成熟度：Playwright 生态成熟 + LLM coding 能力足够改定位 + swc/ast 闸门能锁住「不碰断言」。三件套齐了，缺的是产品化集成。
- 成本效益：传统回归 2.2 天 → 4 小时，ROI 肉眼可见。2-3 年内，自愈会成为前端测试工具链的标准能力（就像 CI 里的自动重试），而不是「AI 功能」。
- 对比其他场景：客服/个人助理的「对」太主观、安全边界太宽、验证成本太高；企业流程自动化需要对接太多异构系统。

**项目印证**：四段闭环——编写(生成用例) → 执行(跑) → 修正(分诊/自愈) → 守护(回归发现力)。自愈只是「修正」一环，但它是整个链条里「唯一可以全自动」的环节，因为只改定位不改断言。

### Q18 请描述你在项目中所进行的数据预处理和清洗工作，包括数据格式转换、去噪、增强、采样等关键步骤及其目的。

**核心**：前端测试域的「冻结世界」——把不可复现的测试执行变成可重复的科学实验，最关键的预处理是请求指纹归一化。

**展开**：
- 数据格式转换：录制浏览器请求/响应，从 Chrome DevTools Protocol (CDP) 裸事件转换成结构化 JSON。关键不是格式，是「同一条请求能不能两次录制后识别为同一个」。
- 去噪与归一化：请求指纹排字段名（`stableStringify`）——`{user,role}` 和 `{role,user}` 归一为同一哈希。不加这一步，字段顺序差异会让回放全穿透真后端（flaky 率飙到 30%+）。
- 采样策略：不是录全部请求，按路由模式去重采样——`/api/user/123` 和 `/api/user/456` 是同一模式，只保留一个模板。
- 目的：不用归一化，同题今天过明天挂，分数无意义——冻结世界是评测可重复性的地基。

**项目印证**：E1 消融实验——直连 22% flaky → 朴素指纹（按 body 哈希）30%（更糟，因为 timestamp 每次不同）→ 归一化 stableStringify 排字段名 3%。

### Q19 请描述你所参与实习项目的实际业务场景及其核心目标，该项目解决了什么样的实际问题？

**核心**：OpsPilot 解决 SRE 值班的「告警风暴 + 排查信息过载」——一条告警触发，值班人要切 4 个平台查日志/指标/K8s/部署历史，20 分钟定位根因。

**展开**：
- 业务场景：生产环境一条 P2 支付服务 5xx 告警触发。传统流程——人工切 ELK 翻日志 → 切 Grafana 查指标 → 切 kubectl 看 Pod → 切 Jenkins 查最近部署 → 翻开 runbook 找同类案例 → 综合判断 → 写复盘。20-30 分钟，夜里值班更慢。
- 核心目标：把「人工串行查 4 个平台」变成「Agent 并行取证 → 自动聚合 → 质量门控 → 出诊断结论」。不是替代人的决策，是替代人的信息收集和初步推理。
- 解决的实际问题：① 证据收集从串行 20min → 并行 <2min；② 证据完整性从「靠值班人经验」→「plan+fanout 系统化」；③ 高危操作有审批中断，不会「半夜自动重启生产服务」。

**项目印证**：13 节点设计中，evidence_fanout 并行 4 路只读取证是整个架构的心脏——查日志/指标/K8s/部署四路并发，asyncio.gather，每路独立超时、失败不串扰。

### Q20 面对大规模业务场景下的原始数据，如何设计自动化流程进行高质量的数据清洗？

**核心**：和 Q18 同一原理但侧重规模——前端测试域的录制回放指纹归一化是「一次录制」的清洗，规模化需要自动化重放 + 差异检测 + 增量更新。

**展开**：
- 去重：请求指纹归一化（stableStringify 排字段名 → 同样路径+结构 = 同一指纹），不是按内容哈希（timestamp 每次不同会被当不同记录）。
- 异常值处理：检测响应体剧烈变化（如列表从 10 条变 10000 条），标记为「可能数据漂移」而非「应用回归」，触发人工确认而非自动判定。
- 格式标准化：所有 HTTP 请求统一为 har 格式 → 内部标准化协议（headers 去动态字段如 authorization/cookie/nonce）→ 生成稳定指纹。
- 缺失值填充：mock API 三档——live（真实后端，高 flaky）/ replay-naive / replay-normalized（冻结回放，低 flaky）。

**项目印证**：前端测试域冻结世界——关键不是一次清洗，是「每次改版后自动重录 + 指纹比对」，把数据清洗从一次性工程变成持续运营，否则题库会腐烂。

### Q21 在大模型项目开发过程中，常见的系统或模型缺陷包括过拟合、推理延迟高、输出不一致等。请列举几类典型缺陷，并阐述相应的解决策略。

**核心**：三个最致命的——不一致（今天判对明天判错）、幻觉（引用不存在的日志）、prompt 过拟合（题库 92% 但用户一句奇怪输入掉到 60%）。

**展开**：
- 不一致（比延迟更致命）：LLM 判卷器今天判对明天判错——解法：钉死裁判模型版本 + 提示词（禁自动最新版）、判三次取多数、kappa 报区间不报裸数、严禁重跑金标到过（p-hacking）。OpsPilot critic 的矛盾检测直接绕过 LLM 用规则表——根除不一致。
- 延迟高：Agent 走 8 步、每步调 LLM → 总延迟 30s+。解法：简单请求走规则快车道（triage 规则前置）；并行取证（fanout 4 路 asyncio.gather）；small-to-big 分块减少上下文注入量。
- Prompt 过拟合：prompt 在 50 道 case 上 92%，但用户一段奇怪的输入当场打回 60%。解法：题库按真实分布采样（含脏数据、边界 case、对抗性输入），不能只用「漂亮」数据测；RAG 召回替代 few-shot，动态选案例而非固定示例。

**项目印证**：应用生成平台不用 few-shot 选 RAG——few-shot 固定几个示例塞 prompt，边界 case 不在示例里就漂移。RAG 从 800 条 → 6000+ chunk 里动态召回，案例随时间增长，边界 case 能从更大语料库命中。

> 🏷️ M19/M23

### Q22 在构建大模型训练或微调数据集时，常见的数据来源有哪些？如何评估不同来源数据的质量与合规性？

**核心**：三个来源——生产日志（真实但脏）、专家造题（干净但少）、LLM 生成（量大但需校准）。质量用 gold set 守门，合规靠脱敏 + 权限隔离。

**展开**：
- 生产日志：OpsPilot 的真实工单数据——含工单描述、证据、诊断结论、处置动作、结果。优势：完全反映真实分布；劣势：有 PII（IP/手机号/用户名），须脱敏后才入库。
- 专家造题：前端测试域题库——手写改版序列 + 已知 bug 注入。优势：答案确切、可控；劣势：慢、贵、覆盖窄。
- LLM 生成（LLM-as-Judge 补覆盖）：在线人工打分覆盖率低 → 用 LLM 批量生产 QA 对。优势：量大、覆盖广；劣势：可能编造幻觉数据。
- 质量评估：gold set 版本化防回退——每条 case 打版本标签，架构变更时同步 review。合规：数据脱敏（正则 + LLM 扫两遍）+ RBAC 权限只读 + 审计日志。

**项目印证**：OpsPilot 数据飞轮——门控写回保质量（只有 `confirmed_by_human=true` 的 RCA 才进向量库）→ LLM-as-Judge 补覆盖 → Golden set 防回退 → 旁路不阻主流程。

### Q23 请阐述大语言模型在典型应用场景中的价值体现，并分析其技术意义与商业意义。

**核心**：技术意义 = 把「需要人读文档 + 查多个系统 + 综合判断」的认知劳动自动化；商业意义 = 降低瓶颈人力的排班压力 + 缩短 MTTR。

**展开**：
- 智能客服（类比）：LLM 替代的不是客服，是「客服翻知识库找答案」那一步——把检索 + 阅读理解自动化。但客服场景的「对」太主观，需要人在 loop。
- 代码生成（前端测试域）：LLM 替代的不是 QA，是「QA 手工写测试脚本 + 每天手工改定位」——把维护自动化。代码的「对」比客服客观得多（编译/测试绿），商业价值更可度量：2.2 天 → 4 小时。
- 内容创作（应用生成平台）：LLM 替代的不是开发者，是「从零规划应用架构」的第一步——RAG 检索相似案例 → Planner 按需裁剪成可执行的 DAG。把「创意模糊阶段」结构化。

**项目印证**：三个项目分别对应三类价值——OpsPilot 降低 MTTR（技术意义）、前端测试域降低维护成本（商业意义）、应用生成平台降低从想法到代码的摩擦（技术+商业）。

**核心**：OpsPilot——从零搭建 SRE 故障处置 Agent。最想讲的可复盘教训：矛盾检测用 LLM 失败了，规则表才是对的。

### Q25 大模型通常依赖海量文本语料进行预训练，而搜索与推荐系统的特征往往高维、稀疏且样本量有限。在这种数据规模和结构差异下，如何有效融合大模型能力与搜推任务的小样本高维特征？

**核心**：不 fine-tune 大模型适配搜推特征，而是把大模型用作「特征提取器 + 推理增强层」，搜推的稀疏特征走传统路径，在决策层融合。

**展开**：
- 特征层：大模型处理非结构化文本（用户评价、商品描述、客服对话），产出语义 embedding；搜推的高维稀疏特征（用户 ID、CTR、曝光次数）走传统双塔/Wide&Deep，不必塞进 LLM。
- 融合层：不是让 LLM 直接做排序（不可控 + 延迟高），而是让 LLM 做「冷启动阶段的语义理解 + 推理」——用户说「帮我找一个适合送礼的 200 以内蓝牙耳机」，LLM 把模糊需求转成结构化过滤条件 + 语义 embedding → 扔给搜推引擎检索 + 排序。
- 迁移学习：搜推的监督信号（点击/转化）回灌给 LLM，不是 fine-tune 参数，而是 fine-tune prompt/检索策略——应用生成平台的 query 改写（58%→84%）就是「不调模型、调输入质量」的思路。

**项目印证**：应用生成平台的「Router 先分类意图 → RAG 检索 scene→plan 范式」——Router 类似搜推里的「意图分类」+ 特征提取，RAG 类似检索段，Planner 类似决策融合层。不用一个 LLM call 搞定，分层才能各自优化。

> 🏷️ M23
