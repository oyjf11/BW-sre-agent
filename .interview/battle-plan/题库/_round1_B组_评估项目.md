## M18 · RAG 评估体系

**Kernel**：检索段看 Recall@k/MRR/NDCG，生成段看 faithfulness/answer relevancy/context precision。自动评估(RAGAS/Trulens)+人工评估(badcase 复盘)并用。最易被忽略的：上下文利用率——检索回来了但模型没用上，说明 prompt 或 rerank 有问题。

**展开**：

### Layer 1 — 为什么 / 背景

RAG 是一个两段式 pipeline，故障可能发生在检索段也可能发生在生成段。单纯看端到端效果无法定位瓶颈——是检索没召回？还是召回了但生成没用？因此评估必须拆成两端，分别度量。通用工具链：RAGAS（RAG Assessment）提供 faithfulness / answer relevancy / context precision / context recall 四维自动评分；Trulens 侧重可观测性，记录每一步的中间结果用于回溯。

### Layer 2 — 怎么做 / 机制

**检索段指标**：
- **Recall@k**：标准答案中的相关文档，在 top-k 召回里出现了几条。答「召回全不全」。
- **MRR（Mean Reciprocal Rank）**：`Σ(1/首个正确结果的排名)/N`。答「第一个对的排多前」。
- **NDCG（Normalized Discounted Cumulative Gain）**：`DCG/IDCG`，带位置权重的排序质量。相关文档排得越靠前，NDCG 越高。

**生成段指标**：
- **faithfulness（忠实度）**：生成内容中，有多少是能从上下文里找到依据的。用 LLM-as-Judge 逐句比对 context 和 answer。
- **answer relevancy（答案相关性）**：答案是否扣题。RAGAS 的做法是让 LLM 根据 answer 反向生成问题，再算反向问题与原问题的语义相似度。
- **context precision（上下文精度）**：检索回来的 context 中有多少是真正相关的。高分 = 噪声少。

**评估方式组合**：
- 离线刷数据集：用标注好的 QA 对跑自动指标，发现问题再人工复盘 badcase。
- 在线 A/B：真实流量分组对照，看核心业务指标（解决率/满意度/NPS）。

**上下文利用率**——这是最易被忽略的指标。检索召回了相关文档，但生成模型的回答里没体现——可能性有三：① prompt 没要求引用，模型自由发挥；② rerank 把高相关文档排到了 window 之外，模型没看到；③ context 组织方式差（顺序乱、噪声多），模型跳过了关键段落。查这个指标要靠 LLM-as-Judge 逐条比对回答和 context，看每条上下文里的关键事实是否在回答中出现。

**RAG 错误归因（控制变量法，母题骨架 M19）**：把「人工塞对的文档」喂给生成器——如果还差 → 生成的锅（调 prompt / 上下文组织）；如果喂对就好了 → 检索的锅（调分块 / embedding / rerank）。这是一条极其高效的排查链路。

### Layer 3 — 坑 / 边界 / 追问预判

1. **命中率打满天花板 ≠ 优化没意义**。测试集太简单时，基线已经 100% Top-1，rerank 在命中率维度归因 +0pp。但去看 **margin（top1 分 − top2 分）**：裸向量 0.21 → rerank 0.51，翻 2.4 倍。margin 大意味着系统「敢下结论」，可以在 0.4 设置信度阈值做拒答/转人工。命中率看「召回对不对」，margin 看「敢不敢下结论」——两类指标缺一不可。

2. **不同检索方法的分数不能横比**。cosine 相似度（embedding 内积，0~1）和 cross-encoder 的 relevance_score 是两种尺度，绝对值没有可比性。但 margin 是同尺度内的相对差，可以横比。报表必须标注尺度差异，否则会产生「rerank 退步了」的误读。

3. **自动评估 ≠ 不要人工**。LLM-as-Judge 会抖会错（偏好长答案、一致性差）。必须定期拿人工标注做相关性校准，算 kappa 看判卷器是否漂移。

4. **评测集本身会腐烂**。服务架构变了，老的 QA 对参考答案就不对了。每条 case 打版本标签，架构变更时同步 review 受影响 case。

### Layer 4 — 同类变式

- **评估数据集构建**（母题 M20）：问题来源（真实日志 + 专家造题）→ 双人标注 + 仲裁 → 负样本（难负例）→ 保多样性和难度分层 → 可复现。
- **数据飞轮**：门控写回保质量（只有人工确认的 RCA 才进向量库）→ LLM-as-Judge 补覆盖 → Golden Set 防回退 → 旁路不阻主流程。

**项目印证**：

- **两阶段检索 + A/B 归因**（最直接）：在 rag2 demo 中做了 A（裸向量）/ B（+query 改写）/ C（+rerank）三路单变量递进消融，归因分解出每个手段的 pp 贡献。同时发现了「命中率打满但 margin 翻 2.4 倍」的评测指标陷阱——这让我深刻理解了「评测要分两类指标」这个原则。
- **OpsPilot 数据飞轮**：`retrieve_memory` 做两阶段检索（粗排召回 + rerank 精排）；只有 `confirmed_by_human=true` 的 RCA 才写回向量库（门控防污染）；Golden set 版本化防回退。

## M19 · Agent 评估体系

**Kernel**：分三层——单步准确率(工具选择对不对/参数对不对)、任务完成率(端到端成功率)、效率(token 消耗/步数/耗时)。关键陷阱：不能只看成功率不看效率，Agent 可能靠反复试错成功但实际不可用。

**展开**：

### Layer 1 — 为什么 / 背景

Agent 的评测比 RAG 难一个量级，因为「对」的定义含大量主观成分。RAG 好歹检索有 ground truth 文档，Agent 的动作序列（先调 A 工具再调 B 工具）很难说「唯一正确路径」。评测的复杂度本质上是**被「对有多客观」逼出来的**——客观能判的，一层代码就够了；一旦「对」含主观，才层层加码到模型判 + gold set。

### Layer 2 — 怎么做 / 机制

**三层评测指标**：
1. **单步准确率**：每步工具选择对不对（该调 DB 查慢查询却调了 K8s 查 Pod）、参数对不对。这是最细粒度的诊断指标。
2. **任务完成率**：端到端的成功率。一道工单从 intake 到 rca，诊断结论是否正确、修复是否有效。
3. **效率指标**：token 消耗、步数、wall-clock 耗时。Agent 可能靠反复试错 10 步完成任务（成功率 100%），但 token 烧了 5 倍、延迟不可接受——这在生产上不可用。

**8 步评测方法论（我总结的从零搭评测框架的完整路径）**：
1. 确认能力可被验证（有裁判才评）
2. 按生命周期拆段（多阶段 Agent 不给一个综合分）
3. 拆每段的「对」= 客观内核 + 主观残留（尽量客观化）
4. 按客观程度配裁判（代码判 > 跑一遍 > 模型判）
5. 建题库（藏住结论的历史输入 + 冻结快照 + 标准答案 + 陷阱）
6. 给主观残留建 gold set（3 人独立标 → kappa → 对齐会 → 全员一致入册）
7. 冻结世界（可重复性的地基）
8. 汇总成北极星向量（P/R + 覆盖 + 成本，永不揉成一个数）

**三层裁判塔**（评测架构的核心）：
```
第 0 层  被评测 agent    会抖、会作弊        ← SUT
第 1 层  判卷器(grader)  代码判+模型判       ← 给 agent 打分
第 2 层  gold set 金标集  死数据+算术，不抖  ← 校准判卷器，套娃到此为止
```
题库考 agent，gold set 考判卷器——两条不同的考。判卷器的模型判部分必须过 gold set（kappa ≥ 0.61）才许上岗。

**判卷器的统计纪律**：钉死裁判模型版本 + 评分提示词（禁自动最新版，防厂商升级漂移）、判三次取多数（吸收抖动）、kappa 报区间不报裸数、严禁重跑金标到过（p-hacking）。

### Layer 3 — 坑 / 边界 / 追问预判

1. **只看成功率不看效率**：Agent 靠反复试错成功，真实用户等不了 30 秒。
2. **总分掩盖短板**：「综合 85 分」既会被刷（加噪声题刷分），又看不出到底哪段弱。用分维度打分——根因准确率、方案可操作性、证据完整度、风险评估合理性——各自独立报。
3. **覆盖率不是裁判**（AI 秒级生成空壳测试凑数字，HOLLOW 检测拦）。
4. **判卷器会抖**：模型判卷器今天判对明天判错。用 gold set 校准，且判卷器「过没过 gold set」本身也带噪声，要判三次 + kappa 区间。
5. **金标需要样本量**：几十条只够演示机制、置信区间很宽；要稳需 50–100 条，要窄区间得几百条。
6. **评测体系本质是组织产物**：gold set 需要三位专家独立标注 + 对齐会 + 题库负责人/金标负责人两个常设岗位。单人 demo 在「组织」那一维天然到不了企业级。

### Layer 4 — 同类变式

- **CI 评测门禁**：传统 CI 守代码（编译/单测），AI 系统里提示词/skill/模型版本一改，能力就整体漂移——传统门禁抓不到。评测门禁 = 改提示词/配置 → 跑题库回归 → 北极星掉出误差范围或安全题跌破 100% → 挡住不许合。两层门：题库门禁（守 agent）+ gold set 门禁（守判卷器）。
- **短板飞轮**：跑评测 → 分层看分 + 翻最差题 trace 定短板 → 改进（主战场是喂信息不是换模型：知识包 > 提示词 > 闸门 > 编排 > 最后才换模型）→ 重跑验证 → 涨分题进回归、作弊未遂做成陷阱题。

**项目印证**：

- **OpsPilot 的 critic 门控就是 Agent 评测思想的落地**：critic 不打分，直接读上游算好的 `evidence_quality_score`（avg × penalty × coverage），做 4 路裁决。这本质上就是把 agent 的诊断质量拆成可度量的维度。
- **前端测试域的完整评测体系设计**：把前端测试 agent 拆成编写/执行/修正/守护四段，每段配不同的裁判——编写用三件套（绿 on 正确代码 + 历史缺陷回放 + 变异测试，全代码判、无需 gold set），修正用代码判 + 模型判 + gold set。正是「按客观程度配裁判」这条原则的完整落地。
- **数据反馈飞轮**：在线人工打分（覆盖率低）→ LLM-as-Judge 补覆盖（分维度打分，不用总分）→ Golden set 防回退（版本化防腐烂）→ 写回知识库（`confirmed_by_human` 门控）。四层串起来，在线监测质量、离线保证不回退。

## M20 · A/B 测试

**Kernel**：单变量递进——每次只改一个变量(如 rerank 阈值)，控制组 vs 实验组，看核心指标(解决率/满意度/NPS)的统计显著性。前提：流量正交分桶 + 足够样本量 + 足够跑的时间(至少 1 周覆盖周周期波动)。

**展开**：

### Layer 1 — 为什么 / 背景

Agent/AI 系统的优化手段很多（换提示词、加 skill、换模型、调 rerank 阈值），但「改完效果变好了」和「是哪个手段起了作用」是两回事。不做单变量递进，只能说「加了一堆东西效果变好」，归不到具体优化。更危险的是——改一行提示词修好一道题、悄悄改坏十道，没有评测门禁就直接合进去了。

### Layer 2 — 怎么做 / 机制

**单变量递进消融（我的核心做法）**：A（基线）→ B（+变量1）→ C（+变量2），每列只多一个变量。归因分解直接出来：B − A = 变量1 贡献的 pp，C − B = 变量2 贡献的 pp。再配翻盘案例（A 错→B 对的具体 case）和反向翻车案例（B 对→C 错），每个手段贡献多少、有没有副作用都能讲清楚。

**A/B 实验四件套（让报表有实验味道）**：
1. **逐条明细**：每条 query 多路结果并排对比
2. **翻盘/翻车归因**：列出 A 错→B 对（改写救场）、B 对→C 错（rerank 带偏）的具体例子
3. **置信度对比**：命中项的平均 top1 分、平均 margin 多路对比
4. **命中率汇总 + 归因分解**：Top-1/Top-3 多路百分比 + 每个优化各值多少 pp

**统计显著性**：
- 比例类指标（命中率/准确率）95% 误差半宽 ≈ `1.96 × √(p(1−p)/n)`
- n=200 → ±5.5pp；n=400 → ±3.9pp；n=1000 → ±2.5pp
- 误差砍一半、样本翻四倍 → 加题比加遍数划算
- 报告写 `79% ± 5.5`，不写 `79%`
- 严禁重跑到绿（p-hacking）——分数在及格线晃就加题再考，不是反复重跑撞运气

**CI 评测门禁（A/B 的自动化版）**：改提示词/配置 → 跑题库回归 → 北极星掉出误差范围 → 挡住不许合。

### Layer 3 — 坑 / 边界 / 追问预判

1. **命中率天花板陷阱**：基线已达 100%，A/B 命中率归因 +0pp，但 margin 翻 2.4 倍。优化有没有用不只看命中率，还看置信度。
2. **尺度陷阱**：cosine 分和 cross-encoder 分是两种尺度，绝对值不能横比。只有 margin（同尺度内 top1 − top2 的相对差）可横比。
3. **margin 的工程意义**：margin 大 → 能设置信度阈值做拒答/转人工。裸向量 margin 0.21 根本没法设 gate，rerank 后 0.51 就行。这才是 rerank 上生产的真正理由。
4. **配对比较而非总分差**：看「几道由错变对、几道由对变错」的翻转明细，而不是只看总分差。总分可能不变但内部剧烈翻转——改好了 10 道、改坏了 10 道，总分一样但性质完全不同。
5. **足够跑的时间**：至少覆盖一个完整周周期波动（工作日 vs 周末流量模式不同）。低于一周的数据可能是碰巧。

### Layer 4 — 同类变式

- **内层消融实验**：不只一个 A/B 对照，而是设计多个消融实验解释每个数字的来源。如冻结世界消融（直连 22% flaky → 朴素指纹 30% → 归一化 3%）、自愈 skill 消融（无 skill 60% → 有 skill 83%）。
- **配对比较的约束式结论**：不写「综合分 85」，写「发现力/覆盖不降的前提下，成本降 X%、信噪比 a→b」。

**项目印证**：

- **两阶段检索 rag2 A/B 归因**：设计了 A（裸向量）/ B（+query 改写）/ C（+rerank）三路单变量递进消融。10 条 case 跑出命中率全 100%、归因 +0pp，但深入看 margin（A 0.21 → B 0.31 → C 0.51），发现了 rerank 在置信度维度的真实收益。这个「指标会撒谎」的判断直接改变了我对评测体系的设计——必须分两类指标、不能只看命中率。
- **前端测试域外层主实验**：两臂对照（人工维护 vs 自愈+冻结），四道公平护栏（同战场、单变量、对手不神化、成本诚实），用北极星向量打分 + 配对比较。

## M21 · OpsPilot 全链路

**Kernel**：13 节点 SRE 诊断 Harness——intake→triage→retrieve_memory→planner→evidence_fanout(并行 4 路取证)→aggregate→diagnose→critic(质量门控)→remediation→risk_gate→approval/executor→verify→rca。核心理念：分解胜于强提示，0.9¹ ≪ 0.98¹³ 每步可验证。

**展开**：

### Layer 1 — 为什么 / 背景

SRE 故障处置有三个硬要求：**可控**（每一步能独立验证和回退）、**可解释**（为什么得出这个诊断结论）、**能中途叫人**（高危操作必须人工审批）。一个裸的 ReAct 循环（shallow agent）面对这三个要求会失控——步骤一长上下文塞满、跑偏无法回头、高危操作无法中断。所以选了 LangGraph（而非纯 ReAct 框架），用**确定性图编排包裹非确定性 LLM**，把规划/子代理隔离/HITL/复盘显式拆成 13 个职责单一的节点。核心理念：一个节点 90% 靠谱、链式串 0.9¹₃ ≈ 0.55，把每个节点的可靠性提到 0.98、0.98¹³ ≈ 0.77——**分解胜于强提示，每步可验证**。

### Layer 2 — 怎么做 / 机制

**一句话讲完 13 节点（5 段）**：
一条报障进来 → **入口段**（intake 把三种输入归一成标准工单）→ **定性段**（triage 规则+LLM 分类，retrieve_memory 两阶段检索捞历史相似案例）→ **取证段（心脏）**（planner 排调查计划 → evidence_fanout 并行 4 路只读取证 → aggregate 聚合打质量分 + 跨层矛盾检测 → diagnose 下根因假设 → **critic 质量门控**，证据不够就退回重查）→ **处置段**（remediation 出修复方案 → **risk_gate 风险闸门** → 高危卡 **HITL 审批中断** → executor 幂等执行 → verify 验真，失败可重试 2 次）→ **收尾段**（rca 成败都写复盘归档）。

**节点之间不直接调用**，全靠往共享的 `IncidentAgentState` 写 key、下游读 key 传递。全程「交通规则」是 5 个条件路由器，决定每段走完往哪拐——这是「可退回重来」的物理实现。

**三个最想讲的设计点**：

1. **critic 门控 = 材料审核员，不是方案评委**。评分与决策解耦——会计算分（`质量 = avg × penalty × coverage`），领导拿分做 4 路裁决：
   - `CONTRADICTION`（退两步回 planner）> `NEED_MORE_EVIDENCE`（退一步回 fanout）> `REPLAN`（退两步）> `PASS`
   - loop guard：转 2 圈摆不平 → 转人工 + 压低置信度 ≤0.5
   - 矛盾检测用**有向规则表**（12 条，零 token 成本），不是 LLM 裁判（LLM 判矛盾准确率 71% 但跨层传播场景掉到 53%，且不一致率 39%）
   - 置信度 `min(_, 0.9)` 封顶防自吹，降级结论 ×0.5，LLM_FAILED 不计入

2. **HITL 审批中断（自建，非 LangGraph 原生）**：OpsPilot 用 LangGraph 0.0.55，没有原生 `interrupt()`。审批流程是自建的——compile 时 `checkpointer=None`，自己逐节点把 state_json 落 `IncidentCheckpoint` 表；审批时 `approval_interrupt → END` 图挂起；人 `POST /approvals/{id}/decision` 点头后 `POST /runs/{id}/resume`，从自建 checkpoint 读回 state、设 `_resume_from_node` 重新 astream、白名单路由 + executor 幂等防重跑。

3. **risk_gate 多信号闸门**：把动作风险 + severity + env + requires_immediate_human + 诊断置信度 + loop guard + 能力预检揉成 4 选 1：`LOW_ONLY`（放行）/ `NEEDS_APPROVAL`（卡审批）/ `BLOCKED`（能力预检失败拦死）/ `NEEDS_HUMAN`。real 模式下 `execute_action` 没接真 adapter 就 fail-closed，提前 BLOCKED，不让跑到执行器才崩。

**Deep Agent 范式对标**（OpsPilot 就是手搓的 deep agent）：
- Planning → `planner` 节点
- Sub-agents 隔离上下文 → `evidence_fanout`（扇出取证）
- File system 卸载上下文 → evidence 持久化到 DB / OSS 归档
- HITL → `risk_gate` + `approval_interrupt`
- Long-term memory → `retrieve_memory` + RAG
- 自我纠错 → `critic` 节点回边（= Reflection）

**端到端可观测**：每个节点被 `_wrap_with_events` 包一层，进节点发 `NODE_STARTED`、抛异常发 `NODE_FAILED`，事件落 DB 带 sequence。前端 SSE 实时渲染 Stepper 走到哪，断线凭 Last-Event-ID 从断点补发。

### Layer 3 — 坑 / 边界 / 追问预判

1. **自建 checkpoint ≠ LangGraph 原生**：0.0.55 没有 `interrupt()`/`Command`/`PostgresSaver`/`Send`。被问「checkpoint 怎么做的」必须答自建，按原生答会被当场戳穿。

2. **Pydantic 对象 vs dict 二义性**：checkpoint 存 JSON，回来 Pydantic 变 dict，直接 `.attr` 抛 AttributeError → 必须 `hasattr(x, 'f') else x.get('f', 'default')`。遍历 list 先防 None。

3. **诚实边界（三个缺口，加分项）**：① 入口 NL 抽取没做——前端是结构化表单，插入点明确（intake 前加 `nl_parse` 节点，下游一行不改）；② CONTRADICTION 仲裁——检测在算（规则表产出信号），但没晋升到 critic 读的候选字段，那条分支当前不触发；③ 方案择优——remediation 只出单方案，没有方案评委。共同点：核心骨架和契约先焊死、能独立验证，适配/增强层留作下一刀。

4. **critic 为什么不用投票**：专科 Agent 分头取证、只读、互不抢排他资源。没有抢资源的硬冲突，只有结论冲突的软冲突。退回重查就行，不需要投票那套乐观锁/先到先得。

5. **为什么 13 个节点不嫌多**：取证和处置间硬插 critic、处置里硬插审批、执行后硬插验真——这些「多余」节点本质都是闸门，拿编排复杂度换「不闯祸」，每个职责单一、能独立测。

### Layer 4 — 同类变式

- **Tool Gateway 安全链路**：定义(JSON schema) → 注册 → LLM 选工具+提参 → 网关校验(schema/类型/必填项) → 选适配器(real/mock) → 重试(最多 2 次，指数退避) → 写审计日志 → 结果回灌。安全 = 权限控制 + 参数校验 + 审计日志 + 高危人工审批。
- **数据飞轮**：门控写回保质量 → Judge 补覆盖 → Golden set 防回退 → 旁路不阻主流程。

**项目印证**：

- OpsPilot 本身就是一个从零搭建的 SRE Agent。我负责：整体架构设计（13 节点编排 + 5 段路由）、critic 门控的设计取舍（打分与决策解耦、矛盾检测选规则表不选 LLM、接口先行的矛盾分支预留）、HITL 审批中断的自建方案（绕过 0.0.55 无原生 interrupt 的限制）、端到端可观测（节点事件落 DB + SSE 实时推）。
- 代码由 coding agent 生成，但架构判断、关键取舍、诚实边界是我一手做的。比如「矛盾检测用规则表不用 LLM」——我们跑了 28 组 mock 工单对比，LLM 裁判不一致率 39%、跨层传播准确率 53%，数据不支持上线，所以选了确定性规则表。这种 tradeoff 判断才是我的核心贡献。

## M22 · 前端测试域闭环

**Kernel**：把前端验收从「AI 写完人工点」变成「Playwright 真跑 + 截图取证 + 五分类失败归因 + 缺陷流转」全闭环。2.2 天 → 4 小时，10+ 团队试点。内核：证据高于断言——LLM 会撒谎，浏览器截图不会。

**展开**：

### Layer 1 — 为什么 / 背景

前端测试套件是会腐烂的资产。脚本靠 CSS class（`.submit-btn`）定位元素，前端天天改版，类名/结构一变——**按钮功能没坏、脚本却找不到了，当晚报红**。红分两种：
- **真红**：应用真的坏了/回归了 → 报告它，绝不自愈
- **假红**：应用没坏，只是定位过期 → 更新定位，红消失

不处理假红：① 人工每天上午手改（纯体力）；② **报警疲劳**——一半红是假的，工程师渐渐无视红灯，藏在里面的真 bug 一起被忽略，报告失去意义。

比喻：报告 = 烟雾报警器；UI 改版 = 积灰误报；住户抠电池 → 真着火不响。**自愈 = 自动清灰，让报警器只为真火响，它从不替你灭火。**

### Layer 2 — 怎么做 / 机制

**四段闭环**：
```
编写(生成用例) → 执行(跑) → 修正(分诊/自愈) → 守护(回归发现力)
  ↑                                                      |
  └────────── 需求/AC、改版事件 ───────────────────────┘
```

**五分类失败归因**（自愈 agent 的核心判断）：
- **假红（定位过期）**：页面没坏，只是选择器找不到 → 更新定位，自愈
- **真红（逻辑回归）**：断言失败，功能真坏了 → 绝不自动修，报告给人
- **抖动（flaky）**：同一脚本多次跑结果不一致 → 冻结世界 + 指纹归一化
- **回归**：曾经修好的 bug 又出现 → 历史回放检测
- **环境**：依赖服务不可用 / 数据变化 → 冻结快照

**死规矩：只许改定位，碰断言一律作废**。改定位 = 更新地址（安全可自动），改断言 = 篡改「什么算对」（偷改考题答案，必须锁死）。技术实现：swc AST 锁断言 + 运行时唯一性校验。判「DOM 改动落在定位还是断言」是自愈最核心也最易判错的能力，评测要狠考。

**冻结世界（可重复性的地基）**：录制回放 + 请求指纹归一化——按路径剥离时间戳/随机 ID，序列化前字段名排序（`{user,role}` 和 `{role,user}` 归一到同一哈希，否则回放全穿透真后端）。E1 消融实验：直连 22% flaky → 朴素指纹 30% → 归一化 stableStringify 3%。

**证据高于断言**：LLM 会说「修好了」，浏览器截图不会撒谎。每次自愈产出修复后，沙箱重跑 + swc AST 锁断言 + 选择器唯一性检查（代码判），再加上模型判「修复语义合不合理」（主观残留才上模型判）。客观能判的判到底，主观残留才用模型判 + gold set 校准。

**评测体系（三件套防作弊）**：
- **编写段评测**：三件套纯代码判（绿 on 正确代码 + 历史缺陷回放 + 变异测试），不需 gold set。绝不拿覆盖率当裁判（AI 秒级生成空壳测试凑数字）。
- **修正段评测**：代码判（沙箱重跑 + swc 断言锁 + 选择器唯一性）+ 模型判（修复语义合不合理）+ gold set（校准模型判）。
- **三类题**：业务题（分诊准不准）、行为题（过程规不规矩）、安全题（输入藏指令注入，必须 100% 不中招）。

### Layer 3 — 坑 / 边界 / 追问预判

1. **DOM 定位 vs 断言的分界线**：只影响「怎么定位元素」的 DOM 改动 ≠ 逻辑坏；影响「被断言的可观察行为」的 DOM 改动 = 逻辑坏。判「改动落在哪一边」是自愈最核心的能力，也是评测最狠考的地方。

2. **覆盖率不是裁判**：任何被 AI 优化的指标都会被玩坏。用「历史缺陷回放 + 变异测试」（能抓住真实坏过的 bug 吗？）替代覆盖率。

3. **自愈只服务于报告的信噪比**：自动化的是「维护」（改定位），不是「判定」（对错）。它不替代报告、不替你灭火。

4. **组件库边缘 case**：Teleport 弹窗挂 body 末尾、虚拟滚动表格行锚点漂移、懒渲染 select——这些让自愈定位 skill 必须有「门道」知识。

5. **企业级缺口在组织和规模**：内核（per-stage 裁判、gold set、防作弊、北极星向量）是企业级的；缺的是规模、组织（题库负责人/金标负责人两个常设岗位）、运行时治理（模型网关、权限、沙箱、审计）。

### Layer 4 — 同类变式

- **自愈 skill 消融实验**：agent 无组件库定位 skill → 自愈成功率 60%；有 skill → 83%
- **防作弊闸门演进**：文本匹配 → swc 锁断言 → +运行时唯一性漂移检测
- **评测门禁**：改自愈 agent 提示词/组件库 skill → 跑题库回归 → 北极星掉出误差范围 → 挡住不许合

**项目印证**：

- 这是我主导的完整能力域设计——从「为什么需要自愈」（假红误报 → 报警疲劳 → 红灯失去意义）到四段闭环架构，到评测体系三层塔，到 CI 评测门禁。核心设计判断我都有：① 死规矩「只改定位不碰断言」；② 冻结世界用 stableStringify 排字段名（不是朴素指纹）；③ 覆盖率不当裁判用历史缺陷回放替代；④ 自愈的定位是「自动清灰」不是「自动灭火」。
- 实际数字：传统前端验收 2.2 天 → 4 小时（一个改版周期的回归时间），10+ 团队试点。评测数据为设计推演（60→83、22%→3% 来自方法论复刻，非实测），我讲的时候诚实标注。

## M23 · 应用生成平台 Router-Planner

**Kernel**：修改阶段用「场景化 RAG + Router/Planner 分层」识别用户想改什么。Router 先分类(改样式/改逻辑/改数据)，Planner 按类型调用不同的修改策略。取舍：不用 LLM few-shot（边界 case 漂移），用 RAG 召回相似历史案例更稳。

**展开**：

### Layer 1 — 为什么 / 背景

用户模糊指令（「帮我做个带登录的待办应用」）→ 系统要生成可执行代码。语义鸿沟太大，纯靠模型从零规划不稳定、不可控、不可复用。关键问题在**修改阶段**尤其严重——用户说「把按钮改成蓝色」「增加导出功能」「换一个数据库」，这三个分别是改样式、改逻辑、改数据的完全不同性质的操作，但都是自然语言描述。Router 要做的第一件事是：**分类——你到底想改什么**。

### Layer 2 — 怎么做 / 机制

**三层架构：Router → RAG → Planner → Executor**

1. **Router（管分流 + 意图分类）**：小模型/规则先判断请求类型（改样式 / 改逻辑 / 改数据 / 新增页面 / 简单问答）。简单请求（改颜色、改文案）直接走快车道，「不经过 Planner」，省 token 省延迟。复杂请求进入下一层。

2. **场景化 RAG（检索 scene→plan 范式）**：Router 产出的结构化意图（意图实体 + 约束）刚好是高质量检索 query。RAG 不是检索代码片段，而是检索**历史上验证过的「场景 → 执行方案（plan）」范式**。Query 改写（术语映射 58%→84%，最大收益）→ 混合检索（向量 + BM25 + 元数据过滤）→ small-to-big 分块（父块 = 完整 plan，子块 = 多视角场景描述，召回打子块、按 parent_id MAX 聚合父块）→ rerank 精排。

3. **Planner（融合生成 DAG）**：拿 RAG 召回的范式作为**参考而非指令**，Planner 有权否决低分命中。按需裁剪融合生成可执行的任务图（DAG）。Planner 拆错 → re-plan 循环。

4. **Executor**：按 DAG 依赖拓扑并行执行，单步重试。

**为什么 RAG 放在 Router 之后、Planner 之前**（最核心的架构判断）：
- RAG 需要结构化 query（意图 + 约束），Router 刚好产出这个
- 简单请求直接绕过 RAG 和 Planner，省 token
- 规划阶段的范式经验（scene→plan pattern）与执行阶段代码分离，互不污染
- 成功 plan 回流语料是纯净的「场景 + 方案」，不混代码细节

**为什么选 RAG 召回而非 LLM few-shot**：
- few-shot 固定几个示例塞进 prompt，边界 case 不在示例里就漂移
- RAG 从历史验证过的 800 条范式 → 6000+ chunk 中动态召回最相似的几个案例
- RAG 的好处：① 案例会随时间增长（成功 plan 回流），few-shot 是死的；② RAG 的结果是「参考」不是「指令」，Planner 有权否决，few-shot 在 prompt 里是硬约束；③ 边界 case 能从更大的语料库里命中相似案例，few-shot 只有几个固定示例

### Layer 3 — 坑 / 边界 / 追问预判

1. **不用一个 LLM call 搞定的三层论证**：① 不可控（黑盒一步到位，改不了中间任何一步）；② 不可复用（规划阶段的范式在一步到位里无处安放）；③ 不可中断（分层后用户看到 plan 即可纠正——最便宜的修改时机，一步到位必须生成完）。

2. **RAG 检索不准怎么办？** 多层兜底：query 改写提升输入质量 → 混合检索兜语义 + 精确 + 硬性过滤 → rerank 卡阈值，宁可只给 1 条高质量范式也不给噪声 → Planner 有权否决低分命中。

3. **Router 用什么模型？** 简单分类用规则/小模型（便宜、确定、可解释），复杂/长尾才掉 LLM。和 OpsPilot triage「规则先兜底、LLM 再补」同一个价值观。

4. **DAG 坏了怎么恢复？** Planner 拆错 → re-plan 循环。Executor 按依赖拓扑执行，某步失败可单步重试。

5. **诚实边界**：这个项目是架构推演级别的深度（一次 Claude 深聊对话的产物），不是真实交付项目。58%→84% 来自原文标注的实测数据。架构判断本身经得起推敲——Router→Planner 分层、RAG 放中间、不用 few-shot 选 RAG——但我不会假装它是上线后的实测数据。

### Layer 4 — 同类变式

- **多租户隔离**：平台用户隔离 + 用户生成 App 隔离（租户套租户）。推荐路线 C：共享 Postgres + 每项目独立 Database（天然隔离，DROP DATABASE 干净利落）。控制平面自研补 Supabase 缺失的项目管理层（生命周期/密钥/路由/配额/休眠/分片）。
- **代码质量闭环**：约束 → 检查 → 修复 → 验收（ESLint/TS 约束 → 静态检查 → LLM 修复 → 测试验收）。

**项目印证**：

- 我负责平台的整体架构设计。最核心的判断有三：① Router 和 Planner 分开——不只是「解耦好」，而是因为 Router 产出的结构化意图刚好是高质量检索 query，而简单请求（改颜色）不应该经过 Planner 浪费 token；② RAG 检索 scene→plan 范式而非代码片段——Planner 站在已验证模式上按需裁剪，不是从零编；③ 不用 LLM few-shot 用 RAG 召回——边界 case 漂移是 few-shot 的致命问题，RAG 从 800 条 → 6000+ chunk 的语料库里动态召回更稳。
- 反幻觉机制：5 条 prompt 约束 + LLM-runner 评审兜底 + 成功 plan 回流语料形成数据飞轮。幻觉率约 6%，用户返工率约 18%。
