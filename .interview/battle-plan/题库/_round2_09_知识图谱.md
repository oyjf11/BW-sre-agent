## 知识图谱（35 题）

### Q1 金融领域LLM生成内容的事实准确性和一致性保障

**核心**：RAG 注入外部可信知识 + 引用溯源 + 二次校验三层约束。

- 离线建金融知识库（监管法规/产品文档/交易规则），在线检索后注入 LLM context 约束生成范围；强制标注引用来源，用 cross-encoder rerank 提升检索精度；二次校验 LLM 检查生成内容与引用文档一致性。

> 🏷️ M1+M9 | ⭐ 了解级别

### Q2 医疗/法律等专业垂直领域完整RAG系统链路

**核心**：离线[领域文档清洗→按结构分块→领域 embedding→索引入库] + 在线[query 术语映射→混合检索→rerank 精排→带引用生成]。

- 关键设计：领域术语映射解决用户口语与专业书面语鸿沟；按文档结构分块（法律按条款、医疗按 SOAP）；选领域专用 embedding 模型；门控写回防污染——只有人工确认的内容才入库。

> 🏷️ M1 | ⭐ 了解级别

### Q3 GraphRAG的实现原理、图检索与传统向量检索的区别

**核心**：传统 RAG 查相似片段，GraphRAG 多一步——先抽实体-关系建图，检索时走图遍历拿推理链。

- 离线：LLM 从文档抽实体和关系→实体为节点/关系为有向边→构建文档级知识图谱；在线：query 先定位起始实体→沿关系边多跳遍历（BFS/DFS+剪枝）→收集路径上的关联事实拼 context→LLM 生成。与传统向量检索的核心差异：向量返回「相似片段集合」，图检索返回「事实推理链」，后者能处理多跳推理查询。

> 🏷️ M24 | ⭐ 了解级别

### Q4 大规模知识图谱的更新机制：实时性与一致性

**核心**：增量抽取→实体对齐→冲突解决→增量写入四步流水线，实体对齐是最大难点。

- 增量抽取：新文档走同样实体/关系抽取管线；实体对齐：新实体与现有实体做精确/模糊匹配（embedding+属性匹配），避免同一实体出现多个节点导致图断链；冲突解决：新旧关系矛盾时按时间戳优先或置信度优先裁决；一致性保障靠写入事务+版本标记。

> 🏷️ M25 | ⭐ 了解级别

### Q5 GraphRAG基本原理、与传统RAG差异、图结构的应用方法

**核心**：GraphRAG 用图结构组织知识，检索走图遍历，解决传统 RAG 处理不了的多跳推理。

- 原理：文档→实体关系抽取→构建图；检索时沿边做邻域/路径遍历拿到推理链而非孤岛片段。传统 RAG 向量检索返回相似段落，无法建模「A 依赖 B，B 发生 C」的关系链路。适用：知识量适中、关系密集、查询需要跨文档缝合信息的场景。

> 🏷️ M24 | ⭐ 了解级别

### Q6 GraphRAG处理增量数据更新保证实时性和准确性

**核心**：事件触发增量管道 + 实体对齐防图断裂 + 定时全量校验。

- 新数据入库→自动触发增量抽取→实体对齐（新实体须与现有节点匹配或新增）→关系追加/更新→版本标记。准确性关键在实体对齐：对齐出错（同实体分裂成两节点）会导致图遍历断链丢失信息；冲突时按时间戳优先解决矛盾关系。定时全量重建做一致性兜底。

> 🏷️ M25 | ⭐ 了解级别

### Q7 GraphRAG实际应用中最大的技术挑战和难点及解决方案

**核心**：建图成本高（每文档过 LLM 抽实体关系）和实体对齐精度是两大核心挑战。

- 建图成本：N 个文档×每文档 2-3 次 LLM 调用，比传统 RAG 一次 embedding 贵 100 倍以上→解决方案是只在关系密集、查询复杂度高的场景上 GraphRAG；实体对齐：新实体与已有实体匹配不准会级联污染图结构→解决方案是精确匹配+模糊匹配双通道+人工审核高影响决策。

> 🏷️ M24 | ⭐ 了解级别

### Q8 GraphRAG图结构构建、实体与关系抽取、检索机制的关键实现步骤

**核心**：四阶段——离线建图（实体识别→关系抽取→图构建→可选社区摘要）→在线检索（实体定位→路径遍历→事实链拼接→生成）。

- 实体识别：LLM/NER 模型从文档抽实体（服务/资源/事件类型等）；关系抽取：LLM 判断实体间关系（依赖/导致/包含等）并挂置信度；检索时从起始实体沿边做多跳 BFS 遍历，收集路径上关联事实注入 LLM，生成带推理链引用的答案。优势在于能处理需要跨文档缝合信息的复杂查询。

> 🏷️ M24 | ⭐ 了解级别

### Q9 GraphRAG相比传统RAG的核心优势：结构化建模、语义关联、推理能力

**核心**：GraphRAG 把孤岛片段变成结构化推理链，在复杂查询上检索精度和生成质量都更强。

- 结构化建模：实体关系图比向量空间更精确地表达知识关联；语义关联发现：图遍历自动发现「A→B→C」的间接关系，向量检索只能捕捉直接相似；复杂推理：对多跳查询（如故障依赖链溯源），图检索拿到完整推理路径而非碎片化段落，上下文连贯性更好，LLM 生成质量更高。

### Q10 GraphRAG基本思想与架构设计、优势、挑战与解决方案

**核心**：基本思想是把非结构化文档转为结构化实体关系图，检索时走图遍历替代纯向量匹配。

- 架构分两条线：离线建图（抽取→建图→索引）和在线检索（实体定位→路径遍历→事实拼接→生成）。优势：能处理多跳推理查询、上下文连贯。挑战：建图 LLM 成本高（适合知识量适中/关系密集场景）+ 实体对齐噪声（需人工审核+置信度阈值）+ 可扩展性（万级节点用 NetworkX 可跑，大规模需图数据库）。

> 🏷️ M24 | ⭐ 了解级别

### Q11 伪多模态RAG与真正多模态RAG的差异

**核心**：伪多模态 RAG 依赖文本描述桥接视觉内容，真正多模态 RAG 直接用多模态模型编码和检索视觉特征。

- 伪多模态：图片/视频先转文字描述（OCR/ASR/图像 Caption），文字再走传统 RAG 管线→实现简单但信息损失大（空间关系/视觉细节丢失）；真正多模态：用多模态 embedding 模型直接编码图像/视频特征进向量库，query 跨模态检索→精度更高但模型和工程复杂度高。差异在信息融合层：一个是「文本化后再融合」，一个是「原生跨模态融合」。

> 🏷️ — | ⭐ 了解级别

### Q12 GraphRAG关键技术难点：图谱构建、实体链接、推理路径搜索、可扩展性

**核心**：四类难点——图谱构建噪声、实体对齐不准、路径搜索爆炸、大规模图性能瓶颈。

- 图谱构建：LLM 关系抽取不确定性→用置信度阈值+多源交叉验证；实体链接（对齐）：新实体与已有节点匹配失败导致图断裂→精确+模糊匹配双通道；推理路径搜索：多跳遍历路径数指数增长→剪枝+路径长度限制；可扩展性：万级节点可用 NetworkX，百万级需图数据库（Neo4j/JanusGraph）+分布式图计算。

### Q13 增量场景下GraphRAG动态维护知识图谱与索引结构

### Q14 GraphRAG构建知识图谱和推理过程中的关键技术挑战

**核心**：实体对齐的级联风险、关系抽取噪声、图更新维护的事务一致性是三大挑战。

- 实体对齐：一个对齐错误会级联影响——分裂节点导致图遍历断链，错误归并会产生虚假关联让 LLM 基于错误事实链生成答案；关系抽取噪声：LLM 抽取的不确定性需要置信度阈值过滤+人工审核；图更新维护：增量写入需事务保证图结构不被破坏，矛盾关系需冲突解决策略（时间戳优先或置信度优先）。

### Q15 GraphRAG为何能实现更精准的结构化召回

**核心**：图谱将文档间隐含关系显式化为边，检索时走边遍历直接拿到因果链而非相似片段堆。

- 传统向量检索：返回与 query 语义相似的段落集合，但无法建模段落之间的逻辑关系（依赖、因果、时序）；GraphRAG：实体→节点、关系→有向边，检索时从起始实体沿边遍历，收集的是「有逻辑连接的事实链」而非孤岛片段。这使得召回结果的结构化程度和逻辑连贯性远超纯向量检索。

> 🏷️ M24 | ⭐ 了解级别

### Q16 GraphRAG实现精准召回的核心依赖条件

**核心**：图结构质量（实体识别和关系抽取的准确度）是精准召回的底层依赖。

- 实体识别精度决定检索入口是否正确——起始实体定位错了后面的遍历全偏；关系抽取覆盖度和准确度决定图遍历能走多远、走多准——关系稀疏则断链、关系噪声则走错方向；查询图匹配能力（将用户 query 转化为图上的起始实体+遍历意图）决定能否有效利用图结构。三者缺一不可。

### Q17 哪些场景更适合GraphRAG而非传统RAG

**核心**：知识量适中、关系密集、查询需要多跳推理的场景适合 GraphRAG。

- 适合：故障根因分析（A 依赖 B → B 发生 C → 根因 D）、法律案例推理（法条→判例→关联法案）、药物相互作用分析（药物 A 影响酶 B → 酶 B 代谢药物 C → 相互作用）。不适合：单事实查询（「Redis 最大内存是多少」）、知识量巨大的开放域问答（建图成本不可承受）。适用条件：关系密度高 + 查询复杂度高 + 文档量可控。

> 🏷️ M24 | ⭐ 了解级别

### Q18 传统RAG系统的主要痛点和改进思路

**核心**：检索噪声、上下文稀释、语义鸿沟是三大痛点，可通过混合检索+rerank+query改写逐一缓解。

- 检索噪声：语义相似但实际不相关的段落被召回→用 cross-encoder rerank 精排过滤；上下文稀释：召回过多无关片段挤占 LLM 窗口→压缩去噪+按相关性重排；语义鸿沟：用户口语和知识库书面语词汇不匹配→query 改写（术语映射）收益最大，实测可提升召回率 26%+。

> 🏷️ M1 | ⭐ 了解级别

### Q22 不用图数据库能否称为真正的GraphRAG

**核心**：能。核心是图查询能力而非存储引擎，但规模和性能受限。

- 功能上：用 NetworkX + 内存邻接表可实现图遍历、多跳路径搜索，满足 GraphRAG 的核心能力；性能上：万级节点可行，百万级节点内存不够、遍历耗时不可接受→需图数据库原生图遍历优化；可扩展性：图数据库提供分布式存储、原生图查询语言（Cypher）、索引加速，大规模场景是刚需。规模是关键分界点。

> 🏷️ M24 | ⭐ 了解级别

### Q23 图数据库在GraphRAG中是否不可替代

**核心**：小规模可替代，大规模不可替代——图数据库的核心价值在原生图遍历优化和分布式扩展能力。

- 不可替代体现在三方面：存储层（邻接表天然适合图，比关系型 JOIN 效率高几个数量级）、查询层（Cypher/Gremlin 图查询语言 + 图索引加速多跳遍历）、扩展层（分布式图分区支持亿级节点）。但万级节点以下用 NetworkX/内存数据结构完全够用，图数据库不是硬门槛。

> 🏷️ M24 | ⭐ 了解级别

### Q24 GraphRAG核心思想及相对于传统RAG的主要创新点

**核心**：用实体-关系图组织知识替代向量空间，检索从「找相似片段」变成「沿关系边推理」。

- 创新点：知识组织方式——文档→实体关系图，知识被结构化而非散列化；检索路径——图遍历拿到有逻辑连接的事实链，而非孤立的相似段落；答案生成——LLM 基于结构化事实链推理，比基于碎片化段落生成更连贯、更可溯源。

> 🏷️ M24 | ⭐ 了解级别

### Q25 检索增强生成（RAG）的技术原理和实现流程

**核心**：离线[清洗→分块→embedding→索引入库] + 在线[query改写→混合检索→rerank精排→拼context→LLM生成]。

- 原理：把外部知识库的文档预先转为向量索引，用户提问时检索最相关的文档片段注入 LLM 上下文窗口，让 LLM 基于真实文档而非训练记忆回答。好处：知识可随时更新（不重训模型）、可溯源（标注引用）、成本低（不需要微调）。全流程：离线建库和在线检索生成两条线并行。

> 🏷️ M1 | ⭐ 了解级别

### Q26 RAG基本原理 + GraphRAG如何利用图结构提升检索效果

**核心**：RAG 把外部文档检索注入 LLM 上下文；GraphRAG 在此基础上多用一步——把文档先建成实体关系图，检索走图遍历拿推理链。

- RAG 基本原理：文档预处理→向量化→检索→拼接 context→生成；GraphRAG 的增量：在检索环节用图遍历替代向量匹配，从实体定位出发沿关系边多跳遍历，收集完整推理链而非碎片段落。优势：对需要跨文档缝合信息的复杂查询（如故障链路溯源），图检索比向量检索的召回更结构化和逻辑连贯。

> 🏷️ M1+M24 | ⭐ 了解级别

### Q27 RAG基本原理 + GraphRAG技术方案详解

**核心**：RAG = 检索外部知识 + 注入 LLM 生成；GraphRAG = 在图结构上检索外部知识 + 注入 LLM 生成。

- RAG：离线建向量库（embedding+索引），在线检索（混合检索+rerank）→拼 context→LLM 生成带引用答案。GraphRAG：离线多一步——LLM 从文档抽实体关系建图（实体→节点，关系→有向边）；在线检索——从 query 定位图中起始实体→沿边做多跳路径遍历→收集关联事实拼 context→LLM 生成。图结构使检索结果具有逻辑关联性，能处理多跳推理。

> 🏷️ M1+M24 | ⭐ 了解级别

### Q29 RAG与知识图谱结合的智能体中确保知识实时性和数据新鲜度

**核心**：事件驱动的增量更新 + 版本标记 + 门控写回防污染。

- 实时性：Agent 运行中产出的新知识（如确认的 RCA）→触发增量抽取管线→实体对齐后追加到图谱，无需全量重建；新鲜度：每条边/节点打时间戳，查询可指定时间窗口；防污染：人工确认后才入库（门控写回），避免 LLM 产生的错误知识污染图谱导致后续决策质量下降。

### Q30 GraphRAG基本原理、实现方法、优势、适用场景

**核心**：文档→实体关系图→图遍历检索→结构化事实链生成。适用于多跳推理场景。

- 实现：离线 LLM 抽取实体+关系建图；在线实体定位→路径遍历→事实拼接→LLM 生成。优势：能处理多跳推理、召回结果逻辑连贯、上下文不稀释。适用场景：故障根因分析、法律案例推理、药物相互作用分析等关系密集且需要跨文档推理的场景。不适用：单事实查询和大规模开放域问答。

### Q31 GraphRAG工作原理、核心思想、与传统RAG的区别

**核心**：GraphRAG = 传统 RAG 的检索层用图遍历取代向量匹配，核心思想是让知识以结构化关系图的形式存在。

- 工作原理：建图（文档→实体→关系→图）→查询（query→定位实体→沿边遍历→收集事实链）→生成（LLM 基于结构化事实链回答）。与传统 RAG 区别：传统 RAG 召回孤岛片段，GraphRAG 召回推理链；传统 RAG 依赖语义相似度，GraphRAG 依赖实体间显式关系。前者适合单事实查询，后者适合多跳复杂查询。

### Q32 Agent系统长期记忆存储形式及大规模历史记录的检索优化

**核心**：向量索引（语义检索快）+ 分层存储（热/温/冷分层降成本）+ 摘要压缩（LLM 压缩历史降 token 量）。

- 存储形式：向量数据库存历史经历的 embedding（语义检索），结构化 DB 存元数据（时间/类型/重要性）；优化方法：分层存储——近期热记忆全量向量索引、中期温记忆摘要向量索引、远期冷记忆仅存概要；摘要压缩——用 LLM 将长对话历史压缩为结构化总结，减少存储和检索开销；混合检索——元数据过滤 + 语义检索 + 时间衰减权重。

### Q33 大模型Agent系统长期记忆存储形式和技术方案对比

**核心**：向量数据库（语义灵活）vs 结构化知识库（精确查询）vs 外部符号记忆（持久化外部存储），三者互补。

- 向量数据库：存 history embedding，语义检索灵活但精确查询弱，适合「类似情况怎么处理」类检索；结构化知识库：存实体/关系/事件，支持精确查询和推理，查询快但建库成本高；外部符号记忆：存对话摘要/笔记/RCA 文档等长期文本，持久化但检索效率依赖 RAG 管线。实践中三者组合用：结构化存元数据索引→向量做语义召回→外部符号记忆做原始证据回查。

> 🏷️ M14 | ⭐ 了解级别

### Q34 大模型幻觉现象的缓解方法

**核心**：数据（高质量训练数据）→架构（RLHF/事实性奖励）→推理（CoT/自我校验）→外部工具（RAG注入事实），四层防线。

- 数据层：提升训练语料的事实准确性和时效性；架构层：RLHF 中引入事实性奖励信号；推理层：Chain-of-Thought 逐步推导可减少跳跃性编造 + 自我校验让模型二次检查生成内容；外部知识层：RAG 注入检索到的真实文档约束生成→标注引用→二次校验 LLM 检查一致性。RAG 能治「知识缺失型」幻觉，治不了「推理型」幻觉——后者需要 CoT 等手段。

> 🏷️ M9 | ⭐ 了解级别

### Q35 大模型幻觉缓解：模型训练、推理机制和外部知识集成

**核心**：三大方向——训练时注入事实性信号→推理时让模型自我校验→推理时注入外部真实知识。

- 模型训练：高质量事实性语料 + RLHF 引入事实性奖励 + 对抗训练检测编造行为；推理机制：CoT 逐步推导减少跳跃编造 + 自我校验（让模型检查自己输出的逻辑一致性）+ 输出 token 概率校准（低概率处标记可能编造）；外部知识集成：RAG 注入真实文档约束 + 引用溯源 + 二次校验 + 门控写回防止错误知识回流污染。三者叠加比单点效果显著更好。
