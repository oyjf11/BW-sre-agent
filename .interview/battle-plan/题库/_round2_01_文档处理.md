### Q1 专业领域（医学/法律）长文档如何进行有效的文本切块？列举切分策略、原理、优缺点及场景适用性

**一句核心**：固定长度→滑动窗口+overlap→语义/结构切，专业领域首选按结构切（标题/章节/法条/病历节段），结构本身就是语义边界信号。

**展开**：
- 固定长度切分：最简单，按字符/token 直切，但会在句子中间切断语义——「该函数用于」和「处理支付回调」分到两个块
- 滑动窗口+重叠：overlap 10–20% 让相邻块共享内容，「处理支付回调」同时出现在前后块，提高命中概率
- 语义/结构切分：Markdown 按 ## 标题切、代码按函数/类切、法律按条款切、医疗按 SOAP 切——专业领域召回率比固定长度高 10–30%

**项目印证**：应用生成平台父子分块——父块=完整 plan 范式（不向量化只存），子块=同一范式的 3 种口语化场景描述入库；NestJS RAG 项目 CHUNK_SIZE=600、OVERLAP=80、MIN_CHUNK_LENGTH=20

> 🏷️ M2

### Q2 如何将分块后的文本转换为向量表示并构建高效向量库索引？说明嵌入模型和向量库的技术选型依据

**一句核心**：bi-encoder 把 chunk 编码成定长向量→归一化→写入向量库建 ANN 索引（HNSW/IVF），选型看领域匹配/维度/中英/MTEB 榜。

**展开**：
- 嵌入模型选型四要素：① 领域匹配（通用 BGE vs 医学/法律专用）② 维度（384/768/1024，高维更准更耗资源）③ 中英文（BGE-M3 中文比 OpenAI ada 强 10–20%）④ MTEB 榜对比
- 向量库选型：FAISS（Meta 库，C++底层 Python 封装，适合自建）、Milvus（分布式/混合检索/属性过滤，生产大规模）、Pinecone/Zilliz（全托管零运维）、小规模 pgvector/SQLite 够用
- ANN 索引：HNSW（图搜，快但吃内存）、IVF（聚类省内存）、PQ（量化极致省），常组合 IVF+PQ

**项目印证**：应用生成平台 text-embedding-v4 1024 维区分 query/document text_type；NestJS RAG 项目 SQLite 一张表存 JSON 向量全量算余弦——数据量小不上向量库

> 🏷️ M3 / M4

### Q3 文本分块策略如何影响 RAG 系统的检索精度、召回率和生成质量？结合案例对比分析

**一句核心**：块大=上下文全但噪声多稀释注意力；块小=精准但易断义丢指代——分块是 RAG 地基，切错后面全白做。

**展开**：
- 块大（1000+ token）：上下文完整但检索精度下降，LLM 注意力被无关内容稀释，token 成本高
- 块小（200 token）：精准匹配但语义断裂，「该方法」的指代丢失——small-to-big 解法：小块检索、大块喂 LLM
- 语义切分在专业领域的召回率比固定长度高 10–30%——结构本身就是语义边界，切分时天然对齐逻辑单元

**项目印证**：应用生成平台 small-to-big 父子分块——24 子块入向量库精准检索，命中后用 parent_id 取回 8 个完整父块喂 Planner；NestJS 项目先按段落分、段落太长再滑动窗口切

> 🏷️ M2

### Q4 RAG 系统中从原始文档输入到向量索引构建的完整数据处理流程，包括清洗、分块、嵌入、存储及索引优化

**一句核心**：离线建库四步——清洗（去噪声/格式化）→ 分块（overlap 防断义）→ embedding（bi-encoder 定长向量）→ 向量库建 ANN 索引（HNSW），环环相扣。

**展开**：
- 清洗：去 HTML 标签/页眉页脚、统一格式、处理表格/代码块、去重——脏数据进=脏检索出
- 分块：按文档类型选策略，overlap 10–20%，元数据附加（标题路径/文档类型/章节编号）
- embedding：选型后正常化（L2 norm=1），余弦=点积省除法运算；query 和 doc 必须同模型，换了模型必须全量重建
- 索引优化：HNSW 调 M 和 efSearch 参数平衡精度与延迟；增量追加时 HNSW 天然支持，IVF 新增大量文档后聚类中心偏移需重建

**项目印证**：应用生成平台完整 RAG 离线链——清洗→父子分块→text-embedding-v4 1024 维→向量库索引；回流闭环：成功 plan 经人工确认后写回向量库增量追加

> 🏷️ M1

### Q5 从原始文档到可检索知识库的完整数据处理与索引流程包含哪些关键步骤？

→ 同 Q4，侧重各阶段技术要点及对系统性能的影响。

**一句核心**：清洗质量决定检索下限、分块策略决定召回上限、embedding 选型决定语义对齐、索引结构决定查询延迟。

**展开**：
- 清洗直接影响：去不掉噪声→检索召回大量无关 chunk；格式不统一→分块切出乱码片段
- 分块策略影响：块大小直接 trade off 召回精度和上下文完整性；无 overlap 切到边界→正确答案被腰斩
- embedding 选型影响：领域不匹配→相似度失真；维度选择→存储+延迟成正相关
- 索引结构影响：HNSW 查询 O(log n) 但内存 O(n×M)；IVF 省内存但精度依赖聚类质量

**项目印证**：同 Q4——应用生成平台四步全落地 + 数据回流闭环

> 🏷️ M1

### Q6 如何将文本块转换为向量并构建高效检索索引？列举嵌入模型和向量库的特点及技术选型依据

**一句核心**：bi-encoder 独立编码→L2 归一化→ANN 索引入库；选 embedding 看领域/维度/中英/MTEB，选向量库看规模/运维/混合检索需求。

**展开**：
- 嵌入模型对比：BGE-M3（1024 维中英强）、text-embedding-v4（阿里云托管 1024 维）、OpenAI ada（1536 维英文为主）、国内可本地部署 text2vec 系列
- 向量库对比：FAISS（库不是数据库，需自管持久化，适合嵌入已有系统）、Milvus（分布式+混合检索+属性过滤，适合生产百万级+）、Pinecone（全托管，零运维，按量付费）
- 技术选型铁律：小规模（万级以下）pgvector/SQLite 精确最近邻够快，上 Milvus 徒增运维

**项目印证**：NestJS RAG 项目 SQLite 一张 `kb_chunks` 表存 JSON 向量全量算余弦；应用生成平台用托管 embedding 服务不自行部署模型

> 🏷️ M3 / M4

### Q7 如何利用嵌入模型和向量数据库实现用户问题与文档块之间的语义相似度匹配？说明实现流程和关键参数

**一句核心**：query 和 document 走同一个 bi-encoder 生成归一化向量 → 余弦相似度（≈点积）排序 → 向量库 ANN 索引加速 top-k 召回。

**展开**：
- 实现流程：用户 query → embedding 模型编码 → L2 归一化 → 向量库 ANN 检索 → 返回 top-k（k=50–200）按 cosine 相似度降序
- 关键参数：k 值（召回候选数，取太小漏正确答案、太大精排压力大）、cosine 阈值（低于 0.3 判定无关→降级）、向量维度（1024 维 256 字节存储/条）
- 协同机制：bi-encoder 双塔分开编码——document 向量离线索引入库，query 向量在线实时计算；归一化后余弦=点积省除法，向量库搜索更快

**项目印证**：应用生成平台 text-embedding-v4 区分 query/document 的 text_type 参数——用错 text_type 会降精度；NestJS RAG 项目 cosine 阈值 0.3 做召回过滤

> 🏷️ M3 / M4

### Q8 如何对文本分块并生成向量表示，利用向量数据库建立高效索引以支持精准检索？说明从预处理到索引优化的关键流程

**一句核心**：预处理→按结构切分+overlap→embedding 编码→向量库建 HNSW/IVF 索引，索引参数调优决定查询延迟和召回精度。

**展开**：
- 预处理：去 HTML/特殊字符/页眉页脚，统一换行符，保留代码块/表格结构
- 分块+向量化：按文档类型选切分策略，embedding 模型选型四要素，批量请求限流（25 条/批 200ms 间隔）
- 索引入库：每 chunk 附带 metadata（标题路径/文档类型/章节/domain），建 ANN 索引时选合适参数——HNSW 的 M（每层连接数）和 efConstruction 平衡精度与内存

**项目印证**：NestJS RAG 项目 SiliconFlow BAAI/bge-m3 批量请求 25 条/批+200ms 限流；SQLite 表 `kb_chunks` 存 JSON 向量；应用生成平台 text-embedding-v4 1024 维

> 🏷️ M2 / M3 / M4

### Q9 向量索引是关键组件，列举提升索引效率与召回率的常见优化策略

**一句核心**：HNSW 调参（M/efSearch）、IVF 聚类优化、PQ 量化压缩、混合检索（向量+BM25）、增量更新、元数据预过滤。

**展开**：
- HNSW 调参：M 控制每层连接数（越大精度越高内存越大）、efSearch 控制搜索宽度（越大精度越高延迟越高）；小规模 M=16、大集群 M=48+
- IVF+PQ 组合：IVF 聚类分桶只搜最近 m 个桶 + PQ 量化压向量省内存——极致省资源但精度略降
- 混合检索：向量（语义）+ BM25（精确术语）两路互补，RRF 融合免调权重——覆盖同义词和精确匹配的盲区
- 增量更新：新文档单独 embedding 追加，HNSW 天然支持增量插入，IVF 大量增后需重建聚类

**项目印证**：NestJS RAG 项目混合检索落地——向量 cosine + SQLite FTS5 → RRF 融合（K=60）→ topK=5；应用生成平台向量 + BM25 + 元数据硬过滤

> 🏷️ M4

### Q10 分块策略如何设计以平衡文本上下文完整性与信息检索精确性？有哪些最佳实践？

**一句核心**：small-to-big 父子分块是黄金解法——小块检索精准匹配 + 大块喂 LLM 保上下文完整；overlap 10–20%、按结构切是通用最佳实践。

**展开**：
- 最佳实践一——父子分块：子块（200–500 token 口语化描述）入向量库用于检索，父块（完整 plan/章节）不向量化只存，命中后用 parent_id 取回
- 最佳实践二——按结构切：专业文档按标题/章节/条款切，overlap 10–20% 防边界切断——结构天然是语义边界
- 最佳实践三——元数据附加：每个 chunk 带标题路径/文档类型/复杂度等结构化字段，检索时可硬过滤，生成时可溯源

**项目印证**：应用生成平台 8 父块（plan 范式）× 3 子块（口语场景） = 24 子块入向量库，检索打子块、回取父块；聚合 parent_id MAX 分数

> 🏷️ M2

### Q11 在 RAG 系统中如何合理地对长文档进行文本切块？比较不同切块策略的优劣

**一句核心**：固定长度（简单但断义） < 滑动窗口+overlap（保上下文） < 语义/结构切（最优但需领域知识）；无万能粒度，按场景调。

**展开**：
- 固定长度：实现成本最低，但会在句子中间切断语义，丢失指代关系——适合短文档/摘要类场景，不适合专业领域
- 滑动窗口+overlap：overlap 10–20% 让关键信息在两块中同时出现，召回率提升，但索引膨胀——适用通用文档
- 语义/结构切：召回率最高，但需要文档有清晰结构——适合 Markdown/HTML 文档、法律条文、医疗病历等结构化内容

**项目印证**：NestJS RAG 项目组合策略——先按段落分，段落太长再滑动窗口切（600 字符/80 overlap）；应用生成平台父子分块按范式结构性切

> 🏷️ M2

### Q12 如何设计数据清洗流程？常见文本切块策略有哪些，各自适用什么场景？

**一句核心**：清洗=去噪→去重→格式化→结构化保留；切块策略按场景选——通用用滑动窗口+overlap，专业用结构切，检索精度要求高用 small-to-big。

**展开**：
- 清洗流程：① 去 HTML 标签/特殊字符/页眉页脚 ② 去重（精确重复/近重复）③ 统一格式（换行符/空白）④ 保留表格/代码块原有结构——不破坏语义
- 固定长度：场景：短文档/FAQ/技术摘要；优点：极简；缺点：断语义
- 滑动窗口+overlap：场景：通用长文档/技术博客；优点：保上下文；缺点：overlap 控制不当索引膨胀
- 语义/结构切：场景：Markdown 文档/API 文档/法律合同/医疗病历；优点：召回率高；缺点：依赖文档结构质量

**项目印证**：NestJS RAG 项目清洗含去 HTML/空格规范化后按段落+滑动窗口切；应用生成平台知识库只在录入时做格式化清洗+手工校审

> 🏷️ M2

### Q13 如何根据任务需求选择合适的文本嵌入模型？chunk 向量化后存入向量数据库的完整流程是什么？

**一句核心**：选 embedding 看领域匹配+维度+中英+MTEB 榜；向量化流程=批量编码→归一化→带 metadata 写入向量库。

**展开**：
- 选型决策树：① 通用中英→BGE-M3/bge-large-zh ② 阿里云生态→text-embedding-v4 ③ 英文为主→OpenAI ada ④ 本地化需求→text2vec 系列
- 向量化流程：chunks 分批（25 条/批）→ embedding API 编码 → L2 归一化（norm=1）→ 拼装 `{id, vector, text, metadata}` → 批量写入向量库
- 关键约束：query 和 doc 必须用同模型（不同模型向量空间不同，横比无意义）；多语言场景区分 text_type 参数

**项目印证**：应用生成平台选用 text-embedding-v4 (1024 维)，区分 query/document text_type；NestJS RAG 用 SiliconFlow BAAI/bge-m3，批量 25 条/批+200ms 间隔限流

### Q14 搭建 RAG 系统知识库的主要流程，包括文档预处理、分块、向量化存储及检索索引构建

**一句核心**：离线建库链=预处理→分块(overlap)→embedding→向量库建 ANN 索引；在线检索链=query 改写→混合检索→rerank→拼 context→LLM 生成。

**展开**：
- 离线四步：① 清洗去噪格式化 ② 分块+overlap+元数据附加 ③ embedding 编码+归一化 ④ 向量库建 HNSW/IVF 索引
- 在线六步：① query 改写（术语映射补 vocabulary gap）② 混合检索（向量+Bm25）③ 粗排 top-k ④ 精排 cross-encoder top-n ⑤ 拼 context ⑥ LLM 生成+引用溯源
- 索引构建注意：HNSW 天然支持增量插入，IVF 大量增后需重建聚类；换 embedding 模型必须全量重建旧向量

**项目印证**：应用生成平台完整 RAG 链路——query 改写 58%→84%→混合检索+元数据过滤→small-to-big→rerank→反幻觉 prompt（5 条）→成功 plan 回流语料

> 🏷️ M1

### Q15 构建 RAG 系统中知识库的完整流程，包括预处理、分块、嵌入模型选择、向量库选型与索引构建

**一句核心**：知识库=预处理后的向量化文档集合+ANN 索引+元数据过滤层，是 RAG 的回答能力的物理载体。

### Q16 如何对专业领域文档（医学文献/法律条文）进行有效的文本切块？

→ 同 Q1，侧重医学法律领域的特殊性。

**一句核心**：医疗按 SOAP（主诉/检查/评估/计划）切、法律按条款/章节切——结构切法在专业领域的召回率比固定长度高 10–30%。

**展开**：
- 医学文献：按 SOAP 结构切保证主诉、检查结果、诊断评估各自独立，检索「胸痛原因」不会混入无关的评估段落
- 法律条文：按「编→章→条」层级切，每条独立一个 chunk，metadata 带层级路径——检索「合同解除条件」精准定位
- 额外考虑：术语密度高，embedding 模型需要该领域的微调版本；子块用专业术语的多种口语化同义表述提升检索覆盖率

**项目印证**：应用生成平台范式有明确结构（plan_steps + pitfalls），天然按语义单元切——正是专业领域结构切法的实践版

> 🏷️ M2

### Q18 构建面向专业领域的 RAG 系统时，如何对原始文档进行清洗和预处理以提升检索与生成质量？

**一句核心**：专业领域清洗不是简单去标签——要保留领域结构（表格/公式/引用）、术语归一化、去页眉页脚不过滤章节编号。

**展开**：
- 结构保留：医学文献保留表格（化验结果）、法律条文保留层级编号（第 X 编第 Y 章）——这些是检索信号不是噪声
- 术语处理：领域缩写归一化（「DM」→「糖尿病」/「糖尿病 mellitus」）、同义词映射——否则 embedding 模型无法对齐

- 格式化：统一引用格式、统一计量单位、公式转 LaTeX 保语义可检索

**项目印证**：应用生成平台知识库录入时做格式化清洗+手工校审——专业范式语料的准确性是 RAG 地基

### Q19 文档的 chunk 划分策略有哪些常见方法？固定长度、语义边界分割、滑动窗口等各自的优缺点及适用场景

**一句核心**：三大策略递进——固定长度（极简但断义）→ 滑动窗口+overlap（保上下文，通用首选）→ 语义/结构切（精准，专业领域首选）。

**展开**：
- 固定长度：优点=实现零成本、可预测；缺点=句子中间切断语义；场景=短文档/快速原型
- 滑动窗口：优点=overlap 10–20% 保语义边界+提高命中率；缺点=索引膨胀（块数增加）；场景=通用文档/技术博客
- 语义边界分割：优点=对齐文档逻辑单元、召回率高；缺点=依赖文档结构质量、实现成本最高；场景=Markdown/API 文档/法律/医疗
- 父子分块（进阶）：小块检索（语义切）+大块喂 LLM（结构切），解了「检索要短准」和「生成要长全」的矛盾

**项目印证**：NestJS RAG 项目滑动窗口 CHUNK_SIZE=600 OVERLAP=80 MIN_CHUNK_LENGTH=20；应用生成平台父子分块=语义切+结构切组合

### Q20 如何设计文本分块策略以平衡上下文完整性和检索精确性？常用方法和权衡考量

### Q25 在 RAG 系统中如何优化文档的分段策略以提高检索质量和生成效果？讨论分段长度、重叠策略和语义完整性的权衡

**一句核心**：三段杠杆——① 粒度按场景定最优块大小 ② overlap 控制在 10–20% 防断义 ③ 按语义边界切而不是硬切长度——三个维度单变量调优+评测验证。

### Q26 在 RAG 系统中对文档或查询进行向量化处理的流程，包括文本预处理、嵌入模型选择、向量生成及归一化

**一句核心**：文本预处理→选 embedding 模型（四要素决策）→ API 编码生成向量→L2 归一化（norm=1）使余弦=点积——query 和 doc 必须同模型。

**展开**：
- 预处理：去噪声但保留语义（去 HTML 标签保留文本，去过多空格保留换行）、截断超长文本到模型最大输入长度
- 嵌入模型选型：看领域（通用/医学/法律）+ 维度（1024是中英甜点）+ MTEB 榜；多语言场景区分 text_type 参数
- 向量生成：批量请求（25 条/批）+ 限流（200ms 间隔），失败的批重试
- 归一化：L2 norm=1 后余弦相似度=向量点积，省去除法运算——所有主流向量库假设输入为归一化向量

**项目印证**：应用生成平台 text-embedding-v4 1024 维，区分 query/document text_type；NestJS RAG BAAI/bge-m3 批量+限流

> 🏷️ M3

### Q27 详细说明 RAG 系统中文档和查询的向量化过程，包括嵌入模型、分块策略、向量归一化、向量匹配原理及优化手段

**一句核心**：文档侧——分块后批量 encoding+归一化入库；查询侧——改写后 encoding+归一化在线计算；匹配=余弦相似度≈点积，ANN 索引加速 top-k 召回。

**展开**：
- 分块→向量化链路：文档分块（overlap 10–20%）→ 每块独立编码 → L2 归一化 → 带 metadata 写入向量库
- 查询向量化：query 先改写（术语映射补 vocabulary gap）→ 同 embedding 模型编码 → L2 归一化 → 向量库 ANN 检索
- 向量匹配原理：归一化后余弦相似度 = q · d（向量点积），值域 [−1, 1]，越大越相关。ANN 索引（HNSW/IVF）用精度换速度，不以穷举方式找最近邻
- 优化手段：混合检索（向量+BM25 双路互补）→ RRF 融合；两阶段 rerank（粗排 cosine 召回→精排 cross-encoder 重打分）

**项目印证**：应用生成平台完整链路——query 改写（58%→84%）→ text-embedding-v4（1024 维）→ 向量检索+BM25+元数据过滤→RRF→rerank

> 🏷️ M3
