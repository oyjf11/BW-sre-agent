### A1–A10 · 后端高危 kernel（浓缩自《高危题标准答案.md》，每条 90 秒答完）
| # | 触发 | 核心 kernel（背这句） |
|---|---|---|
| A1 | FastAPI async 底层 | Uvicorn 起一个 asyncio Event Loop，async def 是协程，遇 await 让出 CPU，I/O 完恢复 → 单线程并发上万。LLM 是 I/O 密集异步够用；**同步 DB 驱动会阻塞整个 loop**，要 `run_in_executor`。 |
| A2 | SSE 断了任务停吗 | **不停。连接生命周期 ≠ 任务生命周期。** 任务在后台线程 + 状态落 DB，断开只关推流通道；重连靠 `Last-Event-ID` 从 DB 补发（OpsPilot 当前用 asyncio.Queue 会丢，是已知待补项）。 |
| A3 | Worker 崩溃恢复 | **Task Lease**：RUNNING 任务有 `lease_expires_at`，Worker 每 10s 续租；崩溃不续租 → watchdog 扫 `lease<now()` 重置 PENDING 重领。RUNNING 状态本身不会过期，所以必须要 Lease。 |
| A4 | 并发改 State / Lost Update | **乐观锁**：`UPDATE ... SET version=version+1 WHERE id=:id AND version=:old`，影响行数=0 即冲突→退避重试。高冲突场景才用悲观锁 `FOR UPDATE`。 |
| A5 | 工具调用何时能重试 | **幂等才能重试**。`idem_key=sha256(run_id+tool+input)`，重试命中同键返缓存。超时/429/500 可重试；400、已吐流式 token 的 LLM、不可逆操作必须带幂等键。 |
| A6 | LangGraph checkpoint | ⚠️**分两层答**。通用：thread_id 主键，CheckpointTuple={channel_values 全量快照 / channel_versions 版本向量 / versions_seen 防重 / pending_writes}。**但你项目是自建**（见 A9）！ |
| A7 | Pydantic 反序列化变 dict | checkpoint 存 JSON，回来 Pydantic 变 dict，直接 `.attr` 抛 AttributeError → `x.f if hasattr(x,'f') else x.get('f','default')`；遍历 list 先防 None。 |
| A8 | K8s Pod 重启状态恢复 | **状态外置到 DB**。内存态(InMemorySaver/Queue)全丢 → 生产换 RedisSaver + Redis Pub/Sub；Worker 启动扫过期 Lease 重领，`graph.invoke(None, thread_id)` 从 checkpoint 续跑。 |
| A9 | 1000 并发 Agent 架构 | **API 与 Worker 解耦**：POST 写 tasks 表(PENDING)秒回；Worker 池 `FOR UPDATE SKIP LOCKED` 抢任务；单 Worker `Semaphore(20)` 限并发 LLM；RedisSaver 共享 checkpoint；Redis Pub/Sub 跨实例推 SSE。 |
| A10 | Saga 补偿一致性 | 每步配正向+补偿，失败按反向补偿（关 MR→删分支→回滚部署）。**业务级尽力而为，非精确回滚**，补偿也要幂等。不用 2PC：外部 API 不支持、性能差。 |

> 🔴 **A6/A9 的致命陷阱**：OpsPilot 用 0.0.55，**没有原生 `interrupt()`/`Command`/`PostgresSaver`/`Send`**。被问「你项目里 checkpoint/HITL 怎么做的」**必须答自建**：compile 时 `checkpointer=None`，自己逐节点把 state_json 落 `IncidentCheckpoint` 表；审批走 `approval_interrupt→END`，人工后从自建 checkpoint 读回 state、设 `_resume_from_node` 重新 astream、白名单路由 + executor 幂等防重跑。按原生答会被当场戳穿。详见 `dispatcher-resume机制.md`。

## 🅱️ B 组 · ⭐⭐⭐ 核心母题（理解到能展开 2 层）

### —— RAG 子系列（题库 ~199 题压成 9 个）——

**M1 · RAG 全流程**（最高频，~30 次！）
- 触发：完整流程 / 整体架构 / 从文档到回答 / 知识库搭建流程 / RAG 实现细节
- 核心：**离线**[清洗→分块(overlap)→embedding→向量库(HNSW)] **在线**[query 改写→混合检索(BM25+向量)→召回 top-k →cross-encoder 精排 top-n →拼 context→LLM 生成+引用/校验]
- 钩子：每一段都通向一个母题（分块=M2，向量化=M3，库=M4，混检=M5，rerank=M7）
- 锚点：应用生成平台「场景化 RAG 意图增强」就是这条链的简化版

**M2 · Chunking 策略**（~15 次）
- 触发：文本切块 / 分块策略 / 专业领域(医学法律)长文档 / 粒度与重叠
- 核心：**固定长度**(简单但切断语义)→**滑动窗口+重叠**(保上下文，overlap 10-20%)→**语义/按结构切**(标题/段落，专业领域首选)。权衡：块大=上下文全但召回噪声多，块小=精准但易断义。
- 钩子：父子分块 / small-to-big / 元数据(标题路径)随块存

**M3 · Embedding 选型 + 向量化**（~15 次）
- 触发：嵌入模型怎么选 / Sentence-BERT / 向量化流程 / 维度影响 / 归一化
- 核心：双塔 bi-encoder(SBERT/BGE) 把句子编码成定长向量；**归一化后用余弦≈点积**。选型看：领域匹配、维度(384/768/1024，高维更准但更耗)、中英、MTEB 榜。query 和 doc 必须同模型。
- 钩子：embedding 升级 → 旧向量要全量重算(新旧不可混用)

**M4 · 向量库 + ANN 索引**（~12 次）
- 触发：向量数据库选型 / FAISS Milvus Pinecone / HNSW IVF / 索引优化
- 核心：**HNSW**(图，查得快内存大，主流)、**IVF**(倒排聚类，省内存)、**PQ**(量化压向量省内存掉精度)。选型：FAISS(库/自建)、Milvus(分布式)、Pinecone(托管)。本质是**用精度换速度的近似最近邻**。
- 钩子：召回精度 vs 延迟 vs 内存的三角权衡

**M5 · 混合检索 + RRF**（~12 次）
- 触发：为什么混合检索 / sparse+dense / BM25+向量 / 融合
- 核心：**Sparse(BM25)** 抓精确词项(术语/ID/罕见词)，**Dense(向量)** 抓语义(同义/改述)，两者互补。用 **RRF**(倒数排名融合，`score=Σ1/(k+rank)`)合并，免调权重。
- 钩子：→ BM25 细节走 M6

**M6 · BM25 原理**（~8 次）
- 触发：BM25 公式 / vs TF-IDF / 词频饱和 / 长度归一化
- 核心：在 TF-IDF 上加两件事——**词频饱和**(k1，词出现再多收益递减)+**文档长度归一化**(b，长文不占便宜)。比 TF-IDF 更稳的稀疏检索。
- 钩子：参数 k1≈1.2-2、b≈0.75

**M7 · Rerank / 两阶段检索**（~20 次！）
- 触发：为什么要重排 / 召回-精排两阶段 / 为什么不直接精排全量 / cross vs bi-encoder
- 核心：**bi-encoder** 双塔分开编码、能预存向量→快但糙(召回)；**cross-encoder** query+doc 拼一起过 BERT→准但慢(精排)。**全量跑 cross 算力爆炸**，所以先召回 top-k 再精排 top-n。算力换精度的经典分层。
- 锚点：和你前端测试的「先快筛范围→再精跑用例」是同一种分层思想

**M8 · 召回质量优化**（~15 次）
- 触发：怎么提升检索相关性 / query 改写 / 上下文压缩 / 召回不准
- 核心：**query 端**(改写/扩展/HyDE 假设文档/多查询)→**检索端**(混检+rerank)→**context 端**(压缩去噪、去重、按相关性排序)。
- 钩子：语义相似但实际不相关 → 加 rerank + 元数据过滤

**M9 · RAG 降级 / 增量更新 / 幻觉缓解**（~15 次）
- 触发：检索不到怎么办 / fallback / 索引增量更新 / RAG 怎么缓解幻觉
- 核心：**降级**=阈值兜底(<阈值就答"知识库无相关，转人工/纯模型"+标注低置信)。**增量**=新文档单独 embedding 追加，避免全量重建。**缓解幻觉**=把外部事实塞进 context 约束生成 + 要求带引用 + 二次校验。
- 钩子：RAG 治"知识缺失型"幻觉，治不了"推理型"幻觉(那是 RL/CoT 的活)

### —— Agent / Prompt 子系列（题库 ~220 题压成 8 个）——

**M10 · Agent 核心组件**（~15 次）
- 触发：Agent 由哪些组件构成 / 基础能力 / 规划记忆工具反思
- 核心：**四件套——规划(拆任务)+记忆(短期上下文/长期向量库)+工具调用(感知外部)+反思(自我纠错)**。LLM 是大脑，工具是手脚，记忆是状态，编排是骨架。
- 锚点：OpsPilot 13 节点正是把这四件套拆成可验证的确定性流程
- 扩展阅读：`harness知识地图.md` 主题1（Harness 三层架构/六大组件，作为这四件套的更上层背景框架）

**M11 · ReAct 框架**（~15 次！）
- 触发：ReAct / Reasoning+Acting / Thought-Action-Observation / 比纯 prompt 强在哪
- 核心：**Thought(推理)→Action(调工具)→Observation(看结果)→循环**，直到收敛。比纯 prompting 强：能拿真实反馈纠偏，不靠一次性瞎编。
- 钩子：对比 Plan-and-Execute(先规划全程再执行) → 走 M12

**M12 · 规划方法**（~10 次）
- 触发：CoT / ToT / GoT / 提升规划能力 / Plan-and-Execute vs ReAct
- 核心：**CoT**(线性思维链)→**ToT**(树，多路探索+回溯)→**GoT**(图，节点可合并)。**ReAct**(边想边做、反应式)vs **Plan-Execute**(先出完整计划再批量执行，省 token 但纠错弱)。
- 锚点：OpsPilot 的 planner 节点 + critic 回边 = 带反思的规划

**M13 · 结构化输出 / 封闭标签**（~12 次）
- 触发：稳定输出 JSON / schema 约束 / 封闭标签分类 / 输出格式不稳
- 核心：**四层兜底**——①prompt 给 schema+few-shot ②约束解码/JSON mode(从源头保证合法)③Logit Bias 限定 token(分类)④后处理校验+失败重试。生产级要 ①+②+④。
- 锚点：应用生成平台 Planner 执行前做「结构化输入校验」，错了回退 Router

**M14 · 工具调用链路 + 安全**（~12 次）
- 触发：工具怎么定义注册调用 / function calling / 工具选择 / 调用安全
- 核心：**定义(JSON schema)→注册→LLM 选工具+提参→网关校验→执行→结果回灌→错误处理**。多工具选择靠打分/成本评估。安全=**权限控制+参数校验+审计日志+高危人工审批**。
- 锚点：OpsPilot 的 Tool Gateway(schema 校验+重试+审计) + risk_gate(写操作/生产/删数据必走审批)

**M15 · Prompt 设计 + 优化评估**（~15 次）
- 触发：prompt 设计原则 / 优化技巧 / few-shot 何时用 / 怎么判断 prompt 到上限
- 核心：**原则**=角色+任务+约束+格式+示例；**技巧**=few-shot(任务复杂/要对齐格式时)、CoT(需多步推理时)。**评估**=Golden Case 集 + 自动指标(准确率/格式合规率)+ 人工，指标不再涨即近上限。
- 锚点：你做意图识别就是用 Golden Case 评准确率

**M16 · 多 Agent 系统**（~12 次）
- 触发：多 Agent 协作 / Reflection / Memory / agent 冲突 / 单 vs 多 Agent
- 核心：**Reflection**=对自己输出做批判再改；**Memory**=共享状态/历史。**冲突**靠仲裁/投票/角色优先级解。单 Agent 够用别上多 Agent(通信开销+冲突)。
- 锚点：OpsPilot critic 节点 = 单 Agent 内的 Reflection 落地(专门攻击 diagnose 结论)
- 扩展阅读：`实战-2026-06-10-critic仲裁打分机制.md`（Reflection 的完整打分公式：质量分=avg置信度×截断惩罚×覆盖度，4路裁决 CONTRADICTION>NEED_MORE_EVIDENCE>REPLAN>PASS + loop guard 转人工）；deep agent 整体框架见下方「扩展母题候选」候选③

**M17 · 框架对比**（~10 次）
- 触发：LangChain vs LlamaIndex / LangGraph 优势 / 框架选型
- 核心：**LangChain**=通用链式编排；**LlamaIndex**=RAG/索引特化；**LangGraph**=有状态图、支持环和条件路由、适合复杂可控 Agent。
- 锚点：OpsPilot 选 LangGraph 是要「确定性图编排包裹非确定性 LLM」+ critic 回边自我修正
- 扩展阅读：「shallow agent(纯ReAct循环) vs deep agent(循环外加规划/文件系统/子agent/长期记忆/HITL)」框架，见下方「扩展母题候选」候选③ —— OpsPilot 是手搓的 deep agent，这是「为什么选 LangGraph 不选轻量框架」的更深一层答案

### —— 评估 子系列（题库 ~50 题压成 3 个）——

**M18 · RAG/Agent 评估体系**（~30 次！评估是隐藏高频）
- 触发：怎么评估 RAG/Agent / Recall@k / MRR / NDCG / 离线在线指标
- 核心：**检索段**=Recall@k(召回全不全)、MRR(第一个对的排多前)、NDCG(带位置权重的排序质量)；**生成段**=忠实度/相关性/正确性(常用 LLM-as-judge)；**系统**=延迟/成本。离线刷数据集 + 在线 A/B。

**M19 · RAG 错误归因**（~10 次）
- 触发：RAG 效果差 / 怎么定位是检索还是生成的锅
- 核心：**控制变量**——把「人工塞对的文档」喂给生成器：还差→**生成的锅**(调 prompt/上下文组织/微调)；喂对就好→**检索的锅**(调分块/embedding/rerank)。
- 锚点：和你前端测试的「失败五分类归因」是同一套控制变量思路

**M20 · 评估数据集构建**（~8 次）
- 触发：评估数据集怎么建 / 标注流程 / 负样本 / 多样性
- 核心：问题来源(真实日志+专家造)→答案标注(双人+仲裁)→负样本(难负例)→保多样性和难度分层→可复现。

**候选① · Harness 控制面 vs 写路径治理**（挂在 A6/A9 下）

- 锚点：`实战-2026-06-15-Harness控制面vs写路径治理.md`

**候选② · 如何评测一个 Agent（评测方法论总纲）**（挂在 M18 下）
- 触发：评测体系怎么从 0 搭起来 / 为什么评测会越搭越复杂
- 核心：评测的复杂度是被「对错有多难判断」一步步逼出来的——需要裁判 → 主观/客观分离(客观能用代码判的绝不用模型判) → 三层裁判塔(干活的AI/判卷器/金标集，金标集用 kappa 校准判卷器) → 题库(含陷阱题/行为题/安全题) → gold set → 冻结世界(评测可重复的前提) → 北极星向量(Precision/Recall/有效覆盖/维护成本，**不合并成一个分数**，防 Goodhart)。
- 锚点：`实战-2026-06-14-如何评测一个agent方法论.md`（8步框架母本）+ `推导链-2026-06-15-什么是harness与怎么搭评测.md` Chain B
- 具体案例（三类题怎么落地的活教材）：`e2e测试/何为自愈？.md`——自愈agent该考什么：真红/假红分诊准确率(业务题)、只修假红不碰断言(行为题+防作弊闸门当判卷器)、抗注入删断言(安全题)；"假红=定位过期"这个判断本身就是 Inferential 检查(主题3 Computational vs Inferential 的具体例子)。

**候选③ · Deep Agent Harness 范式**（挂在 M16/M17 下）
- 触发：deep agent 和 shallow agent 区别 / OpsPilot 算不算 deep agent
- 核心：**Shallow agent** = 纯 ReAct 工具循环；**Deep agent** = 循环外面再加 4 件——规划(Planning)、文件系统当外部上下文(卸载窗口)、子 agent(隔离上下文并行)、长期记忆+HITL。OpsPilot 逐项对应：planner 节点=Planning，evidence_fanout=Sub-agents，evidence 持久化到 DB/OSS=File system，risk_gate+approval_interrupt=HITL，retrieve_memory+RAG=Long-term memory，critic 回边=自我纠错(即 M16 的 Reflection)。

## 🅳 D 组 · ⭐ 不露怯就行（一行 kernel，别背细节）

- **M23 Transformer/Attention**：self-attention=QKV，`softmax(QKᵀ/√d)V`；多头=多子空间并行；位置编码 RoPE(旋转，主流)/ALiBi。
- **M24 归一化**：LayerNorm 对单样本归一(不依赖 batch、适合变长序列)，所以 Transformer 用它不用 BatchNorm；**RMSNorm** 去掉均值更省算力，LLaMA 系主流。
- **M25 Decoder-only vs Enc-Dec**：GPT(decoder-only，自回归生成)；BERT(encoder，双向理解)；主流大模型走 decoder-only，训练效率+生成统一。
- **M26 LoRA/PEFT**：冻结原权重，旁路加**低秩矩阵 A·B**，只训这部分，省显存。局限→DoRA/Adapter/IA³。
- **M27 SFT**：在预训练上用指令数据对齐。坑：**复读机**(重复样本)、**置信度下降/灾难性遗忘**。
- **M28 损失函数**：分类用**交叉熵**(梯度好)不用 MSE；类别不平衡用 **Focal Loss**(γ 压低易样本权重)。
- **M29 RLHF 对齐**：**PPO**(在线、要 reward+ref model、稳但重)、**DPO**(免 RM 直接用偏好对、轻)、**GRPO**(组内相对优势、省 critic)。**Reference Model** 不训练，用 KL 约束防跑偏。
- **M30 奖励设计/Long Horizon**：奖励要防稀疏/延迟/reward hacking；长程任务靠**分层 RL+过程奖励+课程学习**解信用分配。
- **M31 多模态**：VLM=视觉编码器(ViT)+连接器(Q-Former/Adapter)+LLM；视频要切帧(均匀/关键帧)；多模态 RAG 难在跨模态对齐。
- **M32 扩散模型**：DDPM 随机采样时间步加噪学去噪(稳定+全程覆盖)；Flow Matching(更直的生成路径)；MoE 按信噪比分高噪/低噪专家。
