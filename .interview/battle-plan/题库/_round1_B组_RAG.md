## M1：RAG 全流程

**Kernel**：离线[清洗→分块(overlap)→embedding→向量库(HNSW)] 在线[query 改写→混合检索(BM25+向量)→召回 top-k →cross-encoder 精排 top-n →拼 context→LLM 生成+引用/校验]

**展开**：

### Layer 1 — 为什么 / 背景

LLM 有两大先天缺陷：① 知识截止日期（训练数据有终点）② 幻觉（会编造不存在的事实）。RAG 的本质是把「外部可信知识」在推理时注入 LLM 的上下文窗口，用检索换记忆——让 LLM 站在真实文档的肩膀上回答问题，而非依赖训练时的记忆。这比微调便宜（不用重训模型）、比纯提示工程可靠（有据可查）。

RAG 拆成两条线：**离线建库**（把文档变成可检索的向量索引）和**在线检索生成**（用户问问题 → 检索相关文档 → 拼成上下文 → LLM 生成带引用的答案）。

### Layer 2 — 怎么做 / 机制

**离线链**：
1. **清洗**：去 HTML 标签、去页眉页脚、统一格式、处理表格/代码块
2. **分块（chunking）**：把长文档切成语义完整的片段，overlap 10–20% 防切断语义边界 → M2
3. **embedding**：用 bi-encoder（如 BGE/text-embedding-v4）把每个 chunk 编码成定长向量 → M3
4. **索引入库**：向量写入向量数据库，建 ANN 索引（HNSW/IVF） → M4

**在线链**：
1. **query 改写**：把用户口语/模糊问题转成适合检索的表达（术语映射、去口语噪声、拆解多意图）→ M8
2. **混合检索**：向量检索（语义相似）+ BM25 关键词检索（精确匹配），RRF 融合 → M5 M6
3. **粗排（召回）**：取 top-k（如 k=50–200），快但不够准
4. **精排（rerank）**：cross-encoder 对粗排结果逐对打分，取 top-n（如 n=5–10），准但慢 → M7
5. **拼 context**：把最终的 top-n chunk 按相关性排序、去重、拼接，塞入 prompt
6. **生成 + 引用/校验**：LLM 基于 context 生成答案，要求标注引用来源；可选二次校验（用 LLM 检查生成内容是否与引用文档一致）

### Layer 3 — 坑 / 边界 / 追问预判

- **「检索到 ≠ 问题答对了」**：检索 top-1 命中不代表生成的答案正确，需要分开评估检索质量和生成质量（控制变量法 → M19）
- **「加了 RAG 幻觉还在」**：RAG 只能治「知识缺失型」幻觉（模型不知道的事实），治不了「推理型」幻觉（知道事实但推错了）。后者需要 CoT/RL/校验 → M9
- **「chunk 大小是 tradeoff」**：切太大 → 召回相关段落但噪声多（LLM 窗口浪费）；切太小 → 精准但丟上下文（「那个函数」指代不明）。需要 small-to-big 策略：用小块检索、大块喂 LLM
- **「embedding 模型换了就要全量重建」**：新旧 embedding 不可混合检索——向量空间不同，余弦相似度无意义 → M3 钩子
- **「query 改写收益最大也最容易被忽略」**：用户口语和知识库书面语的词汇鸿沟（vocabulary gap）是检索质量的第一杀手，术语映射 ROI 极高（实测 58% → 84%）

### Layer 4 — 同类变式

题库中约 30 题属于此母题，触发词：完整流程 / 整体架构 / 从文档到回答 / 知识库搭建流程 / RAG 实现细节 / RAG 和微调区别 / RAG 怎么提升回答质量。

**变式套路**：被问「RAG 整体架构」→ 直接拉离线/在线两条线；被问「RAG vs 微调」→ RAG 知识可更新不重训、可溯源、成本低；微调改变模型行为、适合风格/格式对齐。两者互补不互斥。

**项目印证**：
- 应用生成平台：完整 RAG 链路——query 改写（术语映射 58% → 84%）→ 混合检索（向量 + BM25 + 元数据过滤）→ small-to-big 父子分块（小块检索、大块喂 Planner）→ rerank → 反幻觉 prompt 约束（5 条）→ 成功 plan 回流语料。RAG 结果是「参考」不是「指令」，Planner 有权否决低分命中。
- OpsPilot：`retrieve_memory` 节点用语义 + 元数据过滤检索历史 RCA，作为 context 注入 planner；`writeback_knowledge` 节点将人工确认的 RCA 写回向量库形成闭环。

## M2：Chunking 策略

**Kernel**：固定长度(简单但切断语义)→滑动窗口+重叠(保上下文，overlap 10-20%)→语义/按结构切(标题/段落，专业领域首选)。权衡：块大=上下文全但召回噪声多，块小=精准但易断义。

**展开**：

### Layer 1 — 为什么 / 背景

LLM 的上下文窗口有限（即便现在 128K/1M token，塞太多也会稀释注意力、增加延迟和成本）。原始文档不能直接扔给 embedding 模型——一整本书的向量等于噪声。必须把文档切成合适粒度的片段，每个片段独立编码、独立检索。

切块是 RAG 的「地基」：切得不好，后面的 embedding、检索、rerank 全是白做。

### Layer 2 — 怎么做 / 机制

三种主流策略，复杂度递增：

1. **固定长度切分**：按字符/token 数直接切。最简单的，但会在句子中间切断，「该函数用于」和「处理支付回调」分到两个块，语义断裂。

2. **滑动窗口 + 重叠（overlap）**：在固定长度基础上加 overlap（10–20%），让相邻块共享一部分内容。例：块长 600 字符、overlap 80 字符。这样「处理支付回调」会同时出现在前一块的末尾和后一块的开头，提高了被命中的概率。我的 NestJS RAG 项目用的就是 CHUNK_SIZE=600、OVERLAP=80。

3. **语义/结构切分**：按文档的天然结构切——Markdown 按 ## 标题切、代码按函数/类切、法律合同按条款切。专业领域首选，因为结构本身就是语义边界的信号。

**进阶策略**：
- **父子分块（small-to-big）**：小块检索（子块，聚焦精准匹配）+ 大块喂 LLM（父块，包含完整上下文）。子块是同一内容的多角度口语化描述，向量化入索引；父块不向量化，只通过 parent_id 钩子回取。这解决了「检索块要短要精准」和「喂 LLM 的块要长要完整」的矛盾。我在应用生成平台 demo 里用的就是这套——8 父块 × 3 子块 = 24 子块入向量库。
- **元数据附加**：每个 chunk 带上标题路径、文档类型、章节编号等元数据，检索时可过滤、生成时可溯源。

### Layer 3 — 坑 / 边界 / 追问预判

- **「块大 vs 块小的经典 tradeoff」**：块大（1000+ token）→ 上下文完整但噪声多，LLM 注意力被无关内容稀释；块小（200 token）→ 精准但容易断义，「该方法」的指代丢失。没有万能粒度，按场景调。
- **「overlap 不是越大越好」**：overlap 太大会导致 chunk 数量膨胀、索引变大、检索变慢。
- **「专业领域一定要按结构切」**：医疗病历按 SOAP（主诉/检查/评估/计划）切、代码按函数切、API 文档按 endpoint 切——结构切法比固定长度在专业领域的召回率高 10–30%。
- **「父子分块的聚合逻辑」**：一个父块如果多个子块都命中，取 MAX 分数聚合；多个父块候选时，取 top-k 父块。

**项目印证**：
- 应用生成平台：父子分块 + small-to-big——父块 = 完整 plan_steps + pitfalls，子块 = 同一范式的 3 种口语化场景描述；检索打子块、回取父块。聚合：parent_id MAX 分数。
- NestJS RAG 项目：朴素实现，先按段落分、段落太长再用滑动窗口切，CHUNK_SIZE=600、OVERLAP=80、MIN_CHUNK_LENGTH=20。

## M3：Embedding 选型 + 向量化

**Kernel**：双塔 bi-encoder(SBERT/BGE) 把句子编码成定长向量；归一化后用余弦≈点积。选型看：领域匹配、维度、中英、MTEB 榜。query 和 doc 必须同模型。

**展开**：

### Layer 1 — 为什么 / 背景

计算机不识字——需要把文本转成数字向量才能做相似度计算。embedding 模型的作用就是把「语义相近的文本」映射到「向量空间中距离近的点」。这是整个 RAG 检索的基础：query 和 document 向量距离越近 → 语义越相关。

### Layer 2 — 怎么做 / 机制

**双塔架构（bi-encoder）**：query 和 document 分别经过独立的编码器（或同一编码器）生成定长向量，然后算余弦相似度。两个向量可以独立预计算——document 向量离线索引入库，query 向量在线实时计算。这就是为什么 bi-encoder 快但不够准：query 和 doc 编码时没有交互。

**归一化的意义**：向量归一化（L2 norm=1）后，余弦相似度等于向量点积——省掉除法运算，检索更快。所有主流向量库都假设输入是归一化向量。

**选型四要素**：
1. **领域匹配**：通用模型（BGE、text2vec）vs 领域模型（医学/法律专用），差别巨大
2. **维度**：384/768/1024/1536，高维更准但更耗存储和算力。我的 rag2 demo 用 text-embedding-v4 的 1024 维
3. **中英文**：中文场景 BGE-M3/bge-large-zh 比 OpenAI ada 在中文上强 10–20%
4. **MTEB 榜**：Massive Text Embedding Benchmark，业界标准对比基准

**核心铁律**：query 和 document 必须用同一个 embedding 模型。不同模型映射到不同的向量空间，余弦相似度横比无意义。换模型 → 必须全量重建旧向量。

### Layer 3 — 坑 / 边界 / 追问预判

- **「换了 embedding 模型必须全量重建」**：这是新手最容易掉的坑——以为换个更强的模型只要新文档用新模型就行。实际新旧向量在不同空间，混在一起检索结果完全失真。
- **「embedding 模型对口语不敏感」**：通用 embedding 模型对领域术语的理解有限，「卖东西的网站」和「电商平台」的余弦相似度可能不高——这就是为什么要做 query 改写的原因（补 vocabulary gap）。
- **「多语言场景注意 text_type」**：很多 embedding API 区分 query 和 document 的 text_type 参数，用错会降精度。

**项目印证**：
- 应用生成平台：用阿里云 text-embedding-v4，1024 维，区分 query/document 的 text_type 参数。
- NestJS RAG 项目：用 SiliconFlow 的 BAAI/bge-m3，批量请求 25 条/批、间隔 200ms 限流。

## M4：向量库 + ANN 索引

**Kernel**：HNSW(图，查得快内存大)、IVF(倒排聚类，省内存)、PQ(量化压向量省内存掉精度)。FAISS(库/自建)、Milvus(分布式)、Pinecone(托管)。本质是用精度换速度的近似最近邻。

**展开**：

### Layer 1 — 为什么 / 背景

向量检索不是算 query 和所有 document 的相似度（百万级向量逐个比太慢）——需要近似最近邻（ANN）索引。ANN 的核心思想是：牺牲一点召回精度（比如 99% 而不是 100%）换取数量级的速度提升。

### Layer 2 — 怎么做 / 机制

**三种主流 ANN 索引**：

| 索引 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **HNSW**（主流） | 构建多层导航图，高层做长跳、底层做精搜。搜索时贪心走图 | 查询极快、精度高 | 内存大（存图结构） |
| **IVF** | 先聚类成 n 个桶，查询时只搜最近的 m 个桶 | 省内存 | 精度建聚类质量 |
| **PQ**（量化） | 把向量压成短编码，用查表代替向量计算 | 极致省内存 | 掉精度 |

三者可组合：IVF+PQ 既省内存又保持不错精度。

**向量库选型**：
- **FAISS**：Meta 的库，C++ 底层 Python 封装，适合自建/嵌入已有系统。轻量但需要自己管持久化和分布式
- **Milvus**：云原生分布式向量数据库，支持多种索引 + 混合检索 + 属性过滤，适合生产大规模场景
- **Pinecone/Zilliz Cloud**：全托管，零运维，按量付费
- **简单场景**：PostgreSQL 的 pgvector 插件、Redis Stack、甚至 SQLite 直接存 JSON（小规模足够）

### Layer 3 — 坑 / 边界 / 追问预判

- **「三角权衡」**：召回精度 vs 查询延迟 vs 内存占用，三者不可兼得。HNSW 精度高但吃内存；IVF+PQ 省内存但可能漏召。
- **「小规模不用上向量数据库」**：数据量在万级以下，直接用 pgvector 甚至 SQLite 算精确最近邻也够快，上 Milvus 反而增加运维复杂度。我的 NestJS RAG 项目就是 SQLite 存 JSON 向量，完全够用。
- **「索引参数要调」**：HNSW 的 M（每层连接数）和 efConstruction/efSearch 需要根据数据量和精度要求调。M 越大精度越高但内存越大。

**项目印证**：
- NestJS RAG 项目：SQLite 一张表 `kb_chunks`，embedding 列存 JSON 字符串，全量算余弦相似度——数据量小，没必要上向量数据库。

## M5：混合检索 + RRF

**Kernel**：Sparse(BM25) 抓精确词项(术语/ID/罕见词)，Dense(向量) 抓语义(同义/改述)，两者互补。用 RRF(倒数排名融合，score=Σ1/(k+rank))合并，免调权重。

**展开**：

### Layer 1 — 为什么 / 背景

纯向量检索有一个盲区：精确匹配不行。比如查「error code E10023」或「函数名 handlePaymentCallback」——向量检索可能因为训练数据里没见过这个具体字符串而召回完全无关的内容，但关键词检索（BM25）一击即中。

反过来，用户问「怎么让系统更快」，知识库里写的是「性能优化方案」——关键词匹配完全抓不到，但向量检索能捕捉到语义关联。

所以两路互补：**向量检索管语义（synonym/paraphrase），关键词检索管精确（term/ID/code）**。

### Layer 2 — 怎么做 / 机制

**混合检索流程**：
1. 同一 query 同时走两路——向量检索（Dense）取 top-k1、BM25 检索（Sparse）取 top-k2
2. 两路结果通过 RRF 融合成统一排序

**RRF（Reciprocal Rank Fusion）**：
```
score(doc) = Σ 1 / (k + rank_i)
```
- k 是平滑常数（通常 60），防止排名靠后的文档得分太低
- 优点：不需要调权重（不像加权求和需要一个 α 超参去调 cosine × α + BM25 × (1-α)），对排名尺度不敏感
- 缺点：丢弃了原始分数的信息（分数高低本身有信号价值）

**Hybrid 加权融合**（替代方案）：
```
score = α × cosine_score + (1-α) × keyword_score
```
优点：保留了原始分数信息；缺点：α 需要针对数据集调参，两个分数需各自归一化。

### Layer 3 — 坑 / 边界 / 追问预判

- **「RRF 的 k 值」**：k=60 是经典默认值，太小会导致排名靠后的文档得分几乎为零、等价于只取 top-1；太大则排名差异被抹平。
- **「两路结果集重叠处理」**：同一个文档可能同时出现在向量结果和关键词结果中，RRF 天然处理（累加两个排名分），加权融合则需要去重后重新算分。
- **「关键词检索对口语无效」**：用户说「别给我发邮件了」，关键词命中的可能是 CRM 相关的文档——完全跑偏。这再次说明 query 改写的重要性。

**项目印证**：
- NestJS RAG 项目：完整实现了混合检索——向量检索（cosine 相似度，阈值 0.3）+ SQLite FTS5 关键词检索 → RRF 融合（K=60）→ topK=5。
- 应用生成平台：向量 + BM25 + 元数据过滤（domain/complexity/multi_user 硬过滤）。

## M6：BM25 原理

**Kernel**：在 TF-IDF 上加两件事——词频饱和(k1，词出现再多收益递减)+文档长度归一化(b，长文不占便宜)。比 TF-IDF 更稳的稀疏检索。

**展开**：

### Layer 1 — 为什么 / 背景

TF-IDF 是经典的稀疏检索算法，但有两个致命问题：① 如果一个词在一篇文档里出现了 100 次，TF-IDF 认为它比出现 10 次的文档相关 10 倍——这不对，出现 50 次和 100 次其实差不多；② 长文档天然比短文档有更高的 TF 值（词出现次数多），会在排序中占便宜。

BM25 就是对这两个问题的修正——加了两件事：词频饱和 + 文档长度归一化。

### Layer 2 — 怎么做 / 机制

**BM25 公式（简化理解）**：
```
score(D, Q) = Σ IDF(qi) × TF_effect(qi, D)

TF_effect = tf × (k1 + 1) / (tf + k1 × (1 - b + b × |D|/avgDL))
```

三个关键参数：
- **k1（词频饱和）**：控制词频的收益衰减速度，通常 1.2–2.0。k1 越大词频影响越大；k1=0 则完全忽略词频（退化为 IDF-only）
- **b（长度归一化）**：0 ≤ b ≤ 1，控制文档长度惩罚力度，通常 0.75。b=1 则完全按比例惩罚长文档；b=0 则不惩罚
- **IDF**：与 TF-IDF 的 IDF 类似，衡量词项在全库中的稀有程度

**效果**：一个词在文档中出现 10 次和 100 次，在 BM25 下得分差不大（词频饱和在起作用）；一篇 500 词的文档和一篇 5000 词的文档，在词频相同的情况下，短文档得分更高（长度归一化在起作用）。

### Layer 3 — 坑 / 边界 / 追问预判

- **「BM25 不关心语义」**：纯字面匹配，对同义词/近义词/改述完全失效。这就是为什么需要向量检索做补充 → M5。
- **「BM25 参数需要调吗」**：大部分场景用默认值 k1=1.5、b=0.75 就够。除非有特殊需求（比如极其重视精确匹配、文档长度差异极大）。
- **「BM25 和 TF-IDF 什么时候差不多」**：短文档场景、词频不高的场景下两者差异不大。BM25 的优势在长文档和词频差异大的场景才明显。

## M7：Rerank / 两阶段检索

**Kernel**：bi-encoder 双塔分开编码、能预存向量→快但糙(召回)；cross-encoder query+doc 拼一起过 BERT→准但慢(精排)。全量跑 cross 算力爆炸，所以先召回 top-k 再精排 top-n。算力换精度的经典分层。

**展开**：

### Layer 1 — 为什么 / 背景

bi-encoder（向量检索）之所以快，是因为 query 和 document 是分开编码的——document 向量可以离线索引入库。但代价是 query 和 doc 在编码过程中没有交互，精度有限。

cross-encoder（精排模型）把 query 和 doc 拼接成一对输入，经过完整 Transformer 编码——每一层都有 query-document 的交叉注意力，精度远高于 bi-encoder。但代价是慢：每个 query-doc 对都要重新编码，无法预计算。

如果全量用 cross-encoder，百万级文档逐个打分算力爆炸。所以两阶段是最优解：**粗排（召回）用 bi-encoder 把候选集从百万缩到百级 → 精排用 cross-encoder 把百级缩到十级**。

### Layer 2 — 怎么做 / 机制

**两阶段检索流程**：
1. **粗排（bi-encoder）**：向量检索召回 top-k 候选（k 通常 50–200）
2. **精排（cross-encoder）**：对 k 个候选逐个做 query-doc 联合编码，输出 relevance_score
3. **排序输出**：按 relevance_score 降序取 top-n（n 通常 5–10）喂给 LLM

**Cross-encoder 常用模型**：bge-reranker、gte-rerank、Cohere Rerank。打分尺度因模型而异——cosine 相似度在 [0,1]，但 cross-encoder 的 relevance_score 范围不固定，绝对值不能和 cosine 横比。

**工程兜底**：rerank API 调用失败 → fallback 回 cosine 分数。外部依赖挂了链路不断，但要标记该结果走了 fallback，对评测透明。

### Layer 3 — 坑 / 边界 / 追问预判

- **「rerank 接在哪一层」**：粗排在哪一层聚合，精排就在哪一层打分。如果是父子分块（粗排打子块、按 parent_id 聚合），精排也打子块层，聚合逻辑不变。不要打父块层——父块候选太少，rerank 没意义。这是我的真实判断：rag2 demo 里子块 24 个、召回 8 个送 rerank，父块只 8 个直接聚合不重排。
- **「命中率打满天花板时，rerank 的收益在哪」**：在 margin（top1 − top2 分差）。我的 rag2 消融评测三路命中率全 100%、归因 +0 pp，但 margin 从 0.21 → 0.51 翻 2.4 倍。生产意义：margin 拉大后可以设置信度阈值做拒答/转人工，裸向量的 margin 做不了。**评测要分两类指标：命中率（对不对）+ 置信度（敢不敢下结论）**。
- **「cross-encoder 和 bi-encoder 的分数不能横比」**：cosine 相似度和 relevance_score 是两种尺度，cross-encoder 那路的平均分可能看着更低，但那是尺度错觉，不是退步。只有同尺度内的相对差（margin）才可横比。
- **「recall_k 怎么定」**：太小 → 可能漏掉正确答案（粗排阶段就丢了，精排再强也救不回来）；太大 → 精排成本高，退化成单阶段。通常 3–10 倍于最终需要的 top-n。

### Layer 4 — 同类变式

题库中约 20 题属于此母题，触发词：为什么要重排 / 召回-精排两阶段 / 为什么不直接精排全量 / cross vs bi-encoder / 两阶段检索架构 / rerank 模型选型 / 精排打分原理。

**变式套路**：被问「为什么不直接精排全量」→ 算力爆炸，cross-encoder 逐个编码的成本是 O(n) 且不可预计算，百万级文档不可行；被问「rerank 和 RRF 的区别」→ RRF 是排名融合（不需要额外模型，只是把已有排名重新加权），rerank 是模型重新打分（query-doc 联合编码，更准但需要额外模型调用）。

**项目印证**：
- 应用生成平台 rag2 demo：两阶段检索落地——cosine 粗排 recall_k=8 子块 → gte-rerank cross-encoder 精排 → 仍按 parent_id MAX 聚合。rerank 分替代 cosine 分（不用加权融合，多一个超参要调）。rerank 失败 fallback 回 cosine + 报表打 `*` 标记。核心判断：粗排在哪层聚合，精排就在哪层打分。
- A/B/C 单变量递进消融：A 裸向量 / B +query 改写 / C +rerank，归因分解每步贡献 pp。反直觉结论：命中率天花板 +0 pp 但 margin 翻 2.4 倍。
- OpsPilot：evidence retrieval 目前未接 rerank（待优化点）。

## M8：召回质量优化

**Kernel**：query 端(改写/扩展/HyDE 假设文档/多查询)→检索端(混检+rerank)→context 端(压缩去噪、去重、按相关性排序)。

**展开**：

### Layer 1 — 为什么 / 背景

RAG 的质量瓶颈 80% 在检索段——检索没召回正确文档，后面 prompt 多好、LLM 多强都白费。召回质量优化是一个端到端问题，需要在三个环节同时发力。

### Layer 2 — 怎么做 / 机制

**三端优化框架**：

**① Query 端（入口优化，ROI 最高）**：
- **改写**：术语映射、去口语噪声、补充上下文。「帮我做个能卖东西的网站」→「构建电商在线商城平台」——实测 58% → 84%
- **扩展**：用 LLM 生成多个角度的 query，分别检索后合并去重
- **HyDE（假设文档）**：让 LLM 先生成一个假想的理想答案，用这个假答案去检索——假答案和真实文档的语言风格更接近
- **多查询/子查询拆分**：复杂 query 拆成多个子 query 分别检索，合并去重

**② 检索端（匹配优化）**：
- 混合检索（向量 + BM25）→ M5
- 两阶段 rerank（粗排 + 精排）→ M7
- 元数据过滤：检索前用 domain/complexity/service 等结构化字段过滤

**③ Context 端（后处理优化）**：
- 压缩去噪：用 LLM 压缩检索到的 chunk，去掉不相关片段
- 去重：多个 chunk 可能包含相同信息，去重减少 token 浪费
- 按相关性重排：把最相关的 chunk 放在 prompt 开头（LLM 对开头内容注意力更高）
- 相似但实际不相关的：检索可能召回「语义相似但回答不了问题」的 chunk → 靠 rerank + 元数据过滤解决

### Layer 3 — 坑 / 边界 / 追问预判

- **「query 改写可能引入噪声」**：规则改写（术语映射）可控但覆盖面有限；LLM 改写覆盖面广但可能改偏（问 A 改写成 B）。生产上建议规则 + LLM 双通道，规则做保底。
- **「HyDE 的适用场景」**：适合「文档和 query 风格差异大」的场景（比如用户问口语，文档是学术论文）；不适合「query 本身已经很精确」的场景。
- **「多查询检索的合并策略」**：简单取并集 + RRF 重排，注意去重防止同一文档多次出现。

**项目印证**：
- 应用生成平台：query 端——STOPWORDS 去口语噪声 + TERM_MAP 术语映射（ROI 最高的一步）；检索端——混合检索 + 元数据过滤 + small-to-big + rerank；context 端——Planner 有权否决低分命中，不注入噪声。
- OpsPilot：retrieve_memory 节点——先带 service+env 过滤检索，结果少于阈值时 fallback 到宽检索（解决冷启动问题 → M9）。

## M9：RAG 降级 / 增量更新 / 幻觉缓解

**Kernel**：降级=阈值兜底(<阈值就答"知识库无相关，转人工/纯模型"+标注低置信)。增量=新文档单独 embedding 追加，避免全量重建。缓解幻觉=把外部事实塞进 context 约束生成 + 要求带引用 + 二次校验。RAG 治"知识缺失型"幻觉，治不了"推理型"幻觉。

**展开**：

### Layer 1 — 为什么 / 背景

RAG 不是万能的——三个常见问题：① 知识库里没有相关文档怎么办（降级）；② 知识库更新时怎么不重建整个索引（增量）；③ 有了 RAG 为什么 LLM 还会编（幻觉缓解的边界）。

### Layer 2 — 怎么做 / 机制

**降级策略（阈值兜底）**：
- 检索分数低于阈值（如 cosine < 0.3 或 rerank margin < 0.4）→ 判定「知识库无相关」
- 三种降级路径：
  - 答「抱歉，我目前的知识库中没有相关信息，建议转人工处理」+ 标记低置信度
  - 纯模型回答（不用 RAG context）+ 明确告知用户「以下回答未基于知识库，请核实」
  - 降级到更宽检索（放宽过滤条件、扩大 recall_k）再试一次
- 核心原则：**宁可说不知道，也别硬编**

**增量更新**：
- 新文档 → 清洗 → 分块 → embedding → 追加写入向量库
- 不需要对旧数据做任何操作（每个 chunk 的向量是独立计算的）
- 注意：如果 embedding 模型换了 → 必须全量重建，新旧不可混用
- 文档删除/更新：通过 chunk_id 删除旧向量，插入新向量

**幻觉缓解（RAG 的边界）**：
- RAG 能治的：「知识缺失型」幻觉——模型不知道某个事实所以编。解法：把事实塞进 context。
- RAG 治不了的：「推理型」幻觉——模型知道事实但逻辑推错了。解法在别处：CoT / 自我校验 / RL。
- **缓解手段**：
  1. 反幻觉 prompt 约束：明确要求「基于提供的文档回答」「不知道就说不知道」「每条结论标注引用来源」
  2. 引用溯源：生成答案时标注 [1][2] 对应具体 chunk
  3. 二次校验：用另一个 LLM 检查生成内容是否与引用文档一致
  4. 低置信度标注：检索分数低时，在回答中标注「置信度：低」

**冷启动问题**：新服务/新领域历史经验少 → 两阶段检索：先带过滤（service+env），结果少于阈值时 fallback 到宽检索（只过滤 env 或不过滤）。OpsPilot 的 retrieve_memory 就是这样做的。

### Layer 3 — 坑 / 边界 / 追问预判

- **「降级阈值怎么定」**：不是拍脑袋——需要基于离线评测，画出不同阈值下的 Precision/Recall 曲线，选业务可接受的点。阈值太高 → 太多无辜 query 被拒答；太低 → 低质量 context 进入生成。
- **「增量更新 ≠ 增量索引」**：如果 ANN 索引（如 IVF）依赖全局聚类中心，新增大量文档后可能需要重建索引（聚类中心变了）。HNSW 天然支持增量插入。
- **「知识库污染比没知识库更可怕」**：错误的文档比没文档危害更大——LLM 会「信」错文档然后生成错误答案。所以要门控写回（只有人工确认的内容才入库 → OpsPilot 的 confirmed_by_human）。写入收窄，读取放开。
- **「幻觉率和 retrieval 精度的关系」**：检索精度 ↑ → 幻觉率 ↓，但不是线性关系。检索到了但不对（错误文档），幻觉率反而可能比没检索到更高（模型被错误 context 误导）。

**项目印证**：
- 应用生成平台：反幻觉 prompt（5 条约束）+ 引用溯源 + Planner 否决权（低分命中不注入 context）。
- OpsPilot：
  - 降级：retrieve_memory 检索分数低于阈值 → 空 memory_hits，Agent 继续跑只是没历史加持
  - 增量：RAG 写回（writeback_knowledge 节点），新 RCA 单独 embedding 追加
  - 门控：confirmed_by_human=true 才写回向量库——早期无门控时 LLM 噪声 RCA 污染了知识库，导致下次检索到错误经验，诊断质量下降。加门控后稳定了。
  - 冷启动：两阶段检索——先带过滤，结果少时 fallback 到宽检索
