### Q01 针对知识库频繁更新的场景，设计高效增量式向量索引更新机制，支持实时/准实时更新并保证检索质量。

**一句核心**：新文档单独 embedding 后追加写入向量库，不触碰旧数据；HNSW 天然支持增量插入。

**展开**：
- 离线处理流水线：新文档 → 清洗 → 分块 → embedding → 追加写入向量库，每个 chunk 的向量独立计算，旧数据无需变动
- 索引选择：HNSW 图索引天然支持增量插入；IVF 依赖全局聚类中心，大量新增后可能需要重建
- 检索质量保障：增量更新后定期跑评测基准，监控 top-1 命中率漂移；新文档占比超过阈值时触发索引重建

**项目印证**：OpsPilot `writeback_knowledge` 节点——人工确认的 RCA 单独 embedding 追加写入，不重建全库；门控 confirmed_by_human=true 防污染。

### Q03 请解释 RAG 的基本概念、核心架构组成及工作流程，说明其解决的关键问题，对比 RAG 优于纯参数化记忆的场景。

**一句核心**：RAG = 推理时将外部知识检索后注入 LLM 上下文窗口，用「检索」换「记忆」，解决知识截止和幻觉问题。

**展开**：
- 核心架构：离线链（清洗→分块→embedding→向量库）+ 在线链（query 改写→混合检索→rerank→拼 context→LLM 生成带引用答案）
- 解决关键问题：① 知识截止日期——训练数据有终点，RAG 可接入最新文档；② 幻觉——事实类编造，RAG 用真实文档约束输出
- RAG 优于纯 LLM 的场景：需要最新知识（新闻/股票/ops）、需要溯源（医疗/法律/金融）、长尾知识覆盖（知识库可无限扩展不重训模型）

**项目印证**：应用生成平台完整 RAG 链路——query 改写 58→84% 命中率→混合检索→small-to-big 父子分块→rerank→反幻觉 prompt。

> 🏷️ M1

### Q04 面对线上业务数据分布快速变化，如何通过外接数据库动态增强模型能力以应对分布漂移？说明实现架构、更新机制与落地挑战。

**一句核心**：外接向量库 + 实时数据库作为模型「外部记忆」，增量注入最新数据；RAG 架构天然解耦模型与知识，知识可独立更新不重训模型。

**展开**：
- 实现架构：LLM + RAG 解耦——模型层不变，知识层（向量库/实时 DB/Redis）独立更新。查询时动态检索最新知识拼入 prompt
- 更新机制：流式写入——新数据实时/准实时 embedding 后追加写入向量库 + 过期数据 TTL 淘汰；实时 DB 存最新统计指标/状态快照
- 落地挑战：① 新老数据分布漂移导致 embedding 偏移——定期评测监控；② 错误数据的危害大于无数据——需要门控写回；③ 实时写入对向量库的性能要求——需支持高并发 upsert

**项目印证**：OpsPilot——retrieve_memory 模拟外部记忆，历史 RCA 写入向量库，运行时注入 planner context；冷启动两阶段检索防空库。

### Q05 系统比较 RAG 与传统「先检索后生成」流程在架构设计、信息融合机制和生成效果上的差异。

**一句核心**：传统流水线检索和生成解耦（先搜文档再独立生成），RAG 是端到端——检索结果通过上下文窗口与生成深度融合，且支持联合训练。

**展开**：
- 架构差异：传统流水线检索和生成是两个独立系统、接口是文本拼接；RAG 中检索器为生成器定制（共享 embedding 空间），检索结果通过 attention 融入生成过程
- 信息融合：传统是机械拼接（检索结果放 prompt 开头），生成模型不知道哪些最重要；RAG 可通过 cross-attention 对检索到的文档段做差异化关注
- 生成效果：传统容易拼错上下文、引用断链；RAG 端到端训练可优化检索+生成的联合 loss，引用更精准、事实一致性更高

**项目印证**：应用生成平台——虽未做端到端训练，但通过两阶段检索 + rerank + 反幻觉 prompt，实现检索到生成的紧密耦合。

> 🏷️ M1

### Q07 请解释什么是 RAG，并说明它如何通过引入外部知识提升语言模型生成准确性和事实一致性。

**一句核心**：RAG = 检索增强生成——推理时从外部知识库检索相关文档，注入 LLM 作为生成依据，把「从记忆生成」变成「基于证据生成」。

**展开**：
- 提升准确性：LLM 可能不知道或过时的事实，通过检索实时文档补齐，避免编造（如「现在几点」→ 检索实时数据而非靠训练记忆）
- 事实一致性：context 中的文档充当「锚定」——LLM 被 prompt 约束「基于提供文档回答」，减少自由发挥的空间
- 核心机制：检索到的 chunk 按相关性排序拼入 prompt 开头（LLM 注意力对开头内容权重更高），形成对生成结果的强约束

**项目印证**：应用生成平台 5 条反幻觉 prompt 约束 + 引用溯源——生成内容必须标注源自哪个 chunk。

> 🏷️ M1

### Q08 什么是 RAG，它如何通过引入外部知识来提升大语言模型的生成质量和事实准确性？

**一句核心**：同 Q07——RAG 推理时从外部知识库检索文档注入 LLM 上下文，用「外部事实」约束「模型生成」。

**展开**：
- 生成质量提升：检索提供领域专业术语 + 结构模板（如 SOP / troubleshooting 步骤），LLM 在此基础上改写而不是从零编
- 事实准确性提升：每条结论需标注引用来源；二次校验（用另一个 LLM 检查生成内容与引用文档一致性）
- 与微调互补：RAG 解决「知识更新」问题——不改模型权重、知识库独立维护；微调解决「行为对齐」问题——改变模型回答风格/格式

**项目印证**：应用生成平台 demo——retrieval 检索成功后，Planner 基于 context 生成 plan_steps，人工评测 factuality 从 3.2 提升到 4.1（5 分制）。

### Q09 请说明 RAG 系统中检索模块的工作机制，并列举至少三种提升检索结果相关性的方法及其原理。

**一句核心**：检索 = query 向量化 → ANN 索引搜索 → 返回相似 chunk；提升方法：query 改写、混合检索、rerank。

**展开**：
- 工作机制：用户 query → embedding → ANN 索引（HNSW/IVF）→ 余弦相似度排序 → 返回 top-k chunk
- 方法一：query 改写——术语映射「卖东西的网站」→「电商平台」，消除口语与书面语的 vocabulary gap，ROI 最高（实测 58% → 84%）
- 方法二：混合检索——向量检索抓语义（同义/改述）+ BM25 关键词检索抓精确（术语/ID/代码），RRF 融合两路排名
- 方法三：rerank 两阶段——粗排 bi-encoder 召回 top-k（快但糙）→ 精排 cross-encoder 对 k 个候选逐对打分取 top-n（准但慢），算力换精度

**项目印证**：应用生成平台三招并用——query 改写 + 混合检索 + gte-rerank 精排，margin 从 0.21 翻到 0.51。

> 🏷️ M8

### Q10 在 RAG 系统中，针对索引环节有哪些常见的优化策略以提升检索效率和准确性？

**一句核心**：索引优化三方向：chunking 策略（语义切分 + overlap）、embedding 选型（领域匹配）、索引结构（HNSW 参数调优 + 元数据过滤）。

**展开**：
- Chunking 优化：改固定长度为语义/结构切分（Markdown 按标题、代码按函数），overlap 10-20% 防切断语义边界；父子分块小块检索大块喂 LLM
- Embedding 优化：选领域匹配的模型（中文用 BGE/bge-large-zh 比 OpenAI ada 强 10-20%），query 和 document 用电同模型（换模型必须全量重建）
- 索引结构优化：HNSW 调 M（连接数）和 efSearch 参数平衡精度/速度；元数据过滤在检索前用 domain/complexity/service 等字段硬过滤减少候选集
- 多级索引：粗粒度聚类 + 细粒度 ANN，先定位大类再精搜

**项目印证**：应用生成平台——8 父块 × 3 子块 = 24 子块入向量库，父块通过 parent_id 钩子回取；元数据 domain/complexity 硬过滤。

> 🏷️ M2/M4

### Q11 LangChain 与 LlamaIndex 在架构设计和核心功能上各有侧重，请比较两者的主要差异及各自适用的典型场景。

**一句核心**：LangChain 是「AI 应用编排框架」（链/Agent/工具调用），LlamaIndex 是「数据与 LLM 之间的桥梁」（索引/检索/查询引擎），两者互补不互斥。

**展开**：
- LangChain 核心：Chain 抽象串联多步推理、Agent 自主调用工具、Memory 管理对话历史、模板/Prompt 管理。适合构建复杂 Agent 工作流
- LlamaIndex 核心：数据连接器（20+ 数据源）、索引结构（向量/树/关键词/知识图谱）、查询引擎（检索→合成→回答）。适合做 RAG 数据处理和检索中台
- 选型原则：需要编排复杂多步 Agent 逻辑（如 code interpreter、工具链）选 LangChain；核心需求是高质量文档检索+问答（如企业知识库）选 LlamaIndex；两者可组合使用

**项目印证**：OpsPilot 用 LangGraph（LangChain 生态）编排 14 节点 SRE Agent 工作流——intake→triage→retrieve→planner→evidence→diagnose→remediation。

> 🏷️ M16

### Q12 当用户查询在知识库中无法找到相关文档时，RAG 系统应如何优雅降级处理？描述常见 fallback 策略及其优缺点。

**一句核心**：检索分数低于阈值 → 判定「知识库无相关」→ 三种降级：答不知道 + 转人工 / 纯模型回答 + 标注 / 放宽检索再试；核心原则：宁可说不知道，也别硬编。

**展开**：
- 策略 A（推荐）：明确告知「抱歉，当前知识库暂无相关信息，建议转人工处理」+ 标记低置信度——安全但用户体验差
- 策略 B：纯 LLM 回答（不用 RAG context）+ 明确标注「以下回答未基于知识库，请核实」——流畅但有幻觉风险
- 策略 C：放宽检索条件（降低阈值、去掉元数据过滤、扩大 recall_k）再试一次——可能召回不相关文档引入噪声
- 阈值设定：基于离线评测，画 Precision/Recall 曲线选定业务可接受的 cutoff 点

**项目印证**：OpsPilot retrieve_memory——先带 service+env 过滤检索，结果少于阈值时 fallback 到宽检索；降级后 Agent 继续运行只是没历史加持。

> 🏷️ M9

### Q15 在 RAG 系统中，有哪些方法可以提升检索阶段的质量？除标准向量检索外，还可以采用哪些增强技术？

**一句核心**：三端优化——query 端改写/扩展/HyDE、检索端混合检索 + rerank + 元数据过滤、context 端去噪/去重/重排。

**展开**：
- Query 端：术语映射消除口语-书面语鸿沟（ROI 最高）、HyDE 生成假想答案用答案风格检索、多查询拆分分开检索合并去重
- 检索端：混合检索（向量 + BM25，RRF 融合）解决纯向量精确匹配盲区；cross-encoder rerank 两阶段精排提升精度；元数据过滤减少无效候选
- 增强技术：small-to-big 父子分块（小块检索大块喂 LLM）、假设文档嵌入（HyDE）解决 query-document 风格不一致、多轮对话合并历史上下文改写
- Context 端：压缩去噪去掉不相关片段、去重减少 token 浪费、按相关性排序把最重要 chunk 放 prompt 开头

**项目印证**：应用生成平台——query 改写 58%→84% + 混合检索 + metadata 过滤 + small-to-big + rerank，hit@1 margin 翻 2.4 倍。

> 🏷️ M8

### Q16 为医疗或法律等专业领域构建 RAG 智能助手时，应如何设计完整的数据处理与索引流程，以确保知识准确性、合规性和检索效率？

**一句核心**：数据管道 = 权威源采集 → 按领域结构切块 → 领域 embedding → 版本化管理索引入库；核心：宁可漏召不召回错误文档。

**展开**：
- 数据采集与清洗：仅纳入权威源（官方指南/教科书/法规原文），剔除不可靠来源，标注来源、认证等级、发布时间
- 分块策略：按领域结构切——病历按 SOAP（主诉/检查/评估/计划）、法律按条款/法条编号、API 文档按 endpoint
- 索引与向量化：领域专用 embedding 模型 > 通用模型；元数据完整标注（来源认证等级/有效期限），检索时过滤
- 版本管理：新增 → 增量追加；修订 → 标记旧版过期 + 插入新版；废止 → 标记无效不入选

**项目印证**：OpsPilot——evidence retrieval 从多种工具源采集数据，每种标注 tool_source 和置信度。

### Q17 请解释 RAG 的基本概念及其工作原理，说明它如何提升大模型的事实准确性和知识覆盖能力。

**一句核心**：RAG = 检索增强生成——推理时动态检索外部知识注入 LLM 上下文，把「从模型记忆生成」变成「基于证据生成」。

**展开**：
- 工作原理：用户 query → 向量化 → ANN 检索 top-k 文档 → rerank 精排 → 拼入 prompt → LLM 基于 context 生成带引用答案
- 事实准确性：外部文档充当「锚定」约束 LLM 不编造；引用溯源 + 二次校验进一步兜底
- 知识覆盖能力：知识库可无限扩展（加入新文档就覆盖新领域），不需要重训模型；比微调便宜、比纯 prompt 可靠
- 局限：RAG 只治「知识缺失型」幻觉，治不了「推理型」幻觉（模型知道事实但推错了）

**项目印证**：应用生成平台 demo——RAG 注入成功 plan 语料，factuality 从 3.2 提升到 4.1。

> 🏷️ M1

### Q18 LangChain 和 LlamaIndex 都是流行的 AI 应用开发框架，请比较它们核心设计理念、适用场景和技术侧重点。

### Q19 在 RAG 系统中，当检索结果包含噪声或不相关文档时，有哪些方法可以提升生成质量与鲁棒性？

**一句核心**：多层防线——检索前元数据过滤 + 检索后 rerank 去噪 + context 压缩 + prompt 约束「不知道就说不知道」。

**展开**：
- 检索层：提高检索精度（混合检索 + rerank + 阈值过滤），让噪声在进入 context 前被拦下；设置相关性阈值，低于阈值的文档不注入
- Context 层：LLM 压缩去噪——让 LLM 对检索结果做摘要、剔除不相关片段；去重合并相同信息；按相关性排名，最相关放最前面
- 生成层：反幻觉 prompt——「只基于提供的文档回答」「如果知识库无相关信息请明确说明」；二次校验看生成内容是否符合引用文档
- 兜底：模型本身具备一定「信噪分辨」能力——当 context 包含矛盾信息时，模型倾向于输出「信息不一致」而非胡编

**项目印证**：应用生成平台——Planner 有权否决低分命中、不注入噪声；反幻觉 prompt 5 条约束。

### Q20 在多轮对话场景中，大语言模型驱动产品可能面临哪些典型挑战？有哪些技术手段提升多轮对话一致性和连贯性？

**一句核心**：挑战：上下文遗忘、话题漂移、幻觉累积；技术手段：滑动窗口记忆 + 对话摘要压缩 + RAG 注入关键历史。

**项目印证**：OpsPilot Agent 工作流——memory 节点在每次 planner 运行前检索历史 RCA，作为上下文约束。

### Q21 在构建 RAG 系统知识库时，当新增文档需要加入索引，如何实现高效、低开销的动态增量更新？说明常用策略及实现挑战。

**一句核心**：新文档独立 embedding → 追加写入向量库不碰旧数据；HNSW 天然支持增量；挑战：分布漂移 + 新旧数据一致性。

**展开**：
- 策略：离线流水线处理新文档（清洗→分块→embedding）→ 写入向量库（INSERT）；每个 chunk 向量独立，旧数据无需重算
- 索引选择：HNSW 图索引天然支持增量插入，新增节点不影响已有图结构；IVF 依赖全局聚类，大量新增后聚类中心漂移，可能需重建
- 挑战：① 新旧文档分布漂移——新数据主题变化导致 embedding 相似度分布偏移，需定期评测监控；② 删除困难——ANN 索引通常不支持直接删除，需维护 deletion filter
- 准实时方案：新文档到达后立即 embedding 入缓存层（Redis），定时批量写入向量库；查询时同时查缓存 + 主库合并去重

**项目印证**：OpsPilot writeback_knowledge——人工确认 RCA 后单独 embedding 追加入库，增量不重建。

> 🏷️ M9

### Q22 在搜索排序系统中，如何计算查询与文档之间的相关性得分？说明基于语义匹配、关键词匹配、深度学习模型等方法的实现方式。

**一句核心**：三种范式互补——关键词匹配（BM25）抓精确术语、语义匹配（向量余弦）抓同义改述、深度学习（cross-encoder）抓深度语义交互。

**展开**：
- 关键词匹配（BM25）：基于 TF-IDF 改进——词频饱和 (k1) 防止词频无限放大 + 文档长度归一化 (b) 防止长文档占优；适合精确术语/ID/代码匹配
- 语义匹配（向量）：bi-encoder 将 query 和 doc 分别编码为定长向量 → 余弦相似度。快（可预存 doc 向量）但精读有限（无关交互编码）
- 深度学习排序（cross-encoder）：query+doc 拼接→过 BERT → 输出 relevance_score。准但慢，适合精排阶段（top-k→top-n）
- 融合：RRF（1/(k+rank)）融合多路排名，或加权求和（需调 α 参数 + 各自归一化）

**项目印证**：应用生成平台 = BM25 + 向量 cosine + gte-rerank cross-encoder 三路，粗排 RRF 融合，精排 cross 替代 cosine 分。

### Q23 在面对线上业务数据分布快速变化的场景时，是否可以使用外挂数据库结合大模型的方法来提升响应准确性？说明实现思路和优势。

### Q24 当推荐系统出现推荐不精准的问题时，可以采取哪些具体的算法层面改进措施？

**一句核心**：三方向——模型结构升级（多任务/深度交叉网络）、特征工程强化（实时特征 + 序列建模）、训练策略优化（负采样 + 冷启策略）。

**展开**：
- 模型结构：从协同过滤 → 深度模型（DeepFM/DIN/DIEN），引入交叉特征 + 用户行为序列建模；多任务学习同时优化 CTR + CVR
- 特征工程：加入实时特征（最近 5 分钟行为）+ 序列特征（点击/购买历史）+ 上下文特征（时间/设备/位置）；特征交叉自动化（AutoInt/DCN）

**项目印证**：检索-RAG 思路迁移到推荐——类似于 RAG 的增量知识更新和冷启动两阶段检索（先精确后宽松）。

### Q25 请介绍 RAG 模型的主要分类方式，并比较不同类别之间的架构差异与适用场景。

**一句核心**：两维度分类——按检索时机（前置检索/后置检索/迭代检索）、按训练方式（frozen RAG/retriever-finetuned/end-to-end RAG）。

**展开**：
- 按检索时机：① 前置 RAG——生成前检索一次，最简单、最常用；② 迭代 RAG——生成过程中多轮检索（Self-RAG/FLARE），每次判断是否需要检索；③ 后置检索——先生成答案再检索验证，FactScore 等评测方法
- 按训练方式：① frozen RAG——检索器和生成器都冻结，零训练即用，最实用；② retriever-finetuned——用 LLM 生成数据训练检索器；③ end-to-end RAG——联合训练检索+生成，精度最高但成本大
- 适用场景：frozen RAG 适合通用知识库问答（零门槛）；迭代 RAG 适合开放域长篇生成；end-to-end 适合有大量标注数据的垂直场景

**项目印证**：应用生成平台 = frozen RAG + 前置检索——检索一次后 Planner 根据 context 生成完整 plan，不再回检索。

### Q26 相较于传统「先检索后生成」流水线方法，现代 RAG 系统在架构设计、信息融合方式和端到端训练方面有哪些关键区别？

**一句核心**：传统流水线两段独立、信息机械拼接、无联合训练；现代 RAG 检索-生成耦合、cross-attention 融合、支持联合 loss 端到端优化。

**展开**：
- 架构设计：传统 = 检索器和生成器完全独立（不同模型/不同系统）；现代 = 共享 embedding 空间、检索结果为生成器定制、支持多轮迭代检索
- 信息融合：传统 = 机械拼接（检索文本放 prompt 开头），生成器平等对待所有 chunk；现代 = 通过 attention 机制对相关文档做差异化关注 + rerank 精排后按重要性排序
- 端到端训练：传统不支持（系统割裂）；现代 RAG 可联合训练检索器和生成器，同时优化 LM loss（生成质量）和 retrieval loss（检索准确性）
- 生成效果：现代 RAG 引用更精准、事实一致性更高、可处理 open-domain QA 中的长尾知识

**项目印证**：应用生成平台虽未做端到端训练，但通过 rerank + 反幻觉 prompt + Planner 否决权实现了检索到生成的紧密耦合。

> 🏷️ M1/M7

### Q27 请说明你对带有时间窗口或时间偏移限制的对话系统的理解，这类约束如何影响上下文管理、模型推理与用户体验设计？

**一句核心**：时间窗口 = 仅保留最近 N 分钟/轮对话作为有效上下文；本质是「遗忘机制」，平衡上下文相关性和计算成本。

**展开**：
- 上下文管理：滑动窗口淘汰超时对话——超过 N 分钟的轮次不再注入 prompt；定时截断 + 摘要备份（关键信息压缩保留）。实现：每轮记录时间戳，构造 prompt 时过滤
- 模型推理：短上下文 = 更快推理（token 少，TTFT 低）+ 更低幻觉风险（信息少不易混淆）。代价：丢失早期关键信息（用户意图/偏好）
- 用户体验：需明确告知用户「对话上下文保留 X 分钟」；超时后主动询问是否重新开始还是继续；关键轮可手动标记「不过期」
- 时间偏移限制：根据会话间隔判断是否为新会话——间隔超过阈值自动断开旧会话、开启新会话

**项目印证**：OpsPilot Agent 每个 run 独立运行，状态通过 checkpoint 持久化；retrieve_memory 检索历史（有时间标签），自然实现了时间维度过滤。

### Q28 在 RAG 系统中，利用大模型对检索内容进行二次生成，能否提升输出安全性？举例说明其作用机制。

**一句核心**：能——二次生成相当于「安全校验层」：过滤敏感内容、验证事实一致性、标注置信度；本质是生成-校验分离。

**展开**：
- 作用机制：① 第一次生成基于检索 context 生成答案 → ② 第二次用独立 LLM prompt「检查生成内容是否与引用文档一致、是否包含敏感信息」→ ③ 不一致的片段标记/滤除/转人工
- 过滤敏感内容：二次 LLM 扫描输出中是否包含 PII/违禁词/不安全建议，命中则拦截
- 增强事实一致性：检查每一条结论是否有对应引用文档支撑；无支撑的声明标记为 [未经证实]
- 优点：生成-校验分离使校验 prompt 可以比生成 prompt 更严格；不同 LLM 做生成和校验可防单点 bias

**项目印证**：应用生成平台反幻觉 prompt 第 5 条约束 + 引用溯源——每条 plan step 标注来源 chunk_id。

### Q30 假设需要为一个内容平台构建用户兴趣模型以识别偏好话题，请从数据收集、特征工程、算法选型及模型更新策略方面阐述设计方案。

### Q31 在信息检索或 RAG 系统中，Query 改写技术有哪些常见的创新方法？各有什么优势？

**一句核心**：四种主流——规则术语映射（可控快）、LLM 改写（覆盖面广）、HyDE 假设文档（风格对齐）、多查询拆分（多角度覆盖）。

**展开**：
- 规则术语映射：维护 TERM_MAP 字典（「卖东西的网站」→「电商平台」），不依赖 LLM、零延迟、完全可控，但覆盖面有限（只能映射预定义词对）
- LLM 改写：用 LLM 将口语 query 转写成正式检索表达式，覆盖面广，但可能改写偏差（问 A 改写成 B），建议规则+LLM 双通道做保底
- HyDE（假设文档嵌入）：让 LLM 先生成一个假想理想答案，用假答案向量检索——假答案和真实文档语言风格接近，解决 query-document 风格鸿沟
- 多查询拆分：复杂多意图 query 拆成多个子 query 分别检索，合并去重 RRF 重排——覆盖每个子意图但 token 成本高

**项目印证**：应用生成平台——规则术语映射（58% → 84%）ROI 最高 + STOPWORDS 去口语噪声，没有用 LLM 改写防偏。

### Q32 在构建 RAG 系统知识库时，动态增量更新可能因新旧文档分布差异引发检索偏差。请分析成因并提出缓解策略。

**一句核心**：成因 = 新旧文档 embedding 分布偏移导致 ANN 检索偏向新数据；缓解 = 周期性评测 + 全量索引重建 + 逐批均衡采样。

**展开**：
- 成因分析：新文档主题/领域与旧库差异大 → 新 embedding 分布在向量空间的不同区域 → ANN 检索可能过度召回新文档或旧文档（取决于 query 分布）
- 缓解策略：① 定期离线评测——用固定 benchmark query set 监控 top-k 命中率和主题分布变化；② 全量重建阈值——当新旧文档占比或分布偏移超过阈值时触发全量索引重建
- 工程策略：③ 逐批次采样——新文档分批入库，每批入库后检索测试集验证偏差；④ 时间权重衰减——检索时对旧文档加衰减因子，让新旧有公平竞争
- 治本方案：定期用最新 embedding 模型全量重建索引（类似搜索引擎全量更新索引）

**项目印证**：OpsPilot——门控写回（confirmed_by_human=true）防止噪声 RCA 污染库，本质上是在控制增量更新的质量。

### Q33 在搜索排序系统中，如何设计和衡量内容的时效性特征？举例说明常用的量化方法。

**一句核心**：时效性 = 内容发布时间与当前时间的距离的函数；量化 = 指数衰减 / 分段权重 / 按内容类型差异化半衰期。

**展开**：
- 设计思路：不同内容类型时效性要求不同——新闻半衰期以小时计、技术文档以月计、法律法条以年计或不过期
- 量化方法：① 指数衰减——score_ts = e^(-λ × days_since_publish)，λ 越大衰减越快；② 分段权重——发布 1 天内 weight=1，1-7 天 weight=0.8，7-30 天 weight=0.5，>30 天 weight=0.3
- 融入排序：作为排序特征输入模型（linear/GBDT/DNN one feature）或直接作为加权因子乘在相关性分数上
- 衡量指标：用时间敏感 query 的 NDCG + 人工标注时效性等级来评测衰减参数

**项目印证**：应用生成平台——plan 方案不直接使用时序衰减，但通过元数据标注「适用版本/最后验证时间」，间接实现时效过滤。

### Q34 在搜索排序系统中，如何对内容的热门程度进行量化？请描述常见的热门度指标及其计算方式。

**一句核心**：热门度 = 多维度用户行为信号的加权综合（点击率、互动、分享、消费完成率），加时间衰减防马太效应。

**展开**：

- 计算方式：① 简单——近 N 天浏览量/互动数按时间衰减求和；② 威尔逊区间——考虑样本量，小样本热度不高估；③ Hacker News 算法——score = (upvotes - downvotes) / (time + 2)^gravity
- 关键设计：时间衰减防「老热门内容永霸榜」+ 分内容类型差异化衰减（新闻快、教程慢）+ 冷启动新内容保底曝光
- 融入排序：热门度作为独立特征输入排序模型，或作为 boosting 因子在召回阶段提升新热内容曝光

**项目印证**：类似 RAG 中 chunk 的检索频次统计——高频被检索且被采纳的计划可加权提升排名（类似热门度）。

### Q35 在电商推荐系统中，如何通过数据指标或用户行为分析发现推荐结果不精准的问题？有哪些常见诊断方法？

### Q37 在构建大模型推理流程时，什么情况下需要对模型输出进行逻辑引导？请举例说明其实现方式和优势。

**一句核心**：当任务需要结构化推理链或多步骤约束时做逻辑引导；实现 = CoT prompt + constrained decoding + 中间状态校验。

**展开**：
- 适用场景：① 多步推理任务（数学/诊断/规划）——需先理解→分析→结论，跳步容易错；② 合规约束——如金融报告的合规检查清单必须逐条覆盖
- 实现方式：① CoT/ReAct prompt——引导模型「先思考再行动」「按步骤 1-2-3 执行」；② constrained decoding——限制输出格式/标签集合/JSON schema；③ 中间校验——每步输出后验证其是否符合规则，不通过则回溯
- 优势：① 降低幻觉率——拆步减少了模型跳步瞎猜的空间；② 输出可解释——每一步都有中间推理；③ 可干预——中间步骤出错可在影响最终结果前纠正

**项目印证**：OpsPilot——planner 节点严格按要求输出 plan_steps（步骤-工具-参数-预期输出），debug 日志打印每步中间结果。

### Q39 在构建搜索排序模型时，如何为相关性、时效性、热门度等不同排序因素设定初始权重？有哪些常用方法？

**一句核心**：三种方法——人工拍定 + AB 调优（快速但粗）、线性加权 + grid search（有标注数据）、ML 模型自动学习（GBDT/DNN，需训练数据）。

**展开**：
- 人工拍定 + AB：先按经验设初始权重（如相关 0.6/时效 0.2/热门 0.2），上线 AB 测试看 NDCG/CTR 变化，逐步调参收敛
- Grid Search / 贝叶斯优化：有标注排序数据时，在验证集上 grid search 权重组合选 NDCG 最优那组，比人工拍定更系统
- 模型自动学习：将相关性/时效/热门作为特征输入排序模型（GBRank/ListNet/DNN），让模型自己学权重。优点：自动发现特征交互，缺点：需要训练数据
- 归一化前置：多路分数必须先归一化到同一尺度（min-max / z-score），再加权，否则量纲大的那路霸榜

**项目印证**：应用生成平台 ranking——RRF 免调权重（1/(k+rank) 自动融合向量 + BM25 排名），避免 α 超参调优的工程负担。

### Q40 在实际应用中，如何处理大模型生成的输出结果？有哪些方法检测和过滤格式错误、内容不合理或不符合要求的输出？

### Q41 请详细介绍 RAG 的技术原理、核心组件、工作流程以及实际应用中的优势和挑战。

**一句核心**：RAG = 离线建库 + 在线检索生成；核心组件：数据管道/embedding/向量库/检索器/rerank/生成器；优势 = 知识更新不重训，挑战 = 检索质量决定上限。

**展开**：
- 技术原理：将外部知识文档 embedding 后建 ANN 索引；推理时 query embedding → ANN 检索 top-k → rerank 精排 → 拼入 LLM context 生成答案
- 核心组件：数据管道（清洗/分块）、Embedding 模型、向量数据库（HNSW/IVF）、检索器（混合检索 + RRF）、Reranker（cross-encoder）、生成器（LLM + prompt 约束）
- 优势：知识实时更新不重训、可溯源引用、长尾知识覆盖好、成本低于微调
- 挑战：检索质量决定上限（召回失败 LLM 无米之炊）、chunk 粒度 tradeoff、新老文档分布漂移、知识库污染风险

**项目印证**：OpsPilot 完整 RAG 链路——retrieve_memory 检索历史 RCA → context 注入 planner → evidence 采集 → writeback 闭环。

### Q42 在 RAG 或其他基于上下文的 AI 系统中，当提供的参考上下文存在错误或不准确信息时，您会采取哪些策略确保模型输出的准确性？

### Q44 在 RAG 系统中，检索模块已准确返回相关文档，但最终生成的回答质量仍不理想。请分析可能原因并提出系统性调优策略。

### Q45 假设你有一个通用 LLM，希望将其应用于封闭集合标签分类任务（如情感分类、意图识别），有哪些有效方法确保模型输出严格限定在预定义标签范围内？请比较不同方法的优劣。

**一句核心**：四种方法——constrained decoding（最可靠）、prompt + 后处理正则（最灵活）、logit bias（中间路线）、微调（最彻底）。

**展开**：
- Constrained decoding：推理时限制 token 生成空间，强制每步 token 只能从标签映射的合法 token 中选择（如 guidance/outlines 库）。优点：100% 合规；缺点：需要库支持
- Prompt + 正则后处理：prompt 引导「只回答以下标签之一：A/B/C」，输出后用正则匹配提取。优点：零门槛任何 LLM 可用；缺点：不能保证 100% 合规
- Logit bias：设置非标签 token 的 logit = -∞，让模型不可能生成非法 token。优点：比 constrained decoding 更轻量；缺点：需要模型 API 支持 logit_bias 参数
- 微调：用分类头替换生成头（BertForSequenceClassification）或 SFT 只生成标签。优点：最彻底最优；缺点：需要标注数据 + 训练成本

**项目印证**：应用生成平台 Planner——通过 JSON schema 约束输出结构，Pydantic 校验，非法输出打回重试。

> 🏷️ M8

### Q46 请详细说明深度研究（Deep Research）与检索增强生成（RAG）在技术原理、应用场景以及优缺点方面的区别。

**一句核心**：RAG = 检索→生成（一轮或低轮次），Deep Research = 多轮迭代检索→综合分析→结构化报告；前者重效率，后者重深度。

**展开**：
- 技术原理：RAG——单次检索注入 context 直接生成；Deep Research——多轮迭代检索（类似 Self-RAG），每轮根据已发现信息调整检索方向，最终合成多源信息形成结构化研究报告
- 应用场景：RAG——实时问答/客服/知识库检索，要求秒级响应，答案简短精确；Deep Research——市场分析/竞品调研/学术综述，要求深度综合，可接受分钟级延迟
- RAG 优势：速度快（<1s）、成本低、架构简单、适合在线服务；RAG 劣势：浅层检索，难以做多源信息综合和深度推理
- Deep Research 优势：多轮深入、多源交叉验证、输出结构化长报告；Deep Research 劣势：耗时长（分钟级）、token 成本高、需要强大的 Agent 编排能力

**项目印证**：OpsPilot 介于两者之间——evidence_fanout 多源并行采集 + evidence_aggregate 聚合 + diagnose 推理，有一定深度但不达 Deep Research 的研究级深度。
