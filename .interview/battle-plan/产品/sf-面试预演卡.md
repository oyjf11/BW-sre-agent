## 0. 一根主线（所有题都挂在这上面）

**结构同构**：
> SRE 故障处置 = 发现异常 → 根因诊断 → 处置建议 → 人工审批 → 执行验证 → 复盘沉淀
> 场地规划优化 = 发现瓶颈 → 归因诊断 → 调度优化建议 → 人工确认 → 执行验证 → 复盘沉淀

**一句话定位**：我做的 OpsPilot 是一个「异常诊断 + 决策建议 + 人在环」的 Agent 母版，
把数据源从「数据库/K8s」换成「场地/班次/快件流」，就是场地规划 Agent。
**我转岗带来的不是物流经验，是「把不稳定的 AI 能力做成企业里可控、可评估、可落地的产品」这套方法论。**

## 2. 🔴 致命题：你没有物流/场地背景，凭什么做这方向的 AI 产品？

【考官想听什么】不心虚、有迁移逻辑、承认短板但给出补齐路径。**这题答崩=出局。**

【答法骨架】三段式：承认 → 桥接 → 补齐
1. **承认**：物流场地的 know-how 我确实需要快速补，这点我不回避。
2. **桥接（核心）**：但场地规划的 AI 问题本质是三类——**异常/瓶颈识别、根因归因、资源动态调度**，
   这三类我在运维 Agent 里全做过，**问题结构是同构的，只是业务变量换了一套**。
3. **补齐**：我学新业务的方式是**先抓约束和指标**——场地的核心约束是什么（产能、时效、成本、人力班次），
   优化目标是什么（快件流转次数、资源利用率、降本）。把这些摸清，AI 方案就是在已知约束下做诊断和调度。

> 钩子句：**「行业知识我两周能补，但『把 AI 诊断决策做成可落地产品』这套方法论，是要攒好几年的——我带来的是后者。」**

⚠️ 防反问「那你怎么快速补行业知识」：答「先找 3 个一线场地主管聊真实痛点，把他们排班/调度的经验规则，
变成 Agent 的诊断规则和评估基线——这正是我做 RAG 意图增强和 Bad Case 迭代的强项。」

## 3. 🔴 现场设计题：设计一个「场地瓶颈调优」Agent，你怎么做？

【考官想听什么】结构化、有节点、有数据源、有人在环、有评估。直接复用你的 13 节点母版。

【答法骨架】照搬 OpsPilot 闭环，换业务皮：
1. **接入/感知**：接入场地实时数据——分拣量、班次排班、设备状态、快件流转路径、积压告警。
2. **分诊 triage**：判断瓶颈类型（产能不足 / 排班错配 / 流转绕路 / 设备异常），定优先级。
3. **多源证据采集（扇出）**：并行拉指标、历史同期、班次表、设备日志——**只读、并行、防误判**。
4. **归因诊断 diagnose**：基于证据链输出**可能根因 + 影响范围 + 置信度**（不是只给结论，要给证据链）。
5. **优化建议 remediation**：给调度方案——调班、资源融通、流转路径优化，每条带**预期收益 + 风险 + 回滚**。
6. **风险门禁 + 人工审批**：高影响动作（改排班、调产能）必须人工确认，**Agent 永远不直接拍板生产操作**。
7. **执行 + 验证**：执行后回采指标验证是否真的降了瓶颈，没降就回滚。
8. **RCA 沉淀**：把这次瓶颈→处置→效果结构化存档，喂回知识库，下次同类问题更快。

> 钩子句：**「核心不是让 AI 替人决策，是让 AI 把『从瓶颈到可执行方案』的链路从几小时压到几分钟，人只做最后确认。」**

加分：主动说「指标我会先定**离线评估基线**——拿历史瓶颈事件回放，看 Agent 的根因命中率和方案采纳率，达标才上灰度。」

## 4. 🔴 业务价值：怎么评估这个 AI 产品的价值 / 指标体系？（你的弱项，重点背）

【考官想听什么】你不是只懂技术，懂怎么衡量产品值不值得做。**用户自评弱项，必须打磨。**

【答法骨架】分三层指标：
- **业务结果层（北极星）**：JD 直接给了——**快件流转次数下降%、资源利用率↑、单票成本↓、时效达成率↑**。这是老板买单的。
- **产品效能层**：瓶颈发现到方案产出的**时长压缩**（如 2 小时→10 分钟）、方案**采纳率**、覆盖场地数。
- **AI 质量层**：根因**命中率/准确率**、误报率、人工修正率——这层决定产品能不能从灰度走向铺开。
- **怎么落地评估**：先离线回放历史事件建基线 → 小范围灰度（1–2 个场地）跑 A/B，对照人工组 → 达标再铺开。

> 钩子句：**「AI 产品的价值不看模型多强，看它在真实约束下，把哪个业务指标实打实推动了多少——其余都是过程指标。」**

⚠️ 防深挖「采纳率低怎么办」：答「采纳率低说明方案不接地气，回去做 Bad Case 迭代——
把一线主管否掉的方案变成新规则/新语料，这是我做研发套件质量门禁时的标准打法。」

## 5. 深挖：你的运维 Agent，根因诊断(RCA)具体怎么做的？

【考官想听什么】技术不是 PPT，你真懂里面怎么转的。

【答法骨架】
- 不是一步出根因，是**证据驱动**：先多源**只读**采集（指标、日志、部署变更、告警上下文），
  再让 LLM 基于**证据链**输出「可能根因 + 影响范围 + 置信度 + 处置步骤 + 回滚」。
- 关键设计：**有 critic 节点做自我质疑**，diagnose 出结论后再过一遍证据是否支撑，降低 LLM 误判。
- 产物是**结构化 RCA**：背景、时间线、证据链、根因判断、处置过程、验证结果、改进建议——可复盘、可沉淀回知识库。

> 钩子句：**「我的设计原则是『LLM 负责假设，证据负责证伪』，不让模型空口下结论。」**

⚠️ 你不熟 Python 实现细节时的护城河：把话题拉回**产品/架构决策**——
「具体 Python 实现是和工程一起落的，我负责的是节点怎么拆、证据怎么采、人在环卡在哪、风险边界怎么定。」
（你是 AI 产品/策略岗，谈设计取舍比谈代码行更对路。）

## 6. 聚类分析在场地场景里用在哪？（JD 第4条点名）

【考官想听什么】你知道这个算法用在哪，不是背名词。

【答法骨架】两个落点：
1. **瓶颈模式聚类**：把历史瓶颈事件按特征（时段、品类、流向、设备）聚类，
   识别出「反复出现的几类典型瓶颈」，针对性出预案——而不是每次从零诊断。
2. **场地/班次画像聚类**：把相似场地或相似时令模式聚成簇，**让一个场地的优化经验迁移到同簇场地**。

> 钩子句：**「聚类在我这是『把一次性诊断变成可复用预案』的工具——把长尾问题收敛成几个母题。」**

## 7. 动态资源调度怎么理解？（加分项 → 对应「资源融通降本」）

【答法骨架】
- 调度本质是**在约束下做分配**：产能、人力班次、设备在不同场地/时段间动态融通，目标是降本 + 保时效。
- AI 的角色：**预测需求波动（时令/高峰）→ 给出调度建议 → 人工确认 → 执行后验证收益**，是辅助决策不是全自动。
- 诚实边界：**强化学习/在线最优调度我了解原理、知道适用边界，但我会先用「规则 + 预测 + 人在环」把闭环跑通，
  再判断哪些环节值得上 RL**——先有可控基线，再谈高级优化。

> 钩子句：**「我不会一上来就上强化学习，先用规则把闭环跑通、把指标立起来，再决定哪里值得更复杂的算法。」**

## 8. HITL / 审批 / 高风险操作安全边界怎么设计？

【答法骨架】
- 分级：**只读诊断**全自动；**写操作/影响生产的调度**必须过**风险门禁 + 人工审批**。
- 机制：Agent 输出建议时强制带**风险等级 + 影响范围 + 回滚方案**，审批人看到这些才决策。
- 执行侧：**幂等执行 + 全程审计**，能模拟执行(dry-run)先验证，再受控执行。

> 钩子句：**「自动化的边界由风险定，不由能力定——能做不等于该让 AI 自己做。」**

## 9. AutoML / 强化学习了解吗？（加分项，重点是不露怯）

【答法骨架】诚实 + 有判断，别硬装：
- **了解原理和适用场景，没有深度调参实战**——这点我直说。
- AutoML 适合**特征/模型选型自动化**，能降算法门槛；RL 适合**序列决策、长期收益**的调度场景。
- 但我的产品判断是：**这些是「优化项」不是「起步项」**。先用可解释的规则 + 预测 + 人在环把业务闭环和指标跑通，
  再用 AutoML/RL 去抠那 10% 的最优——否则一上来上 RL，黑盒、难调、业务不敢用。

> 钩子句：**「我对它们的态度是『知道什么时候该上、什么时候不该上』，这比会调参对产品更重要。」**
