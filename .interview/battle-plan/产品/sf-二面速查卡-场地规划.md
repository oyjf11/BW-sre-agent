### 卡 1 · 自我介绍（场地版，45 秒）
- 【想听】你是谁、为何适配、一句钩子。
- 【可照说】
  > "面试官好。我有 7 年研发背景，最近一年专注在 AI Agent 的落地，做过三个 Agent 产品：一个自然语言生成内部应用的平台、一个带质量门禁的研发提效工具、还有一个生产级的运维诊断 Agent。这一年我形成的能力主线是——把 AI 从一个能跑的 Demo，推到企业里真正可控、可评估、能落地的产品，具体就是任务编排、工具接入、质量验证、人在环、反馈迭代这套工程机制。其中运维诊断那个 Agent，做的就是『异常发现 → 根因诊断 → 处置建议 → 人工审批 → 执行验证』这条闭环。我看到咱们这个场地规划的岗位，本质是同一套闭环换了一套数据源——把数据库、K8s 换成场地、班次、快件流，所以我特别感兴趣。一句话定位我自己：**我擅长的不是某个行业，而是把 AI 的诊断和决策做成企业里可控、可评估的产品闭环。**"

### 卡 2 · 🔴 无场地背景（致命题，答崩=出局）
- 【想听】不心虚、有迁移逻辑、承认短板 + 补齐路径。
- 【骨架】承认 → 桥接 → 补齐。
- 【可照说】
  > "我先坦白，物流场地的 know-how 我确实需要快速补，这点我不回避。但我想说的是，场地规划里的 AI 问题，本质就三类——瓶颈和异常的识别、根因的归因、资源的动态调度——这三类我在运维 Agent 里全做过一遍，问题结构是同构的，只是业务变量换了一套。至于怎么补行业，我的方式是先抓约束和指标：场地的核心约束无非是产能、时效、成本、人力班次，优化目标无非是把流转次数往下压、资源利用率往上提、单票成本降下来。这些摸清了，AI 方案就是在已知约束下做诊断和调度。所以我的总结是——**行业知识我两周能补上，但『把 AI 诊断决策做成可落地产品』这套方法论，是要攒好几年的，我带来的恰恰是后者。**"
- ⚠️ 防"怎么快速补行业知识"：找 3 个一线场地主管聊真实痛点，把他们排班/调度的经验规则变成 Agent 的诊断规则和评估基线——这正是我做 RAG 意图增强和 Bad Case 迭代的强项。

### 卡 3 · 🔴 现场设计：场地瓶颈调优 Agent（8 节点）
- 【想听】结构化、有节点、有数据源、有人在环、有评估。
- 【骨架】感知 → 分诊 → 证据扇出 → 归因 → 建议 → 风险门禁/审批 → 执行/验证 → RCA 沉淀。
- 【可照说】
  > "我会直接复用我运维 Agent 验证过的闭环，换成场地的业务皮，大概八步：第一步**感知接入**，把场地实时数据接进来——分拣量、班次排班、设备状态、快件流转路径、积压告警。第二步**分诊**，先判断这是哪类瓶颈：产能不足、排班错配、流转绕路还是设备异常，定优先级。第三步**多源证据采集**，并行去拉指标、历史同期、班次表、设备日志——这里强调三个词：只读、并行、防误判。第四步**归因诊断**，让模型基于证据链给出『可能根因 + 影响范围 + 置信度』，不是只甩一个结论。第五步**优化建议**，给调班、资源融通、流转路径优化这些方案，每条都带『预期收益 + 风险 + 回滚』。第六步**风险门禁加人工审批**，像改排班、调产能这种高影响动作，必须人确认，Agent 永远不直接拍板生产操作。第七步**执行加验证**，执行完回采指标看瓶颈是不是真降了，没降就回滚。第八步**RCA 沉淀**，把这次『瓶颈→处置→效果』结构化存档喂回知识库，下次同类问题更快。**核心不是让 AI 替人决策，是把『从瓶颈到可执行方案』的链路从几小时压到几分钟，人只做最后确认。**"
- ⚠️ 加分主动补：上线前我会先定**离线评估基线**——拿历史瓶颈事件回放，看根因命中率和方案采纳率，达标才上灰度。

### 卡 4 · 🔴 价值 / 指标体系（你的弱项，重点背）
- 【想听】你不止懂技术，懂怎么衡量产品值不值得做。
- 【骨架】三层：业务结果层（北极星）/ 产品效能层 / AI 质量层 + 离线回放→灰度A/B。
- 【可照说】
  > "我会把指标分三层讲。最上面是**业务结果层，也就是北极星**，JD 里其实直接给了——快件流转次数下降的百分比、资源利用率、单票成本、时效达成率，这层是老板买单的。中间是**产品效能层**——瓶颈从发现到方案产出的时长压缩，比如从两小时到十分钟、方案的采纳率、覆盖了多少个场地。最下面是**AI 质量层**——根因命中率、误报率、人工修正率，这层决定产品能不能从灰度走向全面铺开。怎么落地评估？我会先用历史事件离线回放建一个基线，再挑一两个场地做灰度，AI 组对人工组跑 A/B，达标了再铺开。**一句话：AI 产品的价值不看模型多强，看它在真实约束下，把哪个业务指标实打实推动了多少——其余都是过程指标。**"
- ⚠️ 防"采纳率低怎么办"：采纳率低就是方案不接地气，回去做 Bad Case 迭代——把一线主管否掉的方案变成新规则/新语料，这是我做研发套件质量门禁的标准打法。

### 卡 5 · RCA 根因诊断怎么做（产品视角短版，深度见 2.5-B）
- 【骨架】证据驱动、非一步出结论：多源只读采集 → LLM 基于证据链出「根因+影响+置信度+处置+回滚」→ critic 节点自我质疑 → 结构化 RCA 沉淀。
- 🔑「LLM 负责假设，证据负责证伪。」
- ⚠️ 不熟实现就拉回产品/架构（万能句）。

### 卡 6 · 聚类用在哪（产品视角短版，方法/工具/取舍见 2.5-A）
- 【骨架】① 瓶颈模式聚类——历史瓶颈按时段/品类/流向/设备聚类，识别"反复出现的几类典型瓶颈"出预案；② 场地/班次画像聚类——相似场地聚簇，让一个场地的经验迁移到同簇场地。
- 🔑「聚类在我这是『把一次性诊断变成可复用预案』的工具——把长尾收敛成几个母题。」

### 卡 7 · 动态资源调度（短版，深度见 2.5-F）
- 【骨架】调度本质是约束下做分配；AI 角色 = 预测波动 → 给建议 → 人工确认 → 验证收益，辅助非全自动。
- 🔑「我不会一上来就上强化学习，先用规则把闭环跑通、把指标立起来，再决定哪里值得更复杂的算法。」

### 卡 8 · HITL / 高风险操作安全边界
- 【骨架】**分级**：只读诊断全自动；写操作/影响生产的调度必须过风险门禁 + 人工审批。**机制**：建议强制带「风险等级 + 影响范围 + 回滚方案」。**执行**：幂等执行 + 全程审计，dry-run 先验证再受控执行。
- 🔑「自动化的边界由风险定，不由能力定——能做不等于该让 AI 自己做。」

### 卡 9 · 工程师转 PM（致命题，提前讲不等质疑）— 见 Part 3 角度 4 的可照说
- 【骨架】① 过去一年实际承担的就是产品工作（场景拆解/工作流设计/MVP 范围/试点推进/指标设计）；② AI PM 能力结构在变，懂编排/评估/工具接入约束的 PM 写的需求才可控；我能直接和算法对话。
- 🔑「我是先做了这个角色，再决定追认这个身份，不是凭想象转型。」

- 【骨架】诚实区分：整体规划团队共创；我**主导**的是 ××（前端 + UI 自动化验证能力规划、Sandpack 预览调试体验）；**深度参与并推动**的是 ××（结构化中间产物机制、安全边界设计）。备一个**我个人拍板、有取舍的决策故事**。

### 卡 11 · 大模型幻觉/出错造成客诉（强项题）

### 2.5-A · 聚类分析：你了解哪些方法？用什么工具平台？各自取舍？
- 【可照说】
  > "聚类我一般按数据形态和业务诉求来选，不迷信单一算法，我分三档讲。第一档是 **K-Means**，最快最常用，适合簇大致是球形、数量能预估的场景，比如把场地按吞吐规模分档；缺点是要预先定 K、对噪声和非凸形状很敏感。第二档是密度类的 **DBSCAN 和它的升级版 HDBSCAN**——现在业界做探索性聚类基本首选 HDBSCAN，因为它不用预先定簇数、能识别任意形状、还能把噪声单独拎出来不硬塞进簇，对真实世界那种密度不均、带长尾的脏数据特别稳，参数也就一个最小簇大小、很直观；代价是计算比 K-Means 重一点。第三档是 **高斯混合 GMM 和层次聚类**，GMM 给的是软划分带概率，适合一个场地可能同时属于几类模式的情况；层次聚类能出一棵树，适合我想看不同粒度下的分群。**工具上**，中小数据量我直接用 scikit-learn，HDBSCAN 现在也已经进 sklearn 原生了；数据量上到千万级要分布式，就上 Spark MLlib。**落到场地**，我会先用 HDBSCAN 对历史瓶颈事件做探索性聚类，把反复出现的几类典型瓶颈先聚出来，等模式稳定了、要做线上稳定分档时，再换 K-Means 这种轻量的。**一句话：聚类对我是把一次性诊断变成可复用预案的工具，选型看数据形态，不看算法时髦。**"

### 2.5-B · 根因诊断算法：除了 LLM 看证据，业界还有哪些 RCA 方法？
- 【可照说】
  > "根因诊断我分『传统 AIOps 算法』和『LLM 增强』两条线讲，而且我的观点很明确——**业界正在从相关性走向因果性**。传统方法大概四类：一是**基于异常检测+相关性**的，先对多个指标做异常检测再做事件关联聚类，很多平台的告警降噪就是这个，优点快、缺点只能告诉你『一起出问题了』，分不清因果；二是**拓扑图方法**，用设备、班次、分拣线之间的依赖拓扑去定位故障传播路径；三是**因果图 / 贝叶斯网络**，像阿里的 CloudRCA、eBay 的 Groot 这类，真正去建因果关系做推断，精度高但构图慢、实现重；四是**知识图谱方法**，把历史故障沉淀成运维知识图谱，新故障来了做链接预测。我自己运维 Agent 走的是『**证据驱动 + LLM 归纳**』：多源只读把指标、日志、变更、告警上下文采全，LLM 基于证据链出根因假设，再用一个 **critic 节点做自我质疑**，反过来验证证据撑不撑得住这个结论。**我的设计原则是 LLM 负责提假设、证据负责证伪，不让模型空口下结论。** 落到场地，我会先用规则加拓扑这种可解释的方法把闭环跑通，因果推断这种重武器，留给真正反复出现、值得深挖的瓶颈。"

### 2.5-C · AI 模型监控：模型上线后你怎么监控？（要求4 + 加分项）
- 【可照说】
  > "模型监控我盯三层。第一层是**数据漂移**，就是模型收到的输入特征分布变了——比如某类快件量突然异常，我用 **PSI、KL 散度、JS 散度**这类分布距离指标去衡量。第二层是**概念漂移**，就是输入和结果之间的关系变了、模型悄悄失准；难点是标签往往滞后——一个调度建议好不好要等执行完才知道，所以滞后期我用**预测分布漂移做代理指标**先报警。第三层是**业务质量指标**——根因命中率、方案采纳率、人工修正率，这层直接决定产品能不能从灰度走向铺开。**工具上**：开源我倾向 **Evidently**，Python 库、能塞进流水线、出报告方便；要企业级、尤其要监控 embedding 这种非结构化漂移，业界常用 **Arize、Fiddler**；要强隐私、海量吞吐就 **WhyLabs**；基础设施层接 **Prometheus + Grafana** 做实时告警。**我的原则是：模型不会像服务器宕机那样明着挂，它是悄悄变差的，所以监控必须独立于业务系统单独建，而且要能从『漂移』一路追到『哪个特征、哪个根因』。**"

### 2.5-D · 机器学习 / 深度学习应用：你实际用过哪些？
- 【可照说】
  > "我的定位是 AI 产品和策略，不是天天调参的算法，但我对常用的 ML/DL 有实操级理解，知道什么场景该上什么。我项目里实际用到的：检索增强里的 **embedding 向量召回 + rerank 重排**；意图识别本质是个**分类问题**；Bad Case 迭代其实就是在持续优化训练和评估集。落到场地，我能想到的深度学习落点是**件量的需求和波动预测**——时令、高峰的预测，这种时序问题业界主流是 **LSTM 或时序 Transformer**，再不济用 **XGBoost / LightGBM 这类梯度提升树配特征工程**也很能打，而且更可解释、更好上线。**我的判断一向是：能用可解释的树模型解决的，先别上深度模型；深度模型留给特征实在抓不动的非结构化场景。**"

### 2.5-E · 数据分析方法：你怎么做数据分析？
- 【可照说】
  > "我做数据分析有个固定顺序。先**描述统计**摸清分布——均值、分位数、长尾，场地里很多问题就藏在 P95、P99 那条尾巴上。再做**分组对比和相关性分析**，找哪些因子和瓶颈强相关。但我会特别强调一句：**相关不等于因果**，所以涉及『要不要为这个上方案』的判断，我一定靠**对照实验**说话——这就接到我评估 AI 价值的方法，灰度场地 AI 组对人工组跑 A/B，差值才是真贡献。工具上 SQL 加 Python 的 pandas 是基本盘，可视化 Grafana 或 BI 都行。**一句话：我用数据是为了做决策，不是为了出报表，所以我盯的是『这个差异是不是真的、是不是 AI 带来的』。**"

### 2.5-F · AutoML / 强化学习 / 动态调度（加分项，深一点）
- 【可照说】
  > "这块我诚实——原理和适用边界我清楚，但没有深度调参实战，我直说。**AutoML** 解决的是特征和模型选型的自动化、降算法门槛，业界常用的像 **FLAML、AutoGluon、H2O**，调超参的 **Optuna**；它适合我们这种产品团队快速验证 baseline。**强化学习**适合序列决策、看长期收益的场景，场地的动态资源调度——产能、人力班次在不同场地时段间融通——理论上是 RL 的菜。但我的产品判断很明确：**这些是优化项，不是起步项**。调度我会先用『规则 + 需求预测 + 人在环』把闭环和指标跑通，因为 RL 黑盒、难调、业务一开始不敢用；等基线稳了、确实有那 10% 的最优值得抠，再上 RL。**我对它们的态度是——知道什么时候该上、什么时候不该上，这比会调参对产品更重要。**"

## Part 3 · 🎯 产品总监二面专区

> 总监 ≠ 一面（用人经理/技术）。一面验证"会不会做"，**总监验证：产品 sense、商业 ROI、组织推动力、人靠不靠谱、格局**。
> 通用心法：尽量落到**业务结果 / 取舍判断 / 怎么推动**，少谈技术实现。

### 角度 1 · 产品判断 & 优先级
- **潜台词**：你有没有"价值×可行性×风险"的取舍脑。
- **样题**：「给你这个方向，0-6 个月先做哪个场景？凭什么是它？」「资源只够做一件事，你砍谁？」
- **答法要点**：价值×可行性×风险打分；从能力长板出发先选**瓶颈调优/异常诊断**（同构闭环我做过，知道坑）；第一个项目要把意图识别、知识库、工具接入层沉淀成**中台资产**给后面铺路。
- 🔑「第一波必须选财务能认账的场景，6 个月交出 ROI，团队才拿得到第二波预算和信任。」
- ⚠️ 防"为什么不先做最大价值的"：价值大但工程复杂度高（如语音实时）落地初期 ROI 不确定；先选确定能验证的，用数据赢下一级授权。

### 角度 2 · 商业价值 / ROI 落账（**重点，工程师转 PM 最易露怯**）
- **潜台词**：他要向上汇报，必须看到你能把 AI 翻成财务能认的数。
- **样题**：「这东西做出来到底省了多少钱？怎么证明是你的功劳不是大盘波动？」
- 【可照说】
  > "我算 ROI 不会只说『提效很多』，我会拆成三件事。第一，**收益锚到 JD 给的硬指标**：流转次数下降多少、资源利用率提升多少、单票成本降多少，这些能直接折算成钱。第二，**归因靠对照**——我不会把灰度期间所有的改善都算成 AI 的功劳，我会做对照组，AI 介入的场地对没介入的场地、或者 AI 组对人工组，**两边的差值才是能写进 ROI 报表的那个数**，这样老板问起来我扛得住。第三，**区分一次性投入和持续收益**，把回本周期讲清楚。如果短期还看不到财务收益，我会先交『过程指标 + 影子模式的一致率』当里程碑，让老板看到进展，别让项目在沉默期被砍掉。"
- 🔑「我证明价值的方式是对照组——AI 组对人工组的差值，才是能写进 ROI 报表的那个数。」

### 角度 3 · 组织推动力 / 跨团队协作（**重点**）
- **潜台词**：总监买的是"能在大组织把事推动落地的人"，不是只会画方案的人。
- **样题**：「算法团队不配合、一线主管不买账、数据拿不到，你怎么推？」「你和算法团队怎么分工，会不会因为懂技术越界？」
- 【可照说】
  > "先说和算法团队的分工，我的边界很清楚：**算法回答能不能、怎么实现，我回答值不值、该不该、做到什么程度算可用**。我懂技术不是为了替他们选型，而是三件事：把业务需求翻成算法能接、能评估的指标，比如把『识别要准』翻成『专有名词命中率大于多少、低置信度自动转人工』；在 trade-off 讨论里代表业务和成本拍板；以及识别那些技术上炫但业务上不值的方案，帮团队省力气。再说一线不买账——这其实是好事，说明方案不接地气，我会**把一线主管拉进 Bad Case 迭代**，让他们否掉的方案变成新规则、新语料，让他们从『被替代的人』变成『共建的人』，这是我做研发套件质量门禁时的标准打法。数据拿不到通常是流程和信任问题，我会先用一个小场景、小数据量把价值证出来，拿结果换更大的数据授权。"
- 🔑「我懂技术不是为了替算法选型，是为了让业务需求落到算法能接、能评估的地方。」

### 角度 4 · 转型动机 & 稳定性 / 职级落差（致命）
- **潜台词**：工程师 + OD 转 PM，他在赌你靠不靠谱、是不是跳板、扛不扛得起 owner。
- **样题**：「你工程师背景凭什么做 PM？」「为什么来顺丰、为什么这个方向？」「这岗要扛 owner，hold 得住吗？」
- 【可照说】
  > "我转产品有两个原因。第一，过去一年我实际承担的就是产品工作——场景拆解、工作流设计、MVP 范围决策、试点推进、指标设计；我 title 是工程师，但我在团队里回答的问题是『做什么、做到什么程度算可用、先给谁用』，**我是先做了这个角色，再决定追认这个身份，不是凭想象转型**。第二，AI 产品经理这个岗的能力结构正在变——AI 能力不稳定、边界模糊，PM 如果不懂任务编排、不懂评估、不懂工具接入的约束，写出来的需求要么做不了、要么不可控，我的工程背景让我能直接和算法对话，把业务翻成可验证的方案。至于为什么是这个方向——我做的三个 Agent 都是『诊断 + 决策 + 人在环』，运维场景已经验证了闭环价值，**场地规划是同一套方法论遇到一个数据更密、收益更直接见钱的业务，对我是把能力放到更大杠杆的场景，所以我是主动找这类岗位的**。职级和 owner 我不回避：我做运维智能体时实际就在定义问题、设计闭环、推动落地，title 不代表我承担的责任，我要的就是一个能扛 owner 的位置。"
- 🔑「我做的三个 Agent 都是诊断+决策+人在环，场地规划是同一套方法论遇到一个收益更直接见钱的业务——这是我主动找这类岗位的原因。」

### 角度 5 · 取舍故事 / 失败复盘
- **潜台词**：看你真拍过板、踩过坑、能复盘，而非 PPT 选手。
- **样题**：「讲一个你砍掉的方案 / 坚持加上的设计。」「讲一个你做错的产品判断，后来怎么修的。」
- **答法要点**：备 1 个**取舍故事**（坚持加 critic 自我质疑节点 / 砍掉一步到位生成方案，理由"不追求一步生成，追求每步可承接可验证"）+ 1 个**失败故事**（如检查点持久化没做好，流程转人工后续不回来，复盘后补上）。结构：背景→我的判断→代价→怎么修→沉淀成什么机制。
- 🔑「检查点持久化没做好，流程转人工后就续不回来——这种坑我踩过，所以现在我设计闭环第一件事就是问『断了怎么接回来』。」
- 📝 **待你填**：挑一个**你真拍过板、能被追问两层细节**的，别用我编的。

### 角度 6 · 业务理解速度（没行业经验时他怎么验你）
- **潜台词**：行业知识能补，但你"进入业务的方式"对不对，决定补得快不快。
- **样题**：「你怎么理解场地规划这件事？瓶颈到底从哪来？」「入职前 30 天你做什么？」
- **答法要点**：先抓约束（产能/时效/成本/人力班次）和目标（流转次数↓/利用率↑/降本）；瓶颈来源用假设框架（产能不足/排班错配/流转绕路/设备异常/时令冲击）并坦诚"权重要去一线验证"。30 天：下一线（跟分拣、跟排班、跟一票件全程）+ 拉数据（瓶颈分布、各环节人力成本、系统数据可用性）。
- 🔑「需求洞察必须来自现场——我前 30 天就泡场地，把一线主管的经验规则变成 Agent 的诊断基线。」
- ⚠️ 不确定别硬说：「我目前是公开资料 + 同行案例推演的理解，深度认知需要入职后到一线建立。」诚实在总监这里加分。

### 角度 7 · 格局 / 行业判断（总监爱聊远景）
- **潜台词**：看你认知天花板，能不能在战略层对话。
- **样题**：「2-3 年 Agent 在物流/场地的机会在哪？」「顺丰做 AI 的护城河是什么？」
- **答法要点**：三段式——眼下到明年是语言密集型岗位自动化（客服/单证/调度交互，ROI 清晰）；2-3 年 Agent 成运营系统的"自然语言外壳 + 异常处置副驾"（OR 算法继续优化，Agent 管交互和异常兜底）；更远是跨企业供应链协同。护城河——**不是模型是数据和场景闭环**：顺丰有全国最完整的轨迹/时效/异常/场地数据，谁先把它封装成 Agent 可调用的工具层（MCP 这类标准在做的事），谁就把数据优势变成产品优势。
- 🔑「顺丰的护城河不是模型，是数据和场景闭环——谁先把场地数据封装成 Agent 能调的工具层，谁就赢了。」
