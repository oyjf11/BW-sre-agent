# 实战沉淀 · OpsPilot 全旅程主干（从一句人话到一份复盘）

> 🎯 一句话 kernel（背这句）：**一条报障进来，13 个节点把它收敛成「结构化工单 → 并行取证 → 质量门控 → 受控修复 → 复盘归档」——每一步都能退回重来，关键动作卡人工审批。**

## 30 秒开口（"讲讲你做的这个 Agent" 的标准开场）

OpsPilot 是个 SRE 故障处置 Agent，用 LangGraph 编排 13 个节点。我顺着一条**故障生命线**讲，5 段：

1. **入口**：报障进来，先归一成一张标准工单。
2. **定性**：先分类（triage），再查"以前遇到过没"（捞记忆）。
3. **取证**：排调查计划，多专科 Agent 并行只读取证，聚合打质量分，critic 门控——证据不够就退回重查。
4. **处置**：出方案 → 过风险闸门 → 高危卡人工审批 → 受控执行 → 验真。
5. **收尾**：成败都产出一份 RCA 复盘归档。

## 详细复盘

> 全程的"交通规则"是 **5 个条件路由器**（`builder.py`），决定每段走完往哪拐——"可退回重来"的物理实现，深挖见「条件路由机制」篇。下面每段末尾标了它拐去哪。
>
> 节点之间**不直接互相调用**，全靠往共享的 `IncidentAgentState`（`state.py:36`）里**写 key、下游读 key** 传递——所以"中间产物"＝该节点新写进 state 的字段。每段开头我用 `【吃】→【吐】` 标出它的输入输出 key。

### §1 入口段：一句话 → 一张标准工单（`intake_node`）

`【吃】ticket(原始)` → `【吐】ticket(归一化) / status=NEW / 初始化 evidence_items=[] 等`

入口 API `POST /incidents/runs`（`incidents.py:193`）吃**三种互斥模式**，只能给一种（`:195` 校验，多给少给都 422）：

- 模式 A 手工工单：title/description/service/env/severity 一栏栏填；
- 模式 B `ticket_id`：去工单系统拉（`from_ticket_id`）；
- 模式 C `alert_event`：告警事件转工单（`from_alert_event`）。

三种殊途同归出一个 `IncidentTicket`，喂给 `intake_node`（`__init__.py:49`）。intake 很轻：**归一化**（`normalize_env/service/time_range`，把"生产/prod/PRODUCTION"抹平）+ 初始化几个空列表。

**为什么这段是命脉**：下游 13 节点全假设手里有一张**字段干净的工单**——planner 拿 `service/env` 定计划、risk_gate 拿 `env/severity` 定风险、rca 拿 `title` 写复盘。归一化放最前面，就是别让"生产"和"prod"在 risk_gate 里被当成两个环境。

**诚实缺口（这篇的 steelman 点，跟 critic 篇"接口先行"同款）**：标题写"从自然语言输入开始"，但**代码现状没有任何 NL→工单解析**——前端是结构化表单，那个 description 框只是工单字段、不会被 LLM 抽取。
- 被追问"那不就没做"：**插入点明确、契约现成**。点亮只需在 `intake` 前加一个 `nl_parse` 节点，用一次 LLM tool-calling 把自由文本抽成 `IncidentTicket` 字段，抽完丢给现有 intake，**下游一行不改**——因为下游只认归一化后的工单这个契约。
- 取舍：**先把契约和下游链路焊死、能独立验证，NL 抽取这种入口适配层留作下一刀**，它不改骨架，只是把人填表换成模型填表。

→ 路由：`intake → triage` 无条件直边（`builder.py:219`）。

### §2 定性段：先分类，再问"以前查过没"

`triage【吃】ticket →【吐】triage / requires_immediate_human`
`retrieve_memory【吃】ticket,triage →【吐】memory_hits（并 append evidence_items）`

**triage（`__init__.py:191`）两阶段定性**：
- 阶段 1 规则兜底（`_triage_by_rules:82`）：关键词命中粗分 `deployment_regression`(发布)/`resource_exhaustion`(资源)/`dependency_failure`(依赖)，命中即出 `TriageResult`。
- 阶段 2 LLM 补：规则没命中才掉 LLM。

为什么规则在前：**便宜、确定、可解释**先上，LLM 只兜长尾——和 critic 篇"矛盾检测用纯规则不花 token"同一个价值观。triage 还顺手算 `requires_immediate_human`（prod+P0/P1），一路漂到 risk_gate 影响风险裁决。

**retrieve_memory（`retrieve_memory.py`）捞历史**：拿工单检索过往相似故障/处置（RAG），出 `memory_hits` 供 planner/diagnose 参考。检索是**两阶段（粗排召回 + rerank 精排）**，深挖见 →「两阶段检索接入与AB归因」篇。

→ 路由：`triage → retrieve_memory → planner` 全直边，不分叉。

### §3 取证段：定计划 → 并行取证 → 聚合打分 → 门控（**心脏**）

```
planner   【吃】triage,ticket   →【吐】investigation_plan
fanout    【吃】investigation_plan →【吐】evidence_items[] / specialist_analyses
aggregate 【吃】evidence_items   →【吐】evidence_quality_score / contradiction_signals
diagnose  【吃】证据+记忆        →【吐】root_cause_candidates[]
critic    【吃】quality_score,候选,loop_count →【吐】critic_decision / loop_count+1
```

- **planner（:227）**：拿 triage 的 `incident_type/suspected_services`+env，**模板化**排一组 `InvestigationTask`（category/tool/priority/service）。判发布回归就排 logs+k8s+deployments 并行查。
- **evidence_fanout（:406）**：把任务**并发**打到 Tool Gateway / 专科 Agent（`agent_feature_specialist_pool` 开关：开走 v2 专科池，否则直连工具），每路只读取证、回包成 `EvidenceItem`。**关键性质：专科分头取证、只读、互不抢排他资源**——这是 critic"不投票只退回"的根因。
- **aggregate（:783）= 会计**：揉成 `evidence_quality_score`（`avg×截断惩罚×覆盖度`），同时跑跨层差分定位（6 条 `CONTRADICTION_RULES` 有向规则表），出 `contradiction_signals`。→ 公式/规则表细节全在「critic 仲裁打分」篇 §2/§3。
- **diagnose（:895）**：综合证据+记忆，出排好序的 `root_cause_candidates`（hypothesis+confidence）。
- **critic（:1074）= 门控**：自己不打分，读上游 `evidence_quality_score`+矛盾信号，做 **4 裁决**（CONTRADICTION>NEED_MORE_EVIDENCE>REPLAN>PASS），转 2 圈摆不平转人工。→ 全部裁决逻辑见「critic 仲裁打分」篇。

→ **路由（回退是全链精华，`_route_after_critic` :115）**：PASS→`remediation`；NEED_MORE_EVIDENCE→**退一步**回 `fanout`（计划不变再取证）；REPLAN/CONTRADICTION→**退两步**回 `planner`（重排计划）；NEEDS_HUMAN→`rca`。证据不硬，宁可绕回去再查也不急着下结论。

### §4 处置段：出方案 → 闸门 → 审批 → 执行 → 验真

```
remediation 【吃】候选,证据,ticket →【吐】remediation_plan
risk_gate   【吃】remediation_plan,ticket,候选 →【吐】risk_decision
approval    【吃】remediation_plan →【吐】pending_approval（图挂起→END）
executor    【吃】remediation_plan/approval_result →【吐】action_results/execution_results
verify      【吃】action_results →【吐】verify_decision（并 append 验证证据）
```

- **remediation（:1136）出方案（当前单方案）**：拿 top 候选+证据**规则化**映射动作：判发布回归且有部署证据 → `rollback`(HIGH,需审批)；否则兜底 `restart`(LOW)。
  > 诚实边界：**只出单方案、没有多方案择优**。真正的"方案评委"（多方案按恢复速度/影响面/回滚成本打分+dry-run）是缺口，要补在这加 `_score_remediation()`。
- **risk_gate（:1208）多信号闸门**：把 `动作风险+severity+env+requires_immediate_human+诊断置信度+loop guard+能力预检`揉成 4 选 1：`LOW_ONLY`(放行)/`NEEDS_APPROVAL`(卡审批)/`BLOCKED`(能力预检失败拦死)/`NEEDS_HUMAN`。能力预检细节：real 模式下 `execute_action` 没接真 adapter 就 fail-closed，提前 BLOCKED，不让跑到执行器才崩。
- **approval_interrupt（:1321）HITL 中断**：建 `pending_approval`，**图挂起→END**，run 停半途等人。人 `POST /approvals/{id}/decision` 点头后 `POST /runs/{id}/resume` **从断点续跑**。靠 dispatcher 的 `_resume_from_node`（`builder.py:101`），深挖见 →「dispatcher-resume」篇。
- **executor（:1477）受控执行**：`ControlledExecutor` 幂等+审计——同动作重复 resume 不会执行两次。
- **verify（:1596）验真**：执行完不信"我以为好了"，**再取一轮验证证据**（查 k8s deployment status/指标回落），3 选 1：`SUCCESS/RETRYABLE_FAILURE/FATAL_FAILURE`。
  → 路由 `_route_after_verify`（:146）：SUCCESS→rca；RETRYABLE→回 executor 重试（≤`MAX_EXECUTOR_RETRIES=2`）；耗尽/FATAL→rca。

### §5 收尾段：成败都写复盘（`rca_node` :1772）

`【吃】几乎全 state →【吐】rca_report / final_outcome / status=COMPLETED|NEEDS_HUMAN`

不管 SUCCESS/FAILED/NEEDS_HUMAN **都进 rca** 出 `RcaReport`：先定 `run_outcome`（转人工→NEEDS_HUMAN，失败/验证未过→FAILED，否则 SUCCESS）；再定根因状态（top 置信度 `>=0.7` 算确认、否则 UNKNOWN，转人工一律 UNKNOWN）；报告+证据归档 OSS。
**为什么失败也写**：复盘是给人看的交接物 + 数据飞轮原料（失败案例进 Golden set 防回退，见「数据反馈飞轮」篇）。`rca → END`。

### 🔭 横切：这条旅程怎么被实时看见

每个节点被 `_wrap_with_events`（`builder.py:25`）包一层，进节点发 `NODE_STARTED`、抛异常发 `NODE_FAILED`，事件**落 DB 带 sequence**。前端 **SSE** 实时渲染 Stepper 走到哪、断线凭 Last-Event-ID 从断点补发。深挖见 →「SSE 断线续传」篇。

## 面试问答底稿

**Q：把你这个 Agent 从头到尾讲一遍。**
**A：** 用 30 秒开口起手（5 段），然后甩三个设计点：①全程条件路由可退回重来；②高危 HITL 中断、人不点头不动手；③critic 先过证据门控才进处置。每段留一个钩子等面试官往里钻，钻哪段翻哪篇姊妹篇。

**Q：为什么搞 13 个节点，一个 ReAct 循环不行吗？**
**A：** 通用——shallow agent（裸工具循环）在"高风险、要可控、要可解释、要能中途叫人"会失控；deep agent 把规划/子代理隔离/HITL/复盘显式拆成节点换可控性（见「deepagents 对标」篇）。项目——OpsPilot 是生产运维，错一步搞大故障，所以宁可重：取证和处置间硬插 critic 门控、处置里硬插审批中断、执行后硬插验真。这些"多余"节点本质都是**闸门**，拿编排复杂度换"不闯祸"，每个职责单一、能独立测。

**Q：哪几段是缺口 / 没真跑通？（诚实边界，加分项）**
**A：** 三处，都清楚且有意留：①**入口 NL 抽取**没做，但插入点明确、契约现成（§1）；②**CONTRADICTION 仲裁**——检测在算、信号在灌，但没晋升到 critic 读的候选字段，那条分支当前不触发（critic 篇）；③**方案择优**——只出单方案，没有方案评委（§4）。共同点：**核心骨架和契约先焊死、能独立验证，适配/增强层留作下一刀**，不是没做完。

## 附录 A：核心数据结构字段中英对照（查得到就行，不用背）

> 旅程里 7 个最该看懂的数据结构逐字段中译。**记构词法比记字段省力**：`*_id`=唯一标识；`*_ids`=引用别物的 ID 列表；`*_score`/`confidence`=0~1 的分；`requires_*`=布尔闸门；`*_status`/`*_decision`/`*_type`=枚举。

**① `IncidentTicket` 标准工单**（入口产物）`models/incident.py`
`ticket_id` 工单 ID｜`title`/`description` 标题/描述｜`service` 受影响服务｜`env` 环境｜`severity` 级别(P1~P4)｜`time_range` 排查时间窗｜`source` 来源(pagerduty/alertmanager/manual)｜`metadata` 附加元数据

**② `TriageResult` 分诊结果**（定性产物）`models/triage.py`
`incident_type` 故障类型｜`severity` 调整后级别｜`suspected_services` 怀疑服务列表｜`suggested_time_window` 推荐时间窗｜**`requires_immediate_human` 是否需立即人工**(漂到 risk_gate)｜`rationale` 判断理由

**③ `EvidenceItem` 证据条目**（取证产物，全程累加）`models/evidence.py`
`evidence_id` 证据 ID｜`tool_name` 产出工具｜`category` 类别(logs/metrics/deployments/db/k8s/runbook/verification)｜`source_ref` 来源引用｜`source_timestamp` 证据源时间｜`summary` 摘要｜`raw_payload` 原始数据｜**`confidence` 可信度 0~1**(喂质量分)｜`freshness_score` 新鲜度｜`completeness_score` 完整度｜`tags`/`error_hint` 标签/错误提示

**④ `RootCauseCandidate` 根因候选**（诊断产物）`models/root_cause.py`
`candidate_id` 候选 ID｜`hypothesis` 根因假设｜**`confidence` 置信度 0~1**(critic/risk_gate/rca 都读)｜`incident_type` 封闭枚举分类(**评测主指标**)｜`supporting_evidence_ids` 支持证据｜**`contradicting_evidence_ids` 反驳证据**(CONTRADICTION 读它，待灌装)｜`next_checks` 后续检查建议

**⑤ `ActionSpec` 运维动作**（最小执行单元）`models/action.py`
`action_id` 动作 ID｜`action_type` 类型(restart/rollback/scale)｜`service`/`env` 目标服务/环境｜`params` 参数｜**`risk_level` 风险等级**(LOW/MED/HIGH/CRITICAL)｜**`requires_approval` 是否需审批**｜`idempotency_key` 幂等 Key｜`supporting_evidence_ids` 支撑证据｜`preconditions` 前置条件｜`verification_plan` 验证方案

**⑥ `RemediationPlan` 修复方案**（装一组 ActionSpec）`models/remediation.py`
`summary` 概述｜**`actions` 动作列表**(当前只 1 个)｜`expected_outcome` 期望效果｜`rollback_plan` 回滚步骤｜`risk_notes` 风险说明｜`human_checkpoints` 人工检查点

**⑦ `RcaReport` 复盘报告**（收尾产物）`models/rca.py`
`report_markdown` 完整报告｜`root_cause` 根因描述｜`root_cause_status` 状态(CONFIRMED/SUSPECTED/UNKNOWN)｜`resolution` 解决步骤｜`prevention_items` 预防措施｜`confirmed_by_human` 是否人工确认｜`timeline_summary` 时间线｜`impact_assessment` 影响评估｜`supporting_evidence_ids`/`executed_action_ids` 用到证据/执行动作｜`automation_outcome` 自动化结果｜`manual_next_steps` 人工接管步骤｜`archive_ref` 归档位置(OSS)
