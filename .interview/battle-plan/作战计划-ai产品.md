# AI 产品经理面试作战计划

> 目标岗位：AI 策略产品经理 / AI 平台产品经理，月 base 30k+
> 核心定位：有工程纵深、能把复杂 AI 能力产品化落地的候选人
> 作战原则：不装全能传统 PM，把工程纵深翻译成业务价值、可控流程和可验证结果

## 0. 作战理念

一句话：我不主打“通用型全能 PM”，我主打“懂工程、懂 Agent、能把 AI 从 Demo 推到可控试点的 AI 产品候选人”。

主动打的优势区：

- 复杂 AI 工作流拆解：Router / Planner、13 节点 SRE 流程、前端测试三防线。
- 可控与可信：证据链、质量门禁、Human-in-the-loop、risk_gate、审批与审计。
- 评估与迭代：Bad Case 闭环、离线评测、证据完整率、缺陷归因。
- 企业内部 AI 落地：从 PoC 到试点，从个人提效到团队流程。

只求不露怯的弱项区：

- C 端增长、商业化定价、大规模运营。
- 长期 Axure / 墨刀原型经验。
- 严格 A/B 实验和成熟数据分析平台实战。

统一话术：

> “我的背景不是传统 C 端 PM，而是从研发和 AI 工程化落地切入。我能把复杂业务拆成可执行流程，把不稳定的 AI 能力放进有边界、有证据、有评估、有审批的产品体系里。这是我相对纯业务 PM 的差异化。”

## 1. 两类岗位怎么打

### AI 策略 PM

考察重点：场景判断、业务价值、ROI、推广阻力、从 1 到 100。

打法：少讲底层代码，多讲“为什么这个场景值得上 AI、怎么灰度试点、怎么度量价值、怎么复制到更多团队”。

### AI 平台 PM

考察重点：能力抽象、平台治理、权限模型、评测体系、安全边界、开发者采用率。

打法：把开发套件、应用生成平台和 OpsPilot 都讲成“平台能力分层 + 流程治理 + 质量门禁”的案例。

## 2. 开场定位

### 策略 PM 版

> 我有 7 年研发和企业内部平台经验，近一年重点参与 AI 研发提效、应用生成和运维 Agent，也熟悉企业内部从试点到推广的阻力。我的判断是，AI 产品难点不只是模型能不能生成，而是能不能被放进可控、可验证、可推广的业务流程里。所以我更适合企业内部 AI 策略和落地型产品，把工程纵深翻译成业务价值、流程边界和可复盘结果。

### 平台 PM 版

> 我有前端和全栈研发背景，参与过内部 AI 工具链和应用生成平台建设，也理解业务团队为什么需要低门槛、可配置的能力。我熟悉 Agent Workflow、RAG 意图识别、质量门禁、证据采集和 Human-in-the-loop，知道这些能力要先变成稳定产品流程。做平台 PM 时，我的优势是既能理解业务侧的灵活性，也能守住平台侧的安全、评测、权限和治理边界。

## 3. 三个项目，两种讲法

### 3.1 开发套件：策略 PM 讲法

- 用户 / 场景：企业前端团队和业务研发在 AI 辅助开发后，需要快速确认页面功能、交互流程和关键回归是否真的可用。
- 核心痛点：传统验收依赖人工点击和主观判断，AI 生成代码后更容易出现“看起来完成、实际不可用”的隐性质量风险。
- 产品方案：把前端自动化测试做成可试点推广的质量闭环，用 Playwright 真跑、截图取证、五分类失败归因、质量门禁和 Bad Case 闭环替代口头承诺。
- 价值表达：它不是单个测试脚本能力，而是把 AI 研发提效从“生成更快”推进到“交付可信”，降低推广阻力并让团队敢用。

### 3.2 开发套件：平台 PM 讲法

- 用户 / 场景：内部 AI 开发套件平台、质量平台和多个项目团队，需要把前端测试能力复用到不同技术栈和环境。
- 核心痛点：各团队登录态、端口、路由、选择器和测试数据差异大，如果没有统一契约，平台能力会变成一次性脚本。
- 产品方案：抽象为测试任务输入、Playwright 执行、截图和日志证据、五分类归因、门禁阈值、Bad Case 回流等平台模块。
- 价值表达：平台侧统一证据格式和质量门禁，团队侧配置环境与阈值，既保证治理一致，又保留项目落地弹性。

### 3.3 应用生成平台：策略 PM 讲法

- 用户 / 场景：企业轻应用生成场景中，内部业务团队希望用自然语言生成应用，并在试用后持续做局部修改和流程调整。
- 核心痛点：一次性生成 Demo 容易惊艳，但修改阶段用户表达模糊，系统经常不知道是改样式、改字段、改逻辑还是改数据流。
- 产品方案：用 Router / Planner 拆分意图识别和任务规划，引入场景化 RAG 意图增强，再通过 Schema 中间产物、Sandpack 预览和增量修改稳定交付。
- 价值表达：把轻应用生成从“能生成首版”推进到“能持续迭代”，更适合企业内部从 PoC 走向业务试点。

### 3.4 应用生成平台：平台 PM 讲法

- 用户 / 场景：应用生成平台的内部开发者、模板维护者和业务集成方，需要沉淀可复用的生成、预览和修改能力。
- 核心痛点：如果 Router、Planner、渲染预览和增量修改都混在单次对话里，平台难以治理，也难以定位失败原因。
- 产品方案：把能力拆成意图路由、规划生成、Schema 中间产物、Sandpack 预览、增量修改和 RAG 案例库等可组合模块。
- 价值表达：平台 PM 讲法重点不是“我写了某段逻辑”，而是把不稳定生成能力拆成有边界、有输入输出、有评估口径的平台能力。

### 3.5 OpsPilot：策略 PM 讲法

- 用户 / 场景：SRE 故障处置场景中，值班研发和服务负责人需要快速完成告警接入、证据收集、诊断、修复建议、审批、验证和复盘。
- 核心痛点：故障处置高度依赖个人经验，日志、指标、变更和审批分散在不同系统里，事后也很难还原“为什么这么判断”。
- 产品方案：把 intake 到 rca 工作流做成可控处置链路，用 evidence_fanout 收集证据，用 risk_gate 和 approval_interrupt 控制高风险动作，用 verify 和 RCA 形成闭环。
- 价值表达：它把 SRE Agent 从“会聊天的诊断助手”变成“有证据、有审批、有验证、有复盘的处置流程”，更容易被企业接受。

### 3.6 OpsPilot：平台 PM 讲法

- 用户 / 场景：SRE 平台团队、运维工具团队和服务团队，需要把故障处置 Agent 接入现有监控、日志、工单和审批体系。
- 核心痛点：生产级 Agent 的风险不在于能不能给建议，而在于权限边界、证据可信度、状态恢复、人工审批和执行审计是否可治理。
- 产品方案：把 intake、triage、evidence_fanout、diagnose、risk_gate、approval_interrupt、verify、RCA 拆成平台流程节点，并统一 evidence、approval 和 trace 契约。
- 价值表达：平台 PM 讲法要强调 OpsPilot 是一套 SRE Agent Harness，可复用到不同服务和环境，而不是只服务单个故障场景的脚本。

## 5. 核心母题标准答案

### Q1. 什么场景适合上 AI？

**要害一句**：先看任务是否“高频、有可验证的正确边界、能嵌入现有流程”，再看 AI 能不能补位而不是另起一套。

**三点**：
- 任务高频且重复，人工成本高，AI 替代有规模效应
- 结果可验证、可度量，不是纯主观判断
- 能嵌入现有工作流，不强迫用户改变习惯

**套项目**：开发套件——前端测试是“高频、有明确通过/失败边界、可自动验证”的典型场景，所以适合先上 AI。

### Q2. AI 产品为什么 Demo 惊艳，落地困难？

**要害一句**：Demo 只验证“能不能生成”，落地要解决“稳不稳定、可不可控、值不值得信任”。

**三点**：
- Demo 场景干净，真实环境输入混乱、边界条件多
- 缺少证据链和可验证机制，用户不敢把结果直接用于生产
- 缺少异常恢复和审批机制，一旦出错代价不可控

**套项目**：应用生成平台——首版生成可以很惊艳，但增量修改阶段用户表达模糊，没有 Schema 中间产物和 RAG 意图增强，系统很快露怯。

### Q3. AI 产品北极星指标怎么设计？

**要害一句**：北极星指标要绑定“用户真正完成的业务结果”，而不是模型层面的技术指标。

**三点**：
- 从用户工作流终点往回推，而不是从模型能力往前堆
- 兼顾效率指标（完成时间、人工介入次数）和质量指标（返工率、缺陷率）
- 再配一个“信任指标”，衡量用户敢不敢直接用结果

**套项目**：OpsPilot——不是单看“诊断准确率”，而是看“根因候选命中率 + 高风险操作拦截率 + 审批后执行成功率”的组合。

### Q4. 怎么向 CTO / VP 汇报 ROI？

**要害一句**：用一个清晰的年度收益公式，把模糊的“提效”转成可拆解、可验证的数字结构。

**三点**：
- 公式化：年度收益 = 节省工时价值 + 质量收益 + 复用收益 - AI 运行成本 - 维护成本
- 把试点反馈估算和严格实验结论分开标注，不过度包装
- 给出下一步验证计划，而不是只报一个亮眼总数

**套项目**：开发套件——把“2.2 天到 4 小时”标注为试点反馈估算，放进收益公式里的“节省工时价值”分项，同时说明后续要补对照实验。

### Q5. 怎么定义一个 AI 功能好不好用？

**要害一句**：从业务工作流反推指标，不从模型指标出发。

**三点**：
- 先问用户要完成什么任务、哪一步最容易卡住或返工
- 把“好用”翻译成可观察的行为信号：完成率、返工率、复用率
- 让指标能直接指导产品迭代，而不是只用来对外汇报

**套项目**：应用生成平台——用“一次可预览成功率”和“人工修正轮次”衡量好用程度，而不是单纯看生成代码行数或模型打分。

### Q6. 企业 AI 治理怎么设计？

**要害一句**：治理的核心是把权限、数据隔离、高风险审批、审计、输出可追溯做成默认能力，而不是事后补丁。

**三点**：
- 权限和数据边界先于功能设计，明确谁能看什么、谁能做什么
- 高风险动作必须有审批和审计闭环，留痕可回溯
- 输出要可追溯到证据来源，出问题能定位责任环节

**套项目**：OpsPilot——risk_gate 和 approval_interrupt 把高风险操作挡在执行之前，executor 的幂等执行和审计日志保证事后可追溯。

### Q7. 试点成功后怎么从 1 到 100？

**要害一句**：复制不是简单铺量，而是靠种子团队、标杆案例、模板化复制、能力准入、周度复盘把经验沉淀成可复用资产。

**三点**：
- 先选有代表性的种子团队，跑出一个讲得清楚价值的标杆案例
- 把流程、模板、评测集、准入门槛标准化，降低复制成本
- 用周度复盘持续发现新 Bad Case 并反哺规则

**套项目**：开发套件——一个团队的测试闭环跑通后，把环境配置、阈值、归因规则模板化，新团队按准入清单接入，而不是重新摸索。

### Q8. Prompt 工程怎么产品化？

**要害一句**：把 Prompt 当成可版本管理、可评测、可回滚的产品资产，而不是一次性调试结果。

**三点**：
- 给 Prompt 建立版本和变更记录，关联评测集结果
- 把效果差异量化成可对比的指标，而不是凭感觉判断“更好”
- 出现 Bad Case 时能定位是 Prompt、Schema 还是流程的问题

**套项目**：应用生成平台——Router / Planner 的 Prompt 调整必须配合场景化 RAG 案例库和评测集回归，避免改了一处又破坏另一个场景。

### Q9. 平台统一治理和团队个性化怎么平衡？

**要害一句**：把能力拆成“平台统一的契约层”和“团队可配置的参数层”，治理在契约层兜底，灵活性在参数层释放。

**三点**：
- 平台层统一证据格式、安全边界、审批流程，不可绕过
- 团队层开放环境配置、阈值、模板等可调参数
- 用准入和评测机制约束个性化边界，避免失控

**套项目**：开发套件——平台统一证据格式和质量门禁标准，团队侧自行配置登录态、端口、选择器和测试数据。

### Q10. 怎么设计 Human-in-the-loop？

**要害一句**：不是泛泛设审批，而是精准卡在高风险决策、语义不确定、责任归属这三类节点上。

**三点**：
- 高风险决策：可能造成不可逆后果的操作必须人工确认
- 语义不确定：模型把握不准、置信度低的判断交给人补充信息
- 责任归属：涉及对外承诺或合规边界的结论需要人背书

**套项目**：OpsPilot——approval_interrupt 把高风险修复动作挡住，等待人工审批后才进入 executor，verify 阶段再确认执行结果是否符合预期。

## 6. 防戳穿专章

### 6.1 “2.2 天到 4 小时”是不是人工预估？

**先承认**：这个数字最初确实来自试点团队的工时反馈和人工估算，不是严格对照实验，我不会把它包装成严格 A/B 结论。

**再区分**：效率提升的单点数字不够严谨，但质量度量不是拍脑袋。开发套件真正系统化的是测试质量：Playwright 真跑、截图取证、失败归因、证据完整率和 Bad Case 回流。

**再转化**：如果现在重做，我会用对照组和实验组统一采集任务完成工时、人工返工次数、缺陷逃逸率、回归覆盖率，再看置信区间，而不是只报一个亮眼数字。

**拉回证据**：这次教训反而让我形成了现在的评估观：AI 产品不要只报省时，要同时度量质量、返工和风险。

### 6.2 OpsPilot 到底是不是企业级项目？

**先承认**：OpsPilot 目前是个人主导的工程化项目，还没有在真实企业生产环境里跑完整的故障处置周期，这一点我不回避。

**再区分**：但它不是玩具 Demo——它有 13 节点的完整工作流、evidence 持久化、risk_gate 风险门禁、approval_interrupt 人工审批中断、executor 幂等执行与审计、verify 验证闭环，这些正是企业级 Agent 系统必须具备的骨架。

**再转化**：要把它推向真正的企业级落地，下一步是接入真实工具适配器、对接真实监控告警和工单系统，并补上权限模型和多租户隔离。

**拉回证据**：我更愿意把它定位成“企业级 Agent 架构的工程验证”，证明我理解生产级系统需要的边界和约束，而不是只会做一个能聊天的诊断助手。

### 6.3 你真的懂后端和生产级 Agent 服务吗？

**先承认**：我不是有十年后端经验的资深架构师，分布式系统的极限场景我经验有限。

**再区分**：但在 Agent 服务这个具体领域，我亲手实现了 LangGraph 工作流编排、checkpoint 状态恢复、evidence 扇出聚合、风险门禁与审批中断、工具网关的 schema 校验与重试审计——这些正是生产级 Agent 服务的核心难点，而不是普通 CRUD 后端的难点。

**再转化**：我的优势恰恰是能从产品经理视角理解这些工程约束从何而来、代价是什么、用户会因此受到什么影响，让我在和工程团队对话时不说外行话。

**拉回证据**：这也是我区别于纯业务背景候选人的地方——我能看懂架构图背后的取舍，而不只是看懂功能列表。

### 6.4 “10+ 团队试点”具体是什么口径？

**先承认**：这个数字目前更接近“接触和小范围试用过的团队数”，还没有达到“完整跑通试点周期、签字确认效果”的严格口径。

**再区分**：我会把它拆成几层——真正完整跑通流程并产出可复盘案例的是种子团队，数量更小；接触、试用、给过反馈的团队范围更大，这两类不能混为一谈。

**再转化**：如果继续推进，我会重新定义“试点团队”的准入标准和完成标准，按层级分别统计转化率，而不是用一个笼统数字。

**拉回证据**：这次反思也让我意识到，汇报试点规模时必须先讲清楚口径，否则数字越大，越容易在追问中露怯。

### 6.5 你没有系统写过 PRD / 用户调研，怎么做 PM？

**先承认**：我没有按传统 PM 培训路径系统训练过 PRD 模板和结构化用户调研方法论，这方面确实不是科班出身。

**再区分**：但产品经理的核心能力是“把模糊需求拆解成可执行方案，并验证它是否解决了真实问题”，这件事我在工程化落地里反复做——和试点团队反复沟通需求、把反馈转成功能优先级、把结果讲给决策者听。

**再转化**：PRD 模板和调研方法论是可以快速学习补齐的工具，我更担心有人套着模板却看不清真实问题——这一点我有信心做得不差。

**拉回证据**：我可以用具体材料证明，比如开发套件的能力拆解文档、应用生成平台的方案设计、OpsPilot 的架构决策记录，这些本质上就是 PRD 在工程语境下的呈现。

### 6.6 你的 AI 产品指标是不是不够严谨？

**先承认**：我目前部分指标，特别是效率类数字，更接近试点反馈估算，还没有经过严格对照实验验证，这一点我承认。

**再区分**：但不是所有指标都不严谨——证据完整率、缺陷分流准确率、根因候选命中率这类质量类指标，可以直接从系统日志和产物中客观统计，不依赖主观估算。

**再转化**：我会把指标分成三类管理：客观可统计的系统指标、需要对照实验验证的效果指标、仅作方向参考的试点估算，分别标注口径和置信程度，不混在一起汇报。

**拉回证据**：这正是我现在做评估体系设计时坚持的原则——指标的可信度本身也要可追溯，不能只看数字大小。

## 7. 行为面试软杀招

### 7.1 平时向上汇报多不多，会不会写汇报材料？

> “高频正式 PPT 汇报不算多，我们团队更多用可运行产物、数据和复盘文档同步。但结构化材料我做得不少，比如方案设计、能力拆解、架构决策、试点结果复盘。我的理解是，向上汇报的核心不是 PPT 本身，而是把复杂技术问题翻译成价值、风险和决策项，让决策者快速判断要不要投入、风险在哪里、下一步怎么推进。”

## 8. 面试中主动引导的话题

- AI 落地路径：从 Demo 到可控试点，再到模板化复制。
- 证据驱动评估：答案不重要，证据链、验证和返工率更重要。
- 复杂工作流拆解：把不确定任务拆成可恢复、可审计的节点。
- Human-in-the-loop：不是泛泛审批，而是在高风险决策、语义不确定和责任归属处介入。
- Bad Case 闭环：用户修正和失败案例要回流规则、Prompt、评测集和流程。
