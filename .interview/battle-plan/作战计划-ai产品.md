# AI 产品经理面试作战计划

> 目标：AI 策略产品经理 / AI 平台产品经理，月 base 30k+
> 核心优势：有真实 AI 工程化落地经验，能在产品和技术之间双向翻译
> 核心弱点：缺乏传统产品经验（用户增长、商业化、数据驱动迭代）

---

## 一、两种岗位的区别和应对策略

### AI 策略产品经理
**考察重点**：你能不能判断"什么场景该上 AI、怎么落地、商业价值怎么评估、推广阻力怎么克服"。偏向战略和判断，对技术细节要求适中。

**你的优势**：你有三个真实项目，知道 AI 产品从 PoC 到灰度试点到产品化的真实路径，包括卡在哪、为什么难推广。这是多数 PM 候选人没有的。

**你的弱点**：缺乏纯产品视角——竞品分析、用户调研、增长策略、商业化路径你讲不深。

**应对**：主动把话题引到"AI 落地路径设计 + 价值评估 + 组织推广"，用 Harness 五阶段落地路径（工具化→标准化→流程化→治理→知识闭环）作为分析框架。

---

### AI 平台产品经理
**考察重点**：你能不能设计一个让内部团队/开发者用起来的 AI 平台——能力分层、API 设计、评测体系、安全治理、多租户、运营指标。偏向平台型产品，技术理解要求高。

**你的优势**：你理解 Harness 三层可配置（平台底座/团队配置/任务编排）、能力原子化、评测准入，这正是 AI 平台 PM 需要的思维。

**你的弱点**：没做过对外商业化的平台产品，平台运营、定价策略、开发者生态经验几乎没有。

**应对**：主打"内部 AI 平台"场景，把你在华为的实际经历定位成"内部 AI 平台用户和共建者"的双重视角，这是差异化。

> 我有 7 年研发经验，近一年深度参与了三个企业级 AI 落地项目：研发 Harness、应用生成平台和运维智能体。我对 AI 产品的核心判断是：AI 落地的难点不在模型，在于如何把不稳定的 AI 能力转化成可控、可验证、可规模化的工程体系。我在这三个项目里既是设计参与者，也是内部推广者——踩过从 PoC 惊艳到落地困难的所有坑。

### AI 平台 PM 版本

## 四、两种岗位的核心考题标准答案

### Q5：什么场景适合上 AI？
**一句话**：任务路径动态、需要语义判断、结果可以验证、错误成本可控，才值得上 Agent。

**展开**：判断矩阵 = 任务复杂度 × 路径动态性 × 工具调用需求 × 可验证性 × 风险可控性。规则完全固定的（计税/格式转换/固定审批流）用程序，不要用 AI。

**套项目**：OpsPilot 的故障诊断，告警触发后无法提前写死路径，需要根据现象动态选工具，最后有执行结果可验证，适合 Agent。前端 UI 自动化测试也是——不是简单执行脚本，是先判断测试范围，再选择用例，再执行，再归因，多步骤且路径动态。

---

### Q9：北极星指标怎么设计？
**一句话**：有效任务完成率——不是 DAU、不是 Token 数、不是对话轮次。

**有效完成四要素**：任务完成 + 结果被验证 + 用户接受 + 没有额外返工。

**套项目**：开发套件的北极星是"有效测试任务完成率"——Playwright 执行完 + 结果能判定 + 证据完整 + 阻断缺陷被识别，这才算完成，不是 Agent 返回了内容就算。

---

### Q21/Q30（平台岗）：平台统一和业务个性化怎么平衡？
**框架**：三层可配置。

- **平台层不可变**：安全规范、审计要求、数据隔离、质量结果格式、高风险审批规则
- **团队层可配置**：流程阶段组合、技术栈适配、门禁阈值、审批人
- **任务层可编排**：具体执行参数、并行策略、失败处理

**底线**：管太死→业务跑路自建影子系统；管不住→成本失控、安全漏洞。这是大厂中台的生死线。

**套项目**：开发套件里，平台统一代码规范检查和测试证据格式，项目团队自己配置测试环境、登录方式和阻断阈值。

---

### Q6：Demo 惊艳落地难，为什么？
**5 个根因**：
1. 场景过理想——Demo 输入清晰，真实用户输入模糊
2. 无状态恢复——工具挂了整个任务重来
3. 无验证——LLM 说通过了就信
4. 无成本控制——多 Agent 规模化后账单爆炸
5. 无数据闭环——没有 Bad Case 管理，无法持续优化

**套项目**：前端测试 Demo 很容易做到"生成 Playwright 脚本"，但真实产品要处理项目启动、登录态、动态端口、测试数据、选择器稳定性、环境异常、Flaky、证据收集、失败归因和缺陷流转。

---

### Q39/Q40（弱点区）：你有没有做过用户调研 / 写过 PRD？
**诚实话术**：
> "我没有独立写过完整 PRD 的经历，我的背景更偏向参与需求分析和设计方案。但在开发套件项目里，我参与了能力需求的拆解——比如前端自动化测试需要什么输入、输出什么证据、门禁条件是什么。这个过程跟写 PRD 很接近，只是没有用正式模板。用户调研方面，我们主要通过内部试点反馈和 Bad Case 分析来了解用户痛点，没有做过正式访谈问卷。这是我需要补的。"

## 五、产品弱点补强策略

你最薄弱的几点，以及最低成本的补法：

| 弱点 | 最低成本补法 | 需要多长时间 |
|------|-------------|-------------|
| PRD 写作 | 找一个你最熟悉的功能（前端自动化测试节点），按 PRD 模板写一份，练习用产品语言而不是技术语言描述 | 2-3 小时 |
| 数据分析 / A/B 测试 | 背 A/B 测试的基本概念（对照组/实验组/显著性）+ OpsPilot 离线评测体系就是你的 A/B 经验 | 1 小时 |
| 用户调研 | 把内部试点反馈和 Bad Case 管理包装成"用户研究方法"——你做的事情本质上就是用户研究，只是形式不同 | 不需要补，需要重新叙述 |
| 竞品分析 | 梳理一份 Cursor / Copilot / Claude Code / Dify / Coze 的对比表 | 2-3 小时 |
| 商业化思维 | 了解 API 计费 + Token 成本控制的基本逻辑（这正是 Harness 经济学部分） | 1 小时 |

## 六、面试中主动引导的话题

当面试官问到你不熟的领域（用户增长/商业化/数据驱动），用这个句式把话题引回来：

> "这块我的经验不够深，但我可以从 AI 产品落地的角度说说我的理解……"

然后回到：
- Harness 五阶段落地路径
- 证据驱动的产品评估
- Bad Case 管理和闭环迭代
- 平台三层可配置
- 有效任务完成率作为北极星

这是你的舒适区，主动拉回来比硬答不熟的话题更有说服力。
