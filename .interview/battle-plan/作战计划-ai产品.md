# AI 产品经理面试作战计划

> 目标岗位：AI 策略产品经理 / AI 平台产品经理，月 base 30k+
> 核心定位：有工程纵深、能把复杂 AI 能力产品化落地的候选人
> 作战原则：不装全能传统 PM，把工程纵深翻译成业务价值、可控流程和可验证结果

## 0. 作战理念

一句话：我不主打“通用型全能 PM”，我主打“懂工程、懂 Agent、能把 AI 从 Demo 推到可控试点的 AI 产品候选人”。

主动打的优势区：

- 复杂 AI 工作流拆解：Router / Planner、13 节点 SRE 流程、前端测试三防线。
- 可控与可信：证据链、质量门禁、Human-in-the-loop、risk_gate、审批与审计。
- 评估与迭代：Bad Case 闭环、离线评测、证据完整率、缺陷归因。
- 企业内部 AI 落地：从 PoC 到试点，从个人提效到团队流程。

只求不露怯的弱项区：

- C 端增长、商业化定价、大规模运营。
- 长期 Axure / 墨刀原型经验。
- 严格 A/B 实验和成熟数据分析平台实战。

统一话术：

> “我的背景不是传统 C 端 PM，而是从研发和 AI 工程化落地切入。我能把复杂业务拆成可执行流程，把不稳定的 AI 能力放进有边界、有证据、有评估、有审批的产品体系里。这是我相对纯业务 PM 的差异化。”

## 1. 两类岗位怎么打

### AI 策略 PM

考察重点：场景判断、业务价值、ROI、推广阻力、从 1 到 100。

打法：少讲底层代码，多讲“为什么这个场景值得上 AI、怎么灰度试点、怎么度量价值、怎么复制到更多团队”。

### AI 平台 PM

考察重点：能力抽象、平台治理、权限模型、评测体系、安全边界、开发者采用率。

打法：把开发套件、应用生成平台和 OpsPilot 都讲成“平台能力分层 + 流程治理 + 质量门禁”的案例。

## 2. 开场定位

### 策略 PM 版

> 我有 7 年研发和企业内部平台经验，近一年重点参与 AI 研发提效、应用生成和运维 Agent，也熟悉企业内部从试点到推广的阻力。我的判断是，AI 产品难点不只是模型能不能生成，而是能不能被放进可控、可验证、可推广的业务流程里。所以我更适合企业内部 AI 策略和落地型产品，把工程纵深翻译成业务价值、流程边界和可复盘结果。

### 平台 PM 版

> 我有前端和全栈研发背景，参与过内部 AI 工具链和应用生成平台建设，也理解业务团队为什么需要低门槛、可配置的能力。我熟悉 Agent Workflow、RAG 意图识别、质量门禁、证据采集和 Human-in-the-loop，知道这些能力要先变成稳定产品流程。做平台 PM 时，我的优势是既能理解业务侧的灵活性，也能守住平台侧的安全、评测、权限和治理边界。

## 3. 三个项目，两种讲法

### 3.1 开发套件：策略 PM 讲法

- 用户 / 场景：企业前端团队和业务研发在 AI 辅助开发后，需要快速确认页面功能、交互流程和关键回归是否真的可用。
- 核心痛点：传统验收依赖人工点击和主观判断，AI 生成代码后更容易出现“看起来完成、实际不可用”的隐性质量风险。
- 产品方案：把前端自动化测试做成可试点推广的质量闭环，用 Playwright 真跑、截图取证、五分类失败归因、质量门禁和 Bad Case 闭环替代口头承诺。
- 价值表达：它不是单个测试脚本能力，而是把 AI 研发提效从“生成更快”推进到“交付可信”，降低推广阻力并让团队敢用。

### 3.2 开发套件：平台 PM 讲法

- 用户 / 场景：内部 AI 开发套件平台、质量平台和多个项目团队，需要把前端测试能力复用到不同技术栈和环境。
- 核心痛点：各团队登录态、端口、路由、选择器和测试数据差异大，如果没有统一契约，平台能力会变成一次性脚本。
- 产品方案：抽象为测试任务输入、Playwright 执行、截图和日志证据、五分类归因、门禁阈值、Bad Case 回流等平台模块。
- 价值表达：平台侧统一证据格式和质量门禁，团队侧配置环境与阈值，既保证治理一致，又保留项目落地弹性。

### 3.3 应用生成平台：策略 PM 讲法

- 用户 / 场景：企业轻应用生成场景中，内部业务团队希望用自然语言生成应用，并在试用后持续做局部修改和流程调整。
- 核心痛点：一次性生成 Demo 容易惊艳，但修改阶段用户表达模糊，系统经常不知道是改样式、改字段、改逻辑还是改数据流。
- 产品方案：用 Router / Planner 拆分意图识别和任务规划，引入场景化 RAG 意图增强，再通过 Schema 中间产物、Sandpack 预览和增量修改稳定交付。
- 价值表达：把轻应用生成从“能生成首版”推进到“能持续迭代”，更适合企业内部从 PoC 走向业务试点。

### 3.4 应用生成平台：平台 PM 讲法

- 用户 / 场景：应用生成平台的内部开发者、模板维护者和业务集成方，需要沉淀可复用的生成、预览和修改能力。
- 核心痛点：如果 Router、Planner、渲染预览和增量修改都混在单次对话里，平台难以治理，也难以定位失败原因。
- 产品方案：把能力拆成意图路由、规划生成、Schema 中间产物、Sandpack 预览、增量修改和 RAG 案例库等可组合模块。
- 价值表达：平台 PM 讲法重点不是“我写了某段逻辑”，而是把不稳定生成能力拆成有边界、有输入输出、有评估口径的平台能力。

### 3.5 OpsPilot：策略 PM 讲法

- 用户 / 场景：SRE 故障处置场景中，值班研发和服务负责人需要快速完成告警接入、证据收集、诊断、修复建议、审批、验证和复盘。
- 核心痛点：故障处置高度依赖个人经验，日志、指标、变更和审批分散在不同系统里，事后也很难还原“为什么这么判断”。
- 产品方案：把 intake 到 rca 工作流做成可控处置链路，用 evidence_fanout 收集证据，用 risk_gate 和 approval_interrupt 控制高风险动作，用 verify 和 RCA 形成闭环。
- 价值表达：它把 SRE Agent 从“会聊天的诊断助手”变成“有证据、有审批、有验证、有复盘的处置流程”，更容易被企业接受。

### 3.6 OpsPilot：平台 PM 讲法

- 用户 / 场景：SRE 平台团队、运维工具团队和服务团队，需要把故障处置 Agent 接入现有监控、日志、工单和审批体系。
- 核心痛点：生产级 Agent 的风险不在于能不能给建议，而在于权限边界、证据可信度、状态恢复、人工审批和执行审计是否可治理。
- 产品方案：把 intake、triage、evidence_fanout、diagnose、risk_gate、approval_interrupt、verify、RCA 拆成平台流程节点，并统一 evidence、approval 和 trace 契约。
- 价值表达：平台 PM 讲法要强调 OpsPilot 是一套 SRE Agent Harness，可复用到不同服务和环境，而不是只服务单个故障场景的脚本。
