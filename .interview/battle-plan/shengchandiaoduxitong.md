# 产线智能调度系统设计方案

将 Agent 能力融入产线调度，核心是把传统"规则驱动"的 MES/APS 系统升级为"感知-决策-执行-学习"闭环。下面我从架构、关键 Agent 设计、落地路径三个层面给出方案。

## 一、整体架构:分层多 Agent 协同

建议采用**三层架构**,而不是单一大 Agent 包办所有事:

**感知层(数据底座)**:对接 MES、SCADA、PLC、WMS、ERP、IoT 传感器,统一为实时事件流(Kafka/Flink)。这一层不做决策,只负责把产线状态"翻译"成 Agent 能理解的结构化语义,比如"3 号机床主轴振动异常""SMT 物料库存低于安全水位"。

**Agent 决策层**:多个专职 Agent 各司其职,通过 A2A(Agent-to-Agent)协议或消息总线协作。下面详述。

**执行层**:决策结果通过 API 下发给 MES/PLC/AGV 调度系统,同时所有动作回写到知识库供学习。关键是**人在回路**——高风险决策(如停线、改工艺参数)必须人工确认。

## 二、核心 Agent 角色划分

不要做"万能调度 Agent",而是拆成职责清晰的协作团队:

**订单解析 Agent**:接收订单后,结合 BOM、工艺路线、交期、客户优先级,自动拆解为工单。LLM 在这里的价值是处理非标订单的自然语言描述和异常约束。

**排程 Agent**:核心大脑。融合两种能力——传统 APS 算法(遗传算法、约束规划 OR-Tools)负责硬约束求解,LLM 负责软性权衡和解释。比如算法给出三套排程方案,LLM 解释"方案 A 交期最优但换型成本高 30%,方案 B 均衡",让调度员理解后选择。

**设备健康 Agent**:监听设备数据流,做异常检测和预测性维护。发现 5 号机即将超温时,主动通知排程 Agent 提前把后续工单切到备用设备。

**物料 Agent**:监控物料齐套率、调用 AGV、与供应商系统对接催料。物料不齐时反向触发排程 Agent 重新调整。

**质量 Agent**:对接 SPC/视觉检测,发现质量漂移时联动排程 Agent 暂停相关工单,联动工艺 Agent 分析根因。

**协调 Agent(Orchestrator)**:仲裁 Agent 间冲突。比如设备 Agent 要求停机检修,排程 Agent 想保交期,协调 Agent 基于全局目标函数(交期、成本、OEE)裁决,无法裁决的升级给人。

## 三、Agent 的关键能力实现

**工具调用(Tool Use)**:每个 Agent 必须有明确的工具集——查询数据库、调用排程算法、下发 PLC 指令、读取 SOP 文档。工具粒度要细,描述要准,否则 LLM 会乱调。

**记忆与知识库**:RAG 接入工艺文档、历史故障案例、SOP。短期记忆放当班次的事件,长期记忆放历史排程效果,用于持续学习"哪类订单组合换型损失最小"。

**强化学习闭环**:排程 Agent 的策略可以用 RL 持续优化,奖励函数 = 准时交付率 + OEE - 换型次数 - 加班工时。LLM 负责生成候选策略和解释,RL 负责打分迭代。

**安全护栏**:产线场景容错率极低,必须设置硬规则——比如温度超限直接走 PLC 安全逻辑,不经过 Agent;Agent 只能在"建议-审批-执行"流程内动作,关键写操作要二次确认。

## 四、落地路径建议

不要一上来就全栈替换 MES,风险太大。推荐三步走:

**第一步(1-3 个月):副驾驶模式**。Agent 只读不写,作为调度员的"智能助手"——回答"为什么这个工单延期""现在切换到 B 产品最优顺序是什么",积累信任和数据。

**第二步(3-6 个月):单点闭环**。挑一个低风险场景全自动,比如"物料补料调度"或"非关键设备的预测性维护排程",验证 Agent 决策质量。

**第三步(6-12 个月):多 Agent 协同**。逐步扩展到主排程,但保留人工否决权,目标是把调度员从"救火"中解放出来,专注异常处理和策略优化。

## 五、需要警惕的坑

LLM 幻觉在产线是灾难,所以**所有数值型决策必须走确定性算法**,LLM 只做解释、对话和软约束处理。延迟也是问题——产线毫秒级响应的部分(安全联锁、节拍控制)绝对不能放 Agent,Agent 适合分钟到小时级的调度决策。最后是 ROI,中小批量多品种产线收益最大,大批量流水线传统 APS 已经够用,硬上 Agent 不划算。
