## 一、FastAPI 异步模型

**Q: FastAPI async def 底层怎么工作的？**

**一句核心**：Uvicorn 启动一个 asyncio Event Loop，async def 是协程，遇到 await 让出 CPU，I/O 完成后恢复，单进程并发处理成千上万请求。

**展开**：
- Uvicorn（ASGI 服务器）监听端口，每个请求进来 Event Loop 调度对应 async def 协程
- 协程执行到 `await`（比如等 LLM 返回）时挂起，Event Loop 去处理其他请求
- I/O 完成后恢复协程，继续执行
- 关键：单线程，靠"主动让出 + 回调恢复"实现并发，不是多线程
- LLM 调用是 I/O 密集型（等网络），异步完全够用；CPU 密集型（文档解析/代码扫描）要放进程池，不能用 async

**追问：调用同步 DB 驱动会怎样？**
→ 阻塞 Event Loop，所有并发请求都卡住，退化为单线程串行。解法：`asyncio.run_in_executor` 把同步调用扔进线程池。

**OpsPilot 印证**：OpsPilot 用 FastAPI async def 做路由，LLM 调用用 `complete_sync()` 跑在后台线程的独立 Event Loop 里（`asyncio.run()` 隔离），是"异步框架包裹同步执行"的混合模式。

## 二、SSE 断线后 Agent 要不要停

**Q: 前端 SSE 断了，后端 Agent 任务会停吗？**

**一句核心**：不停。连接生命周期 ≠ 任务生命周期，两者完全解耦。

**展开**：
- Agent 任务在 `BackgroundTasks` 后台线程执行，和 SSE 连接没有绑定关系
- 前端断开只关闭"推流通道"，后台线程继续运行
- 状态持久化到 DB（checkpoint 表 + events 表），任务进度不依赖连接存活
- 前端重连后通过 `GET /runs/{run_id}/events?last_event_id=N` 拉取历史事件

**追问：SSE 的 Last-Event-ID 是什么？**
→ 浏览器断线重连时，自动在请求头带上最后收到的 event_id，服务端从这个 ID 之后续传，避免数据丢失或重复。

**追问：OpsPilot 实现了断线续传吗？**
→ 当前 EventBus 用 asyncio.Queue，重连后会丢失历史。生产级实现应该在 `incident_run_events` 表存 sequence 序号，重连时从 DB 补发。这是我知道的待补项。

**追问：Nginx 代理 SSE 需要什么配置？**
→ `proxy_buffering off`（关闭响应缓冲，否则 Nginx 攒够数据再发，流式失效）+ `proxy_read_timeout 600`（超过任务最大执行时间）

## 三、Worker 崩溃恢复（Task Lease）

**Q: Worker 执行中崩溃，任务怎么恢复？**

**一句核心**：Task Lease 机制——每个 RUNNING 任务有过期时间，Worker 定期续租，崩溃不续租后 watchdog 自动重分配。

**展开**：
```
RUNNING 任务字段：
  lease_expires_at = now() + 30s
  heartbeat_at = now()

Worker 每 10s 续租：
  UPDATE tasks SET lease_expires_at = now() + 30s WHERE id = :id

Watchdog 定时任务：
  SELECT * FROM tasks 
  WHERE status = 'RUNNING' AND lease_expires_at < now()
  → 重置为 PENDING，其他 Worker 重新领取
```

**追问：为什么 RUNNING 状态不够，要 Lease？**
→ RUNNING 状态永远不会自动过期；Lease 有时间戳，Worker 崩溃后超时自动"死亡"，watchdog 能安全判断任务需要重分配。

**追问：重新领取后重复执行怎么办？**
→ 工具调用要设计幂等键（见下一条）。Agent 任务本身要从 checkpoint 恢复，不是从头重跑。

**OpsPilot 现状**：OpsPilot 用 BackgroundTasks 直接跑，没有 Lease 机制。生产化需要引入任务队列 + Lease。

## 四、乐观锁防 Lost Update

**Q: 两个节点并发更新同一份 Agent State 怎么防止数据覆盖？**

**一句核心**：乐观锁——version 字段，更新时带上旧版本号，影响行数为 0 说明被抢先更新了，重试。

**展开**：
```sql
-- 读取时记录 version
SELECT state, version FROM agent_runs WHERE id = :id
-- version = 5

-- 更新时带上 version 作为条件
UPDATE agent_runs 
SET state = :new_state, version = version + 1
WHERE id = :id AND version = 5;

-- 影响行数为 0 → 被其他节点抢先更新了
-- 影响行数为 1 → 更新成功
```

**追问：什么时候用悲观锁？**
→ 并发冲突概率高、冲突代价大时（比如审批决策、关键状态转换）用 `SELECT FOR UPDATE`，宁可等锁不愿重试。

**追问：乐观锁冲突了一直重试会不会死锁？**
→ 不会死锁（没有持有锁），但可能活锁（一直冲突）。需要加退避策略 + 最大重试次数。

**OpsPilot 现状**：state 存在 `incident_checkpoints` 的 JSON 列里，LangGraph 的 checkpointer 内部有版本控制（channel_versions），但业务层没有显式乐观锁。

## 五、工具调用幂等设计

**Q: 工具调用失败了，什么时候可以重试，什么时候不能？**

**一句核心**：可重试的前提是幂等——相同参数重复执行结果一致；不幂等的操作必须带幂等键。

**可重试情况**：
- 网络超时（不确定是否执行）→ 带幂等键重试
- 429 限流 → 指数退避后重试
- 500 服务端错误 → 适当重试

**不可重试情况**：
- 400 参数错误 → 重试没意义
- 已返回部分流式 Token 的 LLM 请求 → 不重试（幂等性破坏，会有重复 Token）
- 发送邮件/触发部署等不可逆操作 → 必须带幂等键

**幂等键设计**：
```python
idempotency_key = sha256(run_id + tool_name + json.dumps(input_sorted))
# 同参数重试命中同一键，服务端返回缓存结果
```

**追问：外部 API 超时后不知道有没有执行，怎么处理？**
→ 带幂等键重试。若已执行，服务端返回已有结果；若未执行，正常执行。关键是接口必须支持幂等键语义。

**OpsPilot 印证**：`executor.py` 里的 ControlledExecutor 用 `call_id` 做幂等键，同一 call_id 只执行一次，重复调用返回缓存结果。

## 六、LangGraph Checkpoint 机制

**Q: LangGraph 的 checkpoint 是怎么工作的？**

**一句核心**：每次节点执行后保存全量状态快照，thread_id 是主键，可从任意检查点恢复。

**展开**：
```
CheckpointTuple 四个核心字段：
  channel_values    → 全量状态快照（当前所有 channel 的值）
  channel_versions  → 各 channel 的版本向量（用于冲突检测）
  versions_seen     → 节点已处理的版本（防重复执行）
  pending_writes    → 未提交的写操作列表

调用方式：
  graph.invoke(input, config={"configurable": {"thread_id": "run_001"}})

恢复方式：
  graph.invoke(Command(resume=decision), config={"configurable": {"thread_id": "run_001"}})
  # LangGraph 从最新 checkpoint 恢复，继续执行
```

**三种 Checkpointer**：
- `InMemorySaver` → 开发/测试用，进程重启丢失
- `RedisSaver` → 生产推荐，`langgraph-checkpoint-redis` 包，支持多 Worker
- `PostgresSaver` → 也可以，与业务 DB 同库

**追问：interrupt() 怎么用？**
→ 节点内调用 `interrupt(payload)` 暂停图执行，payload 通过 stream 的 `interrupts` 字段暴露给调用方；人工决策后用 `Command(resume=<决策值>)` 恢复，LangGraph 从 checkpoint 继续。

**OpsPilot 印证**：`graph_runner.py` 里用 `InMemorySaver`，`approval_interrupt` 节点调用 `interrupt()` 暂停等审批，审批 API 调用 `graph.invoke(Command(resume=decision))` 恢复。

## 七、Pydantic 对象 vs dict 的兼容问题

**Q: LangGraph 里 Pydantic 对象从 checkpoint 反序列化后变成 dict，怎么处理？**

**一句核心**：checkpoint 存的是 JSON，反序列化后 Pydantic 对象变成普通 dict，访问属性会报 AttributeError，必须兼容两种形式。

**错误写法**：
```python
service = ticket.service  # ticket 是 dict 时抛 AttributeError
```

**正确写法**：
```python
service = ticket.service if hasattr(ticket, 'service') else ticket.get('service', 'unknown')
```

**遍历 list 的防 None 写法**：
```python
for action in actions:
    if action is None:
        continue
    if isinstance(action, dict):
        action_type = action.get("action_type")
    else:
        action_type = action.action_type
```

**OpsPilot 印证**：`CLAUDE.md` 里明确标注为"已知陷阱 #1"，`nodes/__init__.py` 里所有节点访问 ticket/actions 时都用了兼容写法。

## 八、K8s 部署后 Agent 状态恢复

**Q: K8s Pod 重启后 Agent 状态怎么恢复？**

**一句核心**：状态外置到 DB，Pod 重启后 Worker 从 DB 领取 PENDING 任务，从 checkpoint 恢复执行。

**展开**：
- Pod 内存状态（InMemorySaver、asyncio.Queue）重启后全部丢失
- 生产必须：checkpointer 换 RedisSaver/PostgresSaver，EventBus 换 Redis Pub/Sub
- Worker 启动时：扫描 `lease_expires_at < now()` 的 RUNNING 任务，重置为 PENDING
- 恢复执行：`graph.invoke(None, config={"thread_id": run_id})` → LangGraph 从最新 checkpoint 继续

**OpsPilot 现状**：用 `InMemorySaver`，Pod 重启后图状态丢失，run 会卡在 RUNNING 状态。生产化必须换持久化 checkpointer + watchdog 任务恢复。

## 九、1000 个并发 Agent 任务的架构设计

**Q: 如果要支持 1000 个并发 Agent 任务，架构怎么改？**

**一句核心**：API 和 Worker 解耦，任务队列 + Worker 池 + 持久化状态存储。

**当前 OpsPilot 问题**：
- BackgroundTasks 直接跑，1000 个任务同时启动 → 1000 个线程 → 内存爆、LLM API 被限流
- InMemorySaver，多 Worker 实例状态不共享

**目标架构**：
```
POST /runs → 写 tasks 表（PENDING）→ 立即返回 run_id
↓
任务队列（PostgreSQL SKIP LOCKED / Redis / RocketMQ）
↓
Worker 池（N 个 Worker，各自 asyncio.Semaphore 限并发）
  ↓
  LangGraph + RedisSaver（多实例共享 checkpoint）
  ↓
  工具调用（幂等键）
↓
SSE 事件推送（Redis Pub/Sub 跨实例）
```

**关键点**：
- `FOR UPDATE SKIP LOCKED` 让多 Worker 不抢同一任务
- Worker 级 `asyncio.Semaphore(20)` 限制单 Worker 并发 LLM 数量
- API 实例和 Worker 实例可独立扩缩容

## 十、Saga 补偿模式

**Q: Agent 调用外部工具，中间某步失败了，怎么保证一致性？**

**一句核心**：Saga 模式——每步定义正向操作和补偿操作，任意步骤失败时按反向顺序执行补偿。

**展开**：
```
正向链：
  创建 Git 分支 → 修改代码 → 创建 MR → 触发部署

任意步骤失败，反向补偿：
  关闭 MR → 删除分支 → 回滚部署
```

**注意**：
- 补偿是业务级"尽力而为"，不是技术级精确回滚
- "发送邮件"无法真正撤回，只能发"取消通知"
- 补偿操作本身也要幂等

**为什么不用 2PC/XA 分布式事务？**
→ 外部 API 不支持两阶段提交协议；性能差；工具调用链天然不适合强事务。

**OpsPilot 印证**：executor.py 里的高风险操作执行前做 dry-run 验证，执行后记录审计；remediation 节点在修复方案里包含回滚步骤，verify 失败时触发回滚序列。
