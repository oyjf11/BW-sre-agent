## 详细复盘

**背景：** GPT 对 OpsPilot 跑了 3 次 E2E mock 工单测试，出了一份报告。用户把报告丢过来问："你能发现什么问题吗？"

**排查过程：**

对照报告结论逐条去读源码 + 原始 JSON 产物（`diagnosis-final.json`、`rca-2-final.json`、`events-*.json`），而不是停在"这份报告说什么"。

**根因 ①：审批分支结构性不可达（HITL 死代码）**

`backend/app/graph/nodes/__init__.py:1144`，`remediation_node` 用这个逻辑决定提案 rollback(HIGH / 需审批) 还是 restart(LOW / 免审批)：

```python
has_deployment_signal = (
    "deployment" in top_candidate.hypothesis.lower()
    and _has_deployment_evidence(evidence_items)
)
```

`hypothesis` 是 LLM 自由文本，但诊断系统提示词强制"请始终使用简体中文回复"。`"deployment" in "本次发布的代码变更引入了...".lower()` **结构性恒为 False**。

讽刺的是：同一个 candidate 对象上就有干净的结构化字段 `incident_type="deployment_regression"`（在 `diagnosis-final.json` 里亲眼验证）——代码舍近求远去匹配多语言自由文本。

连锁后果：`requires_approval=True` 永远生不出来 → `approval_interrupt` 永远不会被路由到 → **HITL 整条链路，在默认中文配置下，结构性不可达**。

GPT 报告的归因："审批分支没测到，需要造一个高风险 fixture"（测试覆盖率问题）。
真实情况：产品代码本身有缺陷，关掉了核心安全机制（生产级问题）。

**根因 ②："unknown: SUCCESS" 重复两行（两个小 bug 叠加）**

`backend/app/graph/nodes/__init__.py:1556`：
```python
action_results = [r.to_dict() for r in valid_results]
execution_results = action_results   # 同一个 list 对象，不是复制
```

再到 `~1818`：
```python
for r in (execution_results or []) + (action_results or []):   # A + A，每条打印两遍
    at = r.get("action_type", "unknown")   # to_dict() 顶层没有 action_type，永远 fallback
```

`ExecutionResult.to_dict()`（`executor.py:33-40`）顶层只有 `action_id/success/result/error/skipped`；`action_type` 在 `result["action_type"]` 一层里。两个 bug 叠加 = 每条执行结果被渲染成两行 `unknown: SUCCESS`。

在 `rca-2-final.json` 里直接验证到背靠背两行。

**关键取舍：** 用结构化字段 vs 自由文本 keyword match。有现成干净字段却选多语言 NLP 猜，是 Agent 系统里典型的"启发式覆盖结构化"反模式。

## 面试问答底稿

**Q：** 举个你在项目里发现/调试问题的例子，越具体越好。

**A（分两层）：**

> 通用层：审查 AI 生成代码或 AI 分析报告时，最有价值的动作不是"读完觉得没问题"，而是找到一个具体的、可以被 yes/no 回答的假设，然后去源头数据验证它。
>
> 项目层：GPT 对 OpsPilot 跑了 E2E 测试，报告说"审批分支没走到，需要补 fixture"——这听起来是测试设计问题。我判断这个结论值得质疑，让 Claude 去读 `remediation_node` 的源码，发现判断是否走高风险路径的那行逻辑，是用英文子串去匹配一个被系统提示词强制成中文输出的自由文本字段——`"deployment" in "本次发布...".lower()` 结构性恒为 False。结果是 HITL 这条安全链路，在默认配置下根本不可达。这是生产级缺陷，不是测试覆盖率的问题；两者处理优先级差很远。我们用 `diagnosis-final.json` 里的原始字段值交叉验证，确认了整个证据链。

**诚实边界：** 我自己读这段代码，第一遍未必能在 5 分钟内独立定位到这个行号——是我带着 Claude Code 一起过：我提质疑方向，Claude 读源码 + 原始 JSON，我判断这个证据链是否成立、是否值得作为缺陷上报。

---

**Q：** 那这是你发现的还是 AI 发现的？

**A：**

> 按角色拆开说更准确：原始 coding agent 写了那两处有缺陷的实现；GPT 跑了 E2E 并观察到了症状（这部分它做对了）；我判断它的归因停得太早、值得往下一层查，Claude 去读代码和产物、提供证据链，我来判断证据链是否立得住。
>
> 如果你问"那两行 fix 是你敲的吗"——不是，改两行 5 分钟的事。但"这个项目里哪条安全机制是静默失效的、为什么、现在该不该修"，这个判断是我做的，我对它负责。**写代码的活越来越多交给 AI；'这个结论站不站得住'的判断，现在和可预见的将来都得是人来收口。**
