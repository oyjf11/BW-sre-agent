## 详细复盘

### 问题背景
rag2 demo 里 config 早就声明了 `RERANK_MODEL = "gte-rerank"`，但检索链路根本没接——属于"埋了接口没启用"。同时原评测只是干巴巴打印 Top-1/Top-3 命中率两行，A/B test 的味道不够，演示时讲不出"每个优化贡献多少"的故事。

### 接 rerank 的位置决策（这次最值钱的判断）

现有架构是 **父子分块 + small-to-big**：父块 8 个不向量化，子块 24 个入库，向量检索打子块，按 `parent_id` MAX 聚合回父块。

rerank 可以接在两个位置：

| 位置 | 怎么做 | 评价 |
|---|---|---|
| **子块层** ✅ | 向量召回 top-N 子块 → gte-rerank 打分 → 用 rerank 分替换 cosine 分 → 按 parent_id MAX 聚合 | 标准两阶段检索，跟 small-to-big 完全兼容 |
| 父块层 ❌ | 向量先聚合出 top-N 父块 → 把父块 title/plan 喂给 gte-rerank 重排 | 父块只 8 个，rerank 没几个候选可比；而且父块是 plan 步骤聚合体，跟用户 query 语义距离大，rerank 收益弱 |

**判断原则**：粗排在哪一层聚合，精排就在哪一层打分。rerank 在子块层是因为 small-to-big 的"粗排"本质是子块级的，rerank 只是把同一层的打分换成更准的模型。

### A/B 实验"味道"怎么调够

光打印 Top-1 提升 pp 不够，真正像 A/B 实验报告的味道靠四件套：

| 段 | 干什么 | 解决什么质疑 |
|---|---|---|
| ① 逐条明细 | 每条 query 三路 Top-1 命中 + 分数并排 | "你拿哪些 case 算的？" |
| ② **翻盘/翻车归因** | 列出哪些 case 在 A 错→B 对（改写救场）、B 对→C 错（rerank 带偏）的具体例子 | "举个改写真起作用的例子？" + "有反例吗？" |
| ③ 置信度对比 | 命中项的平均 top1 分、平均 margin 三路对比 | "命中率没动，rerank 是不是白接？"→margin 拉大就是收益 |
| ④ 命中率汇总 + 归因分解 | Top-1/Top-3 三路百分比 + 条形 + 改写贡献 X pp/rerank 贡献 Y pp | "每个优化各值多少 pp？" |

### 关键取舍
1. **rerank 分数替代 vs 加权融合 cosine** → 选替代。融合多一个超参（权重）要调，且演示时讲"两阶段检索"母题更清楚——就是粗排定候选、精排定排序。
2. **rerank 失败 fallback 回 cosine** → 跟现有"LLM 失败 fallback 到规则兜底"风格一致，demo 永不崩。表里用 `*` 标记该条走了 fallback，对评测公开透明。
3. **recall_k=8** → 子块共 24 个，召回三分之一送精排既演示"缩小候选集"的意义，又不会把全量喂给 rerank（那就退化成单阶段了）。

### 顺手抓的真 bug：CJK 双宽字符 ljust 错位
原 demo 用 `{raw[:22]:<24}` 这种字符数对齐，但中文是全角双宽——`P_ecommerce` 这种 11 字符的列名直接撞进下一列变成 `P_ecommerce✗P_crm`。修法：写一个 `pad(s, width)` 按显示宽度（unicodedata east_asian_width = W/F 算 2）补齐。这是个**只有真跑一遍 demo 才能看出来**的问题，不跑就压根发现不了。

## 面试问答底稿

**Q：你这套 RAG 接了 rerank 之后命中率提升了多少？怎么证明是 rerank 的功劳？**

**A：分两层答。**

通用答案：标准做法是两阶段检索——向量召回 top-N 当粗排、cross-encoder 精排取 top-n，因为 cross 全量跑算力爆炸。命中率提升一般 5-15 pp，但 rerank 真正的强项是拉大 top1/top2 的分差（margin），让置信度变高、下游可以加拒答阈值。

但这个项目我做了一件不太常见的事：**评测设计成单变量递进**。A 路是裸向量基线、B 路只加 query 改写、C 路在 B 基础上加 rerank，每列只多一个变量。最后归因分解就直接出来：B-A 就是改写贡献的 pp，C-B 就是 rerank 贡献的 pp。再配翻盘案例（A 错→B 对的具体 query）和反向的翻车案例，每个手段贡献多少、有没有副作用都能讲清楚。如果只跑"全开 vs 全关"的对照，提升能算出来但归不到具体手段，演示和优化都没抓手。

**诚实边界**：rag2 是个演示脚本（10 条 case，8 个父块），样本量太小做不了显著性检验，归因 pp 是描述性的、不是统计意义上的"显著贡献"。真实生产要扩到几百上千条，最好再加 bootstrap 置信区间。这套思路我能搬到生产环境，但还没在生产规模上跑过。

**Q：rerank 你接在哪一层？为啥不直接对父块 rerank？**

**A：接在子块层。** 因为这套 RAG 是父子分块 + small-to-big：父块=完整 plan（不向量化），子块=多视角场景描述（入向量库），向量检索打子块、再按 parent_id MAX 聚合到父块。rerank 跟着粗排走——粗排在子块层聚合，精排就在子块层打分，最后聚合不变。

不接父块层有两个原因：① 父块本项目只 8 个，rerank 没几个候选可比；② 父块内容是"plan 步骤聚合体"，跟用户口语 query 语义距离大，rerank 收益弱。**位置选错精排就白做**。

**诚实边界**：这部分代码是 Claude 帮我写的，我的角色是判断接入位置、设计 A/B 评测形态、审查 rerank 分数怎么替代 cosine 才不破坏 small-to-big 聚合逻辑、以及发现 demo 输出里中文双宽对齐问题。让我从零手写 dashscope SDK 调用细节、第一遍不查文档不会写得这么干净，但**判断 rerank 该插在哪一层、A/B 怎么单变量递进做归因——这是我的判断**，不是 AI 的。
