## 详细复盘

**背景**：高危题标准答案里 A2 写了"重连靠 Last-Event-ID 从 DB 补发"，但我对具体机制不清楚——浏览器怎么知道？服务端怎么"从 DB 补发"？

**核心机制拆解**：

1. **SSE 协议带 `id:` 字段**  
   服务端推事件时在格式里带序号：
   ```
   id: 3
   data: {"type": "node_done", "node": "diagnose"}
   ```

2. **浏览器原生行为，不用写代码**  
   `EventSource` 会自动记住最后收到的 `id` 值。断线时不需要任何前端代码，浏览器重连时自动在请求头带上：
   ```
   Last-Event-ID: 3
   ```

3. **服务端从 DB 补发**  
   事件表结构需要有 `sequence` 字段（就是 SSE `id:` 用的那个数）：
   ```sql
   CREATE TABLE run_events (
       sequence   INTEGER,   -- 单调递增，对应 SSE id:
       run_id     TEXT,
       payload    JSON,
       ...
   );
   ```
   重连时：
   ```python
   last_event_id = request.headers.get("Last-Event-ID")
   missed = db.query("WHERE run_id=? AND sequence > ?", run_id, last_event_id)
   for e in missed:
       yield f"id: {e.sequence}\ndata: {e.payload}\n\n"
   ```

**为什么 asyncio.Queue 不行**：Queue 是纯内存，事件被消费后就消失了，没有历史可查；重连进来服务端根本不知道你上次到哪里。

**关键取舍**：asyncio.Queue 简单但不可重连；DB sequence + Redis Pub/Sub 可重连但有落盘成本。生产级必须落 DB。

## 面试问答底稿

**Q：SSE 的 Last-Event-ID 是什么？断线重连能续上吗？**

**A（两层）**：
- 通用层：SSE 协议本身支持 `id:` 字段。浏览器的 EventSource 会自动记住最后收到的 id 值，断线重连时在 HTTP 请求头带上 `Last-Event-ID`，这是浏览器原生行为，不需要任何前端代码。服务端只要推事件时带上 `id:`，并把每个事件持久化到 DB（带 sequence 字段），重连时查 `sequence > last_event_id` 的行补发即可。
- 项目层：OpsPilot 目前 EventBus 用 asyncio.Queue，是纯内存的，重连会丢历史。生产化要换成写 DB + Redis Pub/Sub 跨实例推送。这是我知道的已知待补项。

**追问："Nginx 代理 SSE 需要什么配置？"**
→ `proxy_buffering off`（关缓冲，否则 Nginx 攒够数据才发，流式失效）+ `proxy_read_timeout 600`（超过任务最大执行时间）

**追问："Queue 为什么不行？"**
→ 事件被消费后就消失了，没有历史记录。重连进来，服务端不知道你上次到哪了，也没有数据可以补发。

**诚实边界**：这套机制我是在这次对话里理解透的，OpsPilot 目前确实没有实现断线续传，它是我明确知道的生产化待补项，理解方案但未动手实现过。
