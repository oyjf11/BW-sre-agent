# rag2 面试讲法 cheat-sheet

> 用法：先盖住答案，看「母题」自己复述；卡壳了再翻答案。脚本本体是
> `rag_eval.py` / `router_planner.py` / `mock_executor.py`，本文件只管"怎么讲"。

---

## 主线（先背这条，其它都往上挂）

**一句话定位**：把**用户大白话需求 → 可执行任务图(DAG)** 的全链路 demo，三段：
**Router(RAG 检索拿范式) → Planner(LLM 融合规划出 DAG) → Executor(分层并行消费 DAG)**。
RAG 段额外配一套**离线消融评测**(A 裸向量 / B+改写 / C+rerank)量化每个优化的贡献。

**数据流全景（白板能画）**
```
用户口语
  ↓  Router  : ① query 改写  ② small-to-big + rerank 检索  ③ 输出结构化意图
命中"范式父块"(完整 plan + pitfalls)
  ↓  Planner : LLM 为主融合"范式+需求"→JSON DAG；规则兜底；自检；拓扑排序+分层
产出 DAG(任务 + 依赖)
  ↓  Executor: 按层并发；状态机 pending→success/failed/skipped；重试；失败→下游 skip

旁路：rag_eval.py = 给 Router 的 RAG 做离线评测(A/B/C 消融 + 四件套报告)
```
三文件关系：`rag_eval.py` 是地基(数据+检索器+评测) → `router_planner.py` 复用检索器再接 LLM
规划 → `mock_executor.py` 独立演示"DAG 怎么被吃掉"。

## 母题 1：你的 RAG 检索做了哪些设计？（最高频，重点）

**一句话**：父子分块的 small-to-big 检索 + query 改写 + 两阶段 rerank + 元数据过滤。

**① 父子分块 / small-to-big**
- 父块 = 完整 plan_steps+pitfalls，**不向量化、只存**；子块 = 同一范式的 3 种口语化场景描述，**只有子块入库**。
- 检索：**小而准的子块**去匹配用户口语(召回准) → 命中后用 `parent_id` 钩子**取回大而全的父块**喂 Planner(规划要完整)。
- **取舍点**：解决"检索块要短要聚焦"和"喂 LLM 的块要长要全"的矛盾。
- 聚合：一个父块多个子块命中时取 **MAX 分数**。

**② query 改写**（embedding 之前）
- 两步：`STOPWORDS` 去口语噪声(帮我/做个/玩意儿)；`TERM_MAP` 口语→术语("卖东西的网站"→"电商 在线商城"、"管用户权限"→"RBAC 权限控制")。
- **为什么**：补 **vocabulary gap**——用户口语 vs 知识库书面语的词汇鸿沟。

**③ rerank 两阶段**
- 粗排(cosine 召回 `recall_k=8`，快/便宜/广) → 精排(`gte-rerank` cross-encoder 只算这 8 个，准/贵)。
- **工程兜底必说**：rerank 调用失败 → `fallback` 回 cosine + `debug["used_rerank"]` 标记 + 报表打 `*`。外部依赖挂了链路不断且可观测。

**④ 元数据过滤**：metadata(domain/complexity/multi_user/need_auth) 挂子块，检索前可硬过滤。

## 母题 2：评测怎么做的？有什么反直觉结论？（判断力加分项）

**一句话**：对 10 条 case 跑 A/B/C **单变量递进消融**，输出四件套；结论是"命中率 +0 但 margin 翻 2.4 倍"。

**四件套**：① 逐条明细 ② 翻盘/翻车归因 ③ 置信度(平均 Top1 分 + 平均 margin) ④ 命中率汇总+归因 pp 分解。

**🔑 杀手锏故事（背下来）**
> "命中率三路全 100%、归因 +0 pp，看着 rerank 白接了。但 ③ 那栏 **margin 从 0.21→0.51 翻 2.4 倍**。
> 说明**测试集对基线太简单、命中率早打满天花板**，rerank 收益不在『有没有命中』，在『**敢不敢下结论**』——
> margin 拉大后生产里能在 0.4 设**置信度门槛做拒答/转人工**，裸向量 margin 0.21 根本做不了。
> 所以评测要分两类指标：命中率 + 置信度，光看命中率漏掉一半的故事。"

**主动点出的严谨点（尺度陷阱）**
> ③ 里 C 的"平均 Top1 分数"看着比 B 低(0.684 vs 0.787)，是**尺度陷阱**：A/B 是 cosine 相似度、
> C 是 cross-encoder 的 relevance_score，**绝对值不能横比**；但 margin 是同尺度内相对差，横比才有意义。

## 母题 3：怎么保证 LLM 输出能用？（Planner，高频）

**一句话**：LLM 为主 + 规则兜底 + 自检，三层防御。

1. **LLM 为主**：`qwen-plus`，`temperature=0.2`(规划要稳)。System prompt 硬约束：基于范式、每任务带
   `source`(pattern/requirement) 可溯源、`depends_on` 表依赖、不准有环、只输出 JSON。
2. **规则兜底**：LLM 挂了/返回非法 JSON → 范式 plan_steps 线性转 DAG，链路不断。
3. **normalize + 自检**：补缺失字段、去重 id、清理悬空依赖；语义校验——
   需认证必有 auth 任务、多人必有 RLS 隔离(对应范式 pitfall「忘记 team 隔离」)、必有 schema、不准有环。
4. **拓扑排序(Kahn) + 并行分层**：算执行顺序，同层并发。

**取舍点**：LLM 负责"融合创造"给灵活性，规则负责"兜底体检"给确定性下限。

## 母题 4：DAG 怎么被消费 / 失败怎么处理？（Executor）

**一句话**：按层并发，状态机驱动，单步重试隔离偶发故障，彻底失败则下游 skip。

- 按 `parallel_layers` 分层，同层并发。
- 状态机：pending → running → success / failed / **skipped**。
- **单步重试**：偶发失败重试可恢复(场景 B：T2 失败→重试成功，不影响下游)。
- **失败传递**：任务彻底失败 → 所有下游(传递闭包)标记 skipped(场景 C：T3 挂→T6 跳)。
- **取舍点**：局部失败不污染全局——重试隔离偶发故障，skip 传递避免在烂地基上继续盖楼。

## 关键参数速记（被追细节时报得出）

| 项 | 值 |
|----|----|
| embedding | `text-embedding-v4`，1024 维，query/document 分 `text_type` |
| 知识库 | 8 父块(范式) × 3 场景子块 = 24 子块 |
| rerank | `gte-rerank`，粗排 `recall_k=8` → 精排 |
| LLM | `qwen-plus`，`temperature=0.2` |
| 评测 | 10 case，A/B/C 三路，margin = top1−top2 |
| 拓扑 | Kahn 算法 + parallel_layers 分层 |
