实战沉淀 · OpsPilot 全旅程主干 + 仲裁打分

一条报障进来，13 个节点把它收敛成「结构化工单 → 并行取证 → 质量门控 → 受控修复 → 复盘归档」。每一步都能退回重来，关键动作卡人工审批。其中取证段的仲裁负责做质量门控：只打证据质量分、按矛盾优先做 4 路裁决，转 2 圈摆不平就转人工。

30 秒開口

刚看这块的时候我用了一个很朴素的比喻：机房里多个调度员抢机器，最后领导决策。带着这个比喻读 OpsPilot 代码，发现对了一半。

错的一半：这些专科 Agent（DB / K8s / 日志 / 指标 / 网络 / 中间件 / 上游依赖…）是分头取证、只读，根本不抢同一台机器。所以没有"抢资源"的硬冲突，只有"结论冲突"的软冲突。对的一半：确实有个"领导"节点叫仲裁，但它干的事比我以为的窄——只审"证据够不够下结论"，不评"哪个修复方案更好"。

一句话概括：仲裁是材料审核员，不是方案评委。

顺着故障生命线讲，5 段就走完：

  1. 入口：报障进来，先归一成一张标准工单。
  2. 定性：先分类，再问"以前查过没"。
  3. 取证（核心段）：排计划、多专科并行只读取证、聚合打质量分、仲裁门控——证据不够就退回重查。
  4. 处置：出方案 → 过风险闸门 → 高危卡人工审批 → 受控执行 → 验真。
  5. 收尾：成败都产出一份复盘归档。

三个设计要点：①全程可退回重来（仲裁/验证的回路）；②高危人不点头不动手（⑩ → ⑪ 的挂起）；③证据先过门控才下结论（⑧ 仲裁卡在取证和处置之间）。

详细复盘

节点之间不直接互相调用，全靠往共享的 IncidentAgentState（state.py:36）里写 key、下游读 key 传递。每段开头用 【吃】→【吐】 标输入输出。

全程的"交通规则"是 5 个条件路由器（builder.py），决定每段走完往哪拐——"可退回重来"的物理实现。

1 入口段：一句话 → 一张标准工单（intake_node）

【吃】ticket(原始) → 【吐】ticket(归一化) / status=NEW / 初始化 evidence_items=[]

入口 API POST /incidents/runs（incidents.py:193）吃三种互斥模式，只能给一种：

  • 模式 A 手工工单：title/description/service/env/severity 逐栏填；
  • 模式 B ticket_id：去工单系统拉；
  • 模式 C alert_event：告警事件转工单。

三种殊途同归出一个 IncidentTicket，喂给接入节点（__init__.py:49）。接入节点很轻：归一化（normalize_env/service/time_range，把"生产/prod/PRODUCTION"抹平）+ 初始化空列表。

为什么这段重要：下游 13 节点全假设手里有一张字段干净的工单——计划节点拿 service/env 定计划、风险闸门拿 env/severity 定风险、复盘节点拿 title 写复盘。归一化放最前面，就是别让"生产"和"prod"在风险闸门里被当成两个环境。

诚实缺口：标题写"从自然语言输入开始"，但代码现状没有任何自然语言→工单解析——前端是结构化表单。插入点明确、契约现成：在接入节点前加一个 nl_parse 节点，用 LLM 工具调用把自由文本抽成 IncidentTicket 字段，下游一行不改——因为下游只认归一化后的工单。核心骨架和契约先焊死、能独立验证，自然语言抽取这种入口适配层留作下一刀。

2 定性段：先分类，再问"以前查过没"

分诊【吃】ticket →【吐】triage / requires_immediate_human

记忆检索【吃】ticket,triage →【吐】memory_hits（并 append evidence_items）

分诊（__init__.py:191）两阶段定性：

  • 阶段 1 规则兜底（_triage_by_rules:82）：关键词命中粗分 deployment_regression / resource_exhaustion / dependency_failure，命中即出 TriageResult。
  • 阶段 2 LLM 补：规则没命中才掉 LLM。

规则在前的原因：便宜、确定、可解释先上，LLM 只兜长尾——和仲裁段矛盾检测用纯规则不花 token 同一个价值观。分诊还顺手算 requires_immediate_human（prod + P0/P1），一路漂到风险闸门影响风险裁决。

记忆检索（retrieve_memory.py）捞历史：拿工单检索过往相似故障/处置（RAG），出 memory_hits 供计划/诊断参考。检索是两阶段（粗排召回 + 重排序 精排），细节见姊妹篇「两阶段检索接入与 AB 归因」。

3 取证段：定计划 → 并行取证 → 聚合打分 → 门控（核心）

计划     【吃】triage,ticket   →【吐】investigation_plan
并行取证 【吃】investigation_plan →【吐】evidence_items[] / specialist_analyses
聚合     【吃】evidence_items   →【吐】evidence_quality_score / contradiction_signals
诊断     【吃】证据+记忆        →【吐】root_cause_candidates[]
仲裁     【吃】quality_score,候选,loop_count →【吐】critic_decision / loop_count+1

3.1 计划：排调查计划（__init__.py:227）

拿分诊的 incident_type/suspected_services + env，模板化排一组 InvestigationTask（category/tool/priority/service）。例如判发布回归就排 logs + k8s + deployments 并行查。

3.2 并行取证：并行只读取证（__init__.py:406）

把任务并发打到工具网关 / 专科 Agent，每路只读取证、回包成 EvidenceItem。关键性质：专科分头取证、只读、互不抢排他资源——这是后面仲裁"不投票只退回"的根因。不存在乐观锁、先到先得那套，会出问题的只有结论冲突，退回重查就行。

当前 8 个专科覆盖了云原生场景的常见故障面：K8s / DB / 日志 / 指标 / 部署变更 这五个是日常高频，网络 / 中间件 / 上游依赖是后来补齐的。在华为云技术栈下（CCE + DAS + APM + VPC），这八个专科对应的工具适配器都在模拟模式跑通了，切真实模式需要对应的云 API 权限。

专科 Agent 内部的 ReAct 循环（specialist_agent.py:158-252）

并行取证只是把任务分发出去，单个专科 Agent 接到任务后内部怎么跑？答案是 ReAct 循环——思考 → 调工具 → 观察结果 → 再思考，直到觉得证据够了或者被强制终止。

启动：Agent 拿到 AgentTask（service/env/incident_type/时间窗），拼成一条用户提示，加上系统提示（从 agent_configs.yaml 读取该专科的专属提示词），准备好该专科允许调用的工具定义。然后进入主循环。

每一轮（round）干什么：

  1. 把当前完整消息历史（system + user + 之前所有的 assistant/tool 消息）发给 LLM，带上可用工具列表。
  2. LLM 返回两种可能：纯文本（觉得证据够了，开始总结）→ 循环结束；tool_calls（还想查更多东西）→ 进入执行。
  3. 校验 LLM 要调的工具是否在允许列表内（_tool_allowed），过滤掉越权调用。
  4. 用 asyncio.gather 把这一轮所有 tool_call 并行打到工具网关，调用 gateway.call_tool()。
  5. 网关拿到 ToolRequest 后：做结构校验（参数类型/必填项）→ 选适配器（真实模式走真实 API，模拟模式返回仿真数据）→ 执行并重试（最多 2 次，指数退避）→ 写审计日志落 DB。
  6. 工具返回的原始结果作为 role: "tool" 消息追加到消息历史——这就是观察结果，下一轮 LLM 能看到上一轮调工具拿到了什么。

什么时候停：

  • LLM 主动不调工具了（认为证据够了）——正常结束，进入最终分析。
  • 剩余实际耗时 < 2 秒——round_truncated = True，截断。
  • LLM 调用本身超时或报错——round_truncated = True，截断。
  • 跑满 max_tool_rounds 轮，LLM 还在要工具——round_truncated = True，强制截断。

各专科的轮次上限（agent_configs.yaml）：

专科max_tool_rounds原因
k8s / db / 中间件3排查面广，需要多轮深入
logs / metrics / deployments2证据相对聚焦，2 轮够用
网络 / 上游依赖2状态检查为主，少数探测需多轮

加上任务级超时（默认 90s）和单次 LLM 调用超时（30s），两层时间控制防止单路卡住拖死全图。

循环结束后的 4 级降级：

不管 ReAct 怎么停的，最后都会调用 _produce_final_analysis() 收口出 SpecialistAnalysis：

级别场景处理confidence
L0LLM 正常返回 + JSON 解析成功直接用 LLM 输出的结论/异常/置信度min(raw, 0.9)
L1LLM 正常返回但 JSON 解析失败降级，用规则从原始数据里提取异常信号0.25 或 0.0
L2LLM 报错/超时，但有工具原始数据降级，纯规则兜底分析0.25 或 0.0
L3LLM 彻底失败，无任何工具数据返回空壳，不计入质量分0.0

L3 对应的就是前面聚合里"LLM_FAILED 不计入 active_count"的那个 Agent。L1/L2 对应的就是 partial=True、置信度再 ×0.5 的那些。

和仲裁质量分的关系：每个专科 Agent 跑完 ReAct 后产出的 SpecialistAnalysis 带三个关键字段——confidence（自评置信度）、partial（是否降级）、truncated（是否被截断）——这三个字段直接喂给聚合的三因子公式。所以 ReAct 的质量直接决定仲裁收到的分是高是低。

3.3 聚合：质量打分（__init__.py:783）= 会计

聚合是整个仲裁体系的"会计"——负责把各专科 Agent 的置信度揉成一个 0~1 的质量分，同时跑矛盾检测，自己不做裁决。核心是打分与决策解耦：换评分公式不用动仲裁的决策逻辑，反之亦然。

三因子公式：质量 = avg × penalty × coverage

对应代码在 aggregator.py:159-185。三个因子各管一件事，乘法逼迫三项都健康才能拿高分。

avg — 平均置信度（材料硬不硬）

每个专科 Agent 调用 LLM 分析证据后，LLM 输出的 JSON 里带 confidence 字段（0~1），表示对自己结论的确信度。代码对这个值做了两道卡口：

场景confidence 取值
LLM 正常返回min(llm_confidence_raw, 0.9) 封顶（specialist_agent.py:375）
LLM JSON 解析失败，但有规则兜底异常信号固定 0.25（specialist_agent.py:444）
LLM JSON 解析失败，且无任何异常信号固定 0.0
LLM 调用本身失败（LLM_FAILED）固定 0.0，且不计入 avg（aggregator.py:168-169 跳过）

进入聚合器后，partial=True（降级结论）的 Agent 置信度再 ×0.5（第 173 行），最后所有活跃的 Agent 取算术平均。

设计意图：

  • 封顶 0.9：不许任何一个 Agent 自称 100% 确定，防过度自信污染下游决策。
  • 降级 ×0.5：LLM 没正常产出、靠规则兜底的结论天然不可靠，打对折。
  • LLM_FAILED 不计入：彻底坏掉的 Agent 不配参与评分，也不拖累平均。
penalty — 截断惩罚（有没有被中途终止）

每个专科 Agent 内部是 ReAct 循环（思考 → 调工具 → 观察 → 再思考），有最大步数限制。步数上限没出结论就被强制终止的，标记 truncated=True。

penalty = max(1.0 - 0.1 × truncated_count, 0.5)

每多一个被截断的 Agent，总分 ×0.9，封底 0.5。截断意味着 Agent 还没收集完证据就被迫交卷，结论不可信。但和 LLM_FAILED 不同——它至少交了卷，所以不排除，轻微惩罚。

coverage — 覆盖度（人到齐没）
coverage = min(active_count / 8.0, 1.0)

预期 8 个专科（k8s / db / logs / metrics / deployments / 网络 / 中间件 / 上游依赖）全部到齐才满。只来 2 个，coverage = 0.25，直接打到底。强制"证据没收全别急着下结论"——即使已有的两个 Agent 自评置信度再高，缺少的维度本身就是一种信号。

三因子为什么是乘法

加法会掩盖短板：avg 0.9 + coverage 0.25 = 1.15，看不出只来了 2 个专科。乘法迫使三项都健康：

全部满分：0.9 × 1.0 × 1.0 = 0.90
只来 3 个专科：0.9 × 1.0 × 0.375 = 0.338 → 触发 证据不足
一个被截断：0.9 × 0.9 × 1.0 = 0.81 → 尚可放行
两个降级 + 只来 4 个：0.3 × 1.0 × 0.5 = 0.15 → 触发 重排计划

任一短板直接砸穿阈值，不会被其他项"平均掉"。

3.4 矛盾检测：跨层差分定位

所谓矛盾检测，就是多个专科 Agent 各说各的——DB 说有问题、K8s 说没问题、日志说报错、网络说丢包、上游依赖说超时——怎么从这些互相冲突的结论里找到故障到底在哪一层。

这块其实做了三次迭代，虽然代码里只落到了第二版。

第一版：让 LLM 当裁判（跑过分段评测，数据不支持上线）

最直觉的做法：把几个 Agent 的结论一股脑丢给 LLM，让它判断谁对谁错。

A：DB 查询超时，怀疑慢查询
B：Pod 状态正常，K8s 层无异常
C：日志里大量 5xx

→ LLM，你判一下

我们用 28 组模拟工单（覆盖单层故障、跨层传播、全正常、全异常四类场景）跑了一轮对比，LLM 裁判 对比人工标注的真实标签，结果三组数据直接排掉了这个方案：

成本：单次矛盾研判平均消耗 380 token（含系统提示），按一天 200 个 run 算，光这一个环节月烧 2.3M token。这还不算生产峰值的弹性预留。

一致性：同一组输入跑 5 遍，28 组中有 11 组至少出现一次判定翻转（比如第一次判"DB 层问题"、第二次判"需更多证据"）。不一致率 39%，在生产排障场景不可接受——同样的故障两次不同的判断方向，回退路径就乱了。

准确率：和人工标注的真实标签对齐，LLM 裁判的整体准确率 71%。拆开看，单层故障场景（比如纯 DB 慢查询）还行，到了跨层传播（DB 慢查询拖死应用 → 网关 5xx）直接掉到 53%——LLM 倾向于把所有"看起来异常"的层都标成根因，而不是排除掉传播路径上的受害者。

三组数据一拉，结论很直接：LLM 裁判在简单场景凑合能用，但在真正需要矛盾检测的复杂场景（跨层传播、多个 Agent 结论冲突）反而最靠不住。所以这个方案没上线。

但它没有完全废弃——后面演进到第三版的时候，LLM 会作为长尾兜底重新回来。拓扑和规则命中的走确定性路径，都命不中的再由 LLM 收口。这时 LLM 面对的是"已经排除了 90% 常见情况后剩下的疑难杂症"，准确率会比"从头判所有案例"高得多。

第二版：有向规则表（当前代码在跑的）

既然 LLM 不稳，那就回归最朴素的办法——把运维排查的经验写死成规则。

核心思路很简单：分层排除。每层出一个结论（正常/异常），两两比较，如果 A 层异常、B 层正常，就说明问题大概率在 A 层而不在 B 层。

例如"日志异常但 K8s 健康"，能推出的是"别查基础设施了，往应用层看"。这不是断定根因，而是在缩范围。

代码实现就是 aggregator.py:138-156 里的 _detect_cross_agent_contradictions，三步走：

① 每个专科的结论压成 4 选 1 标签：正常 / 异常 / 部分 / 失败。优先认结论文本的开头（约定写 异常: 或 正常:），没写前缀的就兜底判断。

② 拿 12 条硬编码的有向规则去比，每条规则格式是 (A 层, B 层, 定位结论)，命中条件只有一个：A 层异常或部分，同时 B 层正常。

A：异常/部分B：正常推出来的结论
logsk8s应用层故障，跟基础设施无关
k8smetrics可能不是资源层的问题
k8sdeployments可能不是发布导致的
dbk8sDB 自身问题，容器层没挂
dbmetrics慢查询/锁争用，不是 CPU 内存瓶颈
metricsk8s资源压力未导致 Pod 级故障
networkk8s网络层问题，非基础设施故障
networkmetrics网络层问题，非资源瓶颈
middlewarek8s中间件自身问题，非容器层
middlewaremetrics中间件问题，非资源瓶颈
upstreamk8s外部依赖故障，非内部基础设施
upstreamlogs外部依赖导致应用报错，非应用自身问题

注意表是有方向的——(logs, k8s) 和 (k8s, logs) 语义完全不同，所以只挑了有意义的那一向。

为什么方向重要？因为这 8 个专科之间有依赖关系，打个比方：

     外部服务（短信/支付）    ← 对应上游依赖（upstream）
            ↓ 调用
你打开的程序（微信）         ← 对应应用日志（logs）
            ↓ 依赖
      操作系统（Windows）     ← 对应 K8s 容器层
            ↓ 依赖
      硬件（CPU/内存）        ← 对应资源指标（metrics）

  网络（WiFi/网线）           ← 对应网络
  数据库/缓存（Redis）        ← 对应 db / 中间件

程序依赖操作系统，操作系统依赖硬件，同时程序还要通过网络调外部服务和读写缓存。故障只能从下往上传播——硬件不够 → 系统变慢 → 程序卡死。反过来不会：微信自己出缺陷，Windows 和硬件不会被拖垮。

所以同样两个层的组合，方向一换，结论完全不同：

方向 A：上层异常、下层正常 → 能定位。 比如"log 异常但 K8s 正常"：程序崩了，但它依赖的操作系统没问题 → 排除基础设施，问题在程序自己。这是确定性排除，一条规则就能缩范围。表里 (logs, k8s)、(k8s, metrics)、(db, k8s)、(middleware, k8s)、(upstream, k8s) 都属于这一类——"X 层自己的问题，不是下面托着的 Y 层导致的"。

方向 B：下层异常、上层正常 → 推不出结论。 比如"K8s 异常但 logs 正常"：系统报错了但程序还在正常跑。可能的原因太多了——系统报的错影响的是别的程序、程序还没来得及感知、错误已发生但没传播到应用层。给不出一个确定的排查方向，所以表里没有 (k8s, logs) 这条规则。

唯一一个反向进表的例外：metrics 异常 + k8s 正常 → 资源压力未导致 Pod 级故障。这条能进表是因为它给了一个有用的边界判断——"资源紧张，但还没到影响服务的地步"，运维看到可以提前扩容，不需要紧急救火。

本质就是一句话：地基坏了会影响房子，房子坏了不影响地基。所以"房子坏了但地基好的"能让你直接查房子结构（确定性强，收进表）；"地基坏了但房子没塌"解释太多（推不出结论，不收）。

③ 命中就产出一条记录，汇进全局摘要。

这版的优点很清楚：确定性强（同一输入永远同一输出）、零 token 成本、可解释（出问题一眼看到是哪条规则触发的）、和 SRE 手工排查的思路一致。

缺点也实在：规则靠人写，换一个业务场景或者加一个新的专科 Agent，规则要重写；只能判"异常 对比 正常"的极性，判不了异常程度；还有一个边界缺陷——_get_conclusion_prefix 按顺序扫证据、碰到第一条带前缀的就返回，如果某专科第一条证据正常、后面才异常，这个专科会被整体标成正常，对应的跨层矛盾就漏检了。好处是实现简单，代价是漏报偏保守。

第三版：APM 拓扑驱动（下一刀的方向）

有向规则表的根因问题是：规则是人猜出来的。12 条规则覆盖了 8 个层级之间的常见组合，但真实的故障传播路径比这复杂得多。

举个例子：DB 慢查询 → 应用线程池耗尽 → 上游调用超时 → 网关 5xx。从表面现象看是"网关异常、DB 也异常"，但根因在 DB。这种多跳传播，静态规则表是覆盖不到的——它只做了两两比较，没做因果链推理。

以华为云为例，APM 提供了服务拓扑图——自动发现服务间的调用关系，画出谁调谁、延迟多少、错误率多少。这是一张实时更新的有向图，天然带因果方向：A 调 B 延迟高，从调用方向上就能判断是 B 慢了导致 A 超时。加上 CCE 的 K8s 拓扑、DAS 的 DB 指标、VPC 的网络流日志——华为云的观测能力本身不差，缺的不是数据，是把数据拼成推理引擎那一步。

但云厂商给的是观测数据（谁调谁、延迟多少、资源怎么挂的），不是可以直接用来推理的因果图。中间差了三步：

  1. 方向标注：A 调 B 延迟高，是 A 的问题还是 B 的问题？拓扑图只告诉你"有调用关系"，不告诉你谁是因谁是果。这需要叠加时序——先异常的节点大概率是因。
  2. 跨产品合图：K8s 拓扑在 CCE 控制台、DB 拓扑在 DAS 控制台、网络拓扑在 VPC 控制台——三张图没有自动合图服务，要自己拉到一起做数据融合。
  3. 实时性对齐：资源拓扑是小时级快照，服务拓扑是分钟级，告警是秒级。不同时间粒度的数据放在一起比，时序对齐本身就是个脏活。

实操上，做到第三版的团队走的路线是：云厂商 API 拉原料（拓扑 + 指标 + 日志）→ 自建因果图引擎（拼图 + 方向标注 + 时序对齐）→ 故障定位（规则匹配 + LLM 长尾兜底）。云厂商给原料，因果图和推理引擎自己搭。

OpsPilot 当前的架构里，这版的接口已经埋好了——仲裁的矛盾分支读的是 contradicting_evidence_ids，不管是规则表产出还是拓扑引擎产出，只要灌进这个字段，裁决逻辑一行不改。信号灌装这一步是有意留的下一刀：先把核心骨架焊死、能独立验证，数据源升级是增量替换，不动骨架。

3.5 诊断：下根因假设（__init__.py:895）

综合证据+记忆，出排好序的 root_cause_candidates（hypothesis + confidence + contradicting_evidence_ids=[] 预留接口）。

3.6 仲裁：4 路裁决 + 回退深度

仲裁自己不打分，直接读上游聚合算好的 evidence_quality_score + 诊断出的候选，只拿分和矛盾信号做裁决。裁决严格按优先级短路（__init__.py:1109+）：

loop_count >= 2         → 转人工     保险丝，不再裁决，候选置信度压到 ≤0.5
有矛盾候选              → 矛盾       策略已就位，信号灌装后才会触发
quality < 0.4 或缺≥2类  → 证据不足
quality < 0.3 且证据<2   → 重排计划
兜底                    → 放行

裁决 → 跳转的回退深度（builder.py:115 _route_after_critic）：

裁决跳转目标回退深度
证据不足回并行取证退一步（计划不变，再取一轮证）
重排计划 / 矛盾回计划退两步（重做调查计划）
放行修复前进
转人工复盘收尾转人工

为什么不用投票：这些专科 Agent 分头取证、只读、不抢排他资源，不需要乐观锁/先到先得那套。真正会冲突的只有结论冲突，退回重查就行。放行不计数，只有重查类裁决（证据不足 / 重排计划 / 矛盾）才给 loop_count +1。

3.7 路由：回路设计

放行 → 修复；证据不足 → 退一步回并行取证；重排计划/矛盾 → 退两步回计划；转人工 → 复盘。证据不硬，宁可绕回去再查也不急着下结论。

4 处置段：出方案 → 闸门 → 审批 → 执行 → 验真

修复     【吃】候选,证据,ticket →【吐】remediation_plan
风险闸门 【吃】remediation_plan,ticket,候选 →【吐】risk_decision
审批中断 【吃】remediation_plan →【吐】pending_approval（图挂起→结束）
执行     【吃】remediation_plan/approval_result →【吐】action_results
验证     【吃】action_results →【吐】verify_decision（并 append 验证证据）
  • 修复（__init__.py:1136）出方案（当前单方案）：拿最高候选+证据规则化映射动作：判发布回归且有部署证据 → 回滚(需审批)；否则兜底 重启（低风险）。
    诚实边界：只出单方案、没有多方案择优。真正的"方案评委"（多方案按恢复速度/影响面/回滚成本加权打分 + 预演模拟）是缺口，要补就在这加 _score_remediation()。
  • 风险闸门（__init__.py:1208）多信号闸门：把动作风险 + severity + env + requires_immediate_human + 诊断置信度 + 循环保护 + 能力预检揉成 4 选 1：仅低风险(放行) / 需审批(卡审批) / 阻断(能力预检失败拦死) / 转人工。
  • 审批中断（__init__.py:1321）HITL 中断：建 pending_approval，图挂起 → 结束，run 停半途等人。人 POST /approvals/{id}/decision 点头后 POST /runs/{id}/resume 从断点续跑。细节见姊妹篇「dispatcher-resume」。
  • 执行（__init__.py:1477）受控执行：ControlledExecutor 幂等 + 审计——同动作重复续跑不执行两次。
  • 验证（__init__.py:1596）验真：执行完不信"我以为好了"，再取一轮验证证据（查 k8s deployment status / 指标回落），3 选 1：成功 / 可重试 / 严重故障。可重试 → 回执行重试（≤2 次）；耗尽/严重故障 → 复盘。

5 收尾段：成败都写复盘（rca_node，__init__.py:1772）

【吃】几乎全 state →【吐】rca_report / final_outcome / status=完成|转人工

不管 成功 / 失败 / 转人工 都进复盘出复盘报告：先定运行结果，再定根因状态（最高置信度 ≥0.7 算确认、否则未知，转人工一律未知），报告 + 证据归档对象存储。

为什么失败也写：复盘是给人看的交接物 + 数据飞轮原料（失败案例进金标集防回退）。

横切：这条旅程怎么被实时看见

每个节点被 _wrap_with_events（builder.py:25）包一层，进节点发 NODE_STARTED、抛异常发 NODE_FAILED，事件落 DB 带序号。前端 SSE 实时渲染步骤条走到哪、断线凭 Last-Event-ID 从断点补发。细节见姊妹篇「SSE 断线续传」。

面试问答底稿

Q：把你这个 Agent 从头到尾讲一遍。

用 30 秒开口起手（5 段），然后讲三个设计点：①全程条件路由可退回重来；②高危 HITL 中断、人不点头不动手；③仲裁先过证据门控才进处置。每段留一个钩子等面试官往里钻。

Q：多 Agent 结论冲突怎么解？

通用做法是仲裁/投票/角色优先级，单 Agent 内部用反思机制对自己的输出做批判再改。回 OpsPilot——专科 Agent 分头取证、只读、互不抢资源，没有抢资源的硬冲突，只有结论冲突的软冲突。不投票：仲裁做质量门控 + 矛盾检测，矛盾优先级最高、命中就退回计划重做调查计划；同一冲突转 2 圈没解决就转人工，同时把候选根因置信度压到 ≤0.5 防误导。

诚实边界：矛盾检测这条线是有意做的接口先行——跨层差分表已经在算了、产出观测信号了，仲裁策略和数据契约也都就位，只差把信号晋升到仲裁读的候选字段。所以矛盾裁决当前不在真实链路触发，这是我清楚并且有意留的下一刀。被追问"那不就是没做完"，答："核心策略和契约先落地、信号灌装放下一刀，仲裁逻辑就能独立验证了。要点亮就差把规则表结果映射进 contradicting_evidence_ids 这一步。"

Q：仲裁是怎么打这个质量分的？

通用做法是置信度加权 + 覆盖度/完整度惩罚。项目里做了打分与决策解耦——会计算 avg × penalty × coverage，仲裁只拿这个分做 4 路裁决，换公式不碰决策。置信度 min(_, 0.9) 封顶防自吹，降级结论 ×0.5，覆盖度按"8 个专科到齐"归一逼着证据收全。

诚实边界：这是只读分析得出的理解，没动过代码。评分阈值 0.4/0.3 是否合理，还没用离线评测验证过。

Q：这算"领导对多个方案择优"吗？

不算。我比喻里的方案评委角色——按统一评分给多个修复方案打分、分差近就预演模拟——现在的仲裁不干这个。它只评"证据够不够下结论"，不评"哪个修复方案更好"。修复出单方案、风险闸门只做通过/不通过。真正的方案择优是缺口，要补就在修复出多方案 + _score_remediation()（按恢复速度/影响面/回滚成本加权），让仲裁或新 arbitrer 节点变成方案评委。

Q：为什么搞 13 个节点，一个 ReAct 循环不行吗？

浅层智能体（裸工具循环）在"高风险、要可控、要可解释、要能中途叫人"的场合会失控。OpsPilot 是生产运维，错一步搞大故障，所以宁可重：取证和处置间硬插仲裁门控、处置里硬插审批中断、执行后硬插验真。这些"多余"节点本质都是闸门，拿编排复杂度换"不闯祸"，每个职责单一、能独立测。

Q：哪几段是缺口 / 没真跑通？

三处，都清楚且有意留：①入口自然语言抽取没做，但插入点明确、契约现成（1）；②矛盾裁决——检测在算、信号在灌，但没晋升到仲裁读的候选字段，那条分支当前不触发（3.4）；③方案择优——只出单方案，没有方案评委（4）。共同点：核心骨架和契约先焊死、能独立验证，适配/增强层留作下一刀，不是没做完。
