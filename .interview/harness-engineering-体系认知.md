# Harness Engineering 在产研中的落地（体系认知）

> 面试素材 · 宏观篇。配套 `PHASE8_DESIGN_DECISIONS.md`（微观篇，讲具体做过的设计决策）。
> 一份讲"我对 AI 产研工程的体系化理解",一份讲"我落过的具体取舍"。一宏一微，量级互补。
>
> 尺度设定：把它当**阿里这种几万人产研、几千团队、年 token 成本以亿计**的 AI Harness 来谈，
> 而不是单个 agent 的工作流。尺度变了，要回答的问题就根本不同。

---

## 0. 一句话定位

**Harness Engineering，是把企业原本依赖人、文档、经验驱动的产研流程，
重构成一套可执行、可编排、可验证、可审计、可持续学习的工程系统。**

它的第一价值不是"模型更聪明"，而是：**研发过程从松散的人工作业，变成一个受控的运行时。**

但在几万人尺度，这句话要再加一层：

> Harness 真正在沉淀的，是**组织的生产力、知识和治理能力**。模型是租来的、会被全行业快速拉平的
> 外部变量；harness 是你内部复利增长、competitor 抄不走的资产。**模型同质化的时代，谁的 harness
> 强，谁就能靠工程把可靠性与效率的差距拉开。**

## 1. 本质：为什么"几万人尺度"才是它的真问题

小团队问的是：**怎么让一个 agent 跑对？**
几万人产研问的是：**当"AI 完成产研任务"成为全组织标配时，如何让几万人的生产力被系统性放大，
而不是裂解成几万次重复造轮子、几万个安全缺口、和一张失控的账单？**

这不是 AI 问题，是**基础设施问题、平台经济学问题、生产关系问题**。

最准的类比是 **Kubernetes**：

| | K8s 之前 | K8s 之后 |
|---|---|---|
| 被调度的资源 | 机器 | 容器化的服务 |
| 状态 | 私有、利用率低、治理分散 | 统一资源池、平台原语 |

这套 Harness 在做同一件事，只是被调度的资源从"机器"换成了
**"AI 完成一个产研任务的能力"**：

- 过去：三万工程师各自开 IDE 插件、各自接大模型 API、各自写 prompt —— AI 产研的"裸金属时代"。
- Harness：把"分析→设计→实现→测试→交付"这条流水线，变成平台**统一编排、统一治理、统一计量**的托管资源。

**一旦认定它是"研发操作系统"而不是"一个工作流"，架构里所有结构就都解释得通**——
控制面是内核的权限与系统调用，编排面是进程调度器，记忆面是文件系统与缓存，治理面是审计与监控。

## 2. 五个工程动作（每个动作配「取舍」与「失败模式」）

GPT 的版本把"做什么"讲透了。这里每个动作我只补它没讲的：**为什么这样选、做反了会死在哪。**

### 动作一：能力原子化

把研发过程拆成独立、可复用的能力单元（需求理解 / 影响分析 / 架构设计 / 编码 / 单测 / 测试范围分析 /
E2E / 失败归因 / 代码评审 / 缺陷修复 / 知识沉淀…），每个单元可由 Agent / Skill / Tool / Script /
CI Job / Human Task / External Service 任一执行者承载。

- **取舍**：原子化牺牲了"一个全能 agent 端到端搞定"的丝滑感，换来**可组合、可复用、可替换、可评测**。
- **失败模式（反例）**：建一个"全能开发 Agent"。它的致命问题不是能力不够，而是
  **不可观测、不可评测、不可治理**——你无法知道它在哪一步错了、无法单独优化某个环节、
  无法复用它的某段能力。全能 agent 在 demo 里惊艳，在三万人规模的治理面前是黑洞。
- **尺度意义**：原子化之后，一个资深专家封装的 Skill，**三万人都能调用**——
  "一人的能力上限，变成全组织的能力下限"。这是规模化复利的前提。

### 动作二：产物结构化（契约层 = 玩具与生产的分水岭）

每个阶段产出**机器可读的中间产物**，而非一大段自然语言：需求阶段出
`{requirements, acceptance_criteria, scope, out_of_scope, risks}`，设计阶段出
`architecture.md / openapi.yaml / data-model.json / decision-records`，测试阶段出
`test_results / evidence_pack / defect_records`。

- **核心判断**：**Harness 里流动的不是对话，是标准化工程产物。Agent 只是产物的生产者或消费者。**
- **取舍**：结构化牺牲表达自由度（自然语言能说的细微差别，schema 装不下），
  换来**下游可自动消费、可评测、可审计**。
- **失败模式**：让 agent 之间用自然语言"交接"。后果是下游无法机器消费、质量无法评测、
  错误无法定位——你得到一个看起来在协作、实际无法治理的系统。
- **微观印证**：这正是我在 Phase 8 把"自由文本根因"改成"封闭枚举 incident_type"的同一思想——
  **把不可比较变成可比较**，是契约层在一个字段上的体现。

### 动作三：规范变运行时契约（"治理即运行时"）

把"请遵循规范/请补测试/请注意兼容"这类**文档规范**，转换成**可执行约束**：必须过 Lint、
核心路径必须有 E2E、接口变更必须生成兼容性报告、高风险变更必须人工审批、租户数据必须带 tenant_id。
落到 输入/输出 Schema、Agent 权限、工具白名单、质量门禁、状态转换条件、审批规则、回滚策略。

- **取舍**：约束做成**「准入/门禁」而非「事后巡检」**。准入是"不满足就过不去"，巡检是"先放过再抽查"。
  准入牺牲灵活性，换确定性。
- **最危险的失败模式**：**声明了约束，但运行时没人强制**。
  工具 Schema 在控制面定义得很漂亮，可业务团队在执行面直接拿到了工具句柄、绕过了约束。
  → 真正的解法是在控制面和执行面之间加一个**权限执行层**：工具调用时实时鉴权，
  而不是靠注册时的 Schema 声明。**声明式约束必须有命令式强制兜底，否则等于没有。**

### 动作四：用确定性的编排，控制非确定性的 Agent（全套架构最核心的原则）

```
编排引擎 = 确定性的控制系统     Agent = 非确定性的智能执行单元
```

- 编排引擎负责：任务顺序、依赖、并行、状态流转、事件触发、超时重试、失败恢复、权限切换、审批、审计。
- Agent 只负责：理解、分析、生成、归纳、判断、提方案。
- **铁律**：最终决策**不能依据 agent 的一句"测试通过"，必须依据真实执行产物**（trace、screenshot、
  results.json）。Agent 可以撒谎/幻觉，产物不会。
- **可靠性来自结构，不来自模型**：新手赌一个强 prompt 一步到位，老手把任务拆成小步、
  每步可验证可回退。**Agent 的可靠性是编排结构给的，不是 prompt 给的**——这是 harness 最反直觉、
  也最值钱的认知。它意味着：**用一个平庸的模型，靠好的 harness，也能跑出可靠的产研结果。**

### 动作五：知识回流闭环（这才是真正的护城河）

一次任务结束不应只留下代码，还要沉淀：需求↔设计↔代码↔测试↔缺陷↔修复↔发布的**双向关联**、
失败模式、有效修复方案、项目隐含规则、flaky 特征。下次 agent 执行时，不再是"检索几个相似代码块"，
而是知道：这个需求涉及哪些模块、这些模块历史踩过哪些坑、有哪些测试覆盖、改动影响哪些调用方。

- **为什么这是护城河而非普通功能**：在几万人尺度，这构成一个**复利飞轮**——
  `用得越多 → 沉淀越厚 → 检索/规划越准 → 更多人用 → 沉淀更厚`。
- **CTO 视角的战略意义**：模型是租来的、会迭代的、谁都能买；
  但**沉淀在 harness 里的组织知识与能力资产，是复利增长且无法被竞品复制的**。
  这是大厂敢为 harness 砸几个亿、养几百号平台工程师的根本理由——
  **模型拉不开差距时，这套沉淀就是差距本身。**

## 3. 完整产研链路（主链路怎么跑）

```
需求进入 → 自动加载（代码/业务知识/历史需求/历史缺陷/规范/产品特有契约）
  → 需求分析（理解→歧义识别→澄清→范围确认→验收条件提取）   ┐
  → 设计（架构→数据↔接口→一致性评审 + 反证 Agent）          │ 每个阶段
  → 开发（前后端并行，受修改范围/权限/目录边界约束）         │ 产出标准化产物
  → 测试（贯穿全程，前端测试给浏览器真实执行证据）           │ + 过质量门禁
  → 用户验收与交付（终态 = UAT 通过 + 证据包 + 回滚方案）     ┘
```

两个容易被忽略、但在尺度下极重要的点：

- **门禁是"卡在原地"而非"打回重来"**：需求阶段歧义没解决，就**停在需求阶段，绝不直接生成代码**。
  让一个需求不清的任务往下跑，浪费的是后面所有环节的 token 和人力——尺度越大，这种浪费越致命。
- **测试不是开发后的单点，是贯穿全程的属性**：需求阶段提取验收条件、设计阶段检查可测试性、
  开发阶段单测自测、测试阶段集成回归、验收阶段 UAT。**测试左移在 harness 里是结构性的，不是口号。**

## 4. 三层可配置：这是对康威定律的正面强攻（我的差异化视角）

架构里 2.1 / 2.2 / 2.4 那条**可配置粒度递增**的线——平台层不可改 → 团队层可配置 → 任务层可编排——
是整张图**政治上最重要**的设计，也是大厂中台的生死线。它解决平台工程的核心张力：

> **中心化才能治理，去中心化才能创新。两者必须同时成立。**

- 安全 / 合规 / 成本 / 审计 → 必须**中心化**。一处约束三万人受益，一个漏洞中心堵住，
  而不是指望三万人各自记得。
- 业务创新 / 领域知识 / 场景流程 → 必须**去中心化**。平台永远不可能比业务团队更懂他们的场景。

这三层本质上**用架构划定了平台团队与业务团队的权力边界**：平台守住底线（2.1），
把方向盘交给业务（2.2/2.4）。

> 康威定律说"系统结构映射组织结构"。这张图**反过来用**——
> 先用三层可配置定义组织的协作契约，再让系统去承载它。
> **大厂中台的坟场里，死的全是没处理好这条线的：管太死→业务跑路自建；管不住→失控背锅。**

落到具体，平台给通用能力，项目配置：哪些阶段存在、哪些节点必选/可跳过、哪些任务并行、
哪些门禁必过、哪些节点要人工、哪个版本模板生效。最终：
**平台统一底座，项目差异化编排，组织规范统一治理。**

不同项目天然需要不同流水线，这正是三层可配置的用武之地：

```
内部管理系统：  需求 → 设计 → 开发 → 单测 → E2E → 验收
核心交易系统：  需求 → 架构评审 → 安全评审 → 开发 → 单测 → 集成 → 性能 → 灰度 → 观察 → 验收
紧急缺陷修复：  定位 → 最小修复 → 定向测试 → 人工审批 → 紧急发布 → 全量回归 → RCA
```

## 5. 尺度经济学：token / quota / 路由是生存项，不是优化项（我的差异化视角）

小项目里加缓存是"性能优化"。几万人尺度，多级缓存、LLM 路由、那个扎眼的"124 轮 Token/RAG"——
**全是财务和风控议题，直接上 CFO 和 CISO 的桌子。**

- **Token 经济学是 P0**：三万人 × 每天 N 次多轮调用，**年成本以亿计**。缓存命中率提几个点、
  把 80% 请求从旗舰模型路由到小模型——省的是真金白银。
  **模型路由在这个尺度不是技术选型，是成本���构设计。**
- **没有 Quota + 熔断，平台会被一个失控 agent 拖垮**："124 轮"说明系统允许长迭代，
  那就**必须**有预算上限和熔断——一个陷入死循环的 agent（比如 critic 一直说证据不够）
  能烧穿一个团队的季度预算。
- **治理的 ROI 与组织规模成正比**：一条 prompt 注��防御、一个工具白名单、一条审计规则，
  在中心写一次，三万人全部受保护。**这是中心化治理在大厂最硬的经济学理由。**

> 小团队可以"先跑通再补治理"；几万人尺度，**治理没内建进架构，平台第一天就是定时炸弹。**

## 6. 组织分工：四类团队（架构必然映射组织）

| 团队 | 负责 | 对应架构层 |
|------|------|-----------|
| **平台团队** | 编排引擎、状态管理、事件总线、权限系统、模板中心、版本管理、运行观测、审计 | 2.1 通用编排调度 |
| **能力建设团队** | 各领域能力组件（需求分析/设计/代码生成/前后端测试/安全/评审） | 2.4 执行单元 |
| **项目团队** | 选模板、配流水线、接项目知识、定产品契约、配质量门禁、处理人工审批 | 2.2 团队配置层 |
| **治理团队** | 通用规范、安全要求、质量标准、评测体系、Bad Case 管理、能力准入与版本发布 | 右侧治理面（竖贯三层） |

**关键**：治理团队对应的治理面在架构里是**竖着贯穿三层**的，不是横在最后一道。
证据链、质量门、评测反馈在每个环节都存在——**评测是 harness 的控制论，不是 QA 的尾巴。**

## 7. 受控自动化，而非无人化（兼论生产关系迁移）

整张图一个最容易被误读的结论：**Harness 的目标不是消灭人，是明确"什么由系统执行、
什么由 Agent 判断、什么必须由人负责"。**

```
确定性任务（Lint/构建/CI）        → 脚本、工具执行
语义理解与方案生成               → Agent 执行
高风险决策（删生产数据/线上回滚） → 人工审批
最终业务验收                     → 需求方负责
```

Human Task 在编排面是**一等公民**，与 Agent / Tool / Script **平级**——这不是给全自动流程打的补丁，
是架构级的设计。把它放到几万人尺度看，潜台词是一场**生产关系的迁移**：

> 三万工程师的主体工作，正从"自己写"迁移到"**编排 AI 写、在关键节点审核与决策**"。
> 人退到三个位置：**约束的制定者**（控制面）、**流程的编排者**（编排面）、
> **关键节点的审核者**（Human Task）。

这意味着大厂值钱的能力画像在变——**未来稀缺的不是"手速快、API 熟"，
而是"能把复杂任务分解成可被 AI 可靠执行的结构、能设计出好的约束与评测"的人。**
这张图把 Human Task 设成一等公民，等于在架构层面承认并固化了这种新分工。

但这个设计也带工程债，面试时值得主动点出（显成熟）：

- **超时策略**：人工 4 小时没响应，任务超时重试 / 升级 / 取消？
- **stale state**：审批期间外部环境变了（代码库有新提交），恢复执行时怎么处理过期状态？
- **审批原子性**：人批准了但执行失败，回滚责任在谁？

## 8. 落地路径：五阶段，顺序不能乱

企业不可能一次建完整体系。这套图给的是一条**有先后依赖**的组织级路径：

1. **能力工具化** — 高频场景先封装成 Skills/Agents/Commands（代码生成、测试生成、Bug 排查、评审）。
   解决"**能不能用**"。
2. **能力标准化** — 为每个能力定义 输入/输出/权限/工具/契约/证据/失败状态。
   解决"**能不能稳定复用**"。
3. **流程编排化** — 把能力放进流水线，加状态、依赖、事件、人工审批。
   解决"**能不能形成闭环**"。
4. **治理与评测** — 质量门禁、证据链、离线数据集、Bad Case、指标体系、版本管理、审计追踪。
   解决"**能不能规模化推广**"。
5. **知识闭环** — 运行数据/代码/需求/测试/缺陷关联成持续更新的知识图谱。
   解决"**能不能越用越好**"。

**先约束、再分解、后编排、终评测与复利。** 这个顺序本身，就是这群架构专家最想传递的方法论——
它是一条**组织能力的建设路径**，而非一张技术架构的拼图。

> 注意第 4 步「治理与评测」是**规模化推广的前提，不是收尾**。
> 这正是为什么 Phase 8 离线评测在我项目里的定位——它不是"做完验收"，
> 而是"让能力敢被三万人用"的准入条件。CISO 和合规敢不敢放行，看的就是这一层。

## 9. 大厂中台坟场：Harness 的典型失败模式（反面，面试加分）

只讲"应该怎样"是不够的。能讲清"做错会死在哪"，才显出深度。这套架构如果落地走样，
最常见的死法：

1. **造全能 agent** → 不可观测、不可评测、不可治理。demo 惊艳，规模化即黑洞。（违反动作一）
2. **agent 间用自然语言交接** → 下游无法机器消费，错误无法定位。（违反动作二）
3. **约束只声明不强制** → 控制面写得漂亮，执行面随便绕过。治理形同虚设。（违反动作三）
4. **让 agent 自己决定流程终态** → "测试通过"是 agent 说的，不是产物证明的。幻觉直接进生产。（违反动作四）
5. **三层可配置失衡** → 管太死，业务跑路自建影子系统；管不住，成本与安全失控。（违反第 4 节）
6. **没有 quota / 计量** → 一个死循环 agent 烧穿预算；多团队成本无法归因。（违反第 5 节）
7. **评测当尾巴而非内建** → 治理跟不上推广，平台带病规模化，最终被一次安全/合规事故叫停。（违反第 8 节）

**七种死法，对应七个动作/原则。这套架构的每一个设计，都是在堵其中一个坑。**

---

## 10. 收口：六个「变成」+ 面试一句话

这套架构在产研中的落地，可压缩成六件事：

```
把研发知识   变成   上下文（认知层 / RAG / 知识图谱）
把研发规范   变成   契约（Schema / 门禁 / 权限）
把研发能力   变成   组件（原子化 Skill / Agent / Tool）
把研发流程   变成   编排（确定性引擎调度非确定性 Agent）
把研发结果   变成   证据（结构化产物 / trace / evidence pack）
把研发经验   变成   知识（回流闭环 / 双向索引 / 复利飞轮）
```

**Harness Engineering 的完整定义**：

> 以编排引擎为运行骨架，以知识、工具、契约为执行输入，以 Agent 和自动化工具为执行单元，
> 以结构化产物和证据链为阶段交付，以质量门禁和人工审批控制风险，
> 并通过运行数据与知识回流持续优化的 **AI 原生产研工程体系**。

**面试一句话收尾（记忆点）**：

> 几万人尺度下，AI 产研的胜负手不在模型，在 harness。
> 模型是全行业共享、会被快速拉平的外部变量；harness 是把**组织生产力、知识、治理**
> 沉淀成复利资产的内部护城河。
> 它的落地顺序是"**先约束、再分解、后编排、终评测与复利**"——
> 而最反直觉的一点是：**harness 做得好，用一个平庸的模型也能跑出可靠的产研结果。
> 谁的 harness 强，谁就能在模型同质化的时代，靠工程把可靠性与效率的差距拉开。**
