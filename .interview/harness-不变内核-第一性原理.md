# Harness 设计的不变内核（尺度无关的第一性原理）

> 面试素材 · 第一性原理篇。与另两份配套，构成一宏一微一本质的三角：
> - `PHASE8_DESIGN_DECISIONS.md` —— 微观：我落过的具体设计决策
> - `harness-engineering-体系认知.md` —— 宏观：几万人尺度的体系
> - 本篇 —— 本质：剥离规模后，无论大系统还是小 agent 都成立的核心思维
>
> 核心主张：**Harness 设计有一组不随目标大小改变的内核。它们不源于"系统有多大"，
> 而源于你的"建材"——LLM 与现实世界——的固有属性。建材的属性不随规模变，对策也就不随规模变。**

---

## 0. 一个判断：所有 harness 都在解同一个方程

无论是阿里几万人的产研操作系统，还是一个只有三个节点的小 agent，**它们要解的方程完全一样**：

> **如何用一个确定、可信、可审计的系统，去包裹一个概率性、会幻觉、不可靠的算子（LLM）？**

系统的大小，只改变这个方程的**系数**（吞吐、租户、成本量级、组织协作），
不改变方程的**形式**。Harness 设计的不变内核，就是这个方程形式本身的解。

这就是为什么——**抄一个大厂 harness 的"功能列表"是学不到东西的，因为功能随规模膨胀；
但抄它的"内核思维"是通用的，因为内核来自建材，不来自规模。**

---

## 1. 不变性从哪来：内核是建材属性的"对策"

这是整份文档的脊椎，也是它和"最佳实践清单"的根本区别。

你盖楼，混凝土会热胀冷缩、会徐变、抗拉差——这些属性，盖平房和盖摩天楼**完全一样**，
所以"留伸缩缝、加钢筋抗拉"这些对策在两种尺度下都成立。

Harness 也一样。你的建材有两种：**LLM** 和 **它所处的现实世界**。它们的固有属性是固定的，
**每一条 harness 内核，都是针对其中某条属性的直接对策**：

| 建材的固有属性 | 它必然导致的风险 | 对应的不变内核 |
|---|---|---|
| **LLM 概率性**：同输入 ≠ 同输出 | 流程不可复现、不可控 | ① 控制权归确定性系统 |
| **LLM 会幻觉**：自信地给出错误 | 错误悄悄进入下游/生产 | ④ 证据高于断言 |
| **LLM 无界**：乐意永远循环下去 | 卡死、烧穿预算 | ② 约束前置（终止条件 / 预算） |
| **LLM 不透明**：无法内省其决策 | 无法调试、无法治理 | ③ 契约优先 + ⑦ 可观测内建 |
| **LLM 单点不可靠**：一步可能就错 | 一步错，全盘错 | ⑤ 分解胜于强提示 |
| **LLM 无状态**：调用之间无记忆 | 无法暂停 / 恢复 / 长跑 | ⑥ 状态外置且可恢复 |
| **世界：动作不可逆** | 误删、误发布无法挽回 | ① 之特例：人在不可逆之处 |
| **世界：外部环境会变** | 过期状态、过期决策 | ⑥ 状态外置 + 证据即时性 |
| **世界：资源有成本** | 成本失控 | ② 约束前置（quota / 熔断） |

**读这张表的方式**：左列在任何尺度下都不变（LLM 在小 agent 里也会幻觉，在大平台里也会幻觉），
所以右列的内核在任何尺度下都适用。**这就是"尺度无关"四个字的全部证明。**

下面逐条展开七个内核。每一条我都会给出**「大系统长什么样 / 小 agent 长什么样」的双重映射**——
两种尺度并排放在一起，本身就是它尺度无关的实证。

## 1.5 一个前置区分：喂给 LLM 的东西分三类——Context / Capability / Constraint

七个内核讲的是"如何**控制**这个算子"。但在控制之前还有更基础的一步：
**你喂给 LLM 的输入，本身要分三类建模，绝不能糊在一个 prompt 里。**

| 类别 | 回答 | 本质 | 例子 |
|---|---|---|---|
| **Context（知识）** | 它**知道**什么 | 参考信息 | 代码、文档、历史缺陷、运行数据 |
| **Capability（能力）** | 它**能做**什么 | 受权限约束的动作 | 读文件、改代码、调 API、跑测试 |
| **Constraint（契约）** | 它**必须遵守**什么 | 硬约束 | 输出 Schema、目录边界、安全规则、完成判定 |

**为什么必须分离**：同一句"项目用 Vue3"是 Context（参考）；"必须用内部表单组件"是 Constraint（强制）；
"可以调组件检索工具"是 Capability（能力）。三者一旦混在一个 prompt 里，LLM 无法稳定区分
**哪些是参考、哪些是铁律、哪些是可执行动作**——这正是它"不透明 + 概率性"两个属性叠加的必然后果。

**为什么尺度无关**：大平台用认知层 / 工具注册 / 契约层三套独立子系统分别承载；
小 agent 至少要在 prompt 里用三个独立区块把它们写开。**混着喂，是新手最隐蔽的一类 bug——
你以为给的是约束，模型当成了参考。** 这条区分是后面七个内核的地基。

## 2. 七个不变内核

### 内核 ① 控制权永远归确定性系统

**一句话**：LLM 是被编排的算子，不是编排者本身。"下一步做什么、是否结束、走哪条分支"
永远由确定性的代码（或人）决定，绝不交给 LLM 自己。

**对抗的属性**：LLM 概率性 + 无界。

- **大系统**：编排引擎（LangGraph / 状态机 / DAG）掌管任务顺序、依赖、并行、状态流转、
  超时重试、失败恢复。Agent 只负责"理解、分析、生成、提方案"。
- **小 agent**：哪怕只有一个 while 循环，**也是你的代码在判断"要不要再来一轮"**，
  而不是问 LLM"你觉得你做完了吗"。终止条件是代码里的 `if`，不是 prompt 里的祈使句。

**它的关键特例——人在不可逆之处**：当一个动作不可逆（删生产数据、线上发布、转账），
控制权不是交给 LLM，而是**交给人**。人在环不是给全自动流程打的补丁，
它是"控制权归确定性决策者"在**世界不可逆性**下的自然推论——
不可逆处，那个确定性决策者必须是人。

- 大系统：审批节点是一等公民，与 Agent / Tool 平级。
- 小 agent：`rm -rf` 之前停下来问一句"确认？"——同一个原则，缩到一行。

**违反会怎样**：让 agent 自己决定流程终态 → "测试通过"是它说的、不是产物证明的 →
幻觉直接进生产 / 死循环烧穿预算。**这是所有 harness 失败模式里最致命的一种。**

---

### 内核 ② 约束前置，而非事后纠偏

**一句话**：Agent 能做什么、必须遵守什么，是在它**执行之前**就划定的边界，
不是跑起来再纠偏。

**对抗的属性**：LLM 无界 + 世界的成本属性。

- **大系统**：工具白名单（注��才能调）、权限边界、输出 Schema、quota 与熔断、
  状态转换条件。约束做成"准入门禁"，不是"事后巡检"。
- **小 agent**：限定这个 agent 只能调这三个工具、最多跑 10 轮、单次 token 上限 N。
  几行配置，但思想完全一致——**先划牢笼，再放进去跑。**

**为什么"前置"是不变的**：事后纠偏意味着错误已经发生（已经多花了钱、已经调了不该调的工具、
已经产生了副作用）。无论系统大小，**"防止"永远比"补救"便宜**——这条经济学在任何尺度都成立。

**违反会怎样**：约束只写在文档/prompt 里、运行时不强制 → agent 绕过约束 → 治理形同虚设。
真正的解法是**声明式约束必须有命令式强制兜底**：工具调用时实时鉴权，而不是靠注册时的声明。

---

### 内核 ③ 契约优先于对话

**一句话**：系统里流动的应该是**结构化产物**，不是自然语言对话。
每个边界都定义机器可读的输入/输出 Schema。Schema 是组合的基本单位。

**对抗的属性**：LLM 不透明 + 概率性。

- **大系统**：需求阶段出 `{requirements, acceptance_criteria, scope, risks}`，
  测试阶段出 `{test_results, evidence_pack, defect_records}`。
  Agent 只是产物的生产者或消费者，**Agent 之间不靠"自然语言交接"**。
- **小 agent**：强制 LLM 输出 JSON / 调用结构化工具，而不是返回一段需要正则去抠的散文。
  一个工具调用需要 schema，和一次跨团队交接需要 schema，是**同一个理由**。

**为什么尺度无关**：自然语言对人友好、对机器敌对——下游无法稳定消费、无法评测、
无法定位错误。这个困境与系统大小无关，只与"机器要不要消费它"有关。
只要下游是机器，就需要契约。

**微观印证**：我在 Phase 8 把"自由文本根因"改成"封闭枚举 incident_type"——
**把不可比较变成可比较**，就是契约思想在一个字段上的最小体现。
当时只是为了能评测，但它本质上是这条内核。

---

### 内核 ④ 证据高于断言

**一句话**：永远不相信 LLM 的自我报告（"做完了""测试通过了""修复了"）。
决策必须基于**可验证的产物**（工具输出、测试结果、trace、截图）。

**对抗的属性**：LLM 幻觉——它会**自信地**告诉你它成功了。

- **大系统**：质量门禁依据 `test-results.json / trace.zip / screenshots` 判断是否放行，
  而不是依据归因 Agent 的一句"看起来没问题"。
- **小 agent**：agent 说"我跑了测试且通过了"——**不信，去捕获真实的测试退出码和输出**。
  agent 说"文件已修改"——去读文件确认。

**为什么这是 harness 的灵魂**：LLM 最危险的不是"会错"，而是"**会以高置信度错**"。
一个会说"我不确定"的模型是安全的，一个永远说"已完成"的模型是地雷。
证据是唯一能戳破自信幻觉的东西。这与系统大小无关——单 agent 也会对你撒谎。

**微观印证**：Phase 8 的整个设计哲学——"用 fixture 固定证据，让 LLM 推理成为唯一变量，
主指标用枚举等值而非 LLM-judge"——就是"证据高于断言"推到评测层的结果：
**连'判断 agent 做得好不好'这件事，都不能交给另一个 agent 的断言，要交给确定性的证据比对。**

**它的进阶形态——反证，不只验证**：证据高于断言只是下限。成熟的 harness 不止问"有什么证据支持结论"，
还主动问"**有什么证据能推翻它**"——哪里可能错、有没有别的解释、什么条件下结论不成立。
LLM 天生倾向于为自己已经给出的结论找支持(确认偏误)，所以"找反例"必须由 harness **结构性地强制**，
而不能指望模型自觉。

- 大系统：独立的 critic / 反证 Agent，专门攻击主诊断；diagnose 的结论要附 `counter_evidence`。
- 小 agent：哪怕只是在 prompt 里多要一个字段——
  `{hypothesis, supporting_evidence, counter_evidence, remaining_uncertainties}`——
  逼模型把"为什么可能不是它"也写出来。反证可以是独立 Agent、第二轮 prompt、规则检查、对照实验，
  形式不限，**但"必须找反例"这条不能省**。

**微观印证**：OpsPilot 的 `critic` 节点就是这条的落地——它专门审视 diagnose 的结论是否证据充分，
不充分就打回。这不是多余的一步，是"证据高于断言"在**对抗确认偏误**方向上的必然延伸。

---

### 内核 ⑤ 分解胜于强提示

**一句话**：可靠性来自把任务**拆成小的、可验证、可回退的步骤**，
不来自把 prompt 写得更强。

**对抗的属性**：LLM 单点不可靠——任何一次调用都有不低的出错概率。

- **大系统**：把研发拆成需求理解 / 影响分析 / 设计 / 编码 / 单测 / E2E / 失败归因 /
  评审… 每个单元独立、可复用、可单独评测。
- **小 agent**：与其写一个"帮我重构整个模块"的巨型 prompt 赌它一次成，
  不如拆成"先列出要改的文件 → 逐个改 → 每改一个跑一次测试"。
  **TDD 的"小步快跑、红绿循环"，就是这条内核在写代码这件事上的形态。**

**最反直觉、也最值钱的认知**：新手以为"agent 不靠谱 = prompt 不够好"，于是不断堆 prompt；
老手知道"**agent 的可靠性是结构给的，不是 prompt 给的**"。
把 0.9 可靠的一步，拆成五个各自 0.98 可靠、且每步可验证可回退的小步——
整体可靠性反而上去了。这个数学在任何尺度都成立。

**它的终极推论**：**好的 harness，用一个平庸的模型也能跑出可靠结果。**
因为可靠性的来源是结构，而结构是你设计的、不是模型送的。

---

### 内核 ⑥ 状态外置且可恢复

**一句话**：执行状态必须活在 LLM 之外，可被快照、可被恢复。
LLM 的上下文窗口不是你的数据库。

**对抗的属性**：LLM 无状态（调用间无记忆）+ 世界会变（长任务期间环境漂移）。

- **大系统**：checkpoint + 事件溯源 + 状态机持久化。任务跑一半等人审批，
  能从精确的节点恢复；崩溃了能重放。
- **小 agent**：把进度写进一个 dict / 一张表，而不是指望"塞进 prompt 让它记住"。
  一个能"暂停等审批、再继续"的小 agent，需要的状态外置和大系统**一模一样**。

**为什么尺度无关**：LLM 天生健忘——这是它的架构属性，不是它"不够大"。
任何需要跨越多步、需要暂停/恢复、需要审计"它当时是怎么想的"的场景，
都必须把状态拿到 LLM 外面管。一步到位的玩具不需要，但任何"真在干活"的 agent 都需要。

**它带出的一个永恒难题（任何尺度都要回答）**：恢复时世界可能已经变了
（审批期间代码库有了新提交）——**过期状态怎么处理？** 大系统叫 stale state 治理，
小 agent 叫"恢复前重新校验一下环境"。同一个问题，同一个答案方向。

**它的孪生陷阱——记忆必须治理，不能直接累积**：状态外置解决"短期记不住"，
但一旦你想让 agent **跨任务**积累经验（长期记忆），就冒出一个新风险：
**运行记录 ≠ 知识，LLM 的结论也 ≠ 知识。** 如果把每次跑的结论无脑写进记忆库，
错误结论会被后续检索出来、再次影响判断、把错误**越强化越深**——这是记忆污染回路。

所以长期记忆必须经过治理：提取 → 去重 → 验证 → 冲突检查 → 分类 → 版本/有效期管理。

- 大系统：知识入库要过质检和准入，bad case 单独管理，知识带版本和失效期。
- 小 agent：Memory 里不该记"登录模块有 bug"（一次性、会过期的结论），
  而该记"该项目登录态由 localStorage 的 access_token + HttpOnly Cookie 的 refresh_token 共同维护"
  （���定、可复用的事实）。**记忆设计的核心不是"记得更多"，是"只沉淀经过验证、未来可复用的事实"。**

---

### 内核 ⑦ 可观测与闭环内建

**一句话**：你必须能**回放**发生过什么（每次 LLM 调用、每次工具调用的输入输出、耗时、决策路径），
并让结果**反哺**改进。可观测不是事后加的日志，是设计时就内建的能力。

**对抗的属性**：LLM 不透明 + 让系统能"越用越好"。

- **大系统**：全链路 trace、证据链、离线评测数据集、Bad Case 库、指标体系；
  评测结果反哺控制面的 prompt / 工具 / 模型选择，形成闭环。
- **小 agent**：把每一次 LLM 调用和工具调用打进结构化日志，出问题能 debug；
  攒一个 bad case 集合，改完 prompt 用它回归。

**两层含义，缺一不可**：
1. **可观测（看见）**：没有 trace，agent 出错你只能瞎猜。这在任何尺度都是 debug 的前提。
2. **闭环（变好）**：结果不反哺，系统就停在第一天的水平。
   Phase 8 的离线评测，本质就是这条内核在小尺度的落地——
   **给"什么是好输出"建立可复现的基准，让改动能被衡量。**

**为什么内建而非附加**：可观测性补在事后，永远是残缺的（关键中间态没记下来就永远丢了）。
它必须在设计每个执行单元时就一起设计——这条在大系统和小 agent 上**没有任何区别**。

## 3. 一个统一的心智模型：把 LLM 当"聪明但不可靠的外包"

七条内核看起来多，其实可以收敛成**一个比喻**——而比喻天然是尺度无关的
（你管一个外包和管一千个外包，原则一样）：

> **把 LLM 想象成一个聪明、但不可靠、记不住事、不知疲倦也不知停止、做完不留痕迹的外包专家。**

你和这样一个人协作，会自然而然立下的所有规矩，**逐条对应七个内核**：

| 你对外包的直觉规矩 | 对应内核 |
|---|---|
| 公章绝不给他，重要决定我拍板 | ① 控制权归确定性系统 |
| 先说清楚他能动哪些东西、预算多少 | ② 约束前置 |
| 要他交可验收的成果物，不是听他口头汇报 | ③ 契约优先 |
| 他说"搞定了"，我要看到东西才信 | ④ 证据高于断言 |
| 大活拆成里程碑，一个个验收 | ⑤ 分解胜于强提示 |
| 他记不住，所以进度和上下文我来存 | ⑥ 状态外置且可恢复 |
| 他做事不留痕，所以我给他装监控、复盘 | ⑦ 可观测与闭环 |

**这个比喻是整套方法论的"压缩包"**：你不需要记七条，你只需要记住
"**LLM 是个聪明但不可靠的外包**"——七条内核都能从这句话当场推导出来。
而"如何与一个不可靠的聪明人协作"这件事，和你雇一个还是一万个，**道理完全相同**。
这就是尺度无关的最后一层证明。

## 4. 通用自检清单：拿它体检任何一个 harness

无论面对的是组织级平台还是一个周末写的小 agent，都可以用这七问体检。
**全是"否"也能跑，但每个"否"都是一个已知的塌方点**：

1. **控制权**：流程的"下一步/是否结束"是代码决定的，还是 LLM 自己说了算？
2. **约束**：它能调的工具、能花的预算、能改的范围，是跑之前就锁死的吗？
3. **契约**：各环节交接的是结构化产物，还是一段需要猜的自然语言？
4. **证据**：放行/结束的依据是真实产物，还是 LLM 的一句"完成了"？
5. **分解**：靠的是任务拆解 + 逐步验证，还是赌一个巨型 prompt？
6. **状态**：进度活在 LLM 外面、可暂停可恢复吗？还是塞在上下文里？
7. **可观测**：出错时能回放它每一步做了什么吗？结果能反哺改进吗？

> 这七问，问一个 30000 人的平台，和问一个 300 行的 agent，**字面一模一样**。
> 答案的工程量天差地别，但**该不该问、问的是什么，完全不变**。

## 4.5 同一组真理的两种切法：对抗属性 vs 工程学科

本文按"**对抗 LLM/世界的哪条属性**"来切���核（因为这样能**推导**出为什么必然是这些）。
但同一组真理还有另一种常见切法：按"**工程学科**"切，分成七个 "-ing Engineering"。
面试官用哪套词不一定，你得能在两套词之间自由翻译——这张表就是翻译器：

| 工程学科切法 | 回答的问题 | 对应本文内核 |
|---|---|---|
| **Context Engineering** | 它知道什么 | 1.5 节 Context + ⑥ 状态/记忆外置 |
| **Capability Engineering** | 它能做什么 | 1.5 节 Capability + ② 约束前置（工具白名单） |
| **Contract Engineering** | 它必须遵守什么 | ③ 契约优先 + ② 约束前置 |
| **Orchestration Engineering** | 它按什么顺序做 | ① 控制权归确定性系统 + ⑤ 分解 |
| **Evidence Engineering** | 如何证明做对了 | ④ 证据高于断言（含反证） |
| **State & Memory Engineering** | 如何持续执行与积累 | ⑥ 状态外置 + 记忆治理 |
| **Evaluation & Governance** | 如何判断系统长期有效 | ⑦ 可观测与闭环 + ① 之特例人在环 |

**两种切法的分工**：学科切法**好记、好讲、覆盖全**（适合开场快速搭框架）；
属性切法**能解释"为什么是这七条而不是别的"**（适合深聊时顶住追问）。
我自己用属性切法做骨架，是因为——**学科清单会让人追问"你怎么知道清单是全的？"，
而属性推导能回答：因为建材就这么几条属性，对策自然就这么几条，清单的完备性是推出来的，不是凑出来的。**

## 5. 收口

**Harness 设计的不变内核，不是关于"如何造一个大系统"，
而是关于"如何与一个概率性算子安全地协作"。**

系统的规模决定你要写多少代码、养多少团队、烧多少预算；
但**决定一个 harness 是否可信的那组原则，是从 LLM 和现实世界的固有属性里长出来的——
它们不随规模缩放**。

所以：

- 你在一个小 agent 上想清楚的内核，放大到组织级平台依然成立（只是系数变大）；
- 你从一个大厂架构图里读到的内核，缩小到自己的小项目依然适用（只是实现变轻）。

**这正是值得反复打磨"做小项目"的理由**：小项目是把这七条内核想透的最低成本沙盘。
内核一旦内化，面对任何尺度的 harness，你都握着同一把尺。

**面试一句话**：

> 我判断一个 harness 好不好，不看它多大、用了什么框架，
> 我看七件事——**控制权在谁手里、约束有没有前置、交接是不是契约、结论靠不靠证据、
> 任务有没有分解、状态在不在外面、过程可不可回放**。
> 这七条不随系统大小改变，因为它们不是关于系统的，是关于"如何驾驭一个会幻觉的概率算子"的。
> 想透它们最好的办法，是在一个小到能完全掌控的 agent 上，把每一条都亲手实现一遍。
