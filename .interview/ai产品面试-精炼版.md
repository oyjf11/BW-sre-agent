# 产品化 Agent：30 题闪卡

> 格式：**要害一句** → 核心3点 → 套项目一句

---

## 一、场景与产品边界

### Q1. 什么场景适合做 Agent？

**要害：路径不固定、需要工具、结果可验证，才值得上 Agent。**

- 单步骤、规则固定 → 普通程序；多步骤、路径动态 → Agent
- 判断门槛：语义理解 × 工具调用 × 可验证性，三者缺一就先别上
- 最高风险：把"Agent 能做"等同于"应该用 Agent 做"

套项目：OpsPilot 的故障诊断就是典型——告警触发后无法提前写死排查路径，得根据现象动态选 K8s / DB / 日志工具，最后有 Playwright 执行结果可以验证，适合 Agent。

---

### Q2. Agent 与 Workflow 有什么区别？

**要害：Workflow 管确定性流程，Agent 处理流程里的不确定节点。**

- Workflow = 路径预知、状态转换固定；Agent = 输入模糊、路径动态
- 生产系统不是二选一：Workflow 兜底整体，Agent 嵌在需要判断的节点
- 滥用 Agent 的表现：把"选择菜单"也交给 LLM 决定

套项目：OpsPilot 的 intake→triage→planner→executor 是 Workflow，其中 diagnose / critic / remediation 节点交 Agent 处理，rca 总结也是 Agent，但是否执行修复由 risk_gate + approval 确定性裁决。

---

### Q3. 如何设计 Agent 的完整用户旅程？

**要害：任务工作台，不是聊天框。覆盖"发起→执行→干预→验收→异常恢复"全链路。**

- 对话只负责输入和补充上下文，其余全部在任务看板展示
- 每个阶段用户要知道：进展在哪、是否需要我操作、失败了能不能重试
- 结果交付不是一段话，是：结论 + 产物 + 证据 + 风险 + 后续操作

套项目：OpsPilot 用户旅程：提工单 → triage 确认 → planner 给方案（用户可确认）→ executor 执行（审批中断）→ verify 结果 → rca 报告，每步都有 events 流和 trace 可查。

---

### Q4. 如何向用户解释 Agent 的能力边界？

**要害：边界要在任务开始前说清，失败后再解释等于失去信任。**

- 明确"支持 / 暂不支持 / 仅建议不执行"三档
- 执行权限分级：自动执行 / 验证后执行 / 人工批准后执行
- 结果展示：区分"已验证事实"和"模型推断"，不能混在一起

套项目：OpsPilot 明确区分：K8s 只读查询自动执行，重启 Pod 需审批，数据库写操作 risk_gate 拦截，不会告诉用户"Agent 能自��修复一切"。

---

### Q5. 为什么 Agent 产品容易"Demo 惊艳，实际不好用"？

**要害：Demo 证明模型能做到，产品要证明能稳定做到一万次。**

- Demo 的条件：输入清晰、数据完整、环境正常；真实用户：模糊输入、上下文缺失
- 三大坑：①无状态恢复（工具挂了整个任务重来）②无验证（LLM 说对了就信）③无成本控制（多 Agent 规模化后账单爆炸）
- 缺少数据闭环是根本：没有 Bad Case 管理，无法持续迭代

套项目：OpsPilot 的应对：checkpoint 支持中断恢复，risk_gate 独立校验修复方案，evidence 持久化提供真实执行证据，不靠 LLM 自说自话。

---

### Q6. 如何平衡 Agent 自主性和流程确定性？

**要害：让 Agent 在判断的地方自主，让规则在执行的地方兜底。**

- 风险分级：低风险自动、中风险校验后执行、高风险强制人工
- Agent 负责判断，系统负责裁决——两者职责不能倒置
- 自主性的天花板：Agent 可以建议，但不能自己宣布"任务完成"

套项目：OpsPilot 的 critic 节点会反驳 diagnose 的结论，remediation 给方案，但"是否执行"由 risk_gate 规则 + approval_interrupt 人工双重把关，Agent 无法绕过。

---

## 二、Harness 与平台设计

### Q7. Harness 对 Agent 产品价值的核心贡献是什么？

**要害：Harness 不让模型更聪明，让模型能力变成可控、可验证、可复用的工程资产。**

- 控制上下文（只给当前节点需要的信息，不暴露全量）
- 控制边界（权限、工具参数校验、质量门禁）
- 控制状态（中断、恢复、追踪、证据持久化）

套项目：OpsPilot 的 GraphRunner 就是 Harness 的核心——管 checkpoint、管 evidence 持久化、管 approval interrupt，模型只负责推理，Harness 负责把推理结果变成可执行、可审计的工程行为。

---

### Q8. 如何平衡平台统一与项目个性化？

**要害：强制统一安全、审计、数据隔离；开放配置测试策略、门禁阈值、角色审批人。**

- 不可变：权限模型、质量结果格式、高风险审批规则
- 可变：流程阶段组合、技术栈适配、项目特殊规范
- 模板机制：平台提供标准模板，项目调参而非从零配置

套项目：OpsPilot 中不同业务线可以配置自己的 SLO 阈值、K8s namespace 权限、审批人，但 risk_gate 的安全规则和 evidence 格式由平台统一，不允许项目绕过。

---

### Q9. 如何将复杂业务流程拆成 Agent 节点？

**要害：一个节点 = 一个目标 + 一组明确输入输出 + 一套完成条件。**

- 拆分维度：理解 → 决策 → 执行 → 验证，每段是一个节点
- 节点太大的信号：失败不知从哪恢复、同时承担判断和执行
- 节点太小的信号：节点间频繁传递上下文、编排复杂度超过业务价值

套项目：OpsPilot 把故障处置拆成 13 个节点，每个节点输入输出在 state.py 里明确定义，evidence_fanout 失败只重跑该节点，不影响已完成的 triage 和 planner。

---

### Q10. 哪些节点必须设置人工审批？

**要害：审批不由"模型是否聪明"决定，由"风险 × 不可逆性 × 责任归属"决定。**

- 必须审批：生产写操作、数据删除、发布回滚、权限变更、对外正式通知
- 不需要审批：只读查询、Lint、沙箱执行、草稿生成
- 审批本身要说明：为什么需要你、要决定什么、不同选择有什么后果

套项目：OpsPilot 的 approval_interrupt 只拦截 executor 中的写操作（重启、扩容、回滚），只读的 evidence_fanout 全部自动，risk_gate 是规则拦截，不走人工。

---

## 三、交互体验与人机协同

### Q11. Chat UI 还是任务工作台？

**要害：Chat 负责发起，任务工作台负责管理——复杂 Agent 必须两者都有。**

- Chat 适合：自然语言输入、需求澄清、临时追问
- 工作台适合：长任务、多阶段、审批、失败恢复、证据查看
- 设计原型：左列=任务/历史，中间=对话+状态，右侧=产物/证据/Diff

套项目：OpsPilot 的 RunDetailPage 就是工作台——Stepper 展示阶段进度，Tabs 分开展示 events / evidence / diagnosis / remediation，不是一个聊天窗口。

---

### Q12. 如何向用户展示 Agent 当前状态？

**要害：把技术状态翻译成业务语言，让用户知道"现在在干什么、需不需要我介入"。**

- 不要展示：`Node 3 running`；要展示：`正在分析本次变更影响的服务依赖`
- 三层状态：任务级（执行中/等待/失败）、阶段级（triage/诊断/修复）、节点级（高级用户可见）
- 关键区分：系统在执行 vs 系统在等待 vs 系统已阻塞 vs 需要用户操作

套项目：OpsPilot 的 events 流包含 node_name + message，前端按阶段聚合展示，approval_interrupt 触发时前端高亮"等待审批"而不是静默等待。

---

### Q13. Human-in-the-loop 解决什么问题？

**要害：不是加审批，是在三个地方精准介入：高风险决策、语义不确定、责任归属。**

- 不只是执行前审批，可以在输入补充、方案选择、结果验收多处介入
- 设计原则：每个人工节点必须说明"为什么需要你、不同选择有什么后果"
- 副产品：用户修正 = 高价值 Bad Case，要回流到数据集

套项目：OpsPilot 的 approval_interrupt 暂停后展示修复方案、影响范围和回滚选项，审批人的决策记录进 evidence，后续 RCA 可查。

---

### Q14. Agent 置信度不足时应该怎么办？

**要害：先自己补证据，补不了再问用户，不是统一走人工。**

- 判断矩阵：置信度 × 风险等级 × 补证据成本 × 用户等待成本
- 四个选项：继续查证 / 询问用户 / 输出多个候选 / 停止人工接管
- 关键原则：把"最小问题"交给用户，不是把"整个任务"扔给用户

套项目：OpsPilot 的 diagnose 不确定时，critic 节点会追加反驳和补充查证，仍无法收敛再触发 approval_interrupt 附带置信度说明，不是遇到不确定就停。

---

## 四、可信结果与记忆

### Q15. 为什么 Agent 产品需要证据，不只是答案？

**要害：Agent 输出是推断，证据让用户自己判断是否可信、是否可以继续执行。**

- 证据三用：建立信任、支撑决策、支撑审计复盘
- 证据 ≠ 解释：解释是模型生成的语言；证据是日志、截图、工具输出
- 产品要求：从结论可以一键跳转到原始证据，不能只看总结

套项目：OpsPilot 的每个工具调用结果都持久化进 evidence_items，diagnosis 节点输出根因同时附带支撑它的 evidence，用户看到的不是"LLM 认为是 DB 连接池满"，而是"DB 连接数指标 + 慢查询日志 + 接口报错截图"。

---

### Q16. 是否应该向用户展示置信度？

**要害：展示等级和理由，别展示没有校准的百分比——数字没有解释等于虚假精确。**

- 好的表达：`高可信：日志/指标/重试三项一致` vs `中等可信：仅有接口报错，无 DB 侧证据`
- 什么时候才能展示数字：经过历史校准、用户理解含义、有 Ground Truth 支撑
- 最差设计：展示 `82%` 但用户不知道这个数字的参照系

套项目：OpsPilot 的 diagnose 输出根因时标注 `confidence: high/medium/low` 并附 evidence_ids，前端展示理由而不是百分比。

---

### Q17. 执行结果和 Agent 判断冲突，信哪个？

**要害：真实运行事实 > 确定性规则 > 权威数据 > Agent 判断，没有例外。**

- Agent 可以预测"这次修复应该有效"，但工具执行失败就是失败
- 冲突处理：标记冲突 + 不允许通过质量门禁 + 重新分析 + 保留为 Bad Case
- 产品设计：Agent 提议、工具执行、系统裁决——三者角色分清

套项目：OpsPilot 的 verify 节点用真实工具检查修复结果，如果工具返回失败，即使 remediation 节点认为方案应该有效，run 也不会进入 completed 状态。

---

### Q18. Agent 应该记住哪些信息？

**要害：只记"未来仍然有价值、经过验证、用户授权、作用域明确"的信息。**

- 适合记忆：用户偏好、项目技术栈事实、经验证的 Runbook 规律
- 不适合记忆：未验证推断、临时状态、敏感凭据、模型生成但未确认的结论
- 作用域必须分层：会话记忆 / 用户记忆 / 项目记忆 / 组织知识，不能混

套项目：OpsPilot 的 retrieve_memory 节点只拉"项目服务元数据 + 历史经验证 Runbook"，不把 LLM 本次推断的根因直接写进知识库，需经 RCA 人工确认后才入库。

---

### Q19. 如何防止错误信息进入长期记忆？

**要害：记忆写入要有治理流程，不能让 Agent 自由写入。**

- 流程：候选提取 → 来源检查 → 事实验证 → 去重冲突检测 → 审核 → 正式写入
- 记忆状态：candidate / verified / approved / deprecated / conflicted
- 权威层级：用户明确确认 > 系统真实数据 > 多次重复观察 > Agent 单次推断

套项目：OpsPilot 的 RAG 写回（OSS RCA 归档）只在 rca 节点完成且 run 状态为 resolved 后触发，单次 LLM 推断不会直接写入，防止错误结论污染 Runbook。

## 五、指标与评测

### Q20. 产品化 Agent 的北极星指标是什么？

**要害：有效任务完成率——不是调用量、不是 Token 数、不是对话轮次。**

- "有效完成"四要素：任务完成 + 结果被验证 + 用户接受 + 没有额外返工
- 辅助指标：人工干预率、Bad Case 率、平均任务时长
- 最差指标：DAU、对话轮次（能反映流量，不能反映价值）

套项目：OpsPilot 的北极星：一次正确故障解决率（run 进入 resolved 且无人工推翻的比例），辅助：approval 退回率、verify 失败率、RCA 人工修改率。

---

### Q21. 如何定义 Agent 的任务成功？

**要害：系统运行结束 ≠ 任务成功，要分四层验证。**

- 技术完成：工作流无报错结束
- 产物完成：产生规定输出
- 结果正确：通过验证（工具/规则/人工）
- 业务有效：用户实际采用，没有重新做同一件事

套项目：OpsPilot 的 run 有 pending / running / waiting_approval / completed / failed / resolved 六个状态，completed 只代表流程跑完，resolved 代表用户确认问题已解决，是两个不同状态。

---

### Q22. 如何评估 Agent 的准确率？

**要害：先定义"什么的准确率"——意图识别、工具选择、根因判断都要分开测。**

- 不能只看最终成功率，要分阶段：意图识别率 / 工具参数准确率 / 根因判断准确率 / 最终解决率
- 评估方法：建 Ground Truth 标注集 + 分阶段计算 Precision/Recall/F1
- 在线验证：用户修正率 / 人工推翻率 / 重新执行率

套项目：OpsPilot 的评估分层：triage 意图准确率、diagnose 根因分类 F1、remediation 方案采纳率、verify 一次通过率——分别对应不同的优化手段。

---

### Q23. 如何设计 Bad Case 管理机制？

**要害：Bad Case 不是收集失败对话，是结构化定位"哪个节点、为什么、怎么修、有没有回归"。**

- 数据结构：输入 + 上下文 + 模型版本 + Prompt 版本 + 错误节点 + 预期 vs 实际 + 根因分类
- 分类维度：需求理解错误 / 上下文缺失 / 检索错误 / 工具调用错误 / 模型推理错误 / 环境异常
- 闭环：收集 → 分类 → 排优先级 → 修复 → 回归测试 → 灰度监控

套项目：OpsPilot 每次 verify 失败或 approval 被拒，自动生成 Bad Case 候选，标注 node_name + error_type，优先修复高频根因分类（比如"根因诊断漏掉 DB 维度"）。

---

### Q24. 如何平衡效果、速度、成本和体验？

**要害：任务分级——低价值用小模型快速返回，高风险用大模型多源验证。**

- 渐进式结果：先返回"任务已接受 + 初步判断"，完整结果异步更新
- 模型路由：分类/摘要用小模型，复杂规划/根因分析用大模型
- 上下文裁剪：不把整个历史塞进每次请求

套项目：OpsPilot 的 triage 用轻量模型做意图分类，diagnose 和 critic 才用高质量模型；evidence_fanout 并行执行工具调用而不是串行，P99 延迟控制在业务可接受范围。

---

## 六、结合项目

### Q25. 为什么做开发套件而不是直接推通用 Agent？

**要害：通用 Agent 解决"能做什么"，企业套件解决"在公司环境里怎么做、做到什么程度、如何治理"。**

- 通用工具的问题：每人用法不同、产出不稳定、无法审计、经验无法复用
- 套件的价值：把个人最佳实践变成团队标准能力（Rules / Skills / Harness / 验证 / 证据）
- 不是重做一个 Agent，是在 Runtime 之上建工程治理层

套项目：OpsPilot 不是在 Claude Code 之上又做个对话框，而是把 SRE 排障经验固化成 13 个节点的 Harness，新人加入团队也能走标准化处置流程，结果可审计可追溯。

---

### Q26. 为什么某个能力值得成为独立能力域？

**要害：技术复杂度高 + 领域知识强 + 可在多个流程复用 + 是关键质量门禁，满足这四条才拆。**

- 不要因为"功能多"就拆，要因为"责任边界清晰、复用价值高"才拆
- 独立能力域需要定义：输入/输出/工具权限/完成条件/证据格式/版本号
- 拆太细的代价：编排复杂度超过业务收益

套项目：OpsPilot 的故障诊断是独立能力域——因为它同时承担根因分析、证据采集、修复方案生成和验证，复用于告警触发、人工工单、定期巡检三个入口。

---

### Q27. 核心价值是生成方案还是验证结果？

**要害：生成方案是手段，验证结果是目的。**

- 方案可以看起来合理但执行失败、覆盖错误场景、只是形式上存在
- 完整价值链：理解目标 → 设计方案 → 真实执行 → 判断结果 → 提供证据
- 产品指标：不看生成多少，看有效方案率 / 执行成功率 / 问题解决率

套项目：OpsPilot 的 remediation 生成修复方案只是中间步骤，verify 节点用真实工具验证修复效果才是核心价值交付——没有 verify 就没有闭环。

---

### Q28. 执行失败后，如何分类是代码/环境/数据/用例问题？

**要害：要独立的 Failure Triage 流程，不是把报错堆栈直接返回用户。**

- 证据组合：断言结果 + 服务健康状态 + 工具返回 + 重试结果 + 历史记录
- 代码问题：稳定复现 + 多次一致 + 修复后消失
- 环境问题：多个无关用例同时失败 + 健康检查异常 + 依赖超时
- Flaky：重试结果不一致 + 失败集中在时序/异步

套项目：OpsPilot 的 verify 失败后 critic 节点会分析：是修复方案没效果（代码问题）、是环境还未就绪（环境问题）、还是 verify 工具本身有问题（用例问题）。

---

### Q29. 如何衡量 Agent 能力的真实效果？

**要害：分五个维度，不能只看完成率或通过率。**

- 设计质量：核心场景覆盖率 / 有效方案率
- 执行质量：一次成功率 / 平均执行时长 / Flaky 率
- 问题发现价值：有效问题发现率 / 确认率 / 逃逸率
- 归因质量：归因准确率 / 人工修改率 / 平均定位时间
- 效率价值：人工投入降低 / 处置时长降低 / 重复工作减少

套项目：OpsPilot 的 KPI：一次正确解决率（北极星）+ verify 失败后归因准确率 + approval 退回率 + RCA 人工修改率，四个指标覆盖执行、判断、信任、复盘四个维度。

---

### Q30. 如何将 Agent 能力作为标准节点接入编排平台？

**要害：封装成可组合节点，不是绑定一条固定流水线。**

- 每个节点定义：输入/输出/工具权限/超时/重试/成功条件/证据格式/版本号
- 与编排平台的协议：平台触发节点 → 节点读取上下文 → 执行 → 返回结构化结果 → 平台根据结果流转
- 版本绑定：每次执行绑定节点版本 + 流程版本 + 配置版本，避免行为漂移

套项目：OpsPilot 的 builder.py 就是编排层，每个节点（diagnose / remediation / verify）独立封装，上层可以通过配置决定在 staging 告警时跑哪些节点、失败时走哪条边，不耦合固定路径。

## 答题万能框架（2分钟版）

```
1. 明确目标：用户是谁、任务是什么、当前最大痛点
2. 划分确定性/非确定性：哪些给 Agent、哪些给 Workflow/规则
3. 设计完整闭环：输入 → 执行 → 验证 → 异常 → 人工 → 沉淀
4. 明确边界：自动到哪、何时停、何时需要人工
5. 用指标验证：任务完成了吗、人工投入降了吗、质量提升了吗、成本可控吗
```

**最后一句话**：产品化 Agent 不是设计更多 AI 功能，是判断哪里需要智能、哪里必须确定，用流程+工具+证据+人工协同形成完整闭环。
