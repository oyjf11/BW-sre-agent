> 基于 Claude 对话整理，覆盖多租户、Supabase 隔离方案、Router/Planner 分层、场景化 RAG、增量修改、代码质量、LangGraph、Sandpack 等核心考点。

## 一、两层多租户认知（面试开篇加分）

这类产品（Lovable / v0 / Bolt）有**两层租户**，不能混为一谈：

- **第一层**：用平台的开发者之间隔离（普通 SaaS，共享库 + RLS 基本够用）
- **第二层**：每个开发者生成出来的 App 本身又是一个完整应用，有自己独立的数据库、最终用户，也需要隔离 —— **真正难点在这里**

> 面试金句："这类产品的多租户特殊在于，它不只是隔离用户数据，还要隔离用户生成出来的那个 App 的运行时和数据库，相当于租户套租户。"

四个关键决策边界：
| 层面 | 推荐方案 |
|------|---------|
| 平台数据 | 共享库 + RLS，先别过度设计 |
| 代码执行/构建 | **必须强隔离**（沙箱），安全红线 |
| 生成应用的运行时 | 容器池 vs serverless vs 边缘 |
| 生成应用的数据后端 | 每项目独立实例 vs 共享实例逻辑隔离 |

## 二、基于开源 Supabase 的多租户方案

Supabase 开源版是"单项目"设计，需要自研控制平面补上管理层。

### 三种隔离路线（从轻到重）

| 方案 | 做法 | 类比 | 优缺点 |
|------|------|------|--------|
| **A** | 单 Postgres + Schema 隔离 | 合租房大通铺 | 最省钱，但 schema 上千后元数据膨胀、Storage/Realtime 不按 schema 分 |
| **B** | 每项目独立 Supabase 栈 | 独立别墅 | 隔离最强，运维最重，每个 App 一套容器 |
| **C（推荐）** | 共享 Postgres + 每项目独立 Database | 公寓楼各户独立 | **性价比最高，推荐起点** |

### 路线 C 详解

**核心架构：**
```
控制平面（自研：项目生命周期/密钥/路由）
    │
    ├─ 网关层（Kong/自研路由）
    ├─ 共享无状态组件池（GoTrue/REST/Storage）
    └─ Supavisor（连接池）
            │
      共享 Postgres 实例
         ├─ db_project_a
         ├─ db_project_b
         └─ ...
```

**关键要点：**

1. **每项目独立 database**：天然隔离，`DROP DATABASE` 一行删除干净利落；模板 database 秒级新建
2. **Supavisor**：Supabase 官方多租户连接池，事务级复用，解决 N×M×K 连接爆炸
3. **用户体系天然隔离**：每个 database 独立 auth.users，A 应用注册和 B 应用完全无关
4. **休眠机制**：僵尸项目停配套服务、只留 database；冷启动唤醒（秒级）
5. **分片**：单实例有上限，满了开新实例，维护"项目→服务器"映射表
6. **平滑升级**：重点项目流量大/要合规 → 单独迁到独立实例

**控制平面工作：** 项目创建/销毁、密钥生成存储（每项目独立 JWT secret）、路由、休眠回收、配额管理

### 面试追问准备

| 追问 | 答案要点 |
|------|---------|
| 连接数会爆吗？ | Supavisor 事务级连接池，千个客户连接复用少量真实连接 |
| 用户体系怎么隔离？ | 每个 database 独立 auth.users，天然隔离，不用自己写过滤 |
| 僵尸项目浪费资源？ | 休眠机制：停服务、留 database，请求进来秒级唤醒 |
| 一台服务器满了怎么办？ | 分片：开新实例，维护映射表 |

> 收口金句："开源 Supabase 是为单项目设计的，而这类平台需要的是'一个账号管很多项目'的能力——真正的工作量不在数据库本身，而在自己补上那层控制平面。"

## 三、分片（Sharding）vs 弹性扩容（Elastic Scaling）

| | 分片 | 弹性扩容 |
|---|------|---------|
| 解决 | "一台装不下，东西放哪" | "忙时加多少、闲时减多少" |
| 本质 | 布局问题 | 随负载调节问题 |
| 适用 | 有状态的数据库 | 无状态的服务层 |

> 面试点：无状态层弹性扩容容易，有状态的数据库很难弹性，主要靠分片来水平扩展。

## 四、Router & Planner 分层

| | Router | Planner |
|--|--------|---------|
| 职责 | 轻量分类决策 — "走哪条路" | 重任务拆解 — "这条路怎么一步步走" |
| 模型 | 小模型/分类器/规则 | 大模型（强推理） |
| 定位 | 简单请求走快车道 | 复杂任务才走重流程 |

**为什么要分层：** 不是所有请求都需要 Planner，"改个颜色"这种简单请求直接走轻量流程，省 token 省时间。

**追问：**
- "Planner 拆错了怎么办？" → 反馈/重规划机制（re-plan），失败步骤回 Planner 调整 → LangGraph 循环图

## 五、场景化 RAG — 语料积累与效果评估

### 语料来源

| 来源 | 说明 |
|------|------|
| 官方文档/最佳实践 | React、Tailwind、Supabase API 等框架文档 |
| 高质量代码模板 | 平台沉淀的标准写法（登录、CRUD、表单） |
| 历史成功案例 | 用户成功生成、没被反复修改的代码回流 |
| 错误→修复对 | 用户报错、自动修复的 case 沉淀 |

### 效果评估（两层）

| 层面 | 指标 |
|------|------|
| **检索质量**（中间指标） | 召回率、命中率、相关性 |
| **最终生成质量**（最终指标） | 编译通过率、用户改动次数、人工/模型打分 |

> 面试点："评估 RAG 不能只看检索准不准，要看它最终有没有让生成代码更好——能跑通、用户少改，才是真有用。"

## 六、Agent 逻辑写在前端静态代码的隐患（🔴 高危题）

**动机：** MVP 阶段迭代快、少一跳（省中转服务器）

**隐患（得分点）：**
1. **密钥泄露（最致命）** — 前端代码浏览器可见，API Key / 数据库密钥直接暴露
2. **逻辑全暴露** — prompt、工具调用抄袭和逆向
3. **无法服务端管控** — 限流、鉴权、计费、审计不可靠
4. **难统一升级** — 改 prompt 要重新发布前端

> 收口："前端只负责展示和交互。密钥、计费、鉴权、限流、内容安全必须放服务端。前端调用走 BFF 代理，由后端持有密钥、调用模型。"

## 七、LangGraph TS vs Python 版本对比

| 维度 | Python 版 | TS 版 |
|------|-----------|-------|
| 生态成熟度 | "亲儿子"，功能全、更新快 | 跟进慢，某些连接器缺失 |
| 类型系统 | 灵活随意（运行时才暴露问题） | 静态类型安全，但 state 定义繁琐 |
| 流式/异步 | 稳定 | 实现和行为可能有差异 |
| 选型现实 | 很多团队 AI 编排用 Python | 前端是 TS，多语言有架构成本 |

## 八、增量修改 — RAG + 依赖图识别变更范围

**核心矛盾：** 模糊指令（如"把登录按钮改蓝"）要精准映射到代码位置，全量喂给模型既贵又容易改坏。本质是个**定位**问题。

### 完整链路

```
用户指令
   │
   ① 意图理解 ── 操作类型 + 目标实体 + 约束
   │
   ② RAG 检索 ── 混合检索（向量语义 + 关键词精确）
   │              按语义单元切片（组件/函数/export块）
   │              每个chunk带路径、组件名、导出/依赖元数据
   │
   ③ 依赖扩展 ── 顺着 import/调用关系向外扩
   │              谁引用我（反向）+ 我引用谁（正向）
   │
   ④ 范围确定 ── 合并且去重，圈定最小变更集
   │
   ⑤ 增量生成 ── 输出 diff/patch，非整文件重写
   │
   ⑥ 校验回写 ── 编译/类型检查通过才落盘，失败回喂修复
                  成功后增量更新索引和依赖图
```

### 混合检索

| 方式 | 作用 | 优势 |
|------|------|------|
| 向量检索 | 语义匹配："登录按钮" → SignInButton | 理解意图 |
| 关键词 BM25 | 精确命中：函数名、变量名 | 精确标识符 |
| 元数据过滤 | 按文件类型、组件范围先筛 | 降低噪声 |

> 面试金句："RAG 负责'语义定位'（从模糊指令找到最相关代码），依赖图负责'影响扩散'（从一个点找到所有牵连的点）。RAG 解决'改哪里'，依赖图解决'还会连带影响哪里'，两者缺一不可。"

### 增量生成输出 diff 的理由

整文件重写会：① 丢掉用户手动微调 ② 每次重写引入新随机性 ③ 慢且贵。diff 精确可控、可回滚。

## 九、代码质量保证

### 关卡式闭环节

```
生成阶段约束（高质量模板 + RAG 正确用法）
  → 静态检查（ESLint + Prettier + tsc + build）
    → 自动修复闭环（报错喂回模型修正）
      → 用户验收
```

### 反幻觉 Prompt 核心约束

1. 优先基于【参考范式】，不脱离凭空设计
2. 范式未覆盖点标注"来自需求"，不能凭空增加
3. 范式无关步骤要剔除，不硬凑
4. 每步必须溯源到"需求"或"范式"
5. 不确定的用范式默认值并标注"默认方案，可调整"

> 收口："靠'生成时约束 + 生成后自动化检查（ESLint/Prettier/tsc/build）+ 报错自动回喂修复'闭环兜底，把代码质量从靠模型运气变成有流程保障。"

## 十、Sandpack 在线预览性能问题

| 问题 | 原因 | 优化 |
|------|------|------|
| 首次加载慢 | 浏览器内启动打包环境 | 依赖缓存/预构建 |
| 内存吃紧 | 整个运行环境在浏览器 | 大项目降级服务端预览 |
| 依赖安装慢 | 浏览器侧解析安装 | 惰加载/按需编译 |
| 热更新延迟 | 每次改动全量编译 | debounce + 增量编译 |

## 十一、RAG 做在 Router/Planner 层（核心设计）

> 区别于"代码 RAG"（检索代码片段），这里的 RAG 检索的是 **"场景 → plan 范式"** 的经验。

### 解决的问题

用户模糊指令（"做个带登录的待办应用"）和系统可执行的 plan 之间存在语义鸿沟。RAG 检索相似场景下验证过的 plan 范式，喂给 Planner 当参考。

### 完整链路

```
用户模糊指令
   │
   Router（意图识别 + RAG 检索）
   │  产出：结构化意图 + 命中 plan 范式 + 原始需求
   │
   Planner（融合 + 结构化 + 自检）
   │  产出：结构化任务图 DAG
   │
   执行器 / Code Generator
```

### 为什么不在 Planner 内部做 RAG？

放在 Router 之后、Planner 之前，因为 Router 已产出结构化意图（高质量检索 query），且职责清晰：Router 管分流和意图、RAG 管召回经验、Planner 管最终规划，解耦干净。

> 面试金句："RAG 的结果是辅助，不直接执行。它给 Planner 提供'先例和最佳实践'，Planner 负责按当前需求裁剪定制。常见场景有成熟范式兜底，新场景模型仍能灵活规划。成功的 plan 能回流进语料，越用越好。"

### 追问预判

| 追问 | 答案 |
|------|------|
| 检索不准给了错误范式？ | RAG 结果只是参考，Planner 有权否决；设相关度阈值，低分不注入 |
| 意图识别本身不准？ | 用语义匹配容忍模糊；Router 尽量结构化；Planner 根据用户原文兜底修正 |

## 十二、RAG 全流程 — 文档预处理到评估闭环

### 贯穿示例

用户输入：**"帮我做个能让团队记录客户跟进情况的小工具"**

### 阶段 0：文档预处理（离线建库）

| 步骤 | 说明 |
|------|------|
| 清洗 | 去格式、统一术语（"客户管理""CRM""跟进系统"打同义标签） |
| 结构化抽取 | 拆成 scene / stack / plan_steps / pitfalls |
| 打元数据 | domain、complexity、need_auth、multi_user、entity_count |

**数据规模示例：** 800 条范式、6000+ chunk、pgvector（复用 Supabase）

### 阶段 1：分块策略 — Small-to-Big

```
父块：完整范式（带元数据）← 最终喂给 Planner
  ├─ 子块 A：场景描述（~80 字，向量化主力，噪声最小）
  ├─ 子块 B：技术栈
  ├─ 子块 C：plan_steps
  └─ 子块 D：pitfalls
```

> 面试点："用场景子块做向量匹配（精准），命中后取完整父块（上下文完整）。检索精度和上下文完整性这对矛盾，small-to-big 同时兼顾。"

### 阶段 2：检索优化

**2.1 Query 改写（四步）**

用户原话 → 口语化、信息稀疏、缺关键特征

| 步骤 | 做法 | 示例 |
|------|------|------|
| ① 去噪规范化 | 剥掉"帮我做"、"小工具"等口水词 | "团队记录客户跟进情况" |
| ② 术语映射（最关键） | 维护同义词表："客户跟进"→"CRM" | 命中率提升关键 |
| ③ 多视角改写 | 场景/功能/协作各生成一条 query | 降低漏召回 |
| ④ 特征抽取 | 抽关键词 + 元数据 filter | 给 BM25 + 过滤用 |

**实测：** 不做 query 改写 Top-1 命中率 ~58%，加改写后 ~84%（主要收益来自术语映射）

**2.2 混合检索（Hybrid Search）**

- **向量检索**：语义匹配，"跟进≈CRM"
- **关键词 BM25**：精确命中"客户""跟进"
- **元数据过滤**：multi_user=true & need_auth=true，砍掉单人场景

> "向量管语义，BM25 管精确词，元数据 filter 管硬性条件，三者结合。"

**2.3 重排序（Rerank）**

召回管全（Top 8~10），rerank 管准，阈值过滤（如 0.6）：宁可只给 1 条高质量范式，不给噪声。

### 阶段 3：生成环节（Planner + 反幻觉）

**反幻觉 Prompt 约束：**
1. plan 优先基于参考范式，不脱离凭空设计
2. 范式未覆盖点标注"[来自需求]"
3. 范式无关步骤剔除，不硬凑
4. 每步必须溯源
5. 用户没说清的用范式默认值并标注"[默认方案，可调整]"

**Planner 产出示例（每步带来源）：**
```
1. 数据表 customers/follow_ups/team_members  [范式]
2. Supabase Auth 邮箱登录                    [范式]
3. RLS 按 team 隔离                          [范式←对应"团队"需求]
4. 客户列表/跟进时间线/新增跟进页面            [范式]
5. 跟进状态流转 待跟进→跟进中→已成交           [范式]
※ 权限用户未说明 → 默认团队成员均可读写        [默认方案，可调整]
```

### 阶段 4：评估体系闭环

| 评估项 | 谁来做 | 说明 |
|--------|--------|------|
| 召回率/命中率 | 自动化脚本 | 标注测试集跑 Top-1/Top-3 |
| plan 完整性/幻觉率/可执行性 | LLM 裁判（主力） | LLM-as-judge 按 rubric 批量打分 |
| 返工率 | 自动埋点 | 用户拿到后改了几次，最真实信号 |
| 低分/争议 case | 人工（少量精审） | 只抽检 LLM 低分 + 返工异常高的 |

> 收口："有标准答案的用脚本算，要判质量的用 LLM 评，LLM 评不准和要校准的才用人工抽检。人工最贵，只花在刀刃上。"

**闭环回流：**
```
成功且低返工的 plan → 清洗 → 回流成新范式（语料持续增长）
失败/高返工 case  → 定位根因 → 补语料/调 prompt/修订范式
```

### 灵魂记忆点（每阶段一个关键词）

| 阶段 | 关键词 |
|------|--------|
| 预处理 | 不存纯文本，存"场景+plan+元数据" |
| 分块 | 小块检索、大块召回（small-to-big） |
| 检索 | query 改写 → 混合检索 → rerank → 阈值过滤 |
| 生成 | 反幻觉：强制溯源 + 不确定显式标注 + 剔除无关 |
| 评估 | 三层分工：脚本算 / LLM 评 / 人工纠偏；成功回流、失败反推 |

## 十三、Planner 的产出 — 结构化任务图（DAG）

Planner 输出的是**可执行的任务图**，不是文字，不是代码。

### 完整链路

```
用户口语
   │
Router（意图识别 + RAG 检索）
   │  产出：结构化意图 + 命中范式 + 原始需求
   │
Planner
   │  ① 融合（范式 ⊕ 需求 → 自然语言 plan）
   │  ② 结构化（plan → 任务图 DAG）
   │  ③ 自检（不过则回到①）
   │
   执行器（Code Generator / Tools）
   │  按 DAG 顺序生成代码 / 建表 / 配 RLS …
   │
预览 + 用户反馈 →（失败/不满意则回 Planner）
```

### 输出示例（JSON DAG）

```json
{
  "tasks": [
    {
      "id": "T1", "type": "schema", "action": "创建数据表",
      "detail": "customers / follow_ups / team_members",
      "depends_on": [], "source": "pattern"
    },
    {
      "id": "T2", "type": "auth", "action": "配置认证",
      "detail": "Supabase Auth 邮箱登录",
      "depends_on": ["T1"], "source": "pattern"
    },
    {
      "id": "T3", "type": "rls", "action": "权限隔离",
      "detail": "RLS 按 team_id 隔离",
      "depends_on": ["T1", "T2"], "source": "pattern"
```

### 为什么输出 DAG 不是文字

| 特性 | 价值 |
|------|------|
| 有依赖 → 拓扑排序 | 先建表再配认证，严格按序 |
| 无依赖 → 并行 | 两个页面同时生成，提速 |
| 粒度小 → 单步重试 | T4 失败只重跑 T4 |
| 有状态 → 实时进度 | pending → running → success，前端可见 |

### 为什么不是一步到位（分层价值）

| 问题 | 分层方案 |
|------|---------|
| 不可控 | 每层单独评估、单独优化 |
| 不可复用 | 规划阶段的范式经验（RAG）用不上 |
| 改起来贵 | 只重规划受影响任务 → 增量生成 |
| 没法中间干预 | 用户看到 plan 即可纠正，最便宜 |

### LangGraph 串联点

- **分支**：Router 根据意图分流（简单请求走快车道）
- **循环**：自检不过回融合；执行失败回 Planner 重规划；编译失败回模型修复
- **状态传递**：意图、范式、plan、执行状态在节点间流转

> 串联金句："整体是一条'理解→规划→执行→反馈'的链路，但实现成带循环的图。Router 用小模型做意图识别+RAG，Planner 用强模型融合成 DAG，执行器按依赖图生成代码，配合 lint/build 做质量闭环。增量修改只重新规划受影响节点。每一层职责单一、可单独评估，让不确定的生成系统变得可控、可迭代。"

## 十四、Router → Planner → Executor 本地实测要点

### 实现架构

- **Router**：意图识别（规则版提取场景特征）+ RAG 检索（small-to-big 命中父块范式）
- **Planner（LLM 为主 + 规则兜底）**：
  - LLM（通义 qwen）融合范式和需求，强制输出 JSON DAG
  - 规则兜底三道：坏 JSON → 范式直转 DAG、补缺字段去重、自检无环
- **Executor（Mock）**：按并行分层执行，演示状态流转和单步重试

### 执行器验证的三个场景

| 场景 | 演示价值 |
|------|---------|
| 分层并行 | 无依赖任务同层并行，有依赖严格按序 |
| 单步重试 | 某任务失败自动重试，不影响其他任务 |
| 失败传播 | 彻底失败只跳过下游，已完成不受影响 |
