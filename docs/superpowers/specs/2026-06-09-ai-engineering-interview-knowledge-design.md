# AI 工程化面试知识沉淀文档设计

> 生成日期：2026-06-09  
> 目标产物：`.interview/AI工程化项目知识沉淀-面试版.md`  
> 设计状态：待用户审阅  

## 1. 任务理解

本任务要设计一份面向面试的项目知识沉淀文档。文档不是代码实现方案，也不是单纯题库，而是把用户近一年围绕 AI Agent / AI 工程化落地的多个项目，用同一套 Harness Engineering 方法论串起来。

核心目标：

- 用 `.interview/harness-完整认知体系.md` 的七个不变内核作为主线。
- 融合企业级产研 Harness、前端自动化测试验证域、应用生成智能体、OpsPilot、Vibe Dev Toolkit。
- 明确每个项目的事实边界和贡献边界，避免把企业项目、团队试点、本地演示和架构推演混写。
- 服务面试表达：能讲全景，也能在追问时落到可防守的项目锚点。

## 2. 项目边界

| 项目 / 方向 | 面试定位 | 边界标注 |
|---|---|---|
| 企业级产研 Harness | 公司级 AI 研发提效体系背景，体现企业级治理、编排、契约、质量门禁、知识沉淀 | 企业级背景 / 体系认知 |
| 前端自动化测试验证域 | 用户最强可防守主战场，重点讲 AC 驱动 E2E、真执行证据、AC↔断言双向映射、Bad Case 闭环 | 我负责 / 主导做深 |
| 应用生成智能体 | 企业轻应用生成方向，类似 Lovable / Bolt 的企业内部应用生成平台；重点讲 Router/Planner 分层、场景化 RAG、生成质量、多租户、SSO / 数据隔离、预览调试 | 参与规划 / 试点推进 / 架构设计锚点 |
| OpsPilot | 团队基于 SRE 故障处置经验沉淀后的运维智能体尝试，目前在 SIT 环境；重点讲证据采集、诊断、风险门禁、审批执行、RCA 闭环 | 团队 SIT 试点 |
| Vibe Dev Toolkit | 本地可演示复刻项目，用于向面试官证明能把企业 Harness 的关键机制跑成可执行系统 | 本地面试演示复刻 |

## 3. 推荐方案

采用“思想为骨，项目为肉”的文档结构。

文档以七个不变内核作为一级主线，每个内核下挂项目证据：

1. 企业 Harness 如何体现这条内核。
2. 前端测试域如何真实落地。
3. 应用生成智能体如何迁移到生成式应用场景。
4. OpsPilot 如何迁移到 SRE 故障处置场景。
5. Vibe Dev Toolkit 如何做本地可演示复刻。

选择理由：

- 比“项目流水账”更能体现体系化思考。
- 比“纯理论文”更能落到真实项目证据。
- 比“题库手册”更适合作为长期知识沉淀，同时末尾仍可补面试问答锚点。

## 4. 文档结构设计

目标文档建议写入 `.interview/AI工程化项目知识沉淀-面试版.md`。

### 4.1 一句话定位

用一段话讲清楚用户的面试定位：

> 我近一年围绕 AI Agent 工程化落地，参与 / 推动了研发提效 Harness、应用生成智能体、运维智能体三类企业方向。其中我在企业 Harness 的前端自动化测试验证域做得最深；应用生成智能体体现我对产品化 Agent 平台的规划能力；OpsPilot 是团队基于故障处置经验沉淀后的 SIT 试点；Vibe Dev Toolkit 是我为了面试演示做的本地可运行复刻。

### 4.2 诚实边界表

文档前置一张边界表，字段包括：

- 项目 / 方向
- 项目状态
- 我的角色
- 可讲深度
- 面试可用锚点
- 不应夸大的边界

边界类型使用固定枚举：

- 企业级背景
- 我负责 / 主导
- 我深度参与
- 我参与规划
- 团队 SIT 试点
- 本地演示复刻
- 架构推演

### 4.3 七个不变内核 × 项目证据

每个内核使用固定结构：

```text
内核命题
它对抗的 LLM / 现实世界属性
企业 Harness 中的体现
前端测试域中的落地
应用生成智能体中的迁移
OpsPilot 中的迁移
Vibe Dev Toolkit 中的演示价值
面试回答锚点
诚实边界
```

七个内核沿用现有认知体系：

1. 控制权永远归确定性系统
2. 约束前置，而非事后纠偏
3. 契约优先于对话
4. 证据高于断言
5. 分解胜于强提示
6. 状态外置且可恢复
7. 可观测与闭环内建

### 4.4 王牌展开一：前端测试三防线

单独加重前端自动化测试验证域，因为这是最可防守的个人贡献。

重点内容：

- 防线一：真执行 + 机器证据，使用 trace / 退出码 / 截图等产物防止“没跑却说跑了”。
- 防线二：AC 驱动断言，每条断言必须能追溯到结构化验收条件。
- 防线三：AC↔断言双向映射，用确定性脚本识别 COVERED / MISSING / HOLLOW / ORPHAN。
- Bad Case 闭环：失败样本沉淀、检索、复用，避免同类问题重复发生。

### 4.5 王牌展开二：应用生成智能体

应用生成智能体作为产品化 Agent 平台能力的代表，不写成纯架构推演，而是围绕企业轻应用生成场景组织。

重点内容：

- Router / Planner / Executor 分层：Router 管分流和意图识别，Planner 管复杂任务拆解，Executor 管执行闭环。
- 场景化 RAG：检索的是 scene→plan 范式，而不只是代码片段；RAG 结果是 Planner 的参考，不是直接执行指令。
- 生成质量闭环：生成前约束、生成后静态检查、失败回喂修复、成功样本回流。
- 多租户与数据隔离：平台用户隔离 + 生成 App 自身隔离，两层租户不能混淆。
- 面试锚点：为什么 RAG 放在 Router 后、Planner 前；为什么简单请求不进重 Planner；为什么 Supabase 开源版需要补控制平面。

### 4.6 王牌展开三：OpsPilot

OpsPilot 作为 SRE 场景中的团队 SIT 试点，负责证明同一组 Harness 内核可以迁移到故障处置。

重点内容：

- 14 个功能节点 + dispatcher 路由，不把 dispatcher 算作业务功能节点。
- evidence_fanout / evidence_aggregate / diagnose / critic / remediation / risk_gate / executor / verify / rca 的闭环。
- 风险门禁与审批：高风险动作进入审批，低风险动作受控执行。
- 证据驱动诊断：工具证据、诊断候选、critic 复核、verify 真实校验。
- SIT 边界：可讲团队试点和工程链路，不夸大为大规模生产化平台。

### 4.7 Vibe Dev Toolkit 的定位

Vibe Dev Toolkit 不包装成企业生产项目，定位为本地可演示复刻。

它在文档中的作用：

- 证明用户能把企业 Harness 的核心机制从认知落到可运行系统。
- 作为面试现场 demo：AC 输入、任务状态、E2E 执行、覆盖校验、Bad Case 沉淀。
- 与企业 Harness 的关系是“同构复刻”，不是“生产系统本体”。

### 4.8 高频面试问法与回答锚点

文档末尾补一节面试作战索引，覆盖：

- 你到底负责哪部分？
- 这几个项目之间是什么关系？
- 哪些是企业项目，哪些是团队试点，哪些是本地演示？
- 怎么避免大模型没测却说测了？
- 应用生成智能体的 RAG 为什么不直接检索代码？
- Router 和 Planner 为什么拆开？
- OpsPilot 如何防止错误执行？
- 你这套方法论到底是什么？

目标文档完成后，应满足：

1. 读完能回答“你到底做了哪几类 AI 工程化项目”。
2. 每个项目的角色与边界清楚，不互相冒充。

4. 前端测试三防线能作为最强个人贡献独立讲 5-10 分钟。

## 9. 风险说明

| 风险 | 处理方式 |
|---|---|
| 把企业 Harness、应用生成、OpsPilot、Vibe 混成一个项目 | 前置边界表，并在每个章节重复标注项目角色 |
| 把“参与规划”写成“全链路主导” | 使用固定边界枚举，避免泛化表述 |
| 把 Vibe Dev Toolkit 包装成企业项目 | 明确写成本地面试演示复刻 |
| OpsPilot SIT 试点被误读为大规模生产平台 | 写清“团队 SIT 试点”，重点讲工程链路和方法迁移 |
| 文档过长，面试前难用 | 末尾增加高频问法与回答锚点，便于临场检索 |
| 理论太抽象 | 每个内核必须绑定项目证据，不允许只写原则 |
