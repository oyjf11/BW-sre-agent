# 设计：会话沉淀技能 interview-distill

## 背景与目标

用户正备战 Agent 工程师岗位面试（[[interview-prep-study-style]]，目标 30k+，截止约 2026-07-01）。
`.interview/battle-plan/` 已经有一套成熟的"母题（触发器 + kernel + 项目锚点）+ 主动回忆"复习体系
（参考 `母题骨架-一页纸.md`、`dispatcher-resume机制.md`）。

问题：日常修 bug / 做需求的长会话里，经常会产出"原生 vs 项目真实实现""排查到根因的过程""架构取舍"
这类一手实践——这恰恰是用户最缺的面试弹药（背景是前端转型 + 项目代码多由 coding agent 完成，参见
`.interview/核心诉求.md`）。但这些内容目前散落在会话里，会话结束就流失，没有进入复习体系。

目标：做一个可在长会话结束时手动触发的技能，把"这次我们做了什么、怎么排查/取舍的、可复用的
kernel 是什么"提炼成符合现有母题风格的复盘文件，沉淀进 `.interview/battle-plan/`。

## 范围

- 触发：用户一句话手动触发（如"沉淀这次会话"），或我在长技术会话收尾时主动提醒"要不要沉淀"
  （后者通过写入一条 feedback 记忆实现，best-effort，不依赖 hook）。
- 产出：每次生成一个新的复盘文件，并维护一个一行一条的索引文件。
- 不做：不自动触发（不加 Stop hook）、不自动合并进现有母题核心文件（避免改坏正在背的资料）、
  不做面试题库的内容生成（只沉淀"刚发生的这次会话"里的真实素材）。

## 产出位置与文件结构

- 复盘文件：`.interview/battle-plan/实战-YYYY-MM-DD-<主题>.md`（新建，进 git，可追溯）
- 索引文件：`.interview/battle-plan/实战沉淀-索引.md`（每条一行：日期 · 标题 · kernel 一句话 ·
  归属母题编号 · 文件链接）

文件内部结构（"母题格式 + 折叠复盘"，对齐 `母题骨架-一页纸.md` 的"触发→kernel→锚点"三段式
和 `dispatcher-resume机制.md` 的"深挖问答底稿"风格）：

```markdown
# 实战沉淀 · <主题>

> 🎯 一句话 kernel（背这句）：<最核心的可复用原则>

## 母题速记
| 触发（面试官会问） | kernel（背这句） | 项目锚点（这次具体改了/取舍了什么） | 可能追问 |
|---|---|---|---|

<details><summary>📖 详细复盘 + 面试问答底稿（点开）</summary>

## 详细复盘
问题背景 → 排查/实现过程 → 根因或方案 → 关键取舍

## 面试问答底稿
Q：<面试官会怎么问> 
A：分两层答（通用知识 + 项目实际是怎么做的），结尾带"诚实边界"（不夸大，主动说局限）

## 关键文件速查
| 关注点 | file:line |

</details>
```

## 技能内置的提炼方法论（写入 SKILL.md，是本技能的核心价值）

1. **先判断值不值得沉淀**：会话里要有"根因 / 架构决策 / 取舍 / 新机制对比"这类真东西才写一篇；
   纯琐碎修改（改个文案、调个样式）应直接回复"这次没有值得沉淀的素材"，不要硬造内容。
2. **沿固定轴提炼**："发生了什么 → 根因/设计是什么 → 权衡了哪些选项、为什么选这个 →
   能压成哪一句可复用 kernel → 这句话挂到母题骨架的哪个编号（A/M 系列，或标记"新母题候选"）→
   面试官可能怎么问、怎么分两层答、诚实边界是什么"。
3. **诚实话术内置**：用户是前端转型 + 项目代码多由 coding agent 编写
   （[[interview-prep-study-style]]）。提炼时统一按"我和 agent 一起排查 / 我做的架构判断和验证"
   措辞，不要写成"我从零手写实现"；遇到用户对实现细节生疏的部分，在"诚实边界"里如实标注
   "这部分是 agent 实现的，我验证过其行为，但实现细节需要现场补读代码"。
4. **校准到用户的北极星**：目标 30k+ agent/后端 offer；长板是 harness/可靠性思维
   （证据高于断言、分解胜于强提示、确定性编排包裹非确定性）；短板是 Python/后端深度。
   提炼时优先突出能展示长板、补足短板叙事的素材。

## 触发与提醒

- 主触发：用户在收尾时说"沉淀一下""帮我把这次整理进面试库"等类似表达。
- 辅助提醒：在用户的长期记忆中新增一条 feedback 记忆——"在长技术会话（涉及排查/架构决策/取舍）
  收尾时，主动问用户是否要触发 interview-distill 沉淀"。这是 best-effort 的上下文提示，
  不保证 100% 触发；如果用户希望强保证，需要另外配置 Stop hook（本设计不包含，需用户明确要求后
  用 update-config 技能添加）。

## 自检清单（生成内容前必须心里过一遍）

- 这次会话里，最反直觉 / 最有"原来生产是这么做的"含金量的一个点是什么？
- 能不能用一句话讲清 kernel，让用户合上文件也能复述？
- 项目锚点是否具体到文件/函数/取舍，而不是泛泛而谈？
- 面试问答底稿是否区分了"教科书答案"和"项目实际做法"？
- 诚实边界是否真实（不夸大也不过分自我贬低）？

## 不在范围内 / 后续可选项

- 不做内容自动合并进 `母题骨架-一页纸.md`（人工复习时再决定要不要并入）。
- 不做 Stop hook 自动触发（如用户后续要求强保证触发，可加）。
- 索引文件随复盘文件增多，未来可能需要按母题分类重排——本设计先保持"按时间一行一条"最简形态。
