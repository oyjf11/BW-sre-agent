# Harness 构建路线总纲设计规格

## 1. 任务理解

当前 `.interview` 下已有多份 AI Harness 面试材料，但内容分散在方法论、第一性原理、企业认知和案例细节之间。用户的核心问题不是缺少材料，而是缺少一份入口文档，能够同时回答：

1. Harness 应该按什么顺序构建。
2. 个人、小团队和企业分别怎么搭。
3. 具体实施时需要哪些脚本、工具、平台组件和产物。
4. 面试时如何把方法论讲成一条清晰主线。

本次设计的目标是新增一份总纲文档的规格，后续据此生成真正的总纲正文。

## 2. 目标

新增一份入口型总纲文档，暂定路径：

```text
.interview/harness-构建路线总纲.md
```

这份总纲承担三个职责：

1. 把现有材料收敛为一条可讲、可背、可追问的主线。
2. 用同一套步骤同时解释个人版和企业版 Harness 的落地方式。
3. 给出可施工级实施蓝图，包括目录、脚本、工具、平台组件、产物和验收方式。

## 3. 非目标

本次不做以下事情：

1. 不删除或覆盖现有素材文档。
2. 不把所有内容合并成唯一大文档。
3. 不新增脚本、命令、工具链代码。
4. 不修改 OpsPilot 后端、前端或测试代码。
5. 不声称企业级平台已经完整落地。
6. 不把 Vibe Dev Toolkit 的设计推演部分说成已真实上线。

## 5. 内容结构设计

总纲采用“正文讲清主线，后半部分给实施蓝图，附录处理岗位话术”的结构。

```markdown
# Harness 构建路线总纲：从面试主线到实施蓝图

## 0. 这份文档怎么用
## 1. 一句话总脉络
## 2. 10-15 分钟面试主讲版
## 3. Harness 构建的 8 步施工路线
## 4. 双层镜像：个人怎么搭，企业怎么搭
## 5. 个人版可施工蓝图
## 6. 企业版平台蓝图
## 7. 两类岗位讲法
## 8. 与现有文档的映射
## 9. 面试追问索引
```

### 5.1 主线

总纲不再用“七个原则”作为主目录，而用构建顺序作为主线：

```text
选场景 → 定标准 → 建执行骨架 → 接能力 → 设约束 → 拿证据 → 跑评测 → 沉淀知识
```

七个不变内核作为每一步背后的解释出现，而不是继续平铺成原则清单。

### 5.2 两类岗位讲法

只保留两类岗位视角：

1. Agent / AI 工程师版：重点讲编排、状态、工具权限、评测、脚本和证据材料。
2. AI 产品 / 解决方案版：重点讲企业落地、组织分工、价值证明、试点到规模化推广。

不单独保留“后端 / 平台工程师版”。平台能力并入 Agent / AI 工程师版和企业平台蓝图。

## 6. 8 步施工路线设计

每一步统一使用同一模板：

```markdown
### Step N：标题
- 这一阶段解决什么问题
- 个人版怎么做
- 企业版怎么做
- 用到的脚本 / 工具 / 平台
- 产物是什么
- 验收方式
- 对应 Harness 原则
- 面试怎么讲
```

### Step 1：选场景

选择高频、出错影响小、结果容易验证、最好有历史样本的场景。个人版以 Vibe Dev Toolkit 的前端测试生成链路和 OpsPilot 的离线故障诊断评测为锚点；企业版从缺陷初判、测试生成、代码审查、工单摘要等风险可控场景开始。

### Step 2：定标准

先定义“什么算做对”，再让 AI 执行。个人版使用验收条件 JSON、固定样本和离线评测；企业版使用题库、人工确认过的标准答案、回归集和业务指标。

### Step 3：建执行骨架

用确定性流程包住非确定性模型。个人版可以用 `task.json`、`events.jsonl`、`orchestrator.ts` 或 LangGraph；企业版使用工作流引擎、状态检查点、事件总线和人工处理节点。

### Step 4：接能力

把能力拆成 Agent、工具、脚本和人工处理节点，而不是做一个全能 Agent。个人版锚定 `test-generator`、`test-analyst`、`coverage-checker` 和 OpsPilot 节点；企业版锚定能力市场、工具注册中心和内部平台服务。

### Step 5：设约束

约束要在执行前生效，不能只写在提示词里。个人版包括目录边界、重试上限、超时、禁止绕过断言；企业版包括权限系统、工具白名单、模型用量限制、审批策略和契约强校验。

### Step 6：拿证据

不采信 AI 的口头完成声明，只看真实产物。个人版包括 Playwright 测试结果、执行轨迹、截图、测试输出和代码差异；企业版包括持续集成结果、审计日志、灰度指标和证据材料包。

### Step 7：跑评测

把一次演示变成可回归、可比较、可防止指标虚高的评测体系。个人版包括验收条件与断言双向检查、失败案例回放和固定样本评测；企业版包括小规模快速集、全量回归集、隐藏题集和评测执行器。

### Step 8：沉淀知识

让每次失败和修复都能影响下一次执行。个人版包括 `.vibe/badcases/*.yaml`、本地检索和上下文注入；企业版包括失败案例库、项目规则库、文档可信度标注和组织知识库。

## 7. 个人版可施工蓝图

个人版主锚点是 Vibe Dev Toolkit，副锚点是 OpsPilot。

### 7.1 Vibe Dev Toolkit 目录骨架

```text
.vibe/
  specs/acceptance/
  state/
  reports/
  badcases/

.opencode/
  command/
  agent/
  skill/

tools/vibe/
  src/
  tests/

e2e/
```

### 7.2 关键脚本

```text
orchestrator.ts        管任务状态、重试、超时、事件记录
contracts.ts          定义输入输出契约
coverage-checker.ts   检查验收条件是否有断言、断言是否真执行
badcase.ts            失败案例读写和检索
test-generator.md     生成 Playwright 用例的 Agent 定义
test-analyst.md       分析失败原因的 Agent 定义
```

### 7.3 个人版命令流

```bash
$EDITOR .vibe/specs/acceptance/login.ac.json
npm run dev
/opencode command vibe-test login
npx playwright test e2e/login/login.generated.spec.ts --reporter=json
node tools/vibe/dist/coverage-checker.js --ac <ac.json> --result <result.json>
/opencode command learning-compound .vibe/reports/<execution_id>
/opencode command learning-researcher login
```

### 7.4 OpsPilot 映射

```text
执行骨架       LangGraph 节点和条件路由
状态恢复       状态检查点 / 运行状态
工具能力       工具网关 / 适配器
人工确认       审批中断节点 / 审批接口
证据材料       证据条目 / 执行轨迹 / 事件记录
离线评测       Phase 8 离线评测框架
失败复盘       根因分析 / 执行轨迹 / 测试报告
```

实施总纲中需要继续保持中文优先表达。必要的英文专有名词只在第一次出现时括号标注，后文不再中英混写。

## 8. 企业版平台蓝图

企业版不描述为“更大的个人工具”，而描述为统一治理、多团队复用、可计量和可审计的平台。

核心组件：

```text
任务接入层      需求平台、工单系统、代码仓库、持续集成、监控平台
模型调用层      模型调用网关、模型路由、用量计量、缓存
流程编排层      工作流引擎、状态机、状态检查点、事件总线
能力层          Agent、工具、脚本、人工处理节点注册中心
约束层          契约注册中心、权限系统、工具白名单、用量限制、审批策略
证据层          测试结果、执行轨迹、截图、审计日志、证据材料包
评测层          评测执行器、题库、人工标准答案、回归集、隐藏题集
知识层          失败案例库、项目规则库、文档可信度标注、检索服务
治理层          指标看板、成本看板、风险审计、能力准入流程
```

企业版实施阶段：

```text
1. 能力工具化
2. 能力标准化
3. 流程编排化
4. 评测与治理
5. 知识沉淀
```

## 9. 语言风格规范

总纲要像工程师在解释自己怎么做，不像概念百科。

规则：

1. 中文优先，英文术语只在第一次出现时括号标注。
2. 后文统一使用中文名，避免中英文混写。
3. 少用强比喻和口号，多写具体工程动作。
4. 保留必要专有名词，如 LangGraph、Playwright、OpenCode、MCP。
5. 明确区分真实做过、设计推演、企业蓝图。

术语替换：

| 原表达 | 总纲表达 |
|---|---|
| 低爆炸半径 | 出错影响小 / 风险可控 |
| 高爆炸半径 | 一旦出错影响大 / 难恢复 |
| 闸门 | 强制检查点 / 准入规则 / 放行条件 |
| 喂信息 | 准备上下文 / 提供必要信息 |
| 护城河 | 长期差异化能力 / 组织沉淀 |
| 复利飞轮 | 持续积累机制 |
| 脊柱 | 执行骨架 / 主流程 |
| 金标 | 人工确认过的标准答案 |
| Bad Case | 失败案例 |
| AI 作弊 | AI 绕过验收 / 指标被刷高 |
| trace | 执行轨迹 |
| evidence pack | 证据材料包 |
| eval runner | 评测执行器 |
| schema registry | 契约注册中心 |
| tool registry | 工具注册中心 |
| LLM Gateway | 模型调用网关 |
| checkpoint | 状态检查点 |
| Human Task | 人工处理节点 |
| AC | 验收条件 |
| JSON reporter | JSON 格式测试报告 |
| run | 一次执行 / 一次任务执行 |

## 10. 验收方式

总纲完成后按以下标准验收：

1. 从头读一遍，能回答“Harness 到底按什么顺序构建”。
2. 能同时回答“个人怎么搭”和“企业怎么搭”。
3. 列出具体脚本、工具、平台组件和产物。
4. 保留 10-15 分钟面试主线，而不是只剩施工清单。
5. 避免明显中英文混写和模板化表达。
6. 明确标注真实落地与设计推演边界。
7. 能映射回现有核心文档。

## 11. 风险与处理

| 风险 | 处理 |
|---|---|
| 文档过长 | 正文先讲主线，细节放表格和附录 |
| 个人版和企业版混淆 | 每节固定分成个人版、企业版、产物、验收 |
| 术语过多 | 第一次解释，后文使用中文名 |
| 像 AI 话术 | 避免强比喻，改用具体工程动作 |
| 把推演说成实绩 | 明确标注“真实做过 / 设计推演 / 企业蓝图” |

## 12. 回滚方式

如果总纲方向不符合预期，回滚只需要删除后续新增的：

```text
.interview/harness-构建路线总纲.md
```

本设计规格本身也可以单独删除，不影响业务代码和现有面试材料。
